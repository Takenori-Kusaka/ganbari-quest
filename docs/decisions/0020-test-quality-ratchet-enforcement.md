# ADR-0020: テスト品質の劣化を許容しない開発プロセス

| 項目 | 内容 |
|------|------|
| ステータス | accepted |
| 日付 | 2026-04-10 |
| 関連 Issue | #543, #611, #671, #673 |
| 関連 ADR | ADR-0005（Critical修正の品質ゲート）, ADR-0018（Issue起票品質基準） |

## コンテキスト

2026-04-01 にテスト品質監査（#0286）Phase 5 が完了し、カバレッジ閾値を「ラチェット方式」で lines: 48, functions: 37, branches: 41, statements: 43 に引き上げた。

しかし **48時間以内に2回、閾値が引き下げられた**:

| 日付 | イベント | lines | functions | branches | statements |
|------|---------|-------|-----------|----------|------------|
| 04-01 | ラチェットアップ完了 | 48 | 37 | 41 | 43 |
| 04-02 | #0216 きょうだいチャレンジ追加 → 閾値引き下げ | 43 | 31 | 38 | 37 |
| 04-03 | #0276-#0278 大量機能追加 → 再度引き下げ | 38 | 27 | 35 | 32 |

コメントには「テスト追加で再度引き上げ予定」と記載されたが、**引き上げは一度も実行されていない**。

同時期に以下の品質劣化パターンが確認された:

1. **Critical修正（#543）にE2Eテストなし** — ADR-0005 違反。19行のガード条件のみで closes
2. **`clearDialogGhosts()` でアプリバグをテスト側で隠蔽** — Ark UI Dialog の positioner リークをテストヘルパーが `display: none` で強制非表示にし、ダイアログ関連のバグを構造的に検出不可能にした
3. **`test.skip()` 濫用** — dialog-queue.spec.ts の3テスト中4箇所で test.skip。コンボボーナスダイアログのパスは実質未テスト
4. **activity-log-service.test.ts がサービス層をテストしていない** — 19テストの全てが DB の insert/select を直接操作し、`recordActivity()` を一切呼んでいない。プロジェクト最大のコアフローがテストカバレッジ実質ゼロ
5. **レアリティ分布テストが自己参照** — stamp-card-service のテストが実装の `RARITY_WEIGHTS` を呼ばず、テスト内で独自に乱数シミュレーション

**結果**: 4回の既知リグレッション（#543, #611, #671, #673）全てがテストで検出不可能。テストは「存在するが機能しない」状態。

### 根本原因

テスト品質の劣化は偶発的ではなく、**開発プロセスの構造的帰結**:

- 1日60-70コミットの機能量産（全コミットの作者が1名 + Claude Code）
- 機能追加PRでカバレッジが下がったとき、**テスト追加ではなく閾値引き下げでCIを通す**
- セルフレビュー + Copilot のみで品質ゲートが機能しない
- 「ラチェット方式」を宣言しても、引き下げを禁止する仕組みがない

## 決定

### 1. カバレッジ閾値の引き下げを CI で禁止する

- `vite.config.ts` の `thresholds` を引き下げるPRは **CI で自動リジェクト** する
- GitHub Actions の pre-merge チェックに `thresholds` 変更検出スクリプトを追加
- 閾値を引き下げる唯一の方法は、ADR にその理由を記録し、復元計画（日付つき）を同時にコミットすること

### 2. 機能追加PRにはカバレッジ差分チェックを必須とする

- 新規コードの **カバレッジが既存平均を下回るPRはマージ不可**
- 具体的には、PR で追加されたファイル群のカバレッジが全体閾値以上であることを CI で検証
- 「テストは後で追加」は禁止。テストなしの機能追加PRはドラフトのまま据え置き

### 3. テスト回避パターンを禁止する

| 禁止パターン | 検出方法 | あるべき姿 |
|-------------|---------|----------|
| `clearDialogGhosts()` 等のバグ隠蔽ヘルパー | PR レビューで禁止 | アプリ側のバグを修正する |
| `test.skip()` の安易な使用 | CI で test.skip 数を計測、増加を警告 | テストが通るようにアプリを修正するか、テストケースを正しく条件分岐 |
| カバレッジ閾値の引き下げ | CI で thresholds 変更を検出 | テストを追加してからマージ |
| サービスを呼ばないサービステスト | PR レビューチェックリスト | テスト対象の公開 API を直接呼ぶ |
| テスト内で実装ロジックを再実装 | PR レビューチェックリスト | 実装の関数を呼んで結果を検証 |

### 4. カバレッジ閾値の段階的復元計画

| フェーズ | 期限 | lines | functions | branches | statements |
|---------|------|-------|-----------|----------|------------|
| 現状（劣化後） | — | 38 | 27 | 35 | 32 |
| Phase 1: 04-01水準に復元 | 04-17 | 48 | 37 | 41 | 43 |
| Phase 2: 中間目標 | 05-01 | 60 | 50 | 55 | 55 |
| Phase 3: 最終目標 | 06-01 | 80 | 80 | 75 | 80 |

Phase 1 は activity-log-service の正しいテスト追加と test-db.ts の統一で達成可能。

## 結果

- 機能追加のたびにテスト品質が下がる「品質負債の雪だるま」が構造的に防止される
- 「テストは後で」「閾値は一時的に下げる」が CI レベルで不可能になる
- テストが「存在するが機能しない」状態から「リグレッションを検出できる」状態に回復する

## 教訓

- **ラチェットは仕組みで強制しなければ機能しない** — 「引き上げのみ許可」と文書で宣言しても、CI で強制しなければ48時間で破られる
- **テスト数やファイル数は品質の指標ではない** — 117ファイル284テストケースがあっても、コアフロー（recordActivity）のテストがゼロなら意味がない
- **テストを直すのではなくアプリを直す** — テストが落ちたとき、テストの方を緩めれば CI は通る。しかしそれはバグを見えなくしただけ
- **開発速度とテスト品質はトレードオフではない** — テストなしで機能を量産すると、リグレッションの修正コストが蓄積し、総コストが増える。1日60コミットより、テスト付き20コミットの方が速い
