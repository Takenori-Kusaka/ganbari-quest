## コンテキスト

PO（@Takenori-Kusaka）本人からの明示的指摘（2026-04-11）:

> 私自身がエンジニアであることもあり、かなり偏った状態になっているので是正をしていきたいです

実際に 2026-04-11 時点の open issue を集計したところ、偏りは定量的にも確認できた:

| ラベル | 本数 |
|---|---|
| area:billing | **61** |
| area:admin | 25 |
| priority:high | 55 |
| type:fix | 51 |
| type:feat | 29 |
| type:docs | 14 |
| type:test | 13 |
| area:lp | 8 |
| area:child-ui | 6 |
| type:infra | 1 |

一方で **growth / marketing / activation / unit-economics 系のカテゴリは 0 本**。

本プロジェクトは Pre-PMF（Pre Product-Market Fit）フェーズであり、V2MOM Q2 目標は「サインアップ 20 名/月」、Year 1 KPI は「EOY MAU 500 / MRR ¥12,000」である（`docs/design/34-V2MOM.md`, `docs/design/12-事業計画書.md`）。この段階で最も解かなければならない問題は **ユーザーがいない** ことであり、課金基盤の完成度ではない。

### 根本原因

PO が 1 名 + エンジニア出身という体制では、以下の構造的バイアスが機械的に発生する:

1. **慣れた領域を優先する** — バックエンド・課金・管理画面の issue は発想が容易。Growth・Marketing・Onboarding は発想の枠外に置かれやすい
2. **「完璧にしてから売る」思考** — エンジニアは品質ゲートを設けたがるが、Pre-PMF で必要なのは「雑でも早く当てる」こと
3. **Claude Code 等の AI エージェントも同じバイアスを持つ** — PO が起票したパターンを学習し、さらに偏りを増幅する

この偏りは PR テンプレート・レビュー・CI では検出できない（いずれも「起票された issue の品質」を見るだけで、「起票されなかった issue」は見えない）。**起票時点での機械的チェック**が必要である。

## 検討した選択肢

### 選択肢 A: PO の自制に任せる（現状維持）

- メリット: 実装コスト 0
- デメリット: すでに失敗している（上記集計の偏りがその証拠）

### 選択肢 B: 週次レビューで偏りを測定し後追いで調整

- メリット: 既存の workflow を変えない
- デメリット: 起票後の調整は sunk cost バイアスで効かない。一度書いた issue を「やっぱりやらない」にするのは心理的ハードルが高い

### 選択肢 C: Issue テンプレに機械的チェックリストを埋め込み、起票の瞬間にバイアスを検知させる（採用）

- メリット: 起票時点で自問を強制できる。AI エージェントが代理起票するときも同じチェックが効く
- デメリット: チェック項目を形骸化させないための運用ルールが別途必要

## 決定

### 1. Issue テンプレートへの Pre-PMF チェックリスト組込み

`.github/ISSUE_TEMPLATE/feature_request.yml` に `pre_pmf_check` セクションを追加し、`type:feat` の起票時に必須チェックとする（`validations.required: true`）。

チェック項目（5 件）:

1. どのペルソナ（`docs/design/11-ペルソナ定義書.md` の P1/P2）のどの課題を解決するか、本文に明記した
2. `docs/design/34-V2MOM.md` の Method（M1〜M4）に紐づけた
3. この機能がなくても Pre-PMF サインアップ 20 名/月（V2MOM Q2 目標）に到達できるかを自問した（できる → medium 以下 / できない → high 以上 + 本文に根拠記載）
4. 直近に ユーザー獲得 / オンボーディング / 採算性 系 Issue を 1 本以上起票した（連続する新機能 Issue はバイアスの兆候）
5. 工数と `docs/design/12-事業計画書.md` Year 1 KPI（EOY MAU 500 / MRR ¥12,000）への寄与が釣り合っている

### 2. チェック対象

以下の両方に適用する:

- PO 自身が起票する場合
- Claude Code / その他 AI エージェントが代理起票する場合

AI エージェントが「自律的に Issue を起票する」場合も、このチェックを省略してはならない。`.github/CLAUDE.md` にその旨を明記する。

### 3. チェック免除

以下は Pre-PMF チェック対象外とする:

- `priority:critical` の bug fix（緊急対応）
- 法務・セキュリティ・コンプライアンス対応（選択の余地がない）
- `type:fix` の既存機能の保守（新規機能追加ではないため、そもそも Pre-PMF バイアスの対象外）

`type:feat` のみが対象。

### 4. 優先度判定ルール（最重要）

3 番目のチェック項目が最重要である:

> この機能がなくても Pre-PMF サインアップ 20 名/月（V2MOM Q2 目標）に到達できるか

- **できる → `priority:medium` 以下** にする。どれほど欲しくても high 以上を付けない
- **できない → `priority:high` 以上**。かつ本文に「なぜ到達できないか」の根拠を明記する

根拠なしの `priority:high` は、レビュワーが `medium` 以下に降格する。

### 5. 連続起票の警告

新規機能 Issue（`type:feat`）が連続している場合は、Growth / Marketing / Activation 系 Issue を 1 本挟む。これは自律的に Issue を生成する AI エージェントに対しても拘束力を持つ。

## 結果

### 期待される効果

- Pre-PMF の優先順位が「機能の完成度」から「ユーザー獲得への寄与」にシフトする
- PO 1 人体制に起因するエンジニアバイアスを起票時点で機械的に検知できる
- `priority:high` のインフレが抑制される（根拠必須化の副作用）
- AI エージェントが自律的に大量の feat issue を生成する事故を防げる

### トレードオフ

- 起票に 5〜10 分の追加検討コストがかかる
- 「Growth 系 issue を書くために Growth 系 issue を書く」という inversion of responsibility のリスクがある（運用で監視）

### 自己整合性

この issue（#830）自体に本チェックリストを適用する:

1. **ペルソナ**: PO（=運営者ペルソナ）のバイアス是正
2. **V2MOM Method**: M1（新規ユーザー獲得の仕組み構築）に間接的に紐づく（偏りを直すことで M1 系 issue が起票されるようになる）
3. **20 名/月到達に必要か**: **必要**。現状の偏り（growth issue 0 本）では到達不可能。よって `priority:high`
4. **直近の Growth 系起票**: この issue 自体が Growth 系起票の起点となる
5. **工数**: テンプレ 1 ファイル + ADR 1 枚 + CLAUDE.md 追記。工数少・効果大

→ チェック全項目を満たす。自己整合性あり。

## 教訓

- **Pre-PMF で最も価値の高い issue は「まだ起票されていない issue」である** — 起票済み issue だけを見ていると、盲点が永遠に盲点のままになる
- **エンジニアは「完璧主義」をリスクだと認識しにくい** — 「ちゃんと作ってから売る」は Pre-PMF では致命的な遅延になる
- **AI エージェントに自律起票させる場合、人間側のバイアスを AI が忠実に再現する** — 是正メカニズムは AI ワークフローにも埋め込む必要がある
- **Issue の優先度は起票時に決める** — 後から調整する運用は sunk cost バイアスで機能しない
