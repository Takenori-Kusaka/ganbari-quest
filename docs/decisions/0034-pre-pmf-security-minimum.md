## コンテキスト

### なぜこの ADR が必要か

Pre-PMF フェーズ（サインアップ 20 名/月に到達していない段階、PO 1 人 + AI エージェント体制）の防御レベルについて、**「実装コスト・運用コスト・攻撃リスクのバランスで正当化できる基準」** が文書化されていない。

結果として、直近の Issue / PR で次のような過剰防衛設計が連鎖的に提案されている：

- **#1003** — `license_event` テーブル新設（発行・失効・消費を DynamoDB に追記ログとして蓄積）
- **#993 / #1009** — `ops_audit_log` テーブル新設（/ops 画面からの操作を汎用監査ログに記録）
- **#1010** — ライセンスキーブルートフォース検知（IP 単位カウンタ + Discord アラート）
- **#820 周辺議論** — AWS WAF 導入、S3 + Athena 監査ログ基盤

これらはいずれも「PMF 後 or エンタープライズ契約段階では妥当」だが、Pre-PMF 段階の攻撃面・顧客数・コンプラ要件に照らすと **実装コストと運用負担が便益を大きく上回る**。にもかかわらず、PR ごとに同じ議論（「本当に必要か？」「別の手段はあるか？」）が繰り返され、レビュー効率が低下している。

本 ADR は **「このフェーズではこのレベルで十分」** という合意ラインを固定化し、過剰設計を自動的に棄却できる判断基準を提供する。

### 業界的な裏付け

- **YAGNI（You Aren't Gonna Need It）** — Kent Beck / XP。実需が出るまで作らない。
- **Chesterton's Fence**（ADR-0029 参照）— 既存セーフティを安易に剥がすのは禁止。ただし **未導入のセーフティを「念のため追加」することは禁止の対象外**。「入れる / 入れない」の判断自体が設計行為。
- **Threat Model First** — OWASP。脅威モデル（誰が・何を・どう攻撃するか）を明示せずに防御を追加しても、ROI は測れない。Pre-PMF の脅威モデルは「商用有効顧客 1 桁台 + ボット偶発スキャン」であり、エンタープライズ級ではない。
- **Cost of Delay vs Cost of Rework** — Don Reinertsen。Pre-PMF で最大化すべきは学習速度 (Time-to-Learning) であり、早すぎる最適化が opportunity cost を生む。

## 検討した選択肢

### 選択肢 A: 「念のため全部入れる」（DynamoDB 監査ログ + WAF + ブルート検知 + 長期ログ）

- メリット: いかなるインシデントでも事後調査が可能。
- デメリット:
  - 月額 $10〜$30 のランニングコスト増（WAF $5/月 + ルール $1/million req + DynamoDB 書き込み + S3 転送）
  - 初期実装工数 2〜4 週間（個人開発者の稼働を Growth から剥がす）
  - **Pre-PMF では参照されないログが主**（月商 ¥10,000 未満で監査開示要請を受ける確率 ≈ 0）
  - Assertion 配布・env 管理の対象が増え、#911 型の連鎖デプロイ失敗リスクが上がる

### 選択肢 B: 「何も入れない」

- メリット: 最短で機能開発に集中できる。
- デメリット:
  - ライセンスキー総当たり等、鍵強度に依存する攻撃に対して無防備
  - 顧客説明責任（「あなたの購読が解約された理由」「返金タイミング」）を果たす state 情報が欠落する

### 選択肢 C（採用）: 「鍵強度 + マネージド最小構成 + 顧客向け state 保持」

- メリット:
  - 鍵強度（HMAC-SHA256, 2^160 空間, ADR-0026）でブルートフォースをアルゴリズム的に無効化
  - API Gateway 標準スロットリングで DoS を AWS 側でマネージド防御（追加実装 0 行）
  - AWS Budgets で「暴走コスト」の早期検知（無料, 毎月チェック）
  - 顧客説明責任は既存 `licenses` / `license_keys` の state カラム + Stripe webhook 履歴で達成
- デメリット:
  - 「誰が何を操作したか」の粒度ログは残らない（操作者 = PO 本人であることが前提）
  - 長期ログを後から掘ることができない（CloudWatch Logs 3 日 retention）

Pre-PMF では **「操作者 = 開発者本人」「顧客 1 桁台」** という前提が成立するため、C のトレードオフは許容できる。

## 決定

### 採用 / 不採用マトリクス

| レイヤ | 採用（Pre-PMF） | 採用しない（Pre-PMF） | PMF 後再評価トリガ |
|-------|-----------------|----------------------|-------------------|
| **鍵強度** | HMAC-SHA256（ADR-0026） — 2^160 鍵空間、総当たり不能 | ─ | ─ |
| **レート制限** | API Gateway 標準スロットリング（burst/rate 既定値） | AWS WAF（$5/月 + ルール従量） / カスタムミドルウェア | 利用量が throttling 既定値を恒常的に超過 |
| **コスト監視** | AWS Budgets アラート（無料, 月次 + 閾値超過時 SNS） | Cost Explorer API 定期ポーリング / カスタムダッシュボード | Budgets で検知しきれない sudden burst 事例が発生 |
| **顧客向け監査** | 既存 `licenses.status` / `license_keys.status` state カラム + Stripe webhook 履歴 | 汎用監査ログ用 DynamoDB テーブル / S3+Athena | 法的・規約的な監査要請が発生 |
| **ランタイムログ** | CloudWatch Logs 3 日 retention | 長期保管 / S3 転送 / Athena クエリ | インシデント頻度 or コンプラ要件 |
| **ブルートフォース検知** | HMAC 鍵強度のみ（アルゴリズム防御） | IP 単位カウンタ / Discord アラート / 一時ブロック | 鍵強度を無効化する攻撃手法（量子計算等）の現実化 |
| **コスト上限の機械的停止** | なし（Pre-PMF ではコスト暴走リスク > 停止による機会損失） | Budgets Action による自動停止 | 月額コストが想定の 5 倍を超える事象が 3 回以上 |

### 破棄する過剰設計

本 ADR 確定をもって、以下を **明示的に採用しない**（関連 Issue / PR は Close または supersede）：

| 項目 | 出所 | 破棄理由 |
|------|-----|---------|
| `license_event` 追記ログテーブル | PR #1003 | 状態は既存 `license_keys` で表現可能。発行・失効・消費のタイムスタンプは既存カラム（`issued_at`, `revoked_at`, `revoked_reason`, `consumed_at`）で十分 |
| `ops_audit_log` 汎用監査ログテーブル | #993 / #1009 | 操作者 = 開発者本人、顧客説明責任は state + Stripe webhook で達成可能 |
| ブルートフォース検知（IP カウンタ + アラート） | #1010 | HMAC-SHA256 の鍵強度でアルゴリズム的に不能化済み |
| 監査ログ専用 DynamoDB テーブル案 | 周辺議論 | Pre-PMF のクエリ頻度では過剰。TCO（DynamoDB 書き込み + CloudWatch metrics + インデックス設計）が便益を上回る |
| S3 + Athena 監査ログ基盤 | 周辺議論 | 同上。さらに Athena クエリ実行コスト + Glue カタログ運用が加算される |
| AWS WAF 導入 | #820 周辺 | $5/月 + $1/million req。Pre-PMF のリクエスト量（月 10 万 req 未満）では API Gateway 標準で十分 |

### ADR-0029（Safety Assertion Erosion Ban）との関係

本 ADR は ADR-0029 を **無効化しない**。両 ADR の棲み分けは次のとおり：

| | ADR-0029（劣化禁止） | ADR-0034（本 ADR・過剰採用禁止） |
|---|-----|-----|
| **対象** | すでに導入済みのセーフティ | まだ導入されていないセーフティ |
| **判断の向き** | 「剥がしてよいか」 | 「入れるべきか」 |
| **既定値** | 剥がすな（fail-closed を維持） | 入れるな（YAGNI / 脅威モデルで正当化できない限り） |
| **例外手続き** | 新 ADR で ADR-0029 を supersede する | 新 ADR で ADR-0034 を supersede する |

たとえば「`assertLicenseKeyConfigured()` を warn に落とす」は ADR-0029 違反だが、「新規に `assertAuditLogTableExists()` を追加する」は ADR-0034 のマトリクスに照らして不採用となる。**既存セーフティは守るが、新規セーフティは YAGNI で判断する**、が原則。

### API Gateway スロットリング設定の CDK 変更

本 ADR は方針明文化のみを対象とする。CDK での具体的な throttling 値設定・Budgets アラート設定は **別 Issue として分離** する（本 Issue スコープ外）。

## 結果

### ポジティブ

- 新規 Issue 起票時に **「Pre-PMF マトリクスで採用しないに該当するか」** を機械的に判定でき、レビュー時間が短縮する
- `feature_request.yml` / Issue テンプレートに本 ADR へのリンクを載せることで、AI エージェントが代理起票するケースでも過剰設計が抑制される
- DynamoDB テーブル数・env / secret 数が抑制され、#911 型の連鎖配布失敗リスクが低下する
- 月額ランニングコストが Pre-PMF 想定（$5〜$10/月）から逸脱しない

### ネガティブ / トレードオフ

- **PMF 到達後のセキュリティ刷新コスト**は本 ADR を supersede する新 ADR で受け止める必要がある（「あとで入れる」を明記）
- **監査ログが存在しない期間のインシデント調査は限定的**（state + Stripe webhook 履歴 + CloudWatch 3 日分のみ）。PO はこのトレードオフを了承済み
- 「本当はもう少し入れたい」というレビュー心理を機械的に抑制するため、ADR への参照が強制力を持つ。逸脱したい場合は必ず supersede ADR を先に起票すること

### PMF 後の再評価トリガ

以下のいずれかに該当した時点で、本 ADR を supersede する新 ADR を起票する：

1. **規模トリガ** — 月商 ¥10,000 超 または 有料顧客 100 人超
2. **インシデントトリガ** — セキュリティインシデント（不正アクセス・情報漏洩・サービス停止 30 分超）が 1 件でも発生
3. **外部要件トリガ** — 法的 / 規約 / 契約（B2B エンタープライズ契約、GDPR 対応要請、PCI DSS 等）で監査ログや WAF が求められる
