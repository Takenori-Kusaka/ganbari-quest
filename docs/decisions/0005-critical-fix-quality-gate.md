# ADR-0005: Critical バグ修正の品質ゲート

| 項目 | 内容 |
|------|------|
| ステータス | accepted |
| 日付 | 2026-04-09 |
| 関連 Issue | #543, #611, #612 |

## コンテキスト

#543（ダイアログフリーズ）の Critical 修正が以下の理由で不完全だった:
- Issue で提案された3つの対策のうち1つだけ実装して closes
- E2E テストの追加なし（回帰テストが存在しない）
- セルフレビュー + Copilot のみでマージ
- 結果: #611 として再発

## 決定

`priority:critical` のバグ修正は、通常の Done 基準に加えて以下を**全て**満たすこと:

1. **回帰テスト（E2E）を同一PR内で追加** — ガード条件の追加だけでは不十分。テストがなければ次のリファクタリングで消える
2. **Issue の Acceptance Criteria を全項目完了** — 部分実装で closes は禁止。未実装の対策がある場合は別 Issue に切り出してからクローズ
3. **Issue で提案された対策を全て実装** — 提案の一部だけ実装して「直った」とするのは対症療法であり根本解決ではない
4. **全5年齢モードで実機検証** — スクリーンショットを PR に添付
5. **リネーム/リファクタリングとの依存関係を確認** — 直近30日に同じファイルを変更した PR がないかチェック

## 結果

- Critical 修正のマージに時間がかかるが、再発リスクは大幅に低下
- 「速く直す」よりも「正しく直す」を優先する文化が定着する

## 教訓

- **対症療法は根本解決ではない** — ガード条件は「入口を塞ぐ」だけで、構造的な問題は残る
- **テストのない修正は修正ではない** — コードが変更される限り、テストなしの修正は必ず壊れる
- **セルフレビューは検証ではない** — 自分の盲点は自分では見えない
