# 0011. 0-2 歳スコープ判断 — baby モードは「親の準備モード」として扱う

- **Status**: Accepted
- **Date**: 2026-04-21
- **Related Issue**: #1299 (#1298 umbrella 派生 B3-ADR)

## コンテキスト

本プロダクトのコア哲学は「親の願い（子供の現実活動の充実）の可視化 + お金/ポイント変換」であり、daily engagement loop は**子供自身がアプリを操作してポイントを貯める**ことで成立する。

5 段階の年齢モード (baby / preschool / elementary / junior / senior) のうち、**0-2 歳 (baby)** にこの哲学を適用すると以下の構造的問題が発生する:

1. **多重アクション問題**: 0-2 歳は食事・排泄・睡眠等の記録が主で、ぴよログ / ピジョンアプリ等既存育児記録アプリと**記録軸が完全重複**する。本プロダクトを開くこと自体が親の手間の重複になる
2. **電子機器露出問題**: 0-2 歳に対する電子機器利用は医学的に推奨されない (WHO 推奨では 2 歳未満にスクリーンタイムゼロ)。子供がアプリを操作する前提のゲーミフィケーション UI を 0-2 歳に出すこと自体が本プロダクトのブランドに反する
3. **daily engagement loop が成立しない**: 0-2 歳は自律的にアプリを操作できず、親が代理入力する形になるため、本来のゲーミフィケーション効用 (達成感 / 自己効力感 / 継続動機) が子供に還元されない

一方、**0-2 歳の子供を持つ親が本プロダクトに登録したいニーズ**は実在する:

- 3 歳以降に備えて家族登録 / プラン契約を先に済ませたい
- 祖父母からの「おこづかい」を事前にポイント積立して記録したい
- 兄弟姉妹と一緒にまとめて管理したい (上の子が 4 歳、下の子が 1 歳 等)

このニーズを切り捨てると、上の子が preschool 以上の家庭まで取り逃す。

## 決定

### 1. コアターゲットを 3-18 歳に再定義

本プロダクトの **ゲーミフィケーション UI 対象**は **preschool / elementary / junior / senior** の 4 段階とする。LP / 企画書 / 並行実装 doc の「5 段階モード」表記を**「4 段階コア + 準備モード」**に改める。

### 2. baby モードは「親の準備モード」として残す

- 子供登録は可能 (0-2 歳選択可)
- UI は**成長待機画面 + 親による初期ポイント入力画面**に書き換える
- 子供向けゲーミフィケーション UI (活動記録 / スタンプ / クエスト / レベル) は適用しない
- ルート設計は既存 `src/routes/(child)/[uiMode=uiMode]/` の `baby` バリアントで実装 (別ルート起こさない)

### 3. マーケット / seed データの baby 向け削除

- `src/lib/server/marketplace/` の baby 向けコンテンツ (活動パック 3 種 + チェックリスト 3 種) は削除
- `src/lib/server/demo/demo-data.ts` / seed の baby 向け活動 (はいはい等) は削除
- `BABY_FEATURES` フラグは既に全 OFF のため整合、別途のコード変更不要

### 4. データマイグレーション方針

既存 baby レコードは DB TEXT カラム制約なしのため強制マイグレーション不要。新準備モード UI でそのまま表示する。

## 結果

- 0-2 歳の親に対して「準備モードで登録 → 3 歳誕生日で preschool へ自動移行」という自然な導線ができる
- 子供向けゲーミフィケーション UI の品質担保対象が 5 → 4 年齢モードに縮小。E2E / Storybook / variants 設計の保守コスト減
- LP 上「0 歳から使える」訴求は削除、「3-18 歳のお子さまに」を標準表現とする
- 0-2 歳の兄弟姉妹を持つ家庭も「上の子 + 準備モード」で取り逃さない

### トレードオフ

- **既存 baby UI の書き換え**が必要 (スクラップ&ビルド)
- **baby 単独登録**しているテナントは成長待機 UI のみ表示になり、実質 UI 体験が縮退する。ただし既存 B2C ユーザーは 0 人のため実害なし

## Alternatives considered

### Route α (保守的): 既存 baby UI を薄くバリアント

- メリット: 移行コスト小
- デメリット: 現実利用ないゲーミフィケーション UI を残すのは逆にコア哲学 (Anti-engagement / ADR-0012) に反する
- → 却下

### Route β (急進的): childV3toV4 で baby → preschool 強制移行

- メリット: データモデル簡素化
- デメリット: 親の準備ニーズ (子供登録して事前ポイント積立等) を満たせない
- → 却下

### Route γ (採用): 既存ルート内で baby を別 UI に書き換え + 準備モード意味付け

- メリット: 哲学整合 + ニーズ充足
- デメリット: UI 書き換えコスト中程度
- → 採用
