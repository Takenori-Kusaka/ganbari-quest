# ADR-0003: 設計書は実装の Single Source of Truth である

| 項目 | 内容 |
|------|------|
| ステータス | accepted |
| 日付 | 2026-04-09 |
| 関連 Issue | #607, #609, #617 |

## コンテキスト

スタンプカードの正仕様（5スロット・画像アセット・リアルデザイン等）をユーザーが **4回** 説明する事態になった。原因は、会話で決まった仕様が設計書に一度も記載されなかったこと。

監査の結果:
- UI設計書: 14機能が未記載（カバレッジ約65%）
- API設計書: 22エンドポイントが未記載
- DB設計書: 7テーブルが未記載
- アーキテクチャ図: ゼロ（drawio/mermaid/SVG/PNG いずれも存在しない）
- 存在すべき設計書: 16件が欠落（ADR, COPPA, SLI/SLO 等）

## 決定

1. **設計書の更新は Done 基準に含む** — Issue/PR 完了時に設計書が同期されていなければ Done にできない
2. **会話で確定した仕様はその場で設計書に反映** — Issue 起票だけでは不十分
3. **アーキテクチャ図は drawio 形式** で `docs/design/diagrams/` に保存
4. **重要な意思決定は ADR として記録** — `docs/decisions/` に蓄積
5. **Claude Code の memory はユーザーローカルであり、チーム共有には使えない** — 共有すべき知識は全てリポジトリ内に置く

### 設計書更新が必要なケース

| 変更種別 | 更新すべき設計書 |
|---------|---------------|
| API エンドポイントの追加・変更 | `07-API設計書.md` |
| DB テーブル・カラムの追加・変更 | `08-データベース設計書.md` |
| UI 機能・画面・オーバーレイの追加・変更 | `06-UI設計書.md` |
| AWS インフラ構成の変更 | `13-AWSサーバレスアーキテクチャ設計書.md` |
| 認証・セキュリティ関連の変更 | `14-セキュリティ設計書.md` |
| 重要な技術的意思決定 | `docs/decisions/` に ADR を追加 |

## 結果

- 新しい PO/開発者がリポジトリを見るだけで全仕様を把握できる
- 会話のコンパクション時に仕様が消失しない
- PR レビュー時に「設計書が更新されているか」をチェックできる

## 教訓

- **Issue に仕様を書いて「設計書は後で」は必ず忘れられる** — 設計書更新を別タスクに切り出した時点で優先度が下がり、永遠に実行されない
- **Claude Code の memory は個人の記憶補助であり、チームの制度的記憶ではない** — git 管理されないものは共有できない
- CLAUDE.md にルールを書いても守られないなら、PR テンプレートやCI で強制する仕組みが必要
