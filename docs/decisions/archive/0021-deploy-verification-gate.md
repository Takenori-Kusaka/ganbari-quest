## コンテキスト

複数のIssueがCOMPLETEDでクローズされているにもかかわらず、本番環境では修正が反映されていない問題が構造的に再発していた。

### 証拠

| Issue | 内容 | 再発回数 |
|-------|------|---------|
| #405/#700 | Discord招待リンク残存 | 4回 |
| #588/#706 | サインアップ規約閲覧必須化 | 2回 |
| #590 | ライセンスキーフロー改善 | 2回 |

### 根本原因

1. **site/ と SvelteKit の二重管理**: `site/` は静的HTMLでSvelteKitビルドパイプラインの外。Issue対応時にアプリ側（`src/`）だけ修正して`site/`を放置するパターン
2. **デプロイ検証の欠如**: PRマージ=完了と判断し、本番環境での反映確認を行っていない
3. **CIの盲点**: `site/` の変更がCIの変更検知対象外だった

## 決定

### 1. CIに site/ 検証ジョブを追加

- `scripts/check-site-html.mjs` で内部リンク整合性と必須要素を検証
- CI変更フィルタに `site/**` を追加（`site/` 変更時にCIが起動する）
- `check-forbidden-terms.mjs` も site/ 変更時に実行

### 2. PRテンプレートにデプロイ検証セクションを追加

マージ後に以下を確認するまでIssueをcloseしない:
- デプロイワークフローの成功確認
- 本番URLでの変更反映確認

### 3. 禁止語チェックによる再発防止

`check-forbidden-terms.mjs` に無効化済みリンク等を禁止語として追加し、CIレベルで再発を自動検出する。

## 結果

- site/ 変更の見落としがCIレベルで検出される
- Issue完了基準に「本番反映確認」が明示的に含まれる
- 禁止語チェックにより、過去に除去した要素の再混入を自動検出
