## コンテキスト

ダイアログ管理で4回の不具合が発生した:

1. **#543**: 複数オーバーレイが同時 open → 画面操作不能。ガード条件のみで closes
2. **#611**: 再発。ガード条件では不十分と判明。ダイアログキューを Issue で提案
3. **#633**: OverlaysSection に4種のキューを実装。ただしリザルトダイアログはキュー外
4. **#671**: コンボボーナスダイアログが無限ループ。`$effect` + `invalidateAll()` のリアクティブループ

ADR-0002 で「ダイアログキュー必須」と決定したが、キューだけでは不十分だった。根本原因は**ダイアログの表示制御が3つの独立した仕組みに分散し、整合性が保証されていない**こと:

1. 手動の `open` フラグ群（`resultOpen`, `confirmOpen` 等、10個以上）
2. OverlaysSection のキュー（4種のみ）
3. `$effect` による自動表示トリガー（`invalidateAll()` 後に再発火）

パッチの上にパッチを重ねた結果であり、これ以上の修正は不可能。

## 決定

### 既存のダイアログ管理を全て破棄し、有限ステートマシン（FSM）で再実装する

#### 原則

1. **ダイアログの表示は FSM の状態遷移のみで制御する** — `xxxOpen = true` のような直接操作を禁止
2. **同時に表示されるダイアログは常に1つ** — FSM の状態が `idle` 以外のとき、他のダイアログは開けない（キューに入る）
3. **`$effect` からダイアログを開かない** — データ読み込み後のダイアログ判断は FSM の `onDataLoad()` メソッドが行う
4. **`invalidateAll()` はダイアログ状態に影響しない** — FSM が現在のステートを考慮して判断する
5. **FSM は純粋関数で実装する** — テスタブル、予測可能、副作用なし

#### 廃止するもの

- `OverlaysSection.svelte` の内部キュー（`queue`, `activeOverlay`, `enqueueOverlay`, `dequeueOverlay`）
- ホーム画面の個別 `$state` フラグ群（`resultOpen`, `confirmOpen`, `levelUpOpen`, `rewardOpen`, `stampPressOpen`, `birthdayModalOpen`, `messageOpen`, `adventureOpen`）
- ダイアログ表示をトリガーする `$effect` ブロック
- `handleXxxClose` → `checkYyy` → `checkZzz` のハンドラチェーン

#### 新規作成するもの

- `src/lib/features/child-home/dialog-state-machine.ts` — FSM 本体
- `src/lib/features/child-home/dialog-state-machine.test.ts` — 全遷移の単体テスト
- `OverlaysSection.svelte` の書き換え — FSM の状態に基づくレンダリング

## 結果

- ダイアログの状態遷移が1箇所で定義され、予測可能になる
- 「閉じたら次のダイアログ」のロジックが FSM の遷移テーブルに集約される
- `$effect` の副作用によるループが構造的に不可能になる
- 新しいダイアログ追加時、遷移テーブルに1行追加するだけ

## 教訓

- **キューは構造の一部であり、構造そのものではない** — ADR-0002 で「キュー必須」としたが、キューを追加しても3つの独立した制御機構が残った。構造の統一（FSM）が必要だった
- **3回パッチを当てて直らないものは、4回目のパッチでも直らない** — スクラップ＆ビルドの判断を3回目（#633）の時点で行うべきだった
- **Issue 起票者がアーキテクチャを指定しなければ、開発チームは最小工数を選ぶ** — 「キューまたはステートマシン」と書けば必ずキューが選ばれる。FSM を指定すべきだった
