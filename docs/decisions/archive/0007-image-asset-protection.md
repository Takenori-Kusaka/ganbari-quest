# ADR-0007: 画像アセットが存在する機能を絵文字に戻すことは明示的デグレ

## コンテキスト

#607 の根本原因は、#387（おみくじ→スタンプ押印リデザイン）で `static/assets/stamps/` の画像アセット6枚を絵文字表示に置き換えたことだった。画像アセットは `static/assets/stamps/` に残っていたが、コードからの参照がなくなり「孤児ファイル」状態になった。

`asset-catalog.md` には「絵文字NG」基準が明記されていたが、リファクタリング時のチェックリストに含まれておらず、防御策として機能しなかった。

子供向けゲーミフィケーションアプリにおいて、画像アセットの品質は**プロダクトの核心**であり、「絵文字の方がシンプル」は開発者視点の合理化に過ぎない。

## 決定

1. **`static/assets/` に画像アセットが存在する機能は、リファクタリング・リデザイン時にも画像を使用し続ける**
2. 画像→絵文字への変更は「デグレ」として扱い、PR レビューで差し戻す
3. `asset-catalog.md` の「絵文字NG」リストに該当する要素を絵文字で実装した PR はマージ不可
4. 画像アセットを削除する場合は、事前に Issue で「削除理由」を明示する

### PR レビューチェック
- [ ] `static/assets/` に未参照の画像がないか（`grep` で参照チェック）
- [ ] `asset-catalog.md` の「絵文字NG」に該当する要素が絵文字で実装されていないか
- [ ] 画像表示を絵文字に変更するコード差分がないか

## 結果

- 画像アセットが「存在するが未参照」という状態が発生しにくくなる
- リデザイン時に画像品質が維持される
- レビュアーが明確な基準で差し戻しできる

## 教訓

- **「シンプルにする」リファクタリングがユーザー体験を劣化させることがある** — 技術的なシンプルさとユーザー体験の品質は別軸
- **画像アセットの存在そのものが仕様の証拠** — `static/assets/` にファイルがあるなら、誰かが意図を持って作成している
- **`asset-catalog.md` だけでは防げない** — レビュー時のチェックリストとして明示的に組み込む必要がある
