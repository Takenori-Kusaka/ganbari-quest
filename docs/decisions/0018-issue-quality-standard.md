# ADR-0018: Issue 起票は根本原因の特定と構造的解決策の提示を必須とする

## コンテキスト

ダイアログ管理の不具合が **4回** 繰り返された（#543 → #611 → #633 → #671）。

| 回 | Issue | 起票時の提案 | 実装結果 | 再発 |
|----|-------|------------|---------|------|
| 1 | #543 | ガード条件追加 + ダイアログキュー（2案併記） | ガード条件のみ実装、キュー未実装で closes | Yes |
| 2 | #611 | ダイアログキュー実装 | 4種のオーバーレイのみキュー化。リザルトダイアログはキュー外 | Yes |
| 3 | #671 | 「ステートマシンまたは統一キュー」（曖昧な2案併記） | 未着手 | — |

同様にスタンプカードでも、#607 の修正が不完全なまま #673 のデグレを生んだ。

**根本原因**: Issue 起票者（PO / Claude Code）が症状の記述と曖昧な提案に終始し、根本原因の特定と構造的な解決策の設計を怠った。Issue に「ガード条件追加でも良い」「ステートマシンまたはキュー」のような選択肢を併記すると、開発チームは必ず工数の少ない方を選ぶ。

Issue の品質が低ければ、どれだけ PR テンプレート・レビュープロセス・CI を強化しても、**間違った方向に正確に進むだけ**である。

## 決定

### 1. Issue 起票時の必須要件

#### A. 根本原因の特定（「なぜ5回」）

Issue 本文に**根本原因**を明記すること。症状の記述だけでは不十分。

- 同じ領域で過去に Issue がある場合、その Issue がなぜ問題を解決しなかったかを分析する
- 根本原因が特定できない場合は、その旨を明記し「調査 Issue」として切り出す。推測に基づく修正 Issue を起票しない

#### B. 構造的解決策の提示（スクラップ＆ビルド前提）

- **再発している問題**: 既存の実装を修正するのではなく、**スクラップ＆ビルド**（再設計・再実装）を前提とする。パッチの上にパッチを重ねることを禁止
- **設計の指定**: 「何を使うか」（ステートマシン、キュー等）を具体的に指定する。「AまたはB」の曖昧な併記は禁止。どちらが正しいかを Issue 起票者が判断する
- **境界条件の列挙**: 解決策がカバーすべき全てのケースを Acceptance Criteria に列挙する。開発チームが「書かれていなかったから対応しなかった」と言える状態を作らない

#### C. 過去の Issue との関連分析

- 同一領域の過去 Issue を必ず参照する
- 過去の修正がなぜ不十分だったかを記載する
- 「今回こそ根本解決である」と言える理由を明示する

### 2. Issue 起票時のチェックリスト

```markdown
## 起票前チェック（PO必須）
- [ ] 根本原因を特定した（症状ではなく原因を記載している）
- [ ] 同一領域の過去 Issue を確認した（過去の修正がなぜ不十分だったか記載）
- [ ] 解決策は1つに絞った（「AまたはB」の併記をしていない）
- [ ] 再発問題の場合、スクラップ＆ビルドを前提とした（既存コードへのパッチではない）
- [ ] Acceptance Criteria に全境界条件を列挙した
- [ ] 設計上の制約を明記した（使用すべきパターン、禁止事項）
```

### 3. 「曖昧な Issue」の定義と禁止事項

以下に該当する Issue は起票不可。起票前に PO が検討を完了すること:

| 禁止パターン | 例 | あるべき姿 |
|-------------|-----|----------|
| 解決策の併記 | 「ステートマシンまたはキューで」 | 「有限ステートマシンで実装する。理由: ...」 |
| 症状だけの記述 | 「ダイアログが閉じない」 | 「$effect の依存関係ループが原因。具体的には line XX の ...」 |
| 対症療法の許容 | 「ガード条件を追加」 | 「全ダイアログをステートマシン管理に移行。ガード条件は廃止」 |
| 範囲の曖昧さ | 「オーバーレイを修正」 | 「OverlaysSection + ResultDialog + 全 $effect を再設計」 |
| 過去 Issue の無視 | （参照なし） | 「#543, #611, #633 が全て不十分だった理由は ...」 |

### 4. CI による Issue 品質チェック（将来）

Issue テンプレートに上記チェックリストを組み込み、未チェック項目がある場合に bot がコメントで警告する。

## 結果

- Issue 起票に時間がかかるが、手戻り（再発 → 再起票 → 再実装）のコストが大幅に減少する
- 「速く起票する」よりも「正しく起票する」を優先する文化が定着する
- 開発チームが「Issue に書かれた通りに実装したが不十分だった」と言い訳できない品質の Issue が提供される

## 教訓

- **Issue は仕様書である** — Issue の品質が低ければ、実装の品質も低くなる。PR テンプレートや CI でカバーできるのは「正しい方向への実装品質」であり、「方向そのもの」は Issue で決まる
- **選択肢の併記は責任の放棄** — 「AまたはB」と書くことは、判断を開発チームに丸投げすることであり、PO の職務怠慢
- **スクラップ＆ビルドは恐れるべきではない** — パッチの上にパッチを重ねたコードは、新規実装より保守コストが高い。再発3回目以降は必ずスクラップ＆ビルド
- **Issue 起票のコストを惜しむと、プロジェクト全体のコストが増える** — 30分の検討を省いた Issue が、3回の再実装（各数時間）を生む
