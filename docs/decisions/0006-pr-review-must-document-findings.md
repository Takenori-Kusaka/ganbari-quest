# ADR-0006: PR レビューは文書化された指摘を必ず出力する

| 項目 | 内容 |
|------|------|
| ステータス | accepted |
| 日付 | 2026-04-09 |
| 関連 Issue | #619, #613, #614, #615, #616 |

## コンテキスト

2026-04-09 に8件の Draft PR を一括レビュー・マージした際、コンフリクト解消・CI待ち・マージ順序管理のプロセス作業に追われ、コードレベルのレビュー指摘を**一切出力せずに全件マージした**。

結果として以下が本番デプロイされた:

- CSS ハードコード違反 3件（`bg-white/90`, `bg-red-50` 等 — #613, #614）
- テストカバレッジ欠落 3件（showToggle, hasExistingConsent 等 — #615, #616）
- 設計書更新が必要な6件のPRで設計書更新ゼロ（#609）

「指摘ゼロ」は「問題ゼロ」ではなく「レビューしていない」と同義だった。

## 決定

### 1. 全PR レビューで指摘事項を文書化する

PR をマージする前に、レビュアーは以下のカテゴリで指摘を文書化する:

| Severity | 説明 | マージ条件 |
|----------|------|-----------|
| **Critical** | セキュリティ脆弱性、データ損失リスク、本番障害直結 | 修正必須。マージ不可 |
| **High** | コーディングルール違反、テスト欠落、設計書未更新 | 修正必須。マージ不可 |
| **Medium** | パフォーマンス懸念、可読性改善、リファクタ候補 | 修正推奨。follow-up Issue 許容 |
| **Low** | スタイル提案、コメント追加、命名改善 | 任意。マージ可 |

### 2. 「指摘なし」の場合の記録

レビューの結果指摘が見つからなかった場合も、以下を明記する:

```
✅ Reviewed: lint rules, CSS tokens, test coverage, design doc sync
No findings.
```

これにより「レビューしたが指摘なし」と「レビューしていない」を区別する。

### 3. チェック項目

全PR レビューで以下を確認する:

- [ ] CSS: routes 配下に hex カラー・Tailwind デフォルト色のハードコードがないか
- [ ] テスト: 新規/変更コードに対応するテストがあるか
- [ ] 設計書: DB/API/UI 変更があれば `docs/design/` が更新されているか
- [ ] 型安全: `as any` や non-null assertion の不適切な使用がないか
- [ ] CLAUDE.md: プロジェクトルールへの違反がないか

### 4. Medium 以上の指摘がある PR

修正後に再レビューを行い、指摘事項が解決されたことを確認してからマージする。

## 結果

- プロセス作業（CI待ち、コンフリクト解消）にリソースを取られても、レビュー品質を犠牲にしない
- 「CI green = 品質保証」という誤解を排除する。CI は前提条件であり品質の証明ではない
- レビュー記録が残ることで、同種の見落としを防止する学習サイクルが回る

## 教訓

> プロセスの圧力（8件同時マージ、CI待ち、コンフリクト解消）は、品質チェックを省略する正当な理由にはならない。圧力がかかるときほど、チェックリストに忠実であるべき。
