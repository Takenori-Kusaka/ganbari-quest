# ADR-0001: リネーム時は旧識別子の後方互換を必ず確保する

| 項目 | 内容 |
|------|------|
| ステータス | accepted |
| 日付 | 2026-04-06 |
| 関連 Issue | #537, #539, #572 |

## コンテキスト

Issue #537 の要求で `/kinder/home` 等を `/preschool/home` 等にリネームした際、Issue 作成時点で「旧URLからのリダイレクト」を Acceptance Criteria に書き忘れた。PR #539 マージ後、NUC 本番環境で `/kinder/home` が 404 となり「アプリが壊れた」と認識される致命的な障害を引き起こした（#572）。

DB スキーマは `LEGACY_UI_MODE_MAP` で互換維持していたが、**URL も「ユーザー側に保存される識別子」**であることに思い至らなかった。

## 決定

URL/パス・DB識別子・API・Cookie・PWA start_url 等を**リネームする Issue を作成するとき**、Issue 本文の Acceptance Criteria に**旧→新の後方互換ルート**を必ず含める。

### チェックリスト（Issue 作成時に転記）

- [ ] **URL/パスのリネーム** → 旧URLからのリダイレクト経路を追加するか、削除を明示
- [ ] **DB識別子のリネーム** → 後方互換マッピング（`LEGACY_*_MAP`）を追加するか、マイグレーションで全更新
- [ ] **API エンドポイントのリネーム** → 旧エンドポイントのエイリアス or 非推奨期間
- [ ] **Cookie/LocalStorage キーのリネーム** → 旧キーからの読み取り fallback
- [ ] **PWA start_url の変更** → manifest.json 更新 + 旧 start_url の404対策
- [ ] **ブックマーク・外部リンクの影響評価** → ユーザーが保存している可能性のある経路をリストアップ
- [ ] **直近30日の修正PRとの依存関係** → リネーム対象ファイルに対する修正が新ファイルにも適用されていることを確認

## 結果

ユーザーが保存しうる識別子（URL、ブックマーク、ホーム画面ショートカット、Cookie、外部リンク）は git history では追跡できない。Issue 作成者の責任として常に意識すること。

## 教訓

- リネームは「ファイル名を変える」だけではない。ユーザーの手元に残る全ての参照を洗い出す必要がある
- DB の互換は考慮できたのに URL の互換を見落としたのは、「技術的に近い概念でも別々にチェックしないと漏れる」ことの証左
