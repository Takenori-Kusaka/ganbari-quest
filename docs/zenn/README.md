# Zenn コンテンツ戦略と執筆ガイドライン

このドキュメントは、本プロダクト開発で得られた技術的知見を Zenn で公開する際のコンテンツ戦略と、具体的な執筆指針をまとめたものです。

## 1. コンテンツ戦略：Scrap / Article / Book の使い分け

Zenn のコンテンツタイプを、目的と完成度に応じて以下のように使い分け、プロダクトの知見を体系的に蓄積・発信します。

| コンテンツ | 目的・用途 | 格納場所 |
| :--- | :--- | :--- |
| **Scrap** | 作業ログ、試行錯誤の過程、未整理のメモ | `scraps/` |
| **Article** | 完成した単一トピックの技術ノウハウ | `articles/` |
| **Book** | 複数の記事を体系化した技術書 | `books/` |

### Scrap：作業ログ・試行錯誤・メモ
- **目的**: 結論が出る前の「生の状態の知見」を記録・共有するための場所。
- **用途**:
  - 新機能の技術検証（PoC）ログ
  - 障害調査のリアルタイムな記録
  - RAGのパラメータ調整など、試行錯誤のメモ
- **特徴**: スレッド形式で気軽に追記できるため、リアルタイムな情報共有や、個人の学習ログに適しています。

### Article：まとまった技術ノウハウ
- **目的**: 特定のトピックについて完成した知見を、読みやすい形で公開するためのメイン手段。
- **用途**:
  - 特定機能のアーキテクチャ解説
  - バグの根本原因と恒久対策の報告
  - 技術選定の理由と具体的な実装詳細
- **特徴**: Scrapsで育てたネタを、後から読む人を意識して再構成した完成形。1記事1トピックで完結させます。

### Book：体系化された技術書
- **目的**: 複数の記事を組み合わせ、一つのテーマを体系的に解説するための長文コンテンツ。
- **用途**:
  - 「がんばりクエスト開発録」
  - 「Svelte と Drizzle で作るマルチテナントSaaS設計」
  - 「LLM連携機能 実装大全」
- **特徴**: 複数の `Article` として蓄積された知見を、章立てて再編集し、より深い理解を促す「技術白書」としての役割を担います。

**ワークフロー**:
日々の検証・調査メモを `Scrap` に残し、一段落ついた知見を `Article` として公開します。そして、`Article` が十分に溜まった段階で、それらを再編集し `Book` として体系化する、という3段階のフローを基本とします。

## 2. ディレクトリ構造

Zenn のコンテンツは、リポジトリ内の `/docs/zenn` ディレクトリで一元管理します。

```
docs/zenn/
├── README.md      # このファイル
├── articles/      # 記事コンテンツ（.md）
│   └── *.md
├── books/         # 本コンテンツ
│   └── <book-slug>/
│       ├── config.yaml
│       └── *.md
└── scraps/        # スクラップの元ネタ・メモ（.md）
    └── *.md
```

- **`articles/`**: Zenn CLI の `zenn new:article` で作成される Markdown ファイルを格納します。
- **`books/`**: `zenn new:book` で作成される本ごとのディレクトリを格納します。各ディレクトリには `config.yaml` と複数の章（Markdown ファイル）が含まれます。
- **`scraps/`**: Zenn のスクラップに投稿する前のメモや、ローカルでの試行錯誤の記録を Markdown 形式で保存します。ファイル名は `YYYY-MM-DD-topic.md` のような形式を推奨します。

## 3. 記事執筆ガイドライン

`articles/` に格納する記事を作成する際の、具体的なフォーマットやスタイルガイドです。

### 3.1. 目的とターゲット
- **目的**:
  - プロダクト開発の知見を、再利用可能な**実務ドキュメント**として公開する。
  - 社外エンジニアにも再現・応用可能で、プロダクト理解・採用ブランド向上につなげる。
- **想定読者**:
  1. **自社メンバー**: 特にオンボーディング中の開発者。
  2. **類似課題を持つ開発者**: SaaSエンジニアなど。
  3. **外部のエンジニア**: プロダクトや技術スタックに興味がある候補者。
- **ポリシー**:
  - **実務で役立つこと**: 読者の課題解決のヒントとなる情報を提供する。
  - **再現可能なこと**: 読者が実際に試せる具体的な手順やコードを含める。
  - **具体的なケース**: 抽象論でなく、プロダクトでの実例をベースにする。

### 3.2. 記事の種類とポリシー
Zenn の記事は、以下の3カテゴリに分類します。

#### 1. Try系（やってみた・検証系）
- **例**: 「Svelte 5 Runes と Ark UI で実現するアクセシブルなコンポーネント設計」
- **ポリシー**:
  - ゴール、前提環境、手順、結果、考察を必ず含める。
  - 成功だけでなく、失敗やハマりどころも共有する。

#### 2. 解説系（設計・アーキテクチャ）
- **例**: 「我が家のSaaS『がんばりクエスト』で採用しているゲーミフィケーション設計」
- **ポリシー**:
  - 図や擬似コードを活用し、構造を視覚的に伝える。
  - 「なぜその設計にしたか」「採用しなかった選択肢」といった意思決定の背景を説明する。

#### 3. レポート/ナレッジ系（運用・障害）
- **例**: 「Playwright の E2E テストで発生した Flaky Tests の原因と対策」
- **ポリシー**:
  - 「事実 → 説明 → 検証 → 結果」の構成で記述する。
  - 個人の感想と客観的な事実を明確に区別する。

### 3.3. 構成・見出し・粒度
#### 見出しルール
- **見出しレベル**:
  - `#` (h1) は記事タイトルとして Zenn が自動で付与するため、本文中では使用しない。
  - 本文は `##` (h2) と `###` (h3) を基本とし、最大2階層で構成する。`####` (h4) は極力避ける。
- **章立てテンプレート (例)**:
  - `## 背景・目的`
  - `## この記事で得られること`
  - `## 前提・環境`
  - `## 実装・手順`
  - `## 結果と考察`
  - `## まとめ`

#### 粒度の指針
- **1記事1トピック**: テーマを絞り、1記事で説明する新しい概念は3つまでを目安とする。
- **コードの扱い**:
  - 全体は Gist や GitHub リポジトリへのリンクで共有し、本文には要点のみを抜粋する。
  - コピペで動作する最小限のサンプルコードを提供することを推奨する。
- **記事の分割**: `##` レベルの見出しが7つを超える場合は、記事の分割やシリーズ化を検討する。

### 3.4. スタイルガイド
#### 全体方針
- **結論から書く**: 冒頭で「この記事で分かること」や結論を提示する。
- **流し読みへの配慮**: 箇条書き、小見出し、表を多用し、文章の塊を避ける。
- **前提の明示**: 対象読者や技術バージョン、環境などを最初に明記する。

#### 具体的ルール
- **Must (必須)**:
  - 技術用語・製品名は正式名称で記述する (例: TypeScript, SvelteKit)。
  - 事実と意見を明確に分離する (例: 「〜と感じた」)。
  - スクリーンショット等に含まれる機密情報は必ずマスクする。
  - 社の情報公開ポリシーを遵守する。
- **Want (推奨)**:
  - 一文を80字以内に収める。
  - 抽象的な表現 (例: 「高速化された」) は具体的な数値で示す。
  - 図、表、コードブロックにはキャプションを添える。

### 3.5. 公開範囲・タグ・シリーズ
#### 公開範囲
- **原則 Public**: プロダクトの技術的知見は、コミュニティへの貢献として原則公開する。
- **Private の場合**: 機密性が高い情報やセキュリティリスクを含む内容は、社内ドキュメントに留め、Zenn では抽象化して記述する。

#### タグ
- **必須タグ**: メイン技術 (例: `svelte`, `typescript`, `drizzle`) を含める。
- **プロジェクトタグ**: 独自のタグ (例: `ganbari-quest`) を付与し、関連記事を一覧しやすくする。
- **数**: 1記事あたり3〜6個を目安とする。

#### シリーズ
- **用途**: 複数記事にわたるテーマや、時系列で変化する内容をまとめる際に利用する。
- **ルール**:
  - 各記事の冒頭にシリーズ一覧へのリンクを設置する。
  - 各記事が単体でも読めるよう、前提知識の再掲や前回記事へのリンクを記載する。

### 3.6. 運用・レビューフロー
#### 執筆プロセス
1. **下書き**: 箇条書きで「背景」「ゴール」「やったこと」などを書き出す。
2. **骨組み作成**: 先に見出し (`##`, `###`) を作り、全体の構成を練る。
3. **肉付け**: 本文、コード、図を追加する。
4. **セルフレビュー**: 時間を置いてから、「半年後の自分でも再現できるか？」という視点で読み返す。

#### レビュープロセス
- **重要な記事**: 設計思想や障害レポートなどは、チーム内でレビューを行う。
  - **事実確認**: 実装担当者が内容の正しさを確認する。
  - **表現確認**: 別の開発者が読者の視点で分かりやすさを確認する。
- **フィードバック**: レビューコメントは「誤りの指摘」「補足提案」「表現の改善」など、レベル感を分けて行う。

### 3.7. textlint による品質チェック
技術書・記事の品質（誇張・曖昧表現の排除、文体の統一）を担保するため、`textlint` を導入しています。

#### ローカルでの実行
執筆中やコミット前に、以下のコマンドでチェックを実行してください。
```bash
npx textlint "docs/zenn/**/*.md"
```
- **Error**: 文体の混在（である/ですます）や明確なルール違反。原則として修正必須です。
- **Warning**: （今後設定される場合）文脈によっては許容される可能性があるもの。ノイズであれば無視するか、ルール調整を検討してください。

#### CI 統合方針
GitHub Actions (`zenn-lint.yml`) により、`docs/zenn/**` への変更時に自動で lint が実行されます。
- 現在は導入初期のため `continue-on-error: true` (warning レベルの運用) としており、CI は失敗しません。
- 既存ドキュメントの修正が完了し、ノイズが収束した段階で hard-fail（エラー時に CI を落とす）へ移行する予定です（別 Issue で対応）。
