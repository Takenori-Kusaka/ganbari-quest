## 1. 概要

がんばりクエストのエラーハンドリング戦略を定義する。
子供が主要ユーザーであるため、エラーメッセージは**恐怖や不安を与えない**ことを最優先とする。

### 1.1 設計原則

1. **子供を怖がらせない** — 「エラー」「失敗」「問題」等のネガティブワードを子供画面で使わない
2. **親には正確に伝える** — 管理画面では技術的な詳細を適切に表示
3. **サイレント回復** — 可能な限り自動回復し、ユーザーに知らせない
4. **記録フローを止めない** — サイドエフェクト（通知・スタンプ等）の失敗で活動記録を失わない

## 2. エラー分類

### 2.1 レイヤー別エラー

| レイヤー | エラー種別 | 対応方針 |
|---------|----------|---------|
| Presentation | バリデーションエラー | インラインフィードバック |
| Presentation | ネットワークエラー | リトライ + やさしいメッセージ |
| API | 認証エラー (401) | リダイレクト |
| API | 認可エラー (403) | アクセス拒否メッセージ |
| API | バリデーションエラー (400) | フィールド別エラー |
| API | Not Found (404) | カスタム404ページ |
| Service | ビジネスロジックエラー | ユーザー向けメッセージ |
| Service | 外部サービスエラー | リトライ + フォールバック |
| DB | 接続エラー | サイレントリトライ → 503 |
| DB | 一意制約違反 | ビジネスロジックで処理 |

### 2.2 重要度分類

| 重要度 | 定義 | ユーザー通知 | ログ |
|-------|------|------------|------|
| Fatal | システム起動不能 | メンテナンス画面 | CRITICAL |
| Error | 機能不全 | エラーメッセージ | ERROR |
| Degraded | 一部機能低下 | なし（サイレント） | WARN |
| Cosmetic | 表示の乱れ | なし | DEBUG |

## 3. API エラーレスポンス

### 3.1 エラーレスポンス形式

```typescript
// SvelteKit の error() + json() で統一
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "入力内容を確認してください",
    "details": [
      { "field": "name", "message": "名前は必須です" }
    ]
  }
}
```

### 3.2 エラーコード一覧

| HTTP | コード | 意味 | 使用場面 |
|------|--------|------|---------|
| 400 | `VALIDATION_ERROR` | 入力バリデーション失敗 | フォーム送信 |
| 400 | `INVALID_REQUEST` | リクエスト形式不正 | API 呼び出し |
| 401 | `UNAUTHORIZED` | 未認証 | セッション切れ |
| 401 | `INVALID_PIN` | PIN コード不正 | PIN 認証 |
| 401 | `ACCOUNT_LOCKED` | アカウントロック | ロックアウト |
| 403 | `FORBIDDEN` | 権限なし | 他テナント操作 |
| 403 | `PLAN_LIMIT` | プラン上限到達 | 子供追加等 |
| 404 | `NOT_FOUND` | リソース不存在 | 削除済みデータ |
| 409 | `CONFLICT` | 重複・競合 | 重複登録 |
| 429 | `RATE_LIMITED` | レート制限 | API 過剰呼び出し |
| 500 | `INTERNAL_ERROR` | サーバーエラー | 予期せぬ障害 |
| 503 | `SERVICE_UNAVAILABLE` | サービス一時停止 | DB 接続失敗 |

### 3.3 SvelteKit エラーハンドリング

```typescript
// +server.ts でのエラー返却
import { error, json } from '@sveltejs/kit';

// バリデーションエラー
throw error(400, { code: 'VALIDATION_ERROR', message: '...' });

// 正常レスポンス
return json({ data: result });
```

## 4. 子供画面のエラー表示

### 4.1 デザイン原則

- 「エラー」「失敗」「問題」→「うまくいかなかったみたい」「もういちどやってみよう」
- 赤色禁止 → 青・黄・オレンジの暖色系
- テクニカルな情報は一切表示しない
- 年齢モードに応じたひらがな・漢字の使い分け

### 4.2 年齢モード別メッセージ

| 状況 | baby/kinder | lower | upper/teen |
|------|------------|-------|-----------|
| ネットワークエラー | 「つながらないよ〜 もうすこし まってね」 | 「ネットがつながらないみたい。もう少し待ってね」 | 「ネットワークに接続できません。しばらくしてから再度お試しください」 |
| 記録失敗 | 「きろく できなかったよ〜 もういちど やってみてね」 | 「記録できなかったよ。もう一度やってみてね」 | 「記録できませんでした。もう一度お試しください」 |
| ページ不存在 | 「このページは ないみたい」 | 「このページは見つからないよ」 | 「ページが見つかりません」 |

### 4.3 エラー画面コンポーネント

SvelteKit の `+error.svelte` で年齢モードに応じたエラー画面を表示。

## 5. 管理画面のエラー表示

### 5.1 デザイン原則

- 具体的で対処可能なメッセージ
- エラーの原因と対処法を明示
- Alert コンポーネント（`$lib/ui/primitives/Alert.svelte`）を使用

### 5.2 表示パターン

| パターン | コンポーネント | 用途 |
|---------|-------------|------|
| インラインエラー | FormField のエラー表示 | フォームバリデーション |
| トースト通知 | Alert (variant="error") | 操作失敗通知 |
| ページエラー | +error.svelte | ページロード失敗 |
| バナー | Alert (variant="warning") | 機能制限通知（プラン上限等） |

## 6. サイドエフェクトのエラー分離

### 6.1 Hook パターン（activity-log-service）

活動記録の本体処理が成功した後のサイドエフェクトは、個別に try-catch で分離する。

```typescript
// 記録本体（これが失敗したら全体を失敗にする）
const log = await activityRepo.insert(logData);

// サイドエフェクト（個別に失敗しても記録は保持）
const sideEffects = [
  () => processStampGacha(tenantId, childId),
  () => checkLevelUp(tenantId, childId, newTotalPoints),
  () => checkAchievements(tenantId, childId),
  () => notifySiblings(tenantId, childId),
  () => sendPushNotification(tenantId, childName, activityName),
];

for (const effect of sideEffects) {
  try {
    await effect();
  } catch (e) {
    logger.warn(`Side effect failed: ${e.message}`);
    // 記録フローは止めない
  }
}
```

### 6.2 分離対象一覧

| サイドエフェクト | 失敗時の影響 | 対応 |
|---------------|------------|------|
| スタンプガチャ | シール獲得なし | ログ記録のみ |
| レベル判定 | レベルアップ演出なし | ログ記録のみ |
| 実績チェック | 実績アンロックなし | ログ記録のみ |
| きょうだい通知 | 応援演出なし | ログ記録のみ |
| Push 通知 | 親への通知なし | ログ記録のみ |
| Discord 通知 | 開発者通知なし | ログ記録のみ |

## 7. 外部サービスエラー

### 7.1 リトライ戦略

| サービス | リトライ回数 | バックオフ | タイムアウト |
|---------|------------|----------|------------|
| Stripe API | 3回 | 指数 (1s, 2s, 4s) | 10秒 |
| SES | 2回 | 固定 (2s) | 5秒 |
| Cognito | 2回 | 固定 (1s) | 5秒 |
| Web Push | 1回 | なし | 5秒 |
| Discord Webhook | 1回 | なし | 3秒 |

### 7.2 フォールバック

| サービス | フォールバック |
|---------|-------------|
| SES (メール送信失敗) | ログ記録 + 次回 cron で再試行 |
| Web Push (410 Gone) | stale subscription 自動削除 |
| Stripe (Webhook 失敗) | Stripe 側がリトライ（最大3日間） |
| Cognito (認証失敗) | エラーメッセージ表示 |

## 8. ローカルモードのエラー処理

`AUTH_MODE=local` 時の外部サービス呼び出しはスキップまたはモック。

| サービス | ローカルモードの動作 |
|---------|-------------------|
| email-service | ログ出力のみ（実送信しない） |
| notification-service | ログ出力のみ |
| stripe-service | 使用不可（課金機能なし） |
| cognito | 使用しない（PIN 認証のみ） |
| discord-notify | 環境変数未設定時はスキップ |

## 9. データ整合性保護

### 9.1 トランザクション境界

SQLite 環境では Drizzle ORM のトランザクション機能を使用。

```typescript
await db.transaction(async (tx) => {
  await tx.insert(activityLogs).values(logData);
  await tx.update(childStatus).set({ totalPoints: newTotal });
});
```

DynamoDB 環境ではトランザクション書き込み（TransactWriteItems）を使用。

### 9.2 冪等性

| 操作 | 冪等性保証方法 |
|------|-------------|
| 活動記録 | ID ベースの重複チェック |
| ポイント付与 | activity_log_id での一意制約 |
| スタンプ獲得 | stamp_card_id + slot_index での一意制約 |
| Webhook 処理 | Stripe event ID での重複排除 |
