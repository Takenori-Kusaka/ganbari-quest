# docs/rationale/ — 機能別設計経緯ドキュメント 運用ルール

## このディレクトリの目的

がんばりクエストの機能ごとの**設計経緯 (rationale)** を保存する。

| ドキュメント種別 | 粒度・内容 | 保存先 |
|----------------|-----------|--------|
| ADR | 全プロダクト横断の哲学・ポリシー決定 | `docs/decisions/` |
| 設計書 | 機能の結論・仕様（What） | `docs/design/` |
| **rationale** | 機能別の設計経緯・議論の流れ（Why） | **`docs/rationale/`** |
| memory | ユーザーローカルの作業メモ | `.claude/projects/.../memory/` |

ADR は全プロダクト横断の哲学を扱い、設計書は結論・仕様を扱う。その間にある「**なぜそう決めたか**」「**何を検討して棄却したか**」という経緯は従来どこにも保存されず、議論から時間が経つと失われる問題があった（#1328 C1 umbrella で判明）。

rationale ドキュメントはこの欠落層を埋めるために新設する。

---

## いつ rationale を書くか

以下のいずれかに該当する場合は rationale 文書を作成する：

1. **複雑な新機能を実装するとき** — 複数の代替案を比較検討した記録が将来役立つ
2. **既存機能の大きな方向転換** — 「なぜ前の設計を変えたか」を記録しないと、後で同じ議論を繰り返す
3. **過去の議論が再発する兆しが見えたとき** — 「以前も同じ議論をして A を選んだ、その理由はこれ」と即座に示せるようにする

軽微な変更（バグ修正、文言修正、設定値の調整など）には不要。

## ADR との使い分け

| 判断基準 | ADR | rationale |
|---------|-----|-----------|
| 影響範囲 | 全プロダクト横断のポリシー | 特定機能の設計判断 |
| 更新頻度 | 低い（一度決めたら基本変更しない） | 機能進化に応じて更新可 |
| 経緯記述 | 最小限（Alternatives Considered 節のみ） | 経緯・議論フローが中心 |
| 強制力 | ルール（違反は PR でブロック） | 記録（将来の参照用） |

例：「Anti-engagement 原則」は全機能に適用するポリシーなので ADR-0012。  
「コアループ L1/L2/L3 モデルを採用した経緯」は特定の機能設計判断なので rationale。

## 設計書との使い分け

| 問い | 参照先 |
|-----|--------|
| 「この機能の仕様は何か？」 | `docs/design/` の設計書 |
| 「なぜこの仕様になったか？」 | `docs/rationale/` の rationale |
| 「何を検討して棄却したか？」 | `docs/rationale/` の rationale |

設計書は **What** を答え、rationale は **Why** を答える。

---

## memory との使い分け

`.claude/projects/.../memory/` はユーザーローカルの作業メモで、チームと共有されない。  
rationale はリポジトリにコミットされる**チーム共有正本**。

- チームで将来参照する経緯 → `docs/rationale/`
- ユーザー個人の作業メモ → memory

## ファイル命名規則

```
docs/rationale/NN-機能名-rationale.md
```

- `NN`: 2 桁の連番 (01, 02, 03, ...)
- `機能名`: ハイフン区切りの機能識別子（例: `core-loop`, `reward-shop`, `stamp-card`）
- 拡張子: `.md`

例:
- `01-README.md` — 本ファイル（運用ルール）
- `02-core-loop-rationale.md` — コアループ設計経緯
- `03-reward-shop-rationale.md` — ごほうびショップ設計経緯

---

## テンプレート

新規作成時は `_template.md` をコピーして使用する：

```bash
cp docs/rationale/_template.md docs/rationale/NN-機能名-rationale.md
```

テンプレートのセクション構成:

| セクション | 内容 |
|-----------|------|
| 議論の発端 | いつ、誰が、どのような問題意識で始めたか |
| 検討した代替案 | 採用案以外に何を検討したか |
| 棄却理由 | なぜ代替案を選ばなかったか |
| 採用案とその理由 | なぜこれを選んだか（背景文脈とセット） |
| 残された懸念・フォローアップ | 議論中に出たが未解決の論点 |
| 関連 | 議論源 Issue / PR、影響を受ける設計書、関連 ADR |

## 更新ルール

- 新しい rationale を追加したら、上の「rationale 一覧」テーブルも更新する
- 既存 rationale に後続議論が発生したら「残された懸念・フォローアップ」を更新し、新セクション「フォローアップ YYYY-MM-DD」を末尾に追記する
- rationale を廃止する場合は削除せず、先頭に `> **Superseded by**: [XX-機能名-rationale.md](XX-機能名-rationale.md)` を追記する
