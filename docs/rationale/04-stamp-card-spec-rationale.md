# スタンプカード仕様 設計経緯（#1794）

## 議論の発端

- **日時**: 2026-05-01
- **発端 Issue**: #1794
- **問題意識**: LP「ログインボーナスで毎日おみくじが 1 回引けて、スタンプカードが週末に 1 枚完成します」と実装「5 スロット曜日非依存 + 1 日 1 回 cap + 週合計 5 cap + 月曜自動 redeem」の関係が説明不足で混乱を生んでいた（#1794 PO 補足: 「ぽこあポケモン等で採用 UX なのでわかりにくくはないが、混同が続くなら改訂を検討」）

PO 補足から「仕様変更は不要、説明力を補強する」方針が確定。LP 文言・設計書・rationale の 3 層で混乱を防ぐ施策を取る。

## 検討した代替案

| 案 | 概要 | 検討した理由 |
|----|------|-----------|
| 案 A: 仕様変更（5 スロット → 7 スロットに変更） | 「7 日タップで 1 枚」に揃え、文言と仕様を一致させる | LP の混乱解消には最も直接的 |
| 案 B: 仕様変更（曜日依存 — 月〜日の各曜日に固有スロット） | 「週末に完成」が成立する曜日依存設計 | LP 文言の素朴な解釈と一致 |
| 採用案 (C): **仕様維持 + LP 文言補強 + 設計書 §13 にスタンプカード仕様セクション追加 + rationale 起票** | 5 スロット曜日非依存 + 自動 redeem は anti-habituation 整合（§4a.3）。文言を「ポイントに交換できます」に動詞統一し、LP / 設計書 / rationale の 3 層で説明する | PO 補足「仕様維持で文言・rationale 整理」と一致 |

## 棄却理由

- **案 A 棄却理由**:
  - 「タップ忘れ 1 日」の許容（5/7 = 約 71% タップ達成で OK）が anti-habituation 整合の核（§4a.3）
  - 7 スロットにすると「1 日でも休んだらカードが完成しない」プレッシャー設計に逆行する
  - ADR-0012 Anti-engagement 原則の「煽らない設計」と矛盾する
- **案 B 棄却理由**:
  - 曜日依存にすると「月曜だけ忙しい家庭」が永遠に月曜スロットを埋められず体験が壊れる（家族のリズム多様性を破壊）
  - 海外旅行・連休・病欠など曜日要件が崩れる場面で容赦なく redeem 不能になる
  - 5/7 日 OK の柔軟性が消えるため anti-habituation の「タップ忘れ容認」が機能しない

## 採用案とその理由

**案 C: 仕様維持 + LP 文言補強 + 設計書セクション追加 + rationale 起票**

- 選択理由:
1. **anti-habituation 整合**: 5/7 日 OK の柔軟設計は ADR-0012 Anti-engagement と §4a.3 anti-habituation の両方に整合する。
- 仕様を変更すると過去の意思決定 (#1338) を無効化することになる
2. **PO 補足との一致**: 「ぽこあポケモン等で採用 UX なのでわかりにくくはない」「混同が続くなら改訂を検討」 → まず説明補強で対応する段階的アプローチ
3. **LP truth (ADR-0013) 整合**: 文言を実装の事実 (5 スロット + 月曜自動 redeem) に近づける（「完成します」 → 「ポイントに交換できます」）
4. **設計書 SSOT 化**: §7.3 にスタンプカード仕様セクションを追加し、LP 文言マッピング・週次ライフサイクル例・PO 議論再発防止 Q&A を併記して再発を防ぐ
5. **rationale 化**: 本ドキュメントで「なぜ 5 スロット曜日非依存」「なぜ自動 redeem」を文章化し、将来の議論再発時の参照点を作る。
