## 目的

本サービスをSaaS展開するために必要な対応を行う

0. OSSかつSaaSサービスを展開しているもの(gitlabやnotionなど)を参考に、どのような対応が必要か要件を整理し、チケット化してください
1. github pagesで動作するデモ、チュートリアルを作成し、操作方法についてのマニュアルとなるWebアプリを公開する
2. github sponsorsやpatreonなどで支援者を募るためのReadmeを作成する
3. Readmeを拡充し、gifなどでどんな体験ができるWebアプリなのかわかりやすくする
4. AWSでフルサーバレスホスティング可能にし、維持費を限界まで抑えた状態で運用できるようにする(データベース部分をS3/DynamoDBなどのサーバレスデータベースにする、APIをLambdaで動かすなど
5. Web版、andorid版,iOS版を提供し、どの環境からでもアクセスできるようにする
6. 費用プランを作成し、カスタマイズができない無料プラン(親管理画面の自由度を下げるなど)と、親管理画面でフルに管理できるプラン、専用環境を設けられるプランを用意し、一般家庭に合わせた費用体系にする(500円買い切りや年間500円など？もっと高くてもいいかも？)これらは、日本市場を前提とし根拠ある数字(出生数や子供の数、一般的なSaaSサービスの価格など)をもとに設定する。損益分岐点などのビジネスプランの計画を策定し、目指すべき顧客数の目標を設定する。これらの事業計画書などのドキュメントもC:\Users\kokor\OneDrive\Document\GitHub\ganbari-quest\docs\designで公開資料としてください。上記のようなフルサーバレスでの運用における原価計算なども適切に行う必要があり、特にログやデータベースが費用の中心になりそうです。WAFなどセキュリティ対策による費用は個人開発であることも踏まえて初期導入"しない"（WAFのルール費用が高すぎる,,,）ため、Lambda Authenticatorなどで適切にAPIのセキュリティを確保するなどの対応を行うことも必要になると思います。その上での原価としましょう。なるべく無リクエスト時原価は月1000円未満としたいです。
7. 決済システムはなるべく既存のフレームワークを採用し、google playやapple storeなど市場に対してもっとも適した形で提供する。例えば、androidユーザはgoogle playで決済できるようにするなど。Web版はあってもいいが、おそらく多くのメイン顧客に対してはあまり需要がないと思われるため、優先度は低めでいいと思います。(Web版はビジネスプラン的にも別にしてライセンスキーの発行などから特定のユーザのみアクセス可能にするとか、android/iOS版からのアクセスに限定するためにおそらく証明書を独自に発行し、その証明書を使ってしかアプリにアクセスできないようにする、という対応をすることによってWeb版への直接アクセスも防げるかも)
8. 顧客ユーザ管理を可能にするためのユーザ認証機能を設ける(OAuthを前提にし、Email/パスワードといったログインはサポートしない。GoogleやAppleなどのアカウントでログインできるようにする)。これにより、ユーザは自分のアカウントでログインし、データを安全に管理できるようになります。この時重要なこととして、同じアカウントを家族内で共有すること(同じ子供が登録されている環境を子供も親も使う)になるため、共有アクセスできる仕組みを考えておく必要があります。ぴよログなどでは共有のためのQRコードと認証コードを発行し共有先連携を行う形をとっているようで、これも参考にしつつ、家族内での共有アクセスの仕組みを設計してください。新規契約や新規共有だけでなく契約終了や共有解除のフローも設計してください。契約終了時には適切にユーザデータを破棄する(データベースや顔写真など)仕組みを検討してください
9. マーケティング施策として、本アプリ専用のHPや問い合わせのためのDiscordサイト開設など、ユーザがアクセスしやすい環境の構築が必要です。同一リポジトリでHPも用意する(github pagesのデモアプリにHPも兼ねる形でいいと思います)など、ユーザがアクセスしやすい環境を構築してください。また、問い合わせのためのDiscordサイトも開設し、ユーザが気軽に質問やフィードバックを提供できるようにしてください。これにより、ユーザとのコミュニケーションを円滑にし、サービスの改善につなげることができます。
10. アプリ内から問い合わせへのリンクなども設置し、ユーザがアプリを使いながら気軽に問い合わせやフィードバックを提供できるようにしてください。これにより、ユーザはアプリを使用している最中に問題が発生した場合や改善点を思いついた場合にすぐにフィードバックを提供できるようになります。
11. データプライバシーポリシーおよびセキュリティポリシーなどSaaSとして利用者に安心してもらうためのドキュメントを整備し、公開してください。これにより、ユーザは自分のデータがどのように扱われるかを理解し、安心してサービスを利用できるようになります。また、セキュリティ対策についても明確に説明し、ユーザの信頼を得ることが重要です。
12. 上記ポリシーに従ったデータ管理が行われるよう、ユーザデータの保管、管理に関してセキュリティを強化してください。
13. 利用規約を策定し、サービス提供者である私が一切の責任を負わず不利益を保証しないことを担保してください。個人開発である以上、何も責任を取れません。これにより、サービス提供者としてのリスクを最小限に抑えることができます。利用規約には、サービスの利用に関するルールや禁止事項、免責事項などを明確に記載し、ユーザがサービスを利用する前に同意するようにしてください。
14. 可用性の観点から、サービスは適宜アップデートされ、バグ修正や機能追加が行われるべきです。これにより、ユーザは常に最新の機能を利用できるようになり、サービスの品質も向上します。また、アップデートの際にはユーザに対して適切な通知を行い、変更点や新機能について説明することも重要です。加重ルーティングで初週は25%のユーザに新機能を提供し、次週で100%のユーザに提供するなど、段階的に展開することも検討してください。
15. パンフレット、リーフレットになるような、サービスの特徴や利用方法を簡潔にまとめた資料を作成し、配布できるようにしてください。これにより、ユーザはサービスの概要や利用方法をすぐに理解できるようになります。また、オンラインでダウンロードできる形式で提供することも検討してください。これらの資料は、サービスのマーケティングやユーザ教育に役立ちます。
16. ほかに個人開発からSaaSビジネス展開するうえで、家計を圧迫せずに始められる方法や、ビジネス展開するうえでの注意点などがあれば、積極的に取り入れてください。
