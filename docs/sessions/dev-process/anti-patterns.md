# アンチパターン集 — 開発プロセスで繰り返した「逃げ」と是正

> 過去に User / PO から繰り返し是正された開発プロセス上のアンチパターン。「書き始めたら即停止」のシグナルとして使う。

**SSOT 位置付け**: [dev-process/README.md](README.md) の各論。関連: [completion-principles.md](completion-principles.md)（完遂原則の裏返し）

---

## 1. 「scope 外」を言い訳に問題を放置しない

PR レビュー / 実装中に気づいた問題は **必ず同じ PR で修正する**。「PR scope 外」「他の人が直す」「今回の修正とは無関係」を理由に放置しない。品質責任者の責務は商品品質まで仕上げること。レビューの場が一番安く直せるタイミング。

判断フロー:

1. **軽微（30 分以内 / 数行）** → 同じ PR で即修正（冗長な三項演算子 / 未使用 import / 誤字）
2. **大規模（別領域 / 大きな設計変更 / コンフリクトリスク）** → Issue 起票して追跡（起票完了して初めて「対応済み」）
3. **「scope 外として保留」とだけ書いて放置** → 絶対 NG（軽微なら 1、大規模なら 2 に必ず帰着）

---

## 2. 実装の越境をしない（Reviewer / PO は Dev の肩代わりをしない）

| 作業 | 担当 |
|---|---|
| コード実装・修正・削除 / Rebase / 実機動作確認 / SS 生成 | Dev |
| PR レビュー・指摘 / Issue 起票 / ADR 起票 | Reviewer / PO |
| PR close 判断・方針転換 | PO |
| PR base branch 切替（blocker 解消） | Reviewer |

**禁止**: Dev PR への直接 push / 勝手な merge / 実装の肩代わり（rebase / SS / コード修正）/ Dev 未同意の scope 大幅変更。

**方向転換時のプロトコル**: レビュー一時停止（Dev に通知）→ PO と設計ポリシー議論 → ADR 起票 → ADR 合意後 Dev に Issue として明示依頼 → Dev の手が空くまで新 PR を勝手に始めない。

---

## 3. テスト assertion を弱体化しない

assertion を安易に弱めると「テストは通るが本当に欲しい挙動を検証していない」状態になり、リグレッション検出力が低下する（ADR-0006 Safety Assertion Erosion Ban）。以下を必ずチェック:

- `toBeTruthy()` / `toBeDefined()` への置換（具体値検証から逃げていないか）
- `expect(...).toBe(true)` を `.not.toBeNull()` に弱めていないか
- スナップショット差し替えで検証放棄していないか
- `await waitFor()` のタイムアウト延長で「flake 対策」と称してレース根因を放置していないか
- `skip` / `.only` の混入 / `expect.any()` / `expect.anything()` の安易な使用

どうしても弱める場合は理由を PR コメントで明示し、代替手段（より細かい検証 / 別テストでの補完 / 実装側安定化）を併記。理由不十分ならブロック。

---

## 4. ラバースタンプ merge をしない（QA / レビュー時）

PR を「指摘ゼロ」で merge しない。指摘がない場合は「指摘なし」でなく「レビューが不十分」と疑う。マージ = 本番デプロイ。

merge 前に必ず:

1. 全 diff を 1 行ずつ精読（CSS ハードコード / テスト欠落 / 設計違反を探す）
2. レビュー指摘を severity 付きで文書化（問題なしでも「確認済み項目」を明記）
3. Issue の完了条件と 1 項目ずつ突合
4. 横展開チェック（変更が他画面・並行実装に影響しないか grep）
5. プロジェクトルール違反チェック（hex 直書き / primitives 再実装 等）
6. Medium 以上の指摘は修正要求してから merge

「CI green」「コンフリクトなし」は品質の証明ではなく前提条件。詳細手順は [qa-session.md](../qa-session.md)。

---

## 5. QM は CI ゲートではない

CI 緑 + PR body 完備 = approve ではない。Ready PR レビュー時は 5 手順を順に実行してからしか approve しない:

1. `gh issue view <closes #X>` を開き、AC 各項目を PR diff と 1:1 突合
2. PR 本文の `![]()` / `<img>` を全て Read tool で開いて目視
3. 1 画像につき最低 1 行の所見（色 / 形 / tapSize / 違和感）を approve body に残す
4. UI / LP 変更 PR で SS 添付がなければ blocking 指摘
5. 最後に `gh pr checks` を補助情報として確認

CI は「書いた通りにしたか」しか見ない。「書いた内容が Issue 意図と合うか」「UI/UX として妥当か」は人間しか見られない。「見ました」「確認済み」だけの所見は禁止。詳細は [self-review-agent.md](../../operations/self-review-agent.md)。

---

## 6. CI 通過前に Ready にしない

CI 全緑を確認する前に PR を Ready for Review にしない。Ready は QA への「レビュー可能」シグナル。Draft → Ready 変更時は `gh pr checks <num>` で全 PASS を確認してから `gh pr ready`。CI 失敗中は Draft 維持し、修正 → CI 通過 → Ready の順序を厳守。

---

## 7. CI 失敗を「自分のせいでない」で放置しない

CI が失敗していれば原因が自分の変更かに関わらず対応する。

| 状況 | 必須アクション |
|---|---|
| 自分の変更が原因 | 同一 PR 内で修正してから Ready |
| 原因が自分の変更でない | Issue 起票し、PR description / コメントにリンク |
| いずれの場合も | CI 全緑でない PR を Ready にしない |

「原因調査中」「自分のせいではない」のような曖昧記述は禁止。Issue 起票時は `priority:high` 以上でブロッカー明記。

---

## 8. QA / approve / merge を自律実行しない

Dev セッションの役割は「実装して PR を Ready for Review にする」まで。QA レビュー・approve・merge は QA セッション / PO の仕事。Dev が自律的に QA Agent を呼んで merge まで完結させる autonomous フローは越権。

例外: PR 本文の修正（必須セクション欠落など CI fail の修正）は Dev 側責任範囲。

---

## 9. 段階的リリースをしない（stub / no-op merge 禁止）

`main` への merge は即 Lambda 本番反映。

- stub / no-op / TODO 実装の merge 禁止。「follow-up PR で本実装」前提のレビュー依頼は PO クレーム事案
- DynamoDB / SQLite 両対応 repo 追加 PR は両実装完成必須。CDK 定義も同 PR に含める
- follow-up に逃がせるのは「本番に存在しなくても顧客に気付かれない」場合のみ。顧客提供価値（不正検知 / 監査 / 保証）に直結する機能は同一 PR 完結必須

---

## 10. 対症療法を取ったら必ず根本解決 Issue を同時起票する

短期措置（workaround）を取る場合、**同タイミングで根本解決 Issue を起票**する（Issue 化を後回しにしない）。根本解決 Issue には:

- 短期措置の内容 + 適用 PR 番号（撤回ターゲット明確化）
- 根本問題の構造的記述（なぜ短期措置が必要だったか）
- 根本解決の AC（workflow / ADR / SSOT 修正の具体内容）

毎回の短期措置コスト > 根本解決 1 回コスト のため `priority:high` 推奨。

---

## 11. 複雑度最小化の主語は常にユーザ

「複雑度を減らす」の主語は **ユーザが実際に面する複雑度**。内部バリアント数は別物。

- 子供が見るのは「登録済みプリセット活動群」のみ → プリセット数は子供 UX の複雑度に影響しない
- 親はセットアップで年齢・性別入力 → 自動適用されるため総バリアント数は直接見えない
- ユーザ面の複雑度（選択肢の数 / 画面階層の深さ / 新規概念の学習コスト / 同時に覚えるルール数）を最小化する
- 内部バリアント数を「複雑度最小化」を理由に機械的に減らすのは誤り（性別バリアント 10 ファイル廃止提案が却下された事例あり）
- 大規模変更を scope 理由で回避しない。「やるべきか」で判断する

---

## 12. demo / 本番 UI 共通化は実体共通化（shim で済ませない）

demo 画面と本番画面の UI 共通化指示は「描画コードが 1 つになること」を意味する。

- 「UI 等価性維持」を共通化要求の理由に使わない（見た目が変わるのは正常、demo を本番側へ寄せる）
- 「Tier N で統合」「scope 外」「follow-up Issue」と分離する誘惑を排除（並行実装 2 系統を残す PR は「共通化完了」ではない）
- 完了判定: 並行ファイルが残らないことを grep で機械検証
- 同じ指示が 3 回以上ある場合は構造問題として扱う（#2069 系は 6 回再発）
