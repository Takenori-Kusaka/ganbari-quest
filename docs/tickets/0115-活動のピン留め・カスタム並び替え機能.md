# #0115 活動のピン留め・カスタム並び替え機能

## ステータス: Done
## 優先度: 中
## 種別: 新機能（UX改善）

---

## 概要

子供画面の活動一覧で、よく使う活動をピン留め（お気に入り）したり、子供ごとに表示順をカスタマイズできる機能を追加する。現在は `activities.sortOrder`（システム共通）による固定順でしか表示されず、頻繁に使う活動が埋もれてしまう問題がある。

### 課題

- 活動マスタは174件あり、年齢フィルタ後でもカテゴリごとに十数件が並ぶ
- 毎日記録する活動（例: ひらがなれんしゅう、おさんぽ）がリストの中に埋もれる
- 子供・家庭ごとに「よく使う活動」は異なるが、現在のソートはシステム共通の `sortOrder` のみ
- 子供が自分で活動を探して押すUXが「毎日同じものを探す手間」になっている

### ゴール

1. **ピン留め（お気に入り）**: よく使う活動を各カテゴリの先頭に固定表示
2. **使用頻度順ソート**: 過去の記録回数に基づいて自動的に頻出活動を上位に表示
3. **親による並び替え**（オプション）: 親管理画面から子供ごとの活動表示順をカスタマイズ

## 設計

### 方式選定

| 方式 | メリット | デメリット | 採用 |
|------|---------|-----------|------|
| A. ピン留めフラグ（子供×活動） | シンプル、直感的 | 順序の細かい制御不可 | ✅ |
| B. 使用頻度自動ソート | 設定不要、自然なUX | 新しい活動が上がりにくい | ✅ |
| C. 子供ごとの完全ドラッグ&ドロップ並び替え | 最大の自由度 | 174件の並び替えは非現実的、子供には操作困難 | ❌ |
| D. 親管理画面から子供別ソート設定 | 親が適切に管理可能 | 実装コスト高 | Phase 2 |

**→ Phase 1 は「ピン留め + 使用頻度ソート」の組み合わせで実装**

### ソート優先度（新しいロジック）

```
カテゴリ内の活動表示順:

  1. ピン留めされた活動（ピン留め順 → 使用頻度順）
  ─────────────────── 区切り線 ───────────────────
  2. 過去30日の使用頻度順（多い順）
  3. 未使用の活動（sortOrder 順＝従来の固定順）
```

### データモデル

#### 新規テーブル: `child_activity_preferences`

```sql
CREATE TABLE child_activity_preferences (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  child_id   INTEGER NOT NULL REFERENCES children(id) ON DELETE CASCADE,
  activity_id INTEGER NOT NULL REFERENCES activities(id) ON DELETE CASCADE,
  is_pinned  INTEGER NOT NULL DEFAULT 0,   -- ピン留めフラグ (0/1)
  pin_order  INTEGER,                       -- ピン留め内の表示順 (NULL = 未設定)
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now')),
  UNIQUE(child_id, activity_id)
);

CREATE INDEX idx_child_activity_prefs_child 
  ON child_activity_preferences(child_id);
CREATE INDEX idx_child_activity_prefs_pinned 
  ON child_activity_preferences(child_id, is_pinned);
```

#### 使用頻度の算出（既存テーブルから集計）

```sql
-- activityLogs から過去30日の使用回数を集計（新テーブル不要）
SELECT activity_id, COUNT(*) as usage_count
FROM activity_logs
WHERE child_id = ? 
  AND recorded_date >= date('now', '-30 days')
  AND cancelled = 0
GROUP BY activity_id
ORDER BY usage_count DESC
```

### ソートロジックの実装

```typescript
// src/lib/server/services/activity-service.ts

interface SortedActivity extends Activity {
  isPinned: boolean;
  usageCount: number;  // 過去30日
}

function sortActivities(
  activities: Activity[],
  preferences: ChildActivityPreference[],
  usageCounts: Map<number, number>
): SortedActivity[] {
  const pinned = new Map(
    preferences.filter(p => p.isPinned).map(p => [p.activityId, p.pinOrder])
  );

  return activities
    .map(a => ({
      ...a,
      isPinned: pinned.has(a.id),
      usageCount: usageCounts.get(a.id) ?? 0,
    }))
    .sort((a, b) => {
      // 1. ピン留め優先
      if (a.isPinned && !b.isPinned) return -1;
      if (!a.isPinned && b.isPinned) return 1;

      // 2. ピン留め同士: pinOrder → usageCount
      if (a.isPinned && b.isPinned) {
        const orderDiff = (pinned.get(a.id) ?? 999) - (pinned.get(b.id) ?? 999);
        if (orderDiff !== 0) return orderDiff;
        return b.usageCount - a.usageCount;
      }

      // 3. 非ピン: 使用頻度順 → sortOrder（従来の固定順）
      if (a.usageCount !== b.usageCount) return b.usageCount - a.usageCount;
      return a.sortOrder - b.sortOrder;
    });
}
```

## UI設計

### 子供画面（Kinder モード）

#### ピン留め操作

```
ActivityCard 長押し（500ms）→ コンテキストメニュー表示

  └── （将来: 🔽 したにうごかす / 🔼 うえにうごかす）
```

## 注意事項

- ピン留め上限はカテゴリあたり**最大5件**とする（多すぎるとピン留めの意味がなくなる）
- 使用頻度の集計は `activityLogs` の既存データから算出するため、新テーブルは不要
- 使用頻度の集計期間（30日）は将来的に設定可能にしてもよいが、Phase 1 では固定
- `sortOrder` による従来のソートは「兜の緒」として残す（使用頻度0の活動のフォールバック）
