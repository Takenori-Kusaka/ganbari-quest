# 活動記録ボタンの多重送信防止

## Status: Done


### 概要

活動記録時の「きろくする」ボタン押下からレスポンス表示までのタイムラグ中に連打されると、APIが多重送信される問題を防止する。ローディング表示とボタンの無効化で対応する。

### 背景・動機

- 活動をタップ → 「きろくする」を押下 → 「きろくした！」表示までにネットワーク遅延がある
- 子供は反応がないとボタンを連打する傾向がある
- 現在は多重送信を防ぐガードがなく、同じAPIリクエストが複数回送信される可能性がある
- サーバ側にはUNIQUE制約があるため2回目以降はエラーになるが、UX的に問題がある

### ゴール

- [x] 「きろくする」ボタン押下後、レスポンスが返るまでローディング表示
- [x] ローディング中はボタンをdisable（連打防止）
- [x] ローディングスピナーまたはアニメーションで処理中であることを視覚的にフィードバック
- [x] エラー時はローディング解除してリトライ可能に
- [x] 確認ダイアログの「きろくする」ボタンにも同様の対応

### 技術方針

#### フォーム送信のローディング管理

SvelteKit の `enhance` を活用し、送信中の状態管理を行う。

```svelte
<script>
  let submitting = $state(false);
</script>

<form
  method="POST"
  action="?/record"
  use:enhance={() => {
    submitting = true;
    return async ({ update }) => {
      await update();
      submitting = false;
    };
  }}
>
  <button type="submit" disabled={submitting}>
    {#if submitting}
      <span class="spinner"></span> きろくちゅう...
    {:else}
      きろくする
    {/if}
  </button>
</form>
```

#### UIデザイン

**通常状態:**
```
┌─────────────────┐
│   きろくする ✅   │
└─────────────────┘
```

**ローディング中:**
```
┌─────────────────┐
│  ⏳ きろくちゅう  │  ← ボタンdisable + スピナー
└─────────────────┘
```

**完了:**
```
┌─────────────────┐
│  🎉 きろくした！  │  ← 成功アニメーション
└─────────────────┘
```

#### 子供向けの配慮

- 「きろくちゅう...」の表示で処理中であることを明確に伝える
- スピナーは子供にもわかりやすい大きめのアニメーション
- ボタンの色をグレーアウトではなく、柔らかいトーンに変更（子供が壊したと思わないように）

### 影響範囲

- `src/routes/(child)/kinder/home/+page.svelte` - 記録ボタンのローディング対応
- `src/routes/(child)/baby/home/+page.svelte` - 記録ボタンのローディング対応
- 確認ダイアログ（記録確認モーダル）のボタンにも適用

### 依存

なし

### 作業メモ

- SvelteKit の `use:enhance` は `<form>` の送信をクライアントサイドで処理し、`submitting` 状態を制御できる
- サーバ側にもべき等性の確保が望ましい（UNIQUE制約で既に担保済み）
- キャンセルボタン（5秒以内）のUIとも整合性を取る
- #0051（複数回実行対応）と組み合わせる場合、`allowMultiple` な活動でもローディング中は連打防止する

### 成果・結果

- 多重送信防止はUI実装済み（disabled state + loading indicator）
- 「きろくする」ボタン押下後、レスポンスが返るまでローディング表示・ボタンdisable
- エラー時はローディング解除してリトライ可能
- 確認ダイアログのボタンにも同様の対応を適用
