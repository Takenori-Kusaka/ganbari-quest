### 概要

サービス障害・メンテナンス時にユーザーへ適切に状況を伝える仕組みを構築する。CloudFront カスタムエラーページ（500/503 → わかりやすいお知らせ画面）、GitHub Pages ベースのステータスページ、メンテナンスモード切替機能を **追加費用 0円** で実現する。

### 背景・動機

#### ユーザーへの説明手段がない

現状の設計では、Lambda が 500 エラーを返した場合やAWS障害で全面ダウンした場合、ユーザーには**生のエラーレスポンス**が表示される。子供向けアプリとして、「ただいまメンテナンス中です」のような分かりやすい画面が必要。

#### AWS 障害の検知手段がない

AWS 自体が障害を起こした場合（例: ap-northeast-1 の Lambda 障害）、CloudWatch も影響を受けて通知できない可能性がある。AWS Health → EventBridge → SNS の経路で、**AWS公式の障害情報を直接受け取る**仕組みが必要。

#### メンテナンスモードがない

計画メンテナンス時（#0075 SLA: 月1回 JST 2:00-5:00）に、アプリをメンテナンス状態にする仕組みがない。

### 対応方針

#### 1. CloudFront カスタムエラーページ

Lambda や API Gateway がエラーを返した場合、CloudFront がユーザー向けの親切なエラーページに差し替える。

```
CloudFront Custom Error Response:
  ├── 500 → /error/500.html（TTL: 30秒）
  ├── 502 → /error/502.html（TTL: 30秒）
  ├── 503 → /error/maintenance.html（TTL: 60秒）
  └── 504 → /error/504.html（TTL: 30秒）
```

**エラーページの設計:**

```html
<!-- S3: /error/503.html -->
<!-- 子供向けアプリなので、やわらかい表現で -->
<html lang="ja">
<head><meta charset="UTF-8"><title>メンテナンス中 - がんばりクエスト</title></head>
<body style="text-align:center; padding:40px; font-family:sans-serif;">
  <h1>🔧 メンテナンスちゅう</h1>
  <p>いま おてつだいの じゅんびを しているよ</p>
  <p>すこし まっててね！</p>
  <p style="color:#888; font-size:0.8em;">復旧予定: 近日中</p>
</body>
</html>
```

- エラーページは **S3 の別パス** に配置（Lambda 障害時でも S3 から配信できるため）
- TTL を短くすることで、復旧後すぐに正常ページに戻る
- 子供ユーザー向けにひらがな多めの表現

#### 2. AWS Health → EventBridge → SNS

AWS のサービス障害情報を自動で受け取る仕組み。

```
AWS Health
  └── EventBridge Rule（ap-northeast-1, 使用中のサービス）
       └── SNS Topic（#0102 と共通）
            └── メール通知
```

```json
// EventBridge ルール（イベントパターン）
{
  "source": ["aws.health"],
  "detail-type": ["AWS Health Event"],
  "detail": {
    "service": ["LAMBDA", "DYNAMODB", "APIGATEWAY", "CLOUDFRONT"],
    "eventTypeCategory": ["issue", "scheduledChange"]
  }
}
```

- `issue`: AWS 側の障害通知（Lambda 障害等）
- `scheduledChange`: AWS 側の計画メンテナンス通知
- CloudWatch が動かなくても EventBridge + SNS で通知可能（異なるサービスパス）

#### 3. メンテナンスモード

計画メンテナンス時に手動でメンテナンスモードに切り替える。

**方式A: CloudFront のオリジン切替（推奨）**

```
通常時:     CloudFront → API Gateway → Lambda
メンテ時:   CloudFront → S3 /error/maintenance.html
```

- CloudFront のオリジンを S3 のメンテページに切り替え
- AWS CLI 1コマンドで切替可能:

```bash
# メンテナンス開始
aws cloudfront update-distribution \
  --id $DISTRIBUTION_ID \
  --default-root-object maintenance.html
  # ※ 実際にはオリジンの切替。具体的な CLI は CDK 実装時に確定

# メンテナンス終了（元に戻す）
aws cloudfront update-distribution \
  --id $DISTRIBUTION_ID \
  --default-root-object index.html
```

**方式B: Lambda 環境変数フラグ（簡易版）**

```typescript
// Lambda 内のミドルウェアでチェック
if (process.env.MAINTENANCE_MODE === 'true') {
  return json({ status: 'maintenance' }, { status: 503 });
}
```

- Lambda の環境変数を変更するだけ
- CloudFront のカスタムエラーページが 503 → maintenance.html に変換

**推奨:** 方式B（簡易版）を初期導入。方式A は Lambda 自体が起動しない障害向けの保険として将来検討。

#### 4. ステータスページ

ユーザーがサービスの稼働状況を確認できるページ。

**初期版（Phase 4 で構築）:**
- GitHub Pages に静的ステータスページを設置
- 手動更新（障害発生時に GitHub で markdown を編集 → push）
- URL: `https://status.ganbari-quest.com` または同一ドメイン内 `/status`

**自動化（将来検討）:**
- [upptime](https://github.com/upptime/upptime)（GitHub Actions ベースの無料監視ツール）の導入
- `/api/health` の結果をグラフ化
- 過去の障害履歴を一覧表示

**初期は手動更新で十分な理由:**
- 個人開発で障害頻度は低い（月1回未満を目標）
- 自動化の工数に見合わない
- 手動でも markdown を push するだけ（5分以内）

#### 5. 障害対応ランブック

#0078 の障害対応フローを具体的な手順書に落とし込む。

```markdown
## 障害対応ランブック

### Level 1: アラート受信（自動）
1. メールを確認（#0102 の CloudWatch Alarms / AWS Health）
2. UptimeRobot のダウン通知を確認

### Level 2: 状況判断（5分以内）
1. CloudWatch Dashboard で状況確認
2. 障害の種別を判断:
   - A: Lambda エラー → Level 3A
   - B: AWS 障害 → Level 3B  
   - C: DDoS/異常トラフィック → Level 3C

### Level 3A: Lambda エラー対応
1. CloudWatch Logs でエラー内容確認
2. 直近デプロイが原因か判断
3. Yes → ロールバック（#0101 の手順）
4. No → コード修正 → ホットフィックスデプロイ

### Level 3B: AWS 障害対応
1. AWS Health Dashboard で影響範囲確認
2. メンテナンスモード ON（方式B）
3. ステータスページを更新
4. AWS の復旧を待つ
5. 復旧後、メンテナンスモード OFF
6. ステータスページを更新

### Level 3C: DDoS/異常トラフィック対応
1. API Gateway のスロットリング値を一時的に下げる
2. CloudFront のジオブロック範囲を縮小
3. 必要に応じ API Gateway を一時無効化
4. Budget アラート（#0103）で課金状況確認
5. 原因特定後、対策を恒久化
```
