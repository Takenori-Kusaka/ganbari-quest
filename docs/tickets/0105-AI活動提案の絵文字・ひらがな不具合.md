# #0105 AI活動提案の絵文字・ひらがな不具合

## ステータス: Done
## 優先度: 高
## 種別: バグ修正

---

## 概要

親管理画面 → 活動管理 → ✨ AI追加機能において、以下の不具合が発生している:

1. **絵文字がフォールバック固定値になる** — 入力内容に関わらず、カテゴリの先頭アイコンまたは `📝` が返される
2. **ひらがな表記（nameKana）が入力されない** — AI提案結果で `nameKana` / `nameKanji` が null のまま

#0098 で改善が実施済みだが、**本番環境で問題が継続**している。

## 根本原因分析

### 原因1（最有力）: 本番環境で GEMINI_API_KEY が未設定

`activity-suggest-service.ts` L111-112:

```typescript
const apiKey = process.env.GEMINI_API_KEY;
if (!apiKey || apiKey === 'your_gemini_api_key_here') {
    return null;  // → 100% フォールバックに落ちる
}
```

**状況証拠:**
- `.env.example` で `GEMINI_API_KEY` はコメントアウト状態（L12: `# GEMINI_API_KEY=your_gemini_api_key_here`）
- `scripts/setup-server.sh` がプレースホルダー値 `your_gemini_api_key_here` を設定する（L40）
- Docker環境（NUCサーバー）に正しいAPIキーが設定されているか未確認

**検証方法:**

```bash
# NUCサーバーで確認
ssh kusaka-server@192.168.68.79 "cd C:\Docker\ganbari-quest && docker compose exec app printenv GEMINI_API_KEY"
```

APIキーが未設定またはプレースホルダーの場合、**すべてのAI提案がフォールバック経路**を通り、Gemini APIは一切使われない。

### 原因2: フォールバック（suggestByKeywords）の制限

Gemini APIが使えない場合の `suggestByKeywords()` + `inferNames()` の動作:

| 入力例 | icon | nameKana | nameKanji | 備考 |
|--------|------|----------|-----------|------|
| `サッカー` | ⚽ | `サッカー` | null | KEYWORD_ICONS にマッチ、カナのみなのでnameKana設定 |
| `水泳` | 🏊 | null | `水泳` | KEYWORD_ICONS にマッチ、漢字なのでnameKanji設定 |
| `お風呂そうじ` | 🛁 | `お風呂そうじ` | null | マッチ成功、カナのみと判定 |
| `宿題をする` | 📝→📖 | null | `宿題をする` | キーワードマッチ、漢字含むのでnameKanji |
| `abc練習` | 📝 | null | null | キーワード不十分、アイコンはデフォルト |

**`inferNames()` の根本的な制約:**
- かな入力 → `nameKana` のみ設定、`nameKanji` は常に null（漢字変換不可）
- 漢字入力 → `nameKanji` のみ設定、`nameKana` は常に null（読み仮名推定不可）
- **片方しか埋まらない** → UIで「ひらがな表記がない」と見える

### 原因3: Gemini API利用時でもnull返却の可能性

#0098 でプロンプトに「nullにしないでください」と明記したが、Gemini が必ず従う保証はない。`suggestWithGemini()` L232-233:

```typescript
nameKana: obj.nameKana ? String(obj.nameKana).slice(0, 50) : null,
nameKanji: obj.nameKanji ? String(obj.nameKanji).slice(0, 50) : null,
```

Gemini が空文字 `""` や `null` を返した場合、そのまま null になる。

## 修正方針

### Phase 1: 環境設定確認と修正（即時対応）

1. **本番環境のGEMINI_API_KEY確認**
   - NUCサーバーの `docker-compose.yml` または `.env` で APIキーが正しく設定されているか確認
   - プレースホルダー値の場合は正しい Gemini API キーを設定
   - コンテナ再起動で反映

2. **APIキー未設定時のログ強化**
   ```typescript
   function getGeminiClient(): GoogleGenerativeAI | null {
       const apiKey = process.env.GEMINI_API_KEY;
       if (!apiKey || apiKey === 'your_gemini_api_key_here') {
           logger.warn('[activity-suggest] GEMINI_API_KEY未設定: フォールバックモードで動作');
           return null;
       }
       return new GoogleGenerativeAI(apiKey);
   }
   ```

### Phase 2: フォールバック品質の向上

3. **`inferNames()` の改善** — 相互変換テーブルの導入

   よく使われる活動名の漢字↔ひらがな対応表を用意し、片方から他方を推定:

   ```typescript
   const NAME_PAIR_TABLE: Record<string, { kana: string; kanji: string }> = {
       'すいえい': { kana: 'すいえい', kanji: '水泳' },
       '水泳': { kana: 'すいえい', kanji: '水泳' },
       'はみがき': { kana: 'はみがき', kanji: '歯みがき' },
       'おかたづけ': { kana: 'おかたづけ', kanji: 'お片付け' },
       // ... 主要な活動名を網羅
   };
   ```

4. **nameKana/nameKanji のフォールバック値強化** — Gemini応答でnullの場合、入力テキストから補完:

   ```typescript
   // suggestWithGemini() の戻り値で
   nameKana: obj.nameKana ? String(obj.nameKana).slice(0, 50) 
       : inferNames(text).nameKana,
   nameKanji: obj.nameKanji ? String(obj.nameKanji).slice(0, 50) 
       : inferNames(text).nameKanji,
   ```

### Phase 3: UI改善

5. **フォールバック通知の改善** — 現在のUI（`#0098` 実装済み）はフォールバック時にamber色の通知を表示するが、APIキー未設定が原因の場合は管理者向けに分かりやすいメッセージに変更

6. **source フィールドの活用** — レスポンスに含まれる `source: 'gemini' | 'fallback'` を活用し、常にフォールバック状態が続く場合に「API設定を確認してください」と促す
