# #0097 活動削除と記録の安全な扱い

## ステータス: Done
## 優先度: 中
## 種別: 機能改善・不具合

---

## 概要

親の管理画面から活動を削除しようとすると、過去に一度でも活動記録（`activity_logs`）がある場合は「記録があるため削除できません」と表示され、物理削除ができない。現状は「非表示」（ソフトデリート）のみで対処しているが、親目線では「消せない活動が残り続ける」という使い勝手の問題がある。

記録を安全に残しつつ、活動マスタを整理できる仕組みを検討・実装する。

## 背景・現状の実装

### 削除フロー

```
削除ボタン → 確認ダイアログ → hasActivityLogs(id) チェック
  ├─ ログなし → deleteActivity(id) で物理削除 ✅
  └─ ログあり → fail(409) 「記録があるため削除できません」 ❌
```

### コード箇所

| レイヤー | ファイル | 処理 |
|---------|---------|------|
| UI | `(parent)/admin/activities/+page.svelte` L620-657 | 削除確認ダイアログ。「記録がある場合は削除できません」の注記あり |
| アクション | `(parent)/admin/activities/+page.server.ts` L140-161 | `hasActivityLogs` → `deleteActivity` |
| サービス | `services/activity-service.ts` | `deleteActivity`（物理削除）、`hasActivityLogs`（ログ存在チェック）、`setActivityVisibility`（ソフトデリート） |
| リポジトリ | `db/activity-repo.ts` | `setActivityVisibility`（ソフトデリートのみ。物理削除関数なし） |

### FK制約

`activities.id` を参照しているテーブル:

| テーブル | カラム | 制約 |
|---------|--------|------|
| `activity_logs` | `activityId` | FK → `activities.id` |
| `daily_missions` | `activityId` | FK → `activities.id` |

このFK制約により、ログが存在する活動を物理削除すると整合性エラーが発生する。

## 問題点

1. **記録がある活動は削除も非表示の解除もできない** — 間違えて作った活動でも、一度でも記録したら永久に残る
2. **非表示の活動が管理画面から見えにくい** — 非表示にした活動を一覧で確認・復活させるUIが不十分
3. **「非表示」と「削除」の違いがユーザーに分かりにくい** — 削除ダイアログに小さく書いてあるだけ

## 検討事項

### A. 記録付き活動の削除を許可する方式

#### A-1. カスケード削除
活動を消すと、関連する `activity_logs` と `daily_missions` も一緒に削除する。
- ✅ シンプル
- ❌ ポイント履歴・コンボ・実績の整合性が崩れる可能性
- ❌ 子供の過去の記録が完全に消えるため、運用上のリスクが大きい

#### A-2. 活動ログの `activityId` をNULL化して削除
活動を消す前に、関連ログの `activityId` を `NULL` にしてから物理削除。
- ✅ ログ自体は残る（ポイント履歴は保全）
- ❌ 「何の活動だったか」が分からなくなる
- ❌ スキーマ変更（`activityId` のNULL許容化）が必要

#### A-3. ログに活動名スナップショットを保存して削除
活動削除時に、関連ログに活動名・カテゴリ等をスナップショットとして書き込んでから物理削除。
- ✅ 過去の記録に「何をやったか」が残る
- ❌ `activity_logs` テーブルにスナップショット用カラム追加が必要
- ❌ 削除後にカテゴリ集計で正しく機能するか要検討

### B. ソフトデリートの改善（現行方式の拡張）

#### B-1. 非表示活動の管理UIを改善
- 非表示活動のフィルタ表示・一括管理
- 「非表示」と「アクティブ」のステータス切り替えを分かりやすくする
- 非表示活動を折りたたみで下部に表示

#### B-2. アーカイブ状態の導入
`isVisible` の 0/1 に加えて、`archived` 状態を追加。
- アーカイブ：子供の活動選択リストに出ない + 管理画面では「アーカイブ済み」セクションに移動
- 完全削除：ログなしのもののみ許可（現行通り）

### C. 推奨案: A-3 + B-1 の組み合わせ

1. **活動ログにスナップショットカラムを追加**（`activityName TEXT`, `activityCategory TEXT`）
2. **記録付き活動の削除時**: ログにスナップショットを書き込み → FK制約解除 → 物理削除
3. **非表示活動の管理UIを改善**: フィルタ・復活・一括操作を追加
4. **削除確認ダイアログの改善**: 記録件数を表示し、「スナップショットを保存して削除」のオプションを明示
