## 根本原因

**デプロイ時の Docker コンテナ再作成における SQLite WAL 不整合**

### 発生メカニズム

1. `docker compose build && docker compose up -d` を実行
2. `up -d` がコンテナを **Recreate**（旧コンテナ停止→新コンテナ起動）
3. 旧コンテナ内の Node.js プロセスが SQLite を WAL モードで使用中に SIGTERM を受信
4. Node.js の graceful shutdown ハンドリングが未実装のため、`better-sqlite3` のDB接続が適切にクローズされない
5. WAL チェックポイントが未実行のまま `.db-wal` / `.db-shm` がホスト側に残存
6. 新コンテナが同じボリュームマウントから DB を開く際、**不完全な WAL 状態** を読み取り「malformed」と判定

### 寄与要因

| 要因 | 詳細 |
|------|------|
| **Graceful shutdown 未実装** | Node.js プロセスが SIGTERM を受けた際に DB 接続をクローズする処理がない |
| **stop_grace_period 未設定** | docker-compose.yml に `stop_grace_period` がなく、デフォルト10秒だがアプリがその間にシャットダウンしない |
| **デプロイ手順に WAL 考慮なし** | デプロイ前後に WAL チェックポイントを実行するステップがない |
| **SQLite + Docker ボリュームマウント** | SQLite の WAL/SHM ファイルがホスト-コンテナ間で共有されるため、コンテナライフサイクルに影響を受けやすい |

## 混入原因分析

### なぜ今回発生したか

- #0048 キャリアプランニング機能のデプロイ時に `docker compose build && docker compose up -d` を実行
- この時点で子供画面が使用中（`/kinder/home` へのアクセスログあり）であった可能性
- コンテナ Recreate のタイミングで DB への書き込みトランザクションが走っていた

### なぜこれまで発生しなかったか

- 過去のデプロイでは偶然 DB アクセスの隙間でコンテナが切り替わっていた
- または発生していたが、次のヘルスチェック（30秒間隔）前にWALが自動回復していた
- 今回はマイグレーションスクリプト（`add-career-tables.cjs`）をデプロイ直後にホスト側 Node.js から実行したため、Docker 内プロセスとホスト側プロセスが同時に DB にアクセスし WAL 競合が発生した可能性が高い

## 再発防止策

### 【P0: 即座に対応】デプロイ手順の改善

現在の手順:
```bash
ssh server "cd project && git pull && docker compose build && docker compose up -d"
ssh server "node scripts/add-xxx.cjs path/to/db"  # ← ホスト側から直接DB操作
```

改善後の手順:
```bash
# 1. コンテナを停止（WALフラッシュの猶予を与える）
ssh server "cd project && docker compose stop app"
# 2. ホスト側でマイグレーション実行（競合なし）
ssh server "cd project && node scripts/add-xxx.cjs data/ganbari-quest.db"
# 3. ビルド＆起動
ssh server "cd project && git pull && docker compose build && docker compose up -d"
```

### 【P1: 短期】Graceful Shutdown の実装

`src/hooks.server.ts` または Node.js エントリポイントで SIGTERM ハンドリングを追加:

```typescript
process.on('SIGTERM', () => {
  console.log('[SHUTDOWN] SIGTERM received, closing DB...');
  db.exec('PRAGMA wal_checkpoint(TRUNCATE)');
  db.close();
  process.exit(0);
});
```

### 【P1: 短期】docker-compose.yml に stop_grace_period 追加

```yaml
services:
  app:
    stop_grace_period: 15s
```

### 【P2: 中期】マイグレーションの Docker 内実行化

ホスト側から直接 DB ファイルを操作するのではなく、Docker コンテナ内でマイグレーションを実行する:

```bash
docker compose exec app node scripts/add-xxx.cjs /app/data/ganbari-quest.db
```

または、エントリポイント（`docker-entrypoint.sh`）にマイグレーション自動実行を組み込む。

### 【P2: 中期】DB 初期化時の PRAGMA 設定見直し

```sql
PRAGMA busy_timeout = 5000;       -- ロック待ち5秒
PRAGMA wal_autocheckpoint = 100;  -- 100ページごとに自動チェックポイント（デフォルト1000）
PRAGMA synchronous = NORMAL;      -- WALモードではNORMALで十分な耐久性
```

### 【P3: 長期】ヘルスチェックに DB 疎通を含める

現在の `/api/health` が DB クエリを含んでいない場合、DB 障害をヘルスチェックで検知できない。`SELECT 1` レベルの疎通確認を追加。
