# #0114 データインポート・エクスポート機能

## ステータス: Ready
## 優先度: 高
## 種別: 新機能

---

## 概要

親管理画面に、家族のデータを JSON ファイルとしてエクスポートし、別環境（Web版 → ローカル版、ローカル版 → Web版、バックアップ＆リストア等）にインポートできる機能を追加する。

### 動機

- SaaS版（Web版）とローカル版（NUC Docker）間の**データポータビリティ**の確保
- ユーザーが自分のデータの**オーナーシップ**を持てるようにする
- 環境移行時に、子供がこれまで積み上げた**ポイント・活動記録・ステータス・実績**が失われないことが最重要

### ユースケース

| UC | シナリオ | 方向 |
|----|---------|------|
| UC1 | SaaS版を試用 → ローカル版に移行したい | Web → Local |
| UC2 | ローカル版で運用 → SaaS版に乗り換えたい | Local → Web |
| UC3 | バックアップとして定期的にエクスポート | Export only |
| UC4 | 端末故障・DB破損からのリカバリ | Import only |
| UC5 | 開発/テスト環境にデータを複製したい | Export → Import |

### 設計原則

| 原則 | 説明 |
|------|------|
| **名前ベースの参照** | 整数 ID は環境間で不一致になるため、活動名+カテゴリコード等の安定した識別子で参照する |
| **エクスポートID** | 子供の内部 ID は `exportId`（`child-1`, `child-2`）に変換。インポート時に新しい ID を発番 |
| **べき等インポート** | 同じファイルを2回インポートしてもデータが重複しない（日付 + 子供 + 活動の UNIQUE制約でスキップ） |
| **バージョン管理** | `format` + `version` でフォーマットの後方互換性を管理 |
| **チェックサム** | SHA-256 ハッシュでファイル改ざん・破損を検出 |

```
  Query:
```

#### サービス層

```typescript
// src/lib/server/services/export-service.ts
export async function exportFamilyData(options: ExportOptions): Promise<ExportData> {
  // 1. 子供プロフィール取得
  // 2. 活動マスタ取得（IDマッピング用）
  // 3. 子供ごとのトランザクションデータを一括取得
  // 4. IDを名前ベース参照に変換
  // 5. チェックサム計算
  // 6. JSON構造を組み立てて返却
}
```

```
POST /api/v1/import
  Content-Type: multipart/form-data
  Body:
    file: (JSON file)
    conflictStrategy: 'overwrite' | 'skip' | 'create_new'
  Response:
    {
      "success": true,
      "summary": {
        "children": { "imported": 1, "skipped": 0 },
        "activityLogs": { "imported": 1234, "skipped": 12 },
        "pointLedger": { "imported": 567, "skipped": 0 },
        ...
      },
      "warnings": []
    }
```

#### サービス層

```typescript
// src/lib/server/services/import-service.ts
export async function importFamilyData(
  data: ExportData,
  strategy: ConflictStrategy
): Promise<ImportResult> {
  // 1. フォーマット・バージョン検証
  // 2. チェックサム検証
  // 3. スキーマバリデーション（Zod）
  // 4. マスタデータのID再マッピング
  //    - 活動名+カテゴリコード → 移植先のactivity_id
  //    - 未知の活動は警告としてスキップ
  // 5. 子供データの競合チェック
  // 6. トランザクション内で一括INSERT
  //    - activityLogs: UNIQUE(child_id, activity_id, recorded_date) で重複スキップ
  //    - statuses: UPSERT（上書き or スキップ）
  //    - pointLedger: 全件INSERT（台帳は追記型）
  // 7. 画像ファイルの復元（あれば）
  // 8. インポート結果サマリを返却
}
```

## IDマッピング戦略

環境間で自動採番 ID が異なるため、**名前ベースの安定識別子**でマッピングする。

| テーブル | 安定識別子 | マッピング方法 |
|---------|-----------|--------------|
| `categories` | `code`（undou, benkyou...） | 5件固定。コードで完全一致 |
| `activities` | `name` + `categoryCode` | 名前+カテゴリで検索。見つからない場合はカスタム活動として新規作成 |
| `titles` | `name` | 名前で検索 |
| `achievements` | `name` + `type` | 名前+タイプで検索 |
| `avatarItems` | `name` + `type` + `category` | 名前+タイプ+カテゴリで検索 |
| `careerFields` | `name` | 名前で検索 |
| `children` | `nickname` + `birthDate` | ニックネーム+生年月日で競合検出 |

### 未知のマスタデータの扱い

| ケース | 対応 |
|-------|------|
| エクスポート元にあってインポート先にない活動 | **警告を出してスキップ**。該当する activityLogs は `unmapped` としてレポート |
| カテゴリコードが不一致 | フォーマットエラー（5カテゴリは固定のため発生しないはず） |
| 称号/実績がインポート先にない | 警告を出してスキップ。子供の解除履歴は保存しない |

## セキュリティ

| 対策 | 内容 |
|------|------|
| **PIN 除外** | エクスポートファイルに PIN ハッシュやセッション情報を含めない |
| **ファイルサイズ制限** | インポートファイルは最大 50MB まで |
| **バリデーション** | Zod スキーマで全フィールドを厳密に検証。不正なデータは拒否 |
| **チェックサム検証** | SHA-256 ハッシュでファイル改ざん・転送時の破損を検出 |
| **トランザクション** | インポートは単一トランザクションで実行。途中エラー時は全ロールバック |
| **サニタイズ** | テキストフィールドの XSS 対策（HTML エスケープ） |
| **レート制限** | エクスポート: 1分1回、インポート: 1時間3回 |
