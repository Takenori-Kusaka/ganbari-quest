# #0121 個人情報保護・GDPR準拠レビューと設計書反映

## Status: Waiting
## 優先度: 高（SaaS公開前の必須ゲート）
## 種別: セキュリティレビュー・設計改善

---

## 概要

本プロジェクトは子供の個人情報（氏名、年齢、誕生日、活動履歴、アバター画像）を扱う。SaaS展開（#0064〜#0079）を控え、**GDPR（EU一般データ保護規則）をはじめとする各国の個人情報保護法制への準拠状況を網羅的にレビュー**し、不足する要件を特定して設計書群（`docs/design/`）へ反映する。

---

## 2. 各国の個人情報保護法制と要件

### 2.1 EU — GDPR（General Data Protection Regulation）

**適用条件**: EU居住者のデータを処理する場合（サーバ所在地は無関係）

| 条項 | 要件 | 現状 | ギャップ |
|------|------|------|---------|
| **Art. 5** データ最小化 | 目的に必要なデータのみ収集 | △ 活動ログを永久保持 | ログの保持期間ポリシーが必要 |
| **Art. 6** 処理の法的根拠 | 明示的同意 or 正当な利益 | ❌ 同意取得UIなし | Cookie同意バナー + 利用規約同意フロー |
| **Art. 7** 同意の条件 | 自由意志による明確な同意 | ❌ 未実装 | サインアップ時の同意チェックボックス |
| **Art. 8** 子供の同意 | 16歳未満は保護者の同意が必要（加盟国により13歳まで引き下げ可） | ❌ 未実装 | **保護者同意フロー**（本アプリは全ユーザーが子供なので必須） |
| **Art. 12-14** 透明性 | プライバシーポリシーの公開 | ❌ 未作成 | プライバシーポリシーの策定・公開 |
| **Art. 15** アクセス権 | データ主体が自分のデータを閲覧・取得できる | △ UIでは閲覧可能だがエクスポートなし | データエクスポート機能（#0114と統合） |
| **Art. 16** 訂正権 | 不正確なデータの訂正 | △ 一部編集可能 | 全個人情報フィールドの編集機能保証 |
| **Art. 17** 削除権（忘れられる権利） | データ主体の要求でデータを完全削除 | ⚠️ カスケード削除はあるがUI/APIなし | **アカウント削除機能**の実装 |
| **Art. 20** データポータビリティ | 機械可読形式でデータ提供 | ❌ 未実装 | JSONエクスポート機能（#0114と統合） |
| **Art. 25** プライバシー・バイ・デザイン | 設計段階からプライバシー配慮 | △ 部分的 | 設計書への明示的な記載 |
| **Art. 28** 処理者の義務 | サブプロセッサーとのDPA締結 | ❌ 未実施 | AWS, Stripe, Google (Gemini) とのDPA確認 |
| **Art. 30** 処理活動の記録 | データ処理の記録保持義務 | ❌ 監査ログなし | **監査ログシステム**の導入 |
| **Art. 32** セキュリティ | 技術的・組織的な安全管理措置 | △ 基本対策あり | 暗号化・アクセス制御の強化 |
| **Art. 33-34** 侵害通知 | 72時間以内にDPAへ通知、重大な場合はデータ主体にも通知 | ❌ インシデント対応計画なし | インシデント対応手順の策定 |
| **Art. 35** DPIA | 高リスク処理にはデータ保護影響評価が必要 | ❌ 未実施 | **DPIA（データ保護影響評価）実施** |
| **Art. 44-49** 越境移転 | EU域外へのデータ移転にはSCC等の保護措置 | ⚠️ us-east-1リージョン | **EUリージョン or SCC対応** |

### 2.2 日本 — 個人情報保護法（APPI）

**適用条件**: 日本の個人情報を取り扱う全事業者

| 要件 | 条文 | 現状 | ギャップ |
|------|------|------|---------|
| 利用目的の特定・公表 | 第17-21条 | ❌ 未公表 | プライバシーポリシーで明示 |
| 安全管理措置 | 第23条 | △ 基本対策 | 組織的・技術的安全管理措置の文書化 |
| 第三者提供の制限 | 第27条 | △ Gemini APIに画像送信あり | Gemini送信内容の明示と同意 |
| 保有個人データの開示請求 | 第33条 | ❌ 未対応 | データエクスポート機能 |
| 保有個人データの訂正・削除 | 第34-35条 | △ 一部対応 | アカウント削除機能 |
| 漏えい等の報告義務 | 第26条 | ❌ 対応計画なし | インシデント対応手順 |
| **子供の個人情報** | ガイドライン | ❌ 特別対応なし | 保護者同意フロー |

### 2.3 米国 — COPPA（Children's Online Privacy Protection Act）

**適用条件**: 13歳未満の子供からオンラインで情報を収集するサービス（US居住者対象）

| 要件 | 現状 | ギャップ |
|------|------|---------|
| 13歳未満の子供の情報収集前に保護者から検証可能な同意を取得 | ❌ | **保護者同意フロー**（COPPA Safe Harbor推奨） |
| プライバシーポリシーに子供の情報の収集・利用を明示 | ❌ | 子供向け個人情報保護方針の策定 |
| 保護者がデータの閲覧・削除を要求できる | ❌ | アカウント削除 + データエクスポート |
| 必要最小限の情報のみ収集 | △ | データ最小化ポリシー |

### 2.4 その他の主要法制

| 法制 | 地域 | 本アプリへの影響 | 対応優先度 |
|------|------|----------------|-----------|
| CCPA/CPRA | カリフォルニア | 個人情報の販売禁止、削除権 | 中（US展開時） |
| LGPD | ブラジル | GDPRと類似の要件 | 低 |
| PIPA | 韓国 | 厳格な同意要件 | 低 |
| PDPA | シンガポール | 同意ベースの処理 | 低 |

## 3. 現状のセキュリティ課題

### 3.1 認証・認可

| 課題 | 現状 | リスク | 対応 |
|------|------|--------|------|
| API認可チェック欠落 | 一部APIエンドポイントに認証チェックなし | 🔴 Critical | 全 `+server.ts` に `event.locals.authenticated` チェック追加 |
| Cookie secure フラグ | `false` 固定 | 🔴 Critical | 環境判定で `true` 設定 |
| セッション有効期限 | 1年（365日） | 🟠 High | 30日に短縮 + リフレッシュトークン |
| PIN長 | 4-6桁 | 🟠 High | SaaS版ではOAuth必須、PINは補助認証 |

### 3.2 データ保護

| 課題 | 現状 | リスク | 対応 |
|------|------|--------|------|
| birth_date 平文保存 | SQLite / DynamoDB ともに平文 | 🟠 High | DynamoDB移行時にKMS暗号化 |
| アバター画像のアクセス制御 | `static/uploads/` にファイル配置、認証なし | 🔴 Critical | S3 + 署名付きURL or Lambda@Edge認証 |
| 活動ログの永久保持 | 保持期間ポリシーなし | 🟡 Medium | 保持期間の定義（例: 退会後90日で完全削除） |
| バックアップの暗号化 | Git pushによるバックアップ | 🟡 Medium | S3暗号化バックアップ（AES-256-GCM） |

### 3.3 データ削除

| 課題 | 現状 | リスク | 対応 |
|------|------|--------|------|
| アカウント削除UI | なし | 🔴 Critical | 親管理画面に「アカウント完全削除」機能 |
| 子供削除時の画像ファイル | DB上のカスケード削除はあるが物理ファイル未削除 | 🟠 High | `fs.unlinkSync` / S3 `deleteObject` |
| AI生成画像の削除 | `character_images` は削除されるが `static/generated/` のファイル残存 | 🟠 High | 物理ファイル削除の追加 |
| バックアップからの削除 | バックアップ内に削除済みデータが残る | 🟡 Medium | バックアップ保持期間ポリシー（例: 30日でローテーション） |

### 3.4 監査・ログ

| 課題 | 現状 | リスク | 対応 |
|------|------|--------|------|
| 監査ログ | 存在しない | 🟡 Medium | 個人情報アクセス・変更・削除の記録 |
| ログへの個人情報出力 | `console.log` に childId 等が含まれる可能性 | 🟡 Medium | PIIマスカレードの導入 |

## 4. 期待される実装（要件一覧）

### 4.1 アカウント削除・データ完全消去（GDPR Art.17 / APPI 第35条 / COPPA）

```
【ユーザーフロー】
1. 親が管理画面 → 「アカウント削除」を選択
2. 確認ダイアログ: 「すべてのデータが完全に削除されます。この操作は取り消せません。」
3. PIN再入力による本人確認
4. 削除実行

【バックエンド処理】
1. DynamoDB: 該当familyId/childIdの全Itemsを削除（TransactWriteItems）
2. S3: 該当ユーザーのアバター画像・生成画像を削除
3. Cognito: ユーザープール上のアカウント削除（adminDeleteUser）
4. 監査ログ: 「ユーザーXXXXが削除を要求、実行完了」を記録（個人情報はマスク）
5. バックアップ: 30日後のローテーションで自然消去

【削除対象テーブル/リソース】
- children + 全関連テーブル（既存カスケード削除を活用）
- static/uploads/avatars/avatar-{childId}-*.* （物理ファイル）
- static/generated/{childId}/ （AI生成画像の物理ファイル）
- SaaS版: DynamoDB PK=CHILD#{id} の全Item
- SaaS版: S3 avatars/{familyId}/{childId}/ プレフィクス配下
- SaaS版: Cognito User Pool のユーザーレコード
```

### 4.2 データアクセス制御（SaaS版 DynamoDB + S3）

```
【DynamoDBアクセス制御】
- Lambda関数のみがDynamoDBにアクセス可能（IAMポリシー）
- DynamoDBコンソール直接アクセスは管理者IAMロール + MFA必須
- テーブルのServer-Side Encryption (SSE) 有効化（AWS managed key or KMS CMK）
- VPC Endpoint経由でのアクセス（パブリックインターネット非経由）

【S3アクセス制御】
- バケットポリシー: Lambda実行ロール + CloudFront OAC のみ許可
- パブリックアクセス完全ブロック（Block Public Access = ALL ON）
- アバター画像の配信: 署名付きURL（有効期限15分）or CloudFront署名付きCookie
- バケット暗号化: SSE-S3 or SSE-KMS

【管理者アクセス（例外アクセス）】
- AWS IAMロールによるアクセス + MFA必須
- CloudTrailで全API呼び出しを記録
- 承認付きアクセス: 本番DynamoDBへの直接クエリはIAM認証 + 上長承認ワークフロー
```

### 4.3 同意管理

```
【サインアップ時】
- 利用規約への同意チェック（必須）
- プライバシーポリシーへの同意チェック（必須）
- 「子供の個人情報を保護者として管理することに同意します」チェック（必須、COPPA対応）

【Cookie同意バナー】
- 必須Cookie（セッション）: 同意不要
- 分析Cookie（導入する場合）: オプトイン同意

【同意記録の保存】
- 同意日時、同意バージョン、同意内容をDynamoDBに記録
- 同意の撤回機能
```

### 4.4 プライバシー・バイ・デザインの原則適用

| 原則 | 適用内容 |
|------|---------|
| **1. 事前的・予防的** | リスクアセスメントを設計段階で実施 |
| **2. デフォルトの保護** | 最小限のデータ収集がデフォルト（birth_date はオプショナル） |
| **3. 設計に組み込む** | 暗号化・アクセス制御をインフラレベルで適用 |
| **4. 全機能** | セキュリティとユーザビリティの両立 |
| **5. エンドツーエンド** | データ収集から削除までのライフサイクル管理 |
| **6. 可視性と透明性** | プライバシーポリシーの公開、データ処理の説明 |
| **7. 個人の尊重** | ユーザーのデータ管理権限（アクセス・訂正・削除） |

- [ ] 子供削除時にDB上の全関連データが削除されること

- [ ] データエクスポートが全個人情報をJSON形式で出力すること
- [ ] 設計書群に個人情報保護要件が反映されていること

## 注意事項

- **SaaS公開はこのチケットのPhase B完了をゲートとする**: プライバシーポリシー・同意フロー・アカウント削除機能が未実装のままSaaS公開は不可
- **リージョン選定の影響**: 現在 `us-east-1` だが、GDPR Art.44-49 対応のためEUリージョン（`eu-west-1`）への変更、もしくは SCC（標準契約条項）の適用が必要。DynamoDB Global Tables を検討する価値あり
- **子供のデータは特に厳格**: GDPR Art.8 / COPPA / APPI ガイドラインいずれも子供のデータには特別な保護を要求している。本アプリの全ユーザーが子供であるため、デフォルトで最高レベルの保護が必要
- **法的レビュー推奨**: プライバシーポリシーの最終版は法務専門家のレビューを推奨。個人開発でも「準拠を意図した設計」であることが重要
- **既存LAN版への影響**: Phase A のタスクは現行LAN版にも適用可能。物理ファイル削除はセキュリティ上すぐに対応すべき
