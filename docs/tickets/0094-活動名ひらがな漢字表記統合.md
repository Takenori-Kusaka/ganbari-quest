# #0094 活動名のひらがな・漢字表記統合

## Status: Done
## 優先度: 高
## 種別: 機能改善（データモデル変更）

---

## 概要

現在、漢字が読めない年齢向けと読める年齢向けで同一内容の活動を2レコード（ひらがな版・漢字版）に分けて管理している。
活動マスタに「読み（ひらがな）」と「表示名（漢字）」の両方を持たせ、子供の年齢に応じて自動的に表記を切り替えることで、管理を一元化する。

## 背景・課題

### 現状

例: 「おかたづけ」と「お片付け」を別々の活動レコードとして作成し、`ageMin`/`ageMax` で年齢範囲を分けることで表記を出し分けている。

```
id=101  name="おかたづけ"   ageMin=null  ageMax=5   (baby/kinder向け)
id=102  name="お片付け"      ageMin=6     ageMax=null (小学生以上向け)
```

### 問題点

- **管理コスト倍増**: 同じ活動を2つ作成・メンテする必要がある
- **ポイント・実績の分断**: ひらがな版と漢字版は別 activityId のため、活動ログ・コンボ・実績が分断される
- **年齢が上がった時の移行**: 子供がひらがな版の年齢範囲を超えると漢字版に切り替わるが、過去の記録は引き継がれない
- **カテゴリコンボへの影響**: 同じ内容なのに別活動としてカウントされ、コンボ判定に影響する可能性

## 変更内容

### 1. DB スキーマ変更

`activities` テーブルに以下のカラムを追加:

```sql
ALTER TABLE activities ADD COLUMN name_kana TEXT;     -- ひらがな表記（例: "おかたづけ"）
ALTER TABLE activities ADD COLUMN name_kanji TEXT;     -- 漢字表記（例: "お片付け"）
```

- `name`: 従来通りのメイン表示名（デフォルト表示・管理画面用）
- `name_kana`: ひらがな表記（null許容。nullの場合は `name` をそのまま使用）
- `name_kanji`: 漢字表記（null許容。nullの場合は `name` をそのまま使用）

### 2. 年齢による表示名切り替えロジック

表示名の決定ロジックを追加:

```typescript
function getDisplayName(activity: Activity, childAge: number): string {
  const KANJI_AGE_THRESHOLD = 6; // 小学生以上なら漢字表記
  if (childAge >= KANJI_AGE_THRESHOLD && activity.nameKanji) {
    return activity.nameKanji;
  }
  if (childAge < KANJI_AGE_THRESHOLD && activity.nameKana) {
    return activity.nameKana;
  }
  return activity.name; // フォールバック
}
```

配置場所の候補:
- `src/lib/domain/validation/activity.ts` にユーティリティとして追加
- または `src/lib/server/services/activity-service.ts` の `getActivities` の結果に `displayName` を付与

### 3. バリデーションスキーマ更新

`createActivitySchema` / `updateActivitySchema` に新フィールドを追加:

```typescript
nameKana: z.string().max(50).nullable().optional(),   // ひらがな表記
nameKanji: z.string().max(50).nullable().optional(),   // 漢字表記
```

### 4. フロントエンド対応

- 子供用画面（baby/kinder/その他）: `activity.name` の代わりに `displayName` を表示
- 親の管理画面: 活動マスタ編集フォームに「ひらがな表記」「漢字表記」の入力欄を追加
- API レスポンス: `displayName` フィールドを追加するか、表示名決定をフロントで行うか検討

### 5. 既存データのマイグレーション

- ひらがな/漢字ペアの既存レコードを統合するマイグレーションスクリプトを作成
- 統合時にポイント履歴・活動ログの `activityId` を新レコードに寄せる
- 統合後の旧レコードは `isVisible = 0`（非表示）にするか削除
