### 概要

DynamoDB のブリッジモデルに基づき、全データを家族グループ（テナント）単位で管理する。既存エンティティ（CHILD#, ACTIVITY# 等）にテナントプレフィックスを付与し、API レイヤでテナントスコープを強制する。

### 現状

- 既存データ: `CHILD#1`, `ACTIVITY#...`, `SETTING` 等、テナントプレフィックスなし
- Auth エンティティ: `TENANT#t-xxx`, `USER#u-xxx` は既にテナント対応済み（auth-keys.ts）
- 既存の DynamoDB repo は 17 リポジトリ（keys.ts に定義）

### 設計

#### DynamoDB キースキーマ変更

```
変更前:
  PK: CHILD#1          SK: PROFILE
  PK: CHILD#1          SK: ACTIVITY#2024-01-01#act-xxx

変更後（テナントプレフィックス付き）:
  PK: TENANT#t-xxx     SK: CHILD#1
  PK: TENANT#t-xxx     SK: CHILD#1#ACTIVITY#2024-01-01#act-xxx

または（シングルテーブルデザイン維持）:
  PK: TENANT#t-xxx#CHILD#1     SK: PROFILE
  PK: TENANT#t-xxx#CHILD#1     SK: ACTIVITY#2024-01-01#act-xxx
```

#### 家族グループ（テナント）データモデル

```
TENANT#t-xxx / META
  - tenantId: string
  - name: string（家族名）
  - ownerId: string（作成者のuserId）
  - status: 'active' | 'grace_period' | 'suspended' | 'deleted'
  - licenseKey: string
  - plan: 'monthly' | 'yearly' | 'lifetime'
  - createdAt: string
  - memberCount: number

TENANT#t-xxx / MEMBER#u-xxx
  - userId: string
  - role: 'owner' | 'parent' | 'child'
  - joinedAt: string

TENANT#t-xxx / CHILD#1
  - childId: number
  - name: string
  - cognitoUserId?: string（ティーンアカウント紐付け）
  - ...既存の子供プロフィール属性

TENANT#t-xxx / CHILD#1#ACTIVITY#...
  - ...既存の活動データ属性

TENANT#t-xxx / SETTING
  - ...既存の設定データ属性
```

#### テナントスコープ強制

```typescript
// 全 DynamoDB クエリにテナントID を必須化
interface TenantScopedQuery {
  tenantId: string; // 必須
  // ...
}

// repo レイヤで強制
class ActivityRepo {
  async findByChild(tenantId: string, childId: number): Promise<Activity[]> {
    // PK に tenantId を含めることでテナント分離
    const pk = `TENANT#${tenantId}#CHILD#${childId}`;
    // ...
  }
}
```

### ゴール

- [ ] 全既存エンティティのキースキーマにテナントプレフィックス追加
- [ ] keys.ts のキー生成関数を tenantId 必須に変更
- [ ] 全 DynamoDB repo の引数に tenantId 追加
- [ ] API レイヤで JWT の tenantId と一致するデータのみアクセス許可
- [ ] SQLite 版（local モード）は tenantId='local' で固定
- [ ] 既存データのマイグレーションスクリプト
- [ ] テナント分離のユニットテスト
- [ ] 異なるテナントのデータにアクセスできないことを検証する E2E テスト

### マイグレーション方針

```
1. 新キースキーマを並行稼働（読み取り: 旧→新フォールバック）
2. マイグレーションスクリプトで既存データを新スキーマに移行
3. 旧スキーマの読み取りフォールバックを削除
4. 旧データのクリーンアップ
```

### 完了条件

1. 全データが TENANT# プレフィックス付きで保存される
2. 異なるテナントのデータにはアクセスできない
3. local モードでは tenantId='local' で透過的に動作
4. 既存データが新スキーマに移行されている
