### 概要

DynamoDB のブリッジモデルに基づき、全データを家族グループ（テナント）単位で管理する。既存エンティティ（CHILD#, ACTIVITY# 等）にテナントプレフィックスを付与し、API レイヤでテナントスコープを強制する。

### 現状

- 既存データ: `CHILD#1`, `ACTIVITY#...`, `SETTING` 等、テナントプレフィックスなし
- Auth エンティティ: `TENANT#t-xxx`, `USER#u-xxx` は既にテナント対応済み（auth-keys.ts）
- 既存の DynamoDB repo は 17 リポジトリ（keys.ts に定義）

### 設計

#### DynamoDB キースキーマ変更

```
変更前:
  PK: CHILD#1          SK: PROFILE
  PK: CHILD#1          SK: ACTIVITY#2024-01-01#act-xxx

変更後（テナントプレフィックス付き）:
  PK: TENANT#t-xxx     SK: CHILD#1
  PK: TENANT#t-xxx     SK: CHILD#1#ACTIVITY#2024-01-01#act-xxx

または（シングルテーブルデザイン維持）:
  PK: TENANT#t-xxx#CHILD#1     SK: PROFILE
  PK: TENANT#t-xxx#CHILD#1     SK: ACTIVITY#2024-01-01#act-xxx
```

#### 家族グループ（テナント）データモデル

```
TENANT#t-xxx / META
  - tenantId: string
  - name: string（家族名）
  - ownerId: string（作成者のuserId）
  - status: 'active' | 'grace_period' | 'suspended' | 'deleted'
  - licenseKey: string
  - plan: 'monthly' | 'yearly' | 'lifetime'
  - createdAt: string
  - memberCount: number

TENANT#t-xxx / MEMBER#u-xxx
  - userId: string
  - role: 'owner' | 'parent' | 'child'
  - joinedAt: string

TENANT#t-xxx / CHILD#1
  - childId: number
  - name: string
  - cognitoUserId?: string（ティーンアカウント紐付け）
  - ...既存の子供プロフィール属性

TENANT#t-xxx / CHILD#1#ACTIVITY#...
  - ...既存の活動データ属性

TENANT#t-xxx / SETTING
  - ...既存の設定データ属性
```

#### テナントスコープ強制

```typescript
// 全 DynamoDB クエリにテナントID を必須化
interface TenantScopedQuery {
  tenantId: string; // 必須
  // ...
}

// repo レイヤで強制
class ActivityRepo {
  async findByChild(tenantId: string, childId: number): Promise<Activity[]> {
    // PK に tenantId を含めることでテナント分離
    const pk = `TENANT#${tenantId}#CHILD#${childId}`;
    // ...
  }
}
```

#### OneTable 設計方針

`dynamodb-onetable`（SenseDeep 製 npm パッケージ）を採用し、シングルテーブルデザインを型安全に実装する。

**採用理由:**

| 観点 | 効果 |
|------|------|
| コスト最適化 | テーブル数削減によりキャパシティ共有が可能。プロビジョンド容量の無駄が減り TCO 低下 |
| パフォーマンス最適化 | 関連エンティティを同一パーティションに格納し、単一クエリで取得可能（クロステーブル JOIN 不要）|
| 型安全性 | TypeScript スキーマ駆動モデルで PK/SK のテンプレートを宣言、コンパイル時検証 |
| 移行管理 | `onetable-migrate` ライブラリで移行バージョン管理・ロールバックが可能 |

**OneTable スキーマ定義例:**

```typescript
// src/lib/server/db/dynamodb/schema.ts
import { Table } from 'dynamodb-onetable';

export const schema = {

  indexes: {

    GSI1: { hash: 'GSI1PK', sort: 'GSI1SK', project: 'all' },
  },
  models: {
    Tenant: {
      PK: { type: String, value: 'TENANT#{tenantId}' },
      SK: { type: String, value: 'META' },
      tenantId: { type: String, required: true },

      ownerId: { type: String, required: true },

    Child: {

      SK: { type: String, value: 'CHILD#{childId}' },
```

**Dual Write リポジトリ実装例:**

```typescript
// src/lib/server/db/dynamodb/dual-write-repo.ts
// Leader（既存実装）への書き込み成功後、Follower（OneTable）にも書き込む
// Follower 失敗は DLQ に投入し非同期リトライ、読み込み側には影響しない
export class DualWriteActivityRepo implements ActivityRepo {
  constructor(
    private readonly leader: LegacyActivityRepo,
    private readonly follower: OneTableActivityRepo,
    private readonly dlq: DlqClient,
    private readonly featureFlag: FeatureFlagClient,
  ) {}

  async save(activity: Activity): Promise<void> {
    await this.leader.save(activity); // Leader は必ず成功させる
    try {
      await this.follower.save(activity);
    } catch (err) {
      await this.dlq.send({ operation: 'save', payload: activity, error: String(err) });
    }
  }

  async findByChild(tenantId: string, childId: number): Promise<Activity[]> {
    if (await this.featureFlag.isEnabled('onetable-read')) {
      return this.follower.findByChild(tenantId, childId);
    }
    return this.leader.findByChild(tenantId, childId);
  }
}
```

### マイグレーション方針（Strangler Fig + Dual Write）

ゼロダウンタイムを保証するため、PDF「DynamoDB OneTable Migration Strategy」に基づき **Strangler Fig + Dual Write パターン**を採用する。

| フェーズ | 内容 | 期間目安 |
|----------|------|----------|
| **Phase 0**: 分析・設計 | 全アクセスパターン洗い出し、OneTable スキーマ設計、GSI 設計、`onetable-migrate` セットアップ | 2〜3 週 |
| **Phase 1**: 並行実装 | OneTable リポジトリ実装（旧実装と共存）、Dual Write 機構・DLQ 整備、Feature Flag 基盤構築 | 2〜4 週 |
| **Phase 2**: Dual Write 開始 | 本番で Dual Write 有効化、DLQ 監視開始、書き込み整合性確認 | 1〜2 週 |
| **Phase 3**: バックフィル | `onetable-migrate` で既存データを新スキーマに一括移行、整合性チェッカー実行 | 1〜3 週 |
| **Phase 4**: 読み込み切替 | Feature Flag で読み込みを段階的に OneTable へ切替（5% → 50% → 100%）、P95 レイテンシ監視 | 2〜4 週 |
| **Phase 5**: 旧実装廃止 | Dual Write 停止、旧リポジトリ・旧スキーマキー削除、DLQ クリーンアップ | 1〜2 週 |

**判断基準（次フェーズへの移行条件）:**
- DLQ 蓄積率: 直近 1 時間で 0 件
- 整合性不整合率: 0.1% 未満
- P95 レスポンスタイム: ベースラインの 1.5 倍未満

### 監視要件

Dual Write 稼働中は以下のアラートを CloudWatch で設定する:

| メトリクス | 閾値 | アクション |
|-----------|------|------------|
| DLQ メッセージ数 | > 100 件 / 時 | Slack アラート + フェーズ一時停止 |
| ConsumedCapacity | プロビジョンド容量の 80% 超 | スケールアップ検討 |
| P95 レスポンスタイム | ベースラインの 1.5 倍超 | Feature Flag で旧実装に切り戻し |
| 整合性不整合率 | > 0.1% | 即時フェーズ停止・調査 |
