# #0113 本番環境E2Eテスト（Smoke Test）の導入

## ステータス: レビュー確認待ち（GitHub Secrets設定が必要）
## 優先度: 高
## 種別: テスト基盤・CI/CD改善

---

## 概要

GitHub Actions の CI パイプラインに、本番環境（`https://ganbari-quest.com/`）に対する E2E テスト（Smoke Test）フェーズを追加する。デプロイ後に本番環境が正常に動作していることを自動検証し、リグレッションやインフラ障害を早期に検出する。

### 課題

SaaS版（#0066）では OAuth（Google/Apple Sign-In）ベースの認証が導入される予定で、**Email/パスワード認証はサポートしない方針**。しかし E2E テストでは OAuth のインタラクティブフロー（ブラウザリダイレクト → Google/Apple ログイン画面）を自動化するのが極めて困難であるため、**テスト専用の認証バイパス機構**が必要になる。

## 現状

### CI/CD（`.github/workflows/ci.yml`）

```
lint-and-test → e2e-test（ローカルdev server）→ docker-build
```

- E2E テストはローカルの `npm run dev`（`localhost:5173`）に対して実行
- 本番環境に対するテストはなし

### E2E テスト構成

| ファイル | 内容 |
|---------|------|
| `tests/e2e/global-setup.ts` | テスト用DB初期化（PIN「1234」設定、テスト子供2名作成） |
| `tests/e2e/smoke.spec.ts` | アプリ起動確認、子供切り替え、活動記録、ログインボーナス等 |
| `tests/e2e/features.spec.ts` | Babyモード、チェックリスト、特別報酬API |
| `playwright.config.ts` | `baseURL: http://localhost:5173`、`webServer: npm run dev` |

### 認証の現状

- 現行: PIN 認証（ローカル SQLite、`hooks.server.ts` で Cookie ベースのセッション検証）
- SaaS版: Cognito + OAuth（Google/Apple）予定（#0066、未実装）
- テスト用E-mail/パスワード認証: なし

## 設計

### アーキテクチャ全体像

```
GitHub Actions CI
├── lint-and-test（既存）
├── e2e-test（既存：ローカル dev server）
├── docker-build（既存）
└── e2e-production（🆕 本番 Smoke Test）
    ├── トリガー: main ブランチ push 後 or 手動 dispatch
    ├── テスト対象: https://ganbari-quest.com/
    ├── 認証方式: テスト用 Email/Password ユーザー
    └── テストスコープ: Smoke Test（非破壊的な読み取り系テスト中心）
```

### Phase 1: テスト専用 Email/Password 認証の導入

OAuth 認証を E2E テストで自動化するのは以下の理由から現実的でない:
- Google/Apple のログイン画面はサードパーティドメインのため Playwright で完全制御不可
- MFA やキャプチャが介在する可能性
- OAuth トークンの有効期限管理が煩雑

そのため、**Cognito User Pool に Email/Password 認証を内部テスト専用で追加**する。

#### Cognito 側の設定

```
Cognito User Pool
├── Google Identity Provider（本番ユーザー向け）
├── Apple Identity Provider（本番ユーザー向け）
└── Email/Password 認証（テスト専用・制限付き）
    ├── サインアップ: 無効化（管理者のみ作成可能）
    ├── テストユーザー: e2e-test@ganbari-quest.internal
    └── App Client: e2e-test-client（テスト専用、本番UIから不可視）
```

#### テストユーザーの仕様

| 項目 | 値 |
|------|-----|
| Email | `e2e-test@ganbari-quest.internal` |
| パスワード | GitHub Secrets に格納（`E2E_TEST_PASSWORD`） |
| ロール | `parent`（管理画面テスト用） + `child`（子供画面テスト用） |
| テナント | E2E テスト専用テナント（本番データと完全分離） |
| ライセンス | テスト専用ライセンスキー（無期限） |

#### セキュリティ対策

| 対策 | 内容 |
|------|------|
| **サインアップ無効** | Email/Password でのセルフサインアップを禁止。テストユーザーは管理者が手動作成 |
| **専用 App Client** | テスト用 App Client ID は本番 UI に露出しない。CI/CD Secrets 経由のみ |
| **IP 制限（推奨）** | Cognito の高度なセキュリティ機能で、Email/Password 認証を GitHub Actions IP レンジに制限 |
| **レート制限** | テストユーザーのログイン試行を1分5回に制限 |
| **専用テナント** | テストデータは専用テナントに隔離。本番ユーザーのデータに影響しない |
| **監査ログ** | テストユーザーの全アクティビティを CloudWatch Logs に記録 |

### Phase 2: 本番 Smoke Test の設計

#### テストスコープ

**非破壊的な読み取り系テストを中心に設計。**本番データの変更・削除は行わない。

```
本番 Smoke Test
├── ヘルスチェック
│   ├── GET /api/health → 200
│   ├── GET / → 200（子供選択画面）
│   └── レスポンスタイム < 3秒
├── 子供画面（認証不要）
│   ├── /switch → 子供選択画面が表示
│   ├── /kinder/home or /baby/home → ホーム画面が表示
│   ├── BottomNav → 各タブが遷移可能
│   └── /kinder/history → 記録一覧が表示
├── 親管理画面（テストユーザーでログイン）
│   ├── /login → ログイン画面が表示
│   ├── OAuth バイパス → /admin にアクセス可能
│   ├── /admin/activities → 活動一覧が表示
│   └── /admin/children → 子供一覧が表示
├── API 基本動作
│   ├── GET /api/v1/children → JSON レスポンス
│   ├── GET /api/v1/activities → JSON レスポンス
│   └── GET /api/v1/login-bonus/:childId → JSON レスポンス
└── セキュリティ
    ├── 未認証で /admin → /login へリダイレクト
    └── 不正トークンで API → 401
```

#### テストデータ管理

テスト専用テナントに以下のシードデータを事前投入:

| データ | 内容 |
|-------|------|
| テスト子供 | 「テストちゃん」（kinder, 5歳） |
| テスト活動 | 5カテゴリ × 各1活動 |
| テストチェックリスト | 1テンプレート |

**テストによるデータ書き込みは原則禁止。**活動記録のテストが必要な場合は、テスト後にクリーンアップを実施する。

### Phase 3: GitHub Actions ワークフロー

#### ci.yml への追加

```yaml
  e2e-production:
    runs-on: ubuntu-latest
    needs: docker-build  # デプロイ後に実行
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    environment: production
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm

      - run: npm ci

      - name: Install Playwright browsers
        run: npx playwright install --with-deps chromium

      - name: Run Production Smoke Test
        run: npx playwright test --config playwright.production.config.ts
        env:
          E2E_BASE_URL: https://ganbari-quest.com
          E2E_TEST_EMAIL: ${{ secrets.E2E_TEST_EMAIL }}
          E2E_TEST_PASSWORD: ${{ secrets.E2E_TEST_PASSWORD }}
          E2E_COGNITO_CLIENT_ID: ${{ secrets.E2E_COGNITO_CLIENT_ID }}

      - name: Upload Production E2E results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: e2e-production-results
          path: test-results/
          retention-days: 14

      - name: Notify on failure
        if: failure()
        # SNS or Slack 通知（#0102 監視基盤と連携）
        run: echo "::error::Production Smoke Test FAILED"
```

#### 手動実行の対応

```yaml
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  workflow_dispatch:  # 手動トリガー追加
    inputs:
      run_production_e2e:
        description: '本番E2Eテストを実行'
        type: boolean
        default: false
```

### Phase 4: Playwright 本番用設定ファイル

```typescript
// playwright.production.config.ts
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: 'tests/e2e/production',
  fullyParallel: false,        // 本番は直列実行（負荷軽減）
  retries: 1,                  // 1回のみリトライ
  workers: 1,                  // 1ワーカーのみ
  timeout: 30_000,
  reporter: [['list'], ['json', { outputFile: 'test-results/production.json' }]],
  use: {
    baseURL: process.env.E2E_BASE_URL || 'https://ganbari-quest.com',
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
  },
  projects: [
    {
      name: 'production-smoke',
      use: {
        ...devices['Desktop Chrome'],
        viewport: { width: 1280, height: 800 },
      },
    },
  ],
  // webServer は不要（本番環境を直接テスト）
});
```

## 注意事項

- **テストは非破壊的であること**: 本番データの作成・更新・削除は原則禁止
- **テスト専用テナント**: テストデータは専用テナントに隔離し、本番ユーザーのデータに影響しないこと
- **E-mail/Password 認証のセキュリティ**: テスト専用に限定し、セルフサインアップ不可、App Client 非公開、IP 制限を推奨
- **負荷**: 本番 Smoke Test は worker=1、直列実行でサーバーへの負荷を最小限に
- **実行頻度**: main push ごとに自動実行 + 手動 dispatch 可能。将来的にはスケジュール実行（日次）も検討
- **#0066 が未実装の間**: 現行の PIN 認証環境（NUC Docker）で先行して本番 Smoke Test のフレームワークだけ構築し、OAuth 移行後にテスト認証を切り替えることも可能
