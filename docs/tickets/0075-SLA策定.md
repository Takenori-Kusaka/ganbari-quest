### 概要

サービスの可用性・パフォーマンス目標を規定するSLA（サービスレベルアグリーメント）を策定する。

### 背景・動機

個人開発のSaaSとして現実的かつ誠実なSLA設定が必要。過度な保証は避けつつ、ユーザーの信頼を確保する最低限のコミットメントを示す。

### 対応方針

#### SLA項目（個人開発向けの現実的設定）

| 項目 | 目標値 | 備考 |
|------|--------|------|
| サービス可用性 | 99.5%/月 | 月間約3.6時間のダウンタイム許容 |
| 計画メンテナンス | 月1回、JST 2:00-5:00 | 48時間前に通知 |
| データバックアップ | 日次 | DynamoDB PITR (Point-in-Time Recovery) |
| バックアップ保持期間 | 35日間 | DynamoDB PITR標準 |
| データ暗号化 | TLS 1.2+（通信時）、AES-256（保存時） | AWS標準 |
| サポート応答時間 | 48時間以内 | Discord/メール |
| 障害復旧目標（RTO） | 4時間 | |
| 障害復旧地点（RPO） | 1時間 | |

**注意**: 99.9%は個人開発では現実的でないため、99.5%に設定。
AWSのサービス自体は99.99%の可用性だが、デプロイ時のダウンタイムやバグ対応を考慮。

#### 稼働率計算式

```
稼働率 = (月間総時間 - 計画外ダウンタイム) / 月間総時間 × 100

※ 計画メンテナンスは稼働率計算から除外
※ 部分的な機能障害は0.5ダウンタイムとして計算
```
