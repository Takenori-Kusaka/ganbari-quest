### 概要

AI ビジョン機能を持つ Browser Use CLI 2.0 を活用し、Playwright では対応できない**視覚的なデザイン品質検証**（レイアウト崩れ、フォーマット不備、デザイントークン逸脱、子供向け UI の可読性）を自動化できるか調査・検証する。

### 背景・動機

#### 現状のテスト体制の限界

現行の E2E テスト（Playwright）は機能テスト（ボタン押下→画面遷移→データ反映）に特化しており、**視覚的な品質**は検証していない:

- `screenshot: 'only-on-failure'` — 失敗時のデバッグ用スクリーンショットのみ
- `toMatchSnapshot` / `toHaveScreenshot` — 未使用（ピクセル比較ベースの VRT は導入していない）
- デザイン崩れ、フォント不一致、余白ズレ、色の逸脱などはすべて**目視確認**に依存

#### Playwright VRT（Visual Regression Testing）の課題

Playwright にも `toHaveScreenshot()` によるピクセル比較ベースの VRT 機能があるが、以下の点で本プロジェクトには課題がある:

| 課題 | 詳細 |
|------|------|
| ピクセル完全一致の脆さ | フォントレンダリング、アンチエイリアス、OS 差でフォールスポジティブが頻発 |
| しきい値調整の手間 | `maxDiffPixelRatio` の適切な値をページごとに調整し続ける必要がある |
| 意味的な判断ができない | 「ボタンが見づらい」「文字が小さすぎる」といった UX 品質は判定不可 |
| ベースライン管理コスト | スナップショット画像が大量に Git リポジトリに蓄積する |
| 子供向け特有の基準 | 「幼児が見て分かるか」「タップしやすい大きさか」は機械的比較では判定不可 |

#### Browser Use CLI 2.0 の可能性

Browser Use CLI 2.0 は LLM ビジョン機能を内蔵しており、スクリーンショットを AI が**意味的に評価**できる:

- 「このページのレイアウトは崩れていないか？」
- 「ボタンのコントラスト比は十分か？」
- 「子供向け画面としてフォントサイズは適切か？」
- 「タブレット表示とモバイル表示で情報が欠落していないか？」

こうした**人間の主観的判断を必要とするデザイン検証**をAIに委譲することが、本チケットの狙いとなる。

### 技術調査メモ

#### Browser Use CLI 2.0 の概要

| 項目 | 内容 |
|------|------|

#### がんばりクエストでの活用シナリオ

**シナリオ 1: デザインリグレッション検出**

UI 変更後に主要画面のスクリーンショットを取得し、LLM に「このページにデザイン崩れはないか？」と評価させる。Playwright VRT のようなピクセル比較ではなく、意味的な評価。

```bash
# 例: 子供のホーム画面を検証
browser-use open http://localhost:5173/child/home
browser-use screenshot child-home.png
# → LLM に画像を渡して評価（Agent モード or 外部スクリプト）
```

**シナリオ 2: デザインガイドライン準拠チェック**

[幼児・子供向けデザインガイドライン](../reference/幼児%20子供向けデザインガイドライン.md) に基づくチェックを自動化:

| チェック項目 | 判定基準（LLM に指示） |
|-------------|----------------------|
| フォントサイズ | 子供向け画面で 16px 未満のテキストがないか |
| タップターゲット | ボタン・リンクが 44px × 44px 以上か |
| コントラスト比 | WCAG AA（4.5:1）以上を満たしているか |
| 配色 | デザイントークンの色と著しく異なる色が使われていないか |
| 情報量 | 1 画面あたりの情報量が子供に過剰でないか |
| アイコン視認性 | アイコンが直感的に理解できるデザインか |

**シナリオ 3: レスポンシブ検証**

タブレット（1280×800）とモバイル（Pixel 7: 412×915）で同じ画面を撮影し、LLM に情報の欠落や操作性の問題を判定させる。

**シナリオ 4: ダークモード / テーマ検証（将来）**

テーマ切替時のコントラスト不足や、ダークモードでの可読性低下を LLM で検出。

#### 既存テストとの住み分け

| テスト種別 | ツール | 対象 | 判定方法 |
|-----------|--------|------|---------|
| 機能テスト | Playwright | 画面遷移・データ操作・フロー | DOM 状態のアサーション |
| ユニットテスト | Vitest | ドメインロジック・コンポーネント | 関数の入出力・レンダリング結果 |
| **視覚デザインテスト** | **Browser Use CLI 2.0** | **レイアウト・配色・可読性** | **LLM ビジョンによる意味的評価** |

※ Browser Use CLI は Playwright を**置き換える**のではなく、Playwright では検出できない視覚品質の層を**補完**する位置づけ。

#### コスト・リスク分析

| 項目 | 評価 |
|------|------|
| LLM API コスト | 1画面あたり約 $0.01〜0.05（モデル・画像サイズ依存）。50画面 × 2デバイス = $1〜5/回 |
| 実行時間 | 画面キャプチャ + LLM 判定で 1画面あたり 5〜15秒。全画面で 5〜10分程度 |
| フォールスポジティブ | LLM の主観判断のため、再現性にばらつきがある。プロンプト品質に大きく依存 |
| 環境依存 | Python 3.11+ が必要。既存の Node.js ベースの CI に Python ランタイム追加が必要 |
| セキュリティ | スクリーンショット画像が外部 LLM API に送信される。個人情報を含むテストデータに注意 |
| メンテナンス | プロンプトのチューニングが継続的に必要。LLM モデル更新で判定基準が変動する可能性 |

#### 導入判断基準

以下の条件を **Phase 1 の PoC で検証**し、すべて満たす場合に Phase 2（本格導入）に進む:

| 条件 | 基準 |
|------|------|
| 検出精度 | 意図的に崩したレイアウトを 80% 以上検出できること |
| フォールスポジティブ率 | 正常な画面に対する誤検出が 20% 以下であること |
| 実行時間 | 全画面チェックが 15 分以内に完了すること |
| コスト | 1 回の全画面チェックが $10 以下であること |
| 環境構築 | Windows（NUC）環境で安定動作すること |
| 再現性 | 同一画面に対する 3 回連続実行で判定結果が一貫すること |
