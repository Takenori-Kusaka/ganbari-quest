### 概要

現行のDocker/NUC構成から、AWSフルサーバレスアーキテクチャに移行する。月額無リクエスト時1,000円未満を目標とする。

### 背景・動機

SaaS展開には自動スケーリング・高可用性のインフラが必要。個人開発の費用制約から、フルサーバレスで固定費を最小化する。WAFは初期導入しない（費用対効果が悪い）。

### 対応方針

#### アーキテクチャ構成

```
[ユーザー]
    │
    ▼
[CloudFront] ─── [S3: 静的ファイル (SvelteKit client)]
    │
    ▼
[API Gateway (HTTP API)]
    │
    ├── [Lambda Authorizer (Cognito JWT検証)]
    │
    ▼
[Lambda (SvelteKit SSR + API)]
    │
    ├── [DynamoDB (メインDB)]
    ├── [S3 (アバター画像・バックアップ)]
    └── [Cognito (認証)]

[Route 53] → CloudFront（独自ドメイン + ACM証明書）
[CloudWatch] → Lambda/API Gateway ログ（最小保持）
```

#### ドメイン取得・DNS設定

**ドメイン取得（外部レジストラ推奨）:**

Route 53 でもドメイン購入可能だが、個人開発ではコスト面で外部レジストラが有利。

| レジストラ | .com 年額 | .jp 年額 | 備考 |
|-----------|----------|---------|------|
| お名前.com | ~1,400円 | ~3,100円 | 初年度割引あり。自動更新に注意（解除推奨→手動更新） |
| ムームードメイン | ~1,700円 | ~3,300円 | GMO系、UIがシンプル |
| Google Domains → Squarespace | ~1,800円 | — | 移管済み、.jp 非対応 |
| Route 53 | ~$13(≈2,000円) | — | AWS統合が楽だが割高 |

**推奨:** `お名前.com` で `.com` ドメイン取得（年額 ~1,400円）
- 候補: `ganbari-quest.com`, `ganbariquest.com`, `ganbari.app` 等
- Whois代行: 必ず有効化（個人情報保護）
- 自動更新: 意図しない課金防止のため初期は無効→手動更新

**Route 53 への紐付け手順:**

```
1. Route 53 でホストゾーン作成
   → ganbari-quest.com のホストゾーンが生成される
   → NS レコード4つが発行される

2. お名前.com 側でネームサーバー変更
   → デフォルトNSを削除
   → Route 53 の NS レコード4つを設定
   ※ 反映まで最大48時間（通常は数時間）

3. Route 53 に必要なレコードを追加
   → A レコード (Alias): CloudFront ディストリビューション
   → AAAA レコード (Alias): CloudFront（IPv6）
   → CNAME: www → apex ドメインへリダイレクト

4. ACM (Certificate Manager) で SSL証明書発行
   → リージョン: us-east-1（CloudFront用は必須）
   → DNS検証: Route 53 に CNAME 自動追加
   → 数分で検証完了・証明書発行

5. CloudFront に証明書とドメインを紐付け
   → Alternate Domain Name: ganbari-quest.com, www.ganbari-quest.com
   → SSL Certificate: ACM で発行した証明書を選択
```

**Cognito との連携:**
- Cognito Hosted UI のカスタムドメイン: `auth.ganbari-quest.com`
  → Route 53 に CNAME 追加、ACM 証明書が別途必要
- OAuth コールバックURL: `https://ganbari-quest.com/auth/callback`

**コスト:**
- Route 53 ホストゾーン: $0.50/月（≈75円）← **これが唯一の固定費**
- Route 53 クエリ: $0.40/100万クエリ（実質無料レベル）
- ACM 証明書: 無料（AWS発行の場合）
- ドメイン更新: 年1回 ~1,400円（お名前.com .com の場合）

#### SvelteKit → Lambda 変換

**選択肢:**
1. **svelte-adapter-aws** (コミュニティ製): SvelteKit をLambda@Edge + S3に変換
2. **adapter-node + Lambda Web Adapter**: Node.jsサーバーをLambda互換にラップ
3. **adapter-node + API Gateway + Lambda**: 標準的なNode.js Lambda

**推奨**: 選択肢2（Lambda Web Adapter）
- AWS公式のWeb Adapterを使用
- SvelteKitのadapter-nodeをそのまま利用可能
- SSR対応、ストリーミング対応
- Cold start: ~500ms（SnapStart非対応だが許容範囲）

```dockerfile
FROM public.ecr.aws/awsguru/aws-lambda-web-adapter:latest AS adapter
FROM node:22-alpine
COPY --from=adapter /lambda-adapter /opt/extensions/
COPY build/ /app/
WORKDIR /app
CMD ["node", "index.js"]
```

#### SQLite → DynamoDB 移行設計

**シングルテーブル設計（推奨）:**

| PK | SK | データ | 用途 |
|----|-----|-------|------|
| TENANT#t1 | PROFILE | {name, plan, ...} | テナント情報 |
| TENANT#t1#USER#u1 | PROFILE | {name, age, ...} | 家族メンバー |
| TENANT#t1#USER#u1 | ACT#2026-03-06#a1 | {points, ...} | 活動記録 |
| TENANT#t1 | ACHIEVEMENT#ach1 | {unlocked, ...} | 実績 |

**GSI:**
- GSI1: `SK` + `PK`（逆引き用）
- GSI2: `type` + `createdAt`（タイプ別時系列クエリ）

#### デプロイパイプライン（GitHub Actions + CDK）

GitHub から AWS へのデプロイは **GitHub OIDC** でアクセスする。長期アクセスキーを発行せず、セキュアな短命トークンで認証する。

**事前準備（AWS側）:**

```
1. IAM > Identity providers > Add provider
   → Provider type: OpenID Connect
   → Provider URL: https://token.actions.githubusercontent.com
   → Audience: sts.amazonaws.com

2. IAM Role 作成（CDKデプロイ用）
   → Trust policy:
   {
     "Effect": "Allow",
     "Principal": {
       "Federated": "arn:aws:iam::<ACCOUNT_ID>:oidc-provider/token.actions.githubusercontent.com"
     },
     "Action": "sts:AssumeRoleWithWebIdentity",
     "Condition": {
       "StringEquals": {
         "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
       },
       "StringLike": {
         "token.actions.githubusercontent.com:sub": "repo:<OWNER>/<REPO>:ref:refs/heads/main"
       }
     }
   }
   → Permissions: CDKデプロイに必要な権限
     - CloudFormation, Lambda, DynamoDB, S3, CloudFront, Route53, ACM, IAM (PassRole), API Gateway, Cognito

3. GitHub リポジトリにシークレット登録
   → Settings > Secrets and variables > Actions > New repository secret
   → AWS_OIDC_IAMROLE_ARN = arn:aws:iam::<ACCOUNT_ID>:role/ganbari-quest-deploy
```

**GitHub Actions ワークフロー案:**

```yaml
# .github/workflows/deploy.yml
name: Deploy to AWS
on:
  push:
    branches: [main]

permissions:
  id-token: write    # OIDCトークン取得に必須
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'

      - name: Configure AWS credentials (OIDC)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_OIDC_IAMROLE_ARN }}
          aws-region: ap-northeast-1

      - name: Install dependencies
        run: npm ci

      - name: Build SvelteKit
        run: npm run build

      - name: CDK Deploy
        working-directory: infra/
        run: |
          npm ci
          npx cdk deploy --all --require-approval never
```

**❗ GitHub Actions marketplace モジュールのバージョン管理:**

デプロイパイプライン構築時、以下のモジュールは必ず最新メジャーバージョンを確認して使用すること。

| モジュール | 用途 | 現時点最新 | 確認先 |
|---------|------|----------|--------|
| `actions/checkout` | リポジトリチェックアウト | v4 | [GitHub](https://github.com/actions/checkout/releases) |
| `actions/setup-node` | Node.js セットアップ | v4 | [GitHub](https://github.com/actions/setup-node/releases) |
| `aws-actions/configure-aws-credentials` | AWS OIDC 認証 | v4 | [GitHub](https://github.com/aws-actions/configure-aws-credentials/releases) |

※ メジャーバージョンタグ（`@v4` 等）を使用し、マイナー更新は自動で受ける。実装時に Dependabot の GitHub Actions バージョン更新も有効化を検討。
