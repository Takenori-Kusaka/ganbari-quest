### 概要

現行の `.github/workflows/deploy.yml` を本番SaaS運用に耐えうるレベルに引き上げる。開発メンバーが作成した現行版は CDK + OIDC + ECR の基本フローは整っているが、**バージョニング、デプロイ後ヘルスチェック、ロールバック、リリースノート、通知** などSaaSの本番デプロイとして一般的に求められる要素が不足している。

### 対応方針

#### P1: セマンティックバージョニング導入

**方式案: Git Tag ベースのバージョン管理**

```
リリースフロー:
  1. 開発者が git tag v1.2.3 を作成・push
  2. deploy.yml の on.push.tags トリガーでデプロイ開始
  3. Docker イメージに v1.2.3 + github.sha の両タグ付与
  4. Lambda に v1.2.3 のエイリアス作成
```

```yaml
# deploy.yml 修正案（トリガー部分）
on:
  push:
    tags:
      - 'v*.*.*'       # タグプッシュでデプロイ
  workflow_dispatch:     # 手動デプロイも残す
```

```yaml
# Docker タグ付け修正案
tags: |
  ${{ steps.ecr-login.outputs.registry }}/${{ env.ECR_REPOSITORY }}:${{ github.ref_name }}
  ${{ steps.ecr-login.outputs.registry }}/${{ env.ECR_REPOSITORY }}:${{ github.sha }}
  ${{ steps.ecr-login.outputs.registry }}/${{ env.ECR_REPOSITORY }}:latest
```

**検討事項:**
- main への push トリガーを残すか、tag のみにするか
  - **推奨:** tag のみ。main push は ci.yml でテストだけ走らせ、リリースは明示的な tag で制御
  - 理由: チケットの細かい修正コミットごとに本番デプロイが走るのは危険
- package.json の `version` と Git tag を一致させるルール or `npm version` コマンドの活用

#### P2: リリースノート自動生成

```yaml
# deploy ジョブの最後に追加
- name: Create GitHub Release
  uses: softprops/action-gh-release@v2
  with:
    tag_name: ${{ github.ref_name }}
    generate_release_notes: true   # PR タイトル・コミットから自動生成
    body: |
      ## デプロイ情報
      - **バージョン**: ${{ github.ref_name }}
      - **コミット**: ${{ github.sha }}
      - **Docker イメージ**: ${{ steps.ecr-login.outputs.registry }}/${{ env.ECR_REPOSITORY }}:${{ github.ref_name }}
      - **デプロイ日時**: ${{ github.event.head_commit.timestamp }}
```

**補足:**
- GitHub の `generate_release_notes: true` は直前のリリースからの PR・コミットを自動集約する
- `.github/release.yml` でカテゴリ分類（feat/fix/docs/refactor）も可能
- CHANGELOG.md の自動更新は `conventional-changelog` 等で別途検討

#### P3: デプロイ後ヘルスチェック

```yaml
# Lambda 更新後に追加
- name: Smoke test (health check)
  run: |
    # CloudFront 経由でヘルスチェック
    SITE_URL="${{ steps.cdk-outputs.outputs.site-url }}"  # CDK outputs から取得
    for i in 1 2 3 4 5; do
      STATUS=$(curl -s -o /dev/null -w "%{http_code}" "$SITE_URL/api/health")
      if [ "$STATUS" = "200" ]; then
        echo "✅ Health check passed (attempt $i)"
        exit 0
      fi
      echo "⏳ Attempt $i: status=$STATUS, retrying in 10s..."
      sleep 10
    done
    echo "❌ Health check failed after 5 attempts"
    exit 1
```

#### P4: ロールバック

```yaml
# ヘルスチェック失敗時のロールバック
- name: Rollback on failure
  if: failure()
  run: |
    echo "⚠️ Deployment failed, rolling back to previous image..."
    # 前バージョンの latest を再デプロイ
    PREVIOUS_IMAGE=$(aws ecr describe-images \
      --repository-name $ECR_REPOSITORY \
      --query 'sort_by(imageDetails,&imagePushedAt)[-2].imageDigest' \
      --output text)
    aws lambda update-function-code \
      --function-name $LAMBDA_FUNCTION \
      --image-uri $ECR_REGISTRY/$ECR_REPOSITORY@$PREVIOUS_IMAGE \
      --region $AWS_REGION
    aws lambda wait function-updated \
      --function-name $LAMBDA_FUNCTION \
      --region $AWS_REGION
    echo "✅ Rollback complete"
```

#### P6: デプロイ通知

```yaml
# 成功・失敗の通知（Discord or Slack）
- name: Notify deployment result
  if: always()
  uses: sarisia/actions-status-discord@v1  # Discord の場合
  with:
    webhook: ${{ secrets.DISCORD_WEBHOOK }}
    status: ${{ job.status }}
    title: "Deploy ${{ github.ref_name }}"
    description: |
      Commit: ${{ github.sha }}
      Author: ${{ github.actor }}
```

#### P8: Actions モジュール最新化 + Dependabot

```yaml
# .github/dependabot.yml（新規作成）
version: 2
updates:
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
    labels:
      - "dependencies"
      - "github-actions"
```

これにより `docker/build-push-action@v5` → `v6` 等の更新PRが自動生成される。
