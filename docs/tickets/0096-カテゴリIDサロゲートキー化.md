# #0096 カテゴリの日本語テキストキーをサロゲートID化

## ステータス: Open
## 優先度: 中
## 種別: リファクタリング（データモデル変更）

---

## 概要

現在、`activities`・`statuses`・`status_history`・`market_benchmarks` などのテーブルで、カテゴリをUI表示名の日本語テキスト（`'うんどう'`、`'べんきょう'` 等）でそのまま格納・インデックス・ユニーク制約に使用している。
これを `categories` マスタテーブルを新設し、整数サロゲートキー（`categoryId`）で参照する構造に変更する。

## 背景・課題

### 現状の問題点

1. **表示名＝DBキー**: UI表示名をインデックスやユニーク制約のキーに使用しており、表示名変更時にデータ整合性が壊れる
2. **多言語対応不可**: 将来的なSaaS展開（#0010）に伴う国際化で、キー値が日本語テキストでは対応不能
3. **パフォーマンス**: テキスト比較よりも整数比較の方がインデックス効率が良い
4. **表記揺れリスク**: コード中にハードコードされた日本語文字列が多く、typoや全角半角の不一致でバグが入る可能性がある
5. **定数分散**: `CATEGORIES` 配列がドメイン層にあるが、DBのユニーク制約とは独立しており、不一致が起きうる

## 変更方針

### Option A: categories テーブル新設＋全面移行（推奨）

#### 1. `categories` マスタテーブル新設

```sql
CREATE TABLE categories (
  id    INTEGER PRIMARY KEY,
  code  TEXT NOT NULL UNIQUE,  -- 'undou', 'benkyou', 'seikatsu', 'kouryuu', 'souzou'
  name  TEXT NOT NULL,         -- 'うんどう', 'べんきょう', 'せいかつ', 'こうりゅう', 'そうぞう'
  icon  TEXT,                  -- カテゴリアイコン絵文字
  color TEXT                   -- テーマカラーコード
);
```

初期データ:

| id | code | name | icon | color |
|----|------|------|------|-------|
| 1 | undou | うんどう | 🏃 | #FF6B6B |
| 2 | benkyou | べんきょう | 📚 | #4ECDC4 |
| 3 | seikatsu | せいかつ | 🏠 | #FFE66D |
| 4 | kouryuu | こうりゅう | 🤝 | #A8E6CF |
| 5 | souzou | そうぞう | 🎨 | #DDA0DD |

#### 2. 既存テーブルの `category` → `categoryId` 変更

| テーブル | 変更内容 |
|---------|---------|
| `activities` | `category TEXT` → `categoryId INTEGER REFERENCES categories(id)` |
| `statuses` | 同上 + ユニーク制約を `(childId, categoryId)` に変更 |
| `status_history` | 同上 + インデックスを `categoryId` ベースに変更 |
| `market_benchmarks` | 同上 + ユニーク制約を `(ageGroup, categoryId)` に変更 |
| `achievements` | `category TEXT NULL` → `categoryId INTEGER NULL REFERENCES categories(id)` |

#### 3. ドメイン層の変更

```typescript
// src/lib/domain/validation/activity.ts

// 既存（後方互換のため残す）
export const CATEGORIES = ['うんどう', 'べんきょう', 'せいかつ', 'こうりゅう', 'そうぞう'] as const;

// 新規追加
export const CATEGORY_CODES = ['undou', 'benkyou', 'seikatsu', 'kouryuu', 'souzou'] as const;
export type CategoryCode = (typeof CATEGORY_CODES)[number];

export const CATEGORY_DEFS = [
  { id: 1, code: 'undou',    name: 'うんどう',    icon: '🏃', color: '#FF6B6B' },
  { id: 2, code: 'benkyou',  name: 'べんきょう',  icon: '📚', color: '#4ECDC4' },
  { id: 3, code: 'seikatsu', name: 'せいかつ',    icon: '🏠', color: '#FFE66D' },
  { id: 4, code: 'kouryuu',  name: 'こうりゅう',  icon: '🤝', color: '#A8E6CF' },
  { id: 5, code: 'souzou',   name: 'そうぞう',    icon: '🎨', color: '#DDA0DD' },
] as const;

// ユーティリティ
export function getCategoryById(id: number) { ... }
export function getCategoryByCode(code: CategoryCode) { ... }
export function getCategoryByName(name: string) { ... }
```

#### 4. リポジトリ・サービス層

- 全関数の `category: string` パラメータを `categoryId: number` に変更
- `CATEGORIES` ループを `CATEGORY_DEFS` ベースに変更
- `groupBy(activities.category)` → `groupBy(activities.categoryId)` に変更

#### 5. API レスポンス

- カテゴリ情報は `categoryId` + `categoryCode` + `categoryName` を返す
- 旧APIとの互換性のため、移行期間は `category`（テキスト名）も併用可

#### 6. フロントエンド

- `categoryId` / `categoryCode` ベースでアイコン・色を解決
- ハードコードされた日本語カテゴリ名を `CATEGORY_DEFS` 参照に置換

### Option B: categories テーブルなし・code ベース移行（軽量版）

`categories` テーブルは作らず、既存の `category TEXT` を日本語名からローマ字コード（`'undou'` 等）に変更するだけの案。
→ テーブル増加は避けられるが、多言語対応やマスタ管理のメリットは得られない。

## マイグレーション手順

1. `categories` テーブル作成 + 初期データ投入
2. 各テーブルに `category_id INTEGER` カラムを `NULL許容` で追加
3. 既存の `category` テキスト → `category_id` のデータ移行
4. `category_id` を `NOT NULL` に変更
5. 旧 `category` テキストカラムを削除
6. インデックス・ユニーク制約を再作成

※ SQLiteは `ALTER TABLE DROP COLUMN` に制限があるため、テーブル再作成が必要になる場合がある

```bash
# マイグレーションスクリプト
node scripts/migrate-category-ids.cjs

# バックアップは必ず事前に取得
node scripts/backup-db.cjs
```
