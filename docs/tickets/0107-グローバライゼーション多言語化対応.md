# #0107 グローバライゼーション・多言語化対応（i18n）

## Status: Done
## 優先度: 低
## 種別: 機能追加（将来検討）

---

## 概要

がんばりクエストを多言語対応（i18n）し、日本語以外の言語でも利用できるようにする。SaaS展開において海外ユーザーや多言語家庭への対応を見据えた中長期施策。

## 現状分析

### i18n基盤: 未導入

- i18n ライブラリなし（i18next, paraglide, typesafe-i18n 等いずれも未使用）
- `package.json` に国際化関連の依存なし
- SaaS展開ロードマップ（#10）に多言語化の記載なし

### ハードコード日本語の分布

| レイヤー | 状況 | 規模感 |
|---------|------|-------|
| UIコンポーネント（.svelte） | 全面的に日本語ハードコード | 大（全画面対象） |
| フォームバリデーション（+page.server.ts） | 日本語エラーメッセージ | 中（10〜20箇所） |
| APIエンドポイント（+server.ts） | 英語エラーコードで実装済み | 小（ほぼ対応不要） |
| ドメインロジック（domain/） | カテゴリ名等が日本語定数 | 中 |
| DB マスタデータ | 活動名・称号名が日本語 | 大（データ層の設計要） |

### 子供画面特有の課題

- ひらがな表記（nameKana）は日本語固有の概念 → 他言語では不要
- 年齢別の漢字/ひらがな切り替え（`getActivityDisplayName()`）も日本語固有
- 他言語では年齢による表示切り替え自体が不要になるか、別のロジックが必要

## 検討すべき設計方針

### 1. i18n ライブラリの選定

| 候補 | 特徴 | SvelteKit対応 |
|------|------|--------------|
| **Paraglide.js (Inlang)** | コンパイル時翻訳、型安全、ゼロランタイム | SvelteKit 公式推奨 |
| **typesafe-i18n** | 型安全、軽量 | SvelteKit対応 |
| **svelte-i18n** | 実績あり、Svelte向け | SvelteKit対応 |
| **i18next + sveltekit-i18n** | エコシステム広い | アダプタ経由 |

SvelteKit との親和性から **Paraglide.js** が有力候補。

### 2. 翻訳対象の分類

#### Phase A: UI テキスト（優先）

```
// Before
<p>こどもをえらんでね</p>

// After（Paraglide.js の例）
<p>{m.select_child()}</p>
```

- ボタンラベル、見出し、説明文、プレースホルダー
- エラーメッセージ、通知テキスト
- ナビゲーション項目

#### Phase B: マスタデータ

- カテゴリ名（うんどう、べんきょう、せいかつ etc.）
- 称号名・実績名
- 活動名のデフォルトテンプレート

→ DB に locale カラム追加 or 翻訳テーブルの設計が必要

#### Phase C: ユーザー生成データ

- 活動名（ユーザーが自由入力）
- 子供のニックネーム

→ 翻訳対象外（ユーザー自身が各言語で入力）

### 3. ロケール検出と切り替え

- `Accept-Language` ヘッダーによる自動検出
- ユーザー設定での明示的な言語選択（親管理画面）
- URL パス or Cookie でロケールを保持

### 4. 対応言語候補（初期）

| 言語 | コード | 優先度 | 理由 |
|------|-------|-------|------|
| 日本語 | ja | デフォルト | 現行言語 |
| 英語 | en | 高 | グローバル標準 |
| 中国語（簡体字） | zh-CN | 中 | 市場規模 |
| 韓国語 | ko | 中 | 文化的類似性 |

## 実装ステップ（概算）

### Step 1: 基盤構築
- i18n ライブラリ導入・設定
- メッセージカタログのディレクトリ構成確定
- 既存日本語テキストの抽出・カタログ化

### Step 2: UI 層の国際化
- 全 `.svelte` ファイルのハードコード日本語を翻訳キーに置換
- フォームバリデーションメッセージの国際化
- 日付・数値フォーマットの locale 対応

### Step 3: データ層の国際化
- カテゴリ・称号などマスタデータの多言語対応設計
- `nameKana` / `nameKanji` の多言語展開方針決定
- 年齢別表示切り替えロジックの言語別抽象化

### Step 4: 英語翻訳の実装
- メッセージカタログの英語翻訳
- 子供画面の英語 UI 検証（フォントサイズ・レイアウト調整）
- E2E テストの多言語対応

## 影響範囲

- `src/routes/` — 全画面のテキスト置換
- `src/lib/domain/` — カテゴリ定義、バリデーションメッセージ
- `src/lib/server/` — サーバーサイドのエラーメッセージ
- `src/lib/ui/` — 共通UIコンポーネント
- DB スキーマ — マスタデータの多言語対応（要設計）
- テスト — 多言語テストの追加

## 注意事項

- 子供向けの「やさしい日本語」表現は、他言語でも同等の「子供向け表現」が必要
- RTL（右から左）言語への対応は初期スコープ外
- 翻訳品質の担保（機械翻訳 vs 人手翻訳）の方針決定が必要
- ライブラリ選定時は SvelteKit 2 / Svelte 5 との互換性を要確認
