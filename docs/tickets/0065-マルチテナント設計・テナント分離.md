### 概要

複数家庭（テナント）が同一インフラ上で安全にデータを分離して利用できるマルチテナントアーキテクチャを設計する。

### 背景・動機

SaaS展開では複数家庭が同一システムを利用する。テナント間のデータ漏洩は致命的な信頼喪失につながるため、設計段階から堅牢な分離を組み込む必要がある。

### 対応方針

#### テナントモデル選択

| モデル | 説明 | コスト | 分離度 |
|--------|------|--------|--------|
| サイロ（テナント別DB） | テナントごとに別DynamoDBテーブル | 高 | 最高 |
| ブリッジ（共有DB、テナントキー分離） | PKにテナントIDを含む | 低 | 高 |
| プール（完全共有） | テナントIDカラムのみ | 最低 | 低 |

**推奨: ブリッジモデル**
- DynamoDBのPKに `TENANT#{tenantId}` を含める
- コスト効率が高く、個人開発に最適
- IAMポリシーで追加の分離も可能

#### テナント分離の実装レイヤー

```
1. 認証レイヤ（Cognito）
   └ JWTにtenantIdクレーム含む

2. APIレイヤ（Lambda Authorizer）
   └ JWTからtenantId抽出、リクエストコンテキストに注入

3. サービスレイヤ
   └ 全クエリにtenantIdフィルタ強制
   └ tenantIdなしのクエリはエラー

4. データレイヤ（DynamoDB）
   └ PKに TENANT#{tenantId} プレフィックス
   └ クエリは必ずPK条件で実行
```

#### テナントライフサイクル

**プロビジョニング（新規登録）:**
1. OAuth認証でアカウント作成
2. Cognito にテナントID割り当て
3. DynamoDBにテナントメタデータ作成
4. 初期データ（デフォルト活動マスタ等）投入

**デプロビジョニング（解約）:**
1. 解約リクエスト受付
2. 30日間の猶予期間（データ保持）
3. 猶予期間後にデータ完全削除
   - DynamoDB: テナントキー配下の全アイテム削除
   - S3: アバター画像等の削除
   - Cognito: ユーザーグループ削除
4. 削除完了通知

#### プラン別のテナント分離

| プラン | 分離方式 |
|--------|---------|
| Free / ベーシック | 共有テーブル（ブリッジモデル） |
| プレミアム（専用環境） | 専用テーブル or 専用スタック（サイロモデル） |
