## 2. 監視スタック

### 2.1 AWS 環境（SaaS本番）

| ツール | 用途 | コスト |
|-------|------|--------|
| CloudWatch Logs | アプリケーションログ | 無料枠内 |
| CloudWatch Metrics | Lambda/DynamoDB メトリクス | 無料枠内 |
| CloudWatch Alarms | 閾値アラート | 無料枠内（10件） |
| CloudWatch Dashboard | メトリクス可視化 | $3/月/ダッシュボード |
| SNS | アラート通知 | 無料枠内 |
| AWS Budgets | コストアラート | 無料（最初2件） |
| Cost Anomaly Detection | 異常コスト検知 | 無料 |

### 2.3 外部監視

| ツール | 用途 |
|-------|------|
| Discord Webhook | アラート通知（開発者向け） |

## 3. メトリクス定義

### 3.1 インフラメトリクス

| メトリクス | ソース | 閾値 | アラート |
|-----------|--------|------|---------|
| Lambda 実行時間 | CloudWatch | > 10秒 (p95) | ⚠ WARN |
| Lambda エラー率 | CloudWatch | > 1% (5分間) | 🔴 ERROR |
| Lambda コールドスタート | CloudWatch | > 3秒 | ⚠ WARN |
| Lambda 同時実行数 | CloudWatch | > 50 | ⚠ WARN |
| DynamoDB 読み取りスロットル | CloudWatch | > 0 (5分間) | ⚠ WARN |
| DynamoDB 書き込みスロットル | CloudWatch | > 0 (5分間) | 🔴 ERROR |
| CloudFront エラー率 | CloudWatch | > 5% (5分間) | 🔴 ERROR |
| CloudFront キャッシュヒット率 | CloudWatch | < 80% | ⚠ WARN |

### 3.2 アプリケーションメトリクス

| メトリクス | 取得方法 | 用途 |
|-----------|---------|------|
| API レスポンスタイム | ログ (durationMs) | パフォーマンス監視 |
| API エラー率 | ログ (status >= 500) | 障害検知 |
| 認証失敗率 | ログ ([AUTH] 失敗) | セキュリティ監視 |
| アカウントロックアウト | ログ ([AUTH] ロックアウト) | 不正アクセス検知 |

### 3.3 ビジネスメトリクス（OPS ダッシュボード）

| メトリクス | エンドポイント | 用途 |
|-----------|-------------|------|
| DAU (日次アクティブユーザー) | `/ops/api/analytics` | 利用状況 |
| 活動記録数/日 | `/ops/api/analytics` | エンゲージメント |
| サインアップ数 | `/ops/api/analytics` | 成長 |
| トライアル → 有料転換率 | `/ops/api/analytics` | 収益 |
| MRR (月次経常収益) | `/ops/api/analytics` | 収益 |
| チャーン率 | `/ops/api/analytics` | リテンション |

## 4. アラート設定

### 4.1 CloudWatch Alarms（OpsStack）

| アラーム名 | 条件 | 通知先 | 優先度 |
|-----------|------|--------|--------|
| `LambdaErrors` | エラー数 > 5 (5分間) | SNS → Discord | HIGH |
| `LambdaDuration` | p95 > 10秒 (5分間) | SNS → Discord | MEDIUM |
| `DynamoDBThrottle` | スロットル > 0 (5分間) | SNS → Discord | HIGH |
| `CloudFrontErrors` | 5xx率 > 5% (5分間) | SNS → Discord | HIGH |
| `BudgetExceeded` | 月額 > $50 | SNS → メール | CRITICAL |
| `CostAnomaly` | 異常検知 | SNS → メール | HIGH |

### 4.2 Discord 通知チャンネル

| チャンネル | 用途 | 通知内容 |
|-----------|------|---------|
| #アップデート | デプロイ完了 | PR マージ → デプロイ成功 |
| #不具合 | 障害通知 | ERROR/CRITICAL ログ |

### 4.3 通知エスカレーション

| レベル | 条件 | 通知先 | アクション |
|-------|------|--------|----------|
| INFO | 正常デプロイ | Discord #アップデート | ログ記録のみ |
| WARN | 閾値超過 | Discord #不具合 | 1時間以内に確認 |
| ERROR | サービス異常 | Discord #不具合 + メール | 即時対応 |
| CRITICAL | サービス停止 | Discord @everyone + メール + SNS | 即時復旧 |

## 5. ヘルスチェック

### 5.1 エンドポイント

```
GET /api/health
```

レスポンス:
```json
{
  "status": "ok",
  "timestamp": "2026-04-09T10:00:00.000Z",
  "version": "1.0.0",
  "environment": "production",
  "database": "ok"
}
```

### 5.2 チェック項目

| チェック | 内容 | 失敗時 |
|---------|------|--------|
| Database | DB 接続 + 簡易クエリ | status: "error" |
| Environment | 必須環境変数の存在 | status: "error" |

## 8. インシデント対応

### 8.1 障害レベル定義

| レベル | 定義 | 目標復旧時間 |
|-------|------|------------|
| P1 (Critical) | サービス全停止 | 1時間以内 |
| P2 (High) | 主要機能停止 | 4時間以内 |
| P3 (Medium) | 一部機能低下 | 24時間以内 |
| P4 (Low) | 軽微な問題 | 次回リリース |

### 8.2 対応フロー

```
1. 検知（アラート受信 or ユーザー報告）
2. トリアージ（障害レベル判定）
3. 調査（CloudWatch Logs + メトリクス確認）
4. 対応（ロールバック or ホットフィックス）
5. 復旧確認（ヘルスチェック + 手動確認）
6. ポストモーテム（ADR として記録）
```
