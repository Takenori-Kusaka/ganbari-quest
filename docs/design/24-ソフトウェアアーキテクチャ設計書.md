## 2. レイヤードアーキテクチャ

### 2.1 5層構造

```
Layer 1: Presentation (src/routes/)
    ↓
Layer 2: Feature & UI (src/lib/features/, src/lib/ui/)
    ↓
Layer 3: Domain (src/lib/domain/)
    ↓
Layer 4: Service (src/lib/server/services/)
    ↓
Layer 5: Data Access (src/lib/server/db/)
```

**依存ルール**: 上位層は下位層にのみ依存する。逆方向の依存は禁止。

### 2.2 Layer 1: Presentation（`src/routes/`）

SvelteKit のファイルベースルーティングによるプレゼンテーション層。

| ルートグループ | 対象 | 認証 |
|--------------|------|------|
| `(child)/baby/` | 乳幼児（0-2歳） | なし（LAN）/ セッション（SaaS） |
| `(child)/kinder/` | 園児（3-5歳） | 同上 |
| `(child)/lower/` | 小学校低学年（6-9歳） | 同上 |
| `(child)/upper/` | 小学校高学年（10-12歳） | 同上 |
| `(child)/teen/` | 中高生（13-15歳） | 同上 |
| `(parent)/admin/` | 保護者管理画面 | PIN（LAN）/ Cognito（SaaS） |
| `demo/` | デモモード | 不要 |
| `api/v1/` | REST API | Bearer / Session |
| `ops/` | 運営管理画面 | OPS_SECRET_KEY |

**ルール**:
- `+page.svelte` にビジネスロジックを書かない
- データ取得は `+page.server.ts` の `load` 関数で行う
- コンポーネント内の直接 `fetch` 禁止
- `<style>` ブロックは原則50行以下
- インラインスタイル (`style="..."`) は動的な値のみ許容

### 2.3 Layer 2: Feature & UI（`src/lib/features/`, `src/lib/ui/`）

#### Features（`src/lib/features/`）

機能単位のコンポーネントとクライアントロジック。

| ディレクトリ | 責務 |
|------------|------|
| `features/child/` | 子供画面のコンポーネント（ActivityCard, StampCard, LevelUpOverlay 等） |
| `features/admin/` | 管理画面のコンポーネント（ChildForm, RewardForm, SettingsPanel 等） |
| `features/demo/` | デモモード専用コンポーネント |
| `features/shared/` | 子供・管理画面共通コンポーネント |

#### UI Primitives（`src/lib/ui/primitives/`）

Ark UI Svelte のラッパーコンポーネント。routes から Ark UI を直接 import してはならない。

| コンポーネント | 用途 |
|--------------|------|
| `Button.svelte` | 全てのボタン UI |
| `Card.svelte` | カード型コンテナ |
| `Alert.svelte` | アラート・通知表示 |
| `FormField.svelte` | フォーム入力要素 |
| `Dialog.svelte` | モーダルダイアログ |
| `Badge.svelte` | バッジ・ラベル |

#### UI Components（`src/lib/ui/components/`）

レイアウトとナビゲーション。

| コンポーネント | 用途 |
|--------------|------|
| `AdminLayout.svelte` | 管理画面レイアウト（デスクトップナビ） |
| `AdminMobileNav.svelte` | 管理画面モバイルナビ |
| `BottomNav.svelte` | 子供画面ボトムナビゲーション |
| `Header.svelte` | 子供画面ヘッダー |
| `ConfettiEffect.svelte` | 紙吹雪エフェクト |

#### CSS トークンアーキテクチャ（`src/lib/ui/styles/`）

3層トークン構造で UI の一貫性を保証する。

```
Layer A: Base Tokens (--color-brand-500, --color-neutral-200, ...)
    ↓ 参照
Layer B: Semantic Tokens (--color-action-primary, --color-surface-card, ...)
    ↓ 参照
Layer C: Component Usage (var(--color-action-primary))
```

**ルール**:
- routes/features 配下で hex カラー直書き禁止
- routes 配下では Semantic トークンのみ使用
- Base トークンは `app.css` の `@theme` ブロック内でのみ定義

### 2.4 Layer 3: Domain（`src/lib/domain/`）

ドメインモデル、バリデーション、ゲーミフィケーションロジック。

| モジュール | 責務 |
|----------|------|
| `labels.ts` | 用語辞書（Single Source of Truth） |
| `level-calc.ts` | レベル計算（ポイント → レベル閾値テーブル） |
| `stamp-gacha.ts` | スタンプガチャ抽選（N:55%, R:30%, SR:12%, UR:3%） |
| `streak.ts` | ストリーク（連続記録日数）計算 |
| `season-pass.ts` | シーズンパスロジック |
| `age-config.ts` | 年齢モード設定（UI テーマ・フォントサイズ・カテゴリマッピング） |
| `validation/` | Zod スキーマによる入力バリデーション |

**ルール**:
- Domain 層はサーバー/クライアント両方で使用可能（`$lib/server` に依存しない）
- 外部サービスへの依存禁止
- 純粋関数を基本とする

| **ゲーミフィケーション** | stamp-card-service | スタンプカード管理 |

- エラーは `@sveltejs/kit` の `error()` で SvelteKit のエラーハンドリングに委譲

### 2.6 Layer 5: Data Access（`src/lib/server/db/`）

Repository パターンによるデータアクセス抽象化。

```
interfaces/           ← リポジトリインタフェース定義
  ├── child-repo.interface.ts
  ├── activity-repo.interface.ts
  └── ...
sqlite/               ← SQLite (Drizzle ORM) 実装
  ├── child-repo.ts
  └── ...
dynamodb/             ← DynamoDB (AWS SDK v3) 実装
  ├── child-repo.ts
  └── ...
factory.ts            ← 環境変数で実装を切替
schema.ts             ← Drizzle テーブル定義
create-tables.ts      ← SQLite テーブル作成 SQL
```

**切替ロジック**（`factory.ts`）:

```typescript
// AUTH_MODE 環境変数で判定
if (AUTH_MODE === 'cognito') → DynamoDB 実装
else → SQLite 実装
```

**デモモード**（`src/lib/server/demo/demo-data.ts`）:
- インメモリデータで全リポジトリをモック
- DB 不要でフロントエンド動作確認可能

## 4. デザインパターン

### 4.1 Repository パターン

データアクセスをインタフェースで抽象化し、SQLite/DynamoDB を切替可能にする。

### 4.2 Factory パターン

`factory.ts` で環境に応じたリポジトリ実装を生成。

### 4.3 Facade パターン

各リポジトリファイル（`src/lib/server/db/child-repo.ts` 等）がファサードとして機能し、factory からの実装を透過的に提供。

### 4.4 Hook パターン（activity-log-service）

活動記録後のサイドエフェクト（スタンプガチャ、レベル判定、通知等）を dynamic import + try-catch で疎結合に実行。1つのフックの失敗が他に波及しない。

```typescript
// 例: activity-log-service.ts 内
try {
  const { processStampGacha } = await import('./stamp-card-service');
  await processStampGacha(tenantId, childId);
} catch { /* スタンプ処理失敗は記録フローを止めない */ }
```

### 4.5 Single Source of Truth パターン

| 対象 | SSoT |
|------|------|
| UI ラベル・用語 | `src/lib/domain/labels.ts` |
| DB スキーマ | `src/lib/server/db/schema.ts` |
| 旧 URL マッピング | `src/lib/server/routing/legacy-url-map.ts` |
| 設計書 | `docs/design/` |

## 6. 並行実装パターン

本プロジェクトは複数の並行実装ペアを持つ。詳細は `docs/design/parallel-implementations.md` を参照。

| パターン | ペア | 同期方法 |
|---------|------|---------|
| 年齢モード | baby/kinder/lower/upper/teen の5ルート | 手動同期（#567/#568 で統合予定） |
| 本番 ↔ デモ | `src/routes/(child)/` ↔ `src/routes/demo/` | 手動同期 |
| アプリ ↔ LP | `src/` ↔ `site/` | 手動同期（labels.ts → shared-labels.js） |
| Desktop ↔ Mobile ナビ | AdminLayout ↔ AdminMobileNav ↔ BottomNav | 手動同期 |
| SQLite ↔ DynamoDB | `sqlite/` ↔ `dynamodb/` | インタフェース準拠 |
