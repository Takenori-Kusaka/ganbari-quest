## 2. SLI 定義

### 2.1 可用性 (Availability)

```
SLI = 成功レスポンス数 / 全リクエスト数
```

- 成功: HTTP 2xx, 3xx レスポンス
- 失敗: HTTP 5xx レスポンス
- 除外: HTTP 4xx（クライアントエラーは可用性に含めない）
- 計測元: CloudFront アクセスログ

### 2.2 レイテンシ (Latency)

```
SLI = p50, p95, p99 のレスポンスタイム
```

- 計測区間: CloudFront 受信 → Lambda レスポンス返却
- 単位: ミリ秒
- 計測元: CloudWatch Lambda Duration メトリクス

### 2.3 エラー率 (Error Rate)

```
SLI = 5xx エラー数 / 全リクエスト数
```

- 計測元: CloudWatch Lambda Errors メトリクス

## 3. SLO 定義

### 3.1 可用性

| 指標 | 目標 | エラーバジェット（月間） |
|------|------|---------------------|
| 可用性 | 99.5% | 3.65時間/月 |

**根拠**: 個人開発 SaaS として 99.9% は過剰。99.5% は月間約3.6時間のダウンタイムを許容し、デプロイやメンテナンスに余裕を持たせる。

### 3.2 レイテンシ

| パーセンタイル | 目標 | 対象 |
|-------------|------|------|
| p50 | < 200ms | 全 API |
| p95 | < 1,000ms | 全 API |
| p99 | < 3,000ms | 全 API（コールドスタート含む） |

**根拠**: Lambda コールドスタートが 2-3 秒発生するため、p99 は 3 秒を許容。

### 3.3 エラー率

| 指標 | 目標 |
|------|------|
| 5xx エラー率 | < 0.5% |

## 4. エラーバジェット運用

### 4.1 エラーバジェットの計算

```
月間エラーバジェット = (1 - SLO) × 月間総分数
99.5% SLO → 0.5% × 43,200分 = 216分 = 3.6時間
```

### 4.2 バジェット消費時のアクション

| 消費率 | アクション |
|-------|----------|
| < 50% | 通常運用。新機能開発を優先 |
| 50-80% | 慎重にデプロイ。リスクの高い変更は延期 |
| > 80% | 機能フリーズ。信頼性改善のみ実施 |
| 100% (超過) | 全変更停止。ポストモーテム実施 |

## 5. 計測・レポート

### 5.1 ダッシュボード

CloudWatch Dashboard で以下を常時表示:
- 直近24時間の可用性
- 直近24時間のレイテンシ (p50/p95/p99)
- 直近24時間のエラー率
- 月間エラーバジェット消費率

### 5.2 レポート周期

| 周期 | 内容 |
|------|------|
| 日次 | CloudWatch アラートによる自動監視 |
| 週次 | 週次レポートに SLO 達成状況を含める |
| 月次 | エラーバジェット消費率のレビュー |

---

## 6. SLO 見直し

四半期ごとに以下を確認し、必要に応じて SLO を調整:
- 実際の可用性・レイテンシ・エラー率の推移
- ユーザー数の変化
- インフラ構成の変更
- コスト制約の変化
