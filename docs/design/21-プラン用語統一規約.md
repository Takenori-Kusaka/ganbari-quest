# 21. プラン用語統一規約（Naming Convention）

## ステータス: 確定（2026-05-29 改訂 — family → premium rename、PO 判断適用）

---

## 目的

プロダクトのプラン名称・用語を全ドキュメント・コード・Stripe・UI で統一する。ロゴ・ブランド素材作成時に名称が揺れないことを保証する。

> **2026-05-29 改訂サマリ** (#2609 解消、Issue #2588 Phase 1 補強 2 反映): 旧「ファミリープラン」を **プレミアムプラン** に rename。3 段階の運用本格度 (無料 / スタンダード / プレミアム) を素直に表す命名へ統一。詳細経緯は §10 改訂履歴参照。

## 4. 禁止用語一覧

| NG用語 | 正しい用語 | 理由 |
|--------|----------|------|
| ベーシックプラン | スタンダードプラン | 「ベーシック」は最低限の印象。「スタンダード」は標準=十分の印象 |
| ファミリープラン（プラン名として、Phase 7 完遂後） | プレミアムプラン | 2026-05-29 改訂で rename 確定。Phase 7 (#2656 系 atom rename) **完遂までは移行期表記として両用許可**、完遂後は禁止語に格上げ |
| 有料プラン（ユーザー向け） | スタンダードプラン / プレミアムプラン | 具体的なプラン名を使う |
| 月額プラン / 年額プラン | スタンダードプラン / プレミアムプラン | 支払い方法はプラン名ではない。年額自体が Phase 1 補強 2 で廃止予定 |
| ライセンス（ユーザー向け） | プラン | 「ライセンス」は技術用語。一般ユーザーには「プラン」 |
| paid（UIラベル） | standard / premium | ユーザーに見える箇所では具体的なプラン名を使う |

> **「プレミアム」の使い方 (2026-05-29 改訂)**: **プラン名として使用可能** (旧版で「ブランド修飾語として温存」としていた方針を撤回)。同時にブランドコピー (「プレミアムな子育て体験」「プレミアム機能」) としても使用可。プラン名兼用が成立する根拠は §7 名称選定の根拠参照。

> **CI gate 連動**: `scripts/check-forbidden-terms.mjs` の禁止語 list から「プレミアムプラン」を 2026-05-29 改訂で削除済 (本 PR #2609)。「ファミリープラン」を Phase 7 完遂までは検出対象に追加しない (移行期両用許可)。

## 7. 名称選定の根拠

### なぜ「スタンダード」か

- 「ベーシック」= 基本的、最低限 → ¥500/月に見合わない安い印象
- 「スタンダード」= 標準、これが基準 → 「十分な機能がある」印象
- 中間 tier として業界汎用 (Notion / Slack / Asana 等で広く採用)

### なぜ「プレミアム」へ rename したか (2026-05-29 改訂、PO 判断)

旧版では「ファミリー」を採用 (子育て・家族向け製品として自然) としていたが、`docs/design/billing-redesign/phase1-plan-naming-pricing-axis-requirements.md` Phase 1 補強 2 (#2588) の deep-research で **業界規範ミスマッチ + 訴求軸乖離** が判明:

1. **業界規範ミスマッチ**: B2C で「Family」プランは **家族構成軸 = メンバー追加・席数増加** を表す確立規範 (Apple One Family / Spotify Family / Microsoft 365 Family / Headspace Family / NYT Family が全て member-focused)。本プロダクトは flat-rate 家族メンバー数固定モデルなので、「ファミリー」命名は **メンバー追加機能の誤期待** を引き起こす
2. **訴求軸乖離**: 本プロダクトの 3 段階は **運用本格度** (無料 = 試用 / スタンダード = 家庭運用 / プレミアム = 人生記録レベルの本格運用) を表す。「ファミリー」は本格度を伝えない
3. **業界汎用性**: 「プレミアム」は Notion Plus / Spotify Premium / Slack Premium 等で「上位 tier の一般名」として確立。choice paralysis を最小化

**棄却案** (deep-research で推奨されたが PO 判断で不採用):
- 「ジャーナル」「おもいで」: 「人生記録」直接表現だが独自性高すぎ、tier 順序が直感的でない
- 「シルバー / ゴールド」: 金属表現が日本市場でやや欧米的、本プロダクト UI トーン不整合

訴求軸 (実態の「人生記録・証跡レベル」) はプラン名ではなく **機能訴求 + LP コピー** で担保する戦略 (PO 確定、#2588 FR-1)。

### なぜ旧版で「プレミアムをプラン名にしない」としていたか (撤回根拠)

旧版 §138 で「3 段階の中間にプレミアムを置くと階層矛盾」「ブランド全体修飾語として温存」と記述していたが:

- (a) 旧版は「中間 = スタンダード、上位 = ファミリー」前提だったため、上位「プレミアム」は階層的に余剰だった
- (b) 本改訂で **上位 = プレミアム** に振り直したため、階層矛盾は解消
- (c) 「ブランドコピー用に温存」も、プラン名 + ブランドコピー兼用が業界汎用 (Spotify Premium が両方で機能) のため不要

→ 旧版方針を撤回し、プレミアムをプラン名として正式採用。

## 9. SSOT 順序ルール (2026-05-29 #2609 解消で明文化)

本規約 (docs/design/21) は **プラン用語の上位 SSOT** として、以下の派生 SSOT より **上位** に位置する:

1. **上位 SSOT** (本規約): プラン名 / 階層 / 命名根拠
2. **派生 SSOT**:
   - `docs/design/billing-redesign/phase1-plan-naming-pricing-axis-requirements.md` (#2588): 訴求軸 / 課金期間 / 顧客不安 framing
   - `src/lib/domain/terms.ts` (ADR-0045 atom SSOT): 実装表示文字列
   - `src/lib/domain/labels.ts` (ADR-0045 compound SSOT): compound 表示文字列
   - `scripts/check-forbidden-terms.mjs` / `scripts/check-no-plan-literals.mjs` (CI gate): 禁止語検出
3. **改訂順序**: 上位 SSOT (本規約) → 派生 SSOT (atom rename、Phase 7) → 配信物 (LP / メール、Phase 7 完遂後)

> **過去の SSOT 衝突事例 (#2609)**: Phase 3 sibling PR (#2605 / #2606) で旧本規約 §4 「プレミアム禁止語」と #2588 「premium 採用確定」の衝突が 2 PR 連続検出された。本改訂で上位 SSOT (本規約) を #2588 整合に update し、衝突解消。
