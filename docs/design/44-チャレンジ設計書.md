## 1. 設計背景

### 1.1 この設計がなかった場合に何が困るか

日々のコアループ（活動を記録 → ポイント獲得 → レベル・称号で可視化、[26-ゲーミフィケーション設計書.md](26-ゲーミフィケーション設計書.md)）だけでは、次の家庭課題が解けない:

1. **記録のマンネリ化** — 毎日同じ活動だけを記録し続けると、子供にとって「ただの作業」になり動機が薄れる。日々のコアループは「同じ行動の反復」を前提とするため、これ単体ではマンネリ化を是正できない。
2. **やる活動の偏り** — 子供は得意・好きなカテゴリ（例: 運動）ばかり記録し、苦手・後回しのカテゴリ（例: 勉強・生活）に手が伸びない。コアループはどのカテゴリを記録するかを誘導しないため、偏りが固定化する。

チャレンジ機能は、これら 2 課題を「期間つきの中期目標」という 1 つの仕組みで解く。日々のコアループ（短期）と長期のゴール（卒業ジャーニー）の **中間にある "中期目標" レイヤ**を担う。子供が苦手分野を底上げし得意分野を伸ばす、日々の活動とは別軸の動機を作る。

## 2. 設計原則

### 2.1 顧客価値（core）

チャレンジ機能が提供する顧客価値は以下の 2 点に集約される:

| # | 顧客価値 | 解決する 1.1 の課題 |
|---|---|---|
| ① | **日々同じ活動の記録によるマンネリ化を是正する** | 1.1-1 |
| ② | **やる活動とやらない活動の偏りを是正する**（苦手カテゴリを底上げし、時々得意カテゴリを深掘りする） | 1.1-2 |

チャレンジはアプリが自動生成する（§3.1）。**ユーザーカスタマイズ（親手動でのチャレンジ作成・marketplace 配信）は提供しない**。自由度を上げると条件設定がプログラミング的に複雑化し、Pre-PMF には過剰（[ADR-0010](../decisions/0010-pre-pmf-scope-judgment.md) / YAGNI）であるため、アプリ自動生成に一本化し、改善はユーザーフィードバック（達成率等の analytics、§3.4）で回す。

**子供同士の競争は採らない**。兄弟間の勝ち負け比較は自己肯定感を毀損する（depression/自傷リスクの相関）ため、チャレンジは各 child 個別の自己ベース（苦手の底上げ・得意の深掘り）に閉じる。設計経緯は [rationale 12](../rationale/12-auto-challenge-generation-rationale.md)。

### 2.2 activity / checklist / reward との差別化

チャレンジは他の 3 リソースと役割が重複しないよう、以下 4 点で差別化する:

| 差別化軸 | チャレンジ | activity / checklist / reward |
|---|---|---|
| **期間スコープ** | 週単位の期間を持つ（始まりと終わりがある） | 期間概念を持たない（恒常的に存在） |
| **複数活動の集計** | 「合計 N 回」のように複数活動記録を集計して達成判定 | 1 件単位の記録 / 交換 |
| **自動生成** | アプリが苦手/得意を分析して自動生成（ユーザー設定なし） | ユーザーが内容を設定 |
| **達成ボーナス** | 期間内達成でボーナスポイントを付与（達成判定を伴う） | 記録ごと / 交換ごとの即時ポイント増減 |

### 2.3 Anti-engagement 整合（[ADR-0012](../decisions/0012-anti-engagement-principle.md)）

チャレンジは「滞在時間を延ばす」ための仕掛けではない。期間つき目標は「子供が自分から次に何をやろうか考える」きっかけ（自律性の育成）であり、連続演出・通知連打で介入時間を延伸する設計は採らない。達成演出は 1 件のみ表示し、連続演出を行わない。

## 3. 仕様

### 3.1 アプリ自動生成への一本化

チャレンジは **auto-challenge（週次自動生成）の 1 系統のみ**で提供される。親が何もしなくても、システムが child ごとに週 1 件のチャレンジを自動生成し、苦手カテゴリの底上げと時々の得意深掘りで偏り是正・マンネリ是正を担う（顧客価値 ①②）。

| 項目 | auto-challenge |
|---|---|
| **生成主体** | システム（週次自動生成、親関与なし） |
| **対象** | child ごとに 1 件 / 週（各 child 個別、競争・兄弟連動なし） |
| **データ** | `auto_challenges` テーブル（[08-DB §auto_challenges](08-データベース設計書.md)） |
| **サービス** | `auto-challenge-service.ts` |
| **アルゴリズム** | §3.4 / [rationale 12](../rationale/12-auto-challenge-generation-rationale.md) |

**親手動でのチャレンジ作成（旧 child-challenge / `/admin/challenges` 作成フォーム）は提供しない**（§2.1、ユーザーカスタマイズ撤去）。「今週はこれを頑張ってほしい」という親の意図伝達や兄弟共同チャレンジは、チャレンジ機能のスコープ外とする（顧客価値を ①② に集約）。

### 3.2 marketplace 非陳列の決定（#2896）

**2026-06-11 PO 判断**: marketplace（みんなのテンプレート）の陳列対象を **活動 / ごほうび / チェックリストの 3 type に絞る**。チャレンジ（challenge-set）は marketplace の陳列対象から外す。

- **機能本体は `/admin/challenges` に保持**する。自作チャレンジ・auto-challenge・`child_challenges` は撤去しない（本設計書 §3.1 の 2 系統は継続）。
- marketplace 唯一の challenge-set preset（`japan-annual-events`、日本年間行事パック）は **地域限定の季節コンテンツ**であり、全ユーザーに恒常陳列する性質ではないため**廃止**する。
- challenge-set の **型 / schema / Registry 登録 / 直リンク経路は互換のため残置**する（型エラー連鎖の回避、`?import=` 受領経路の互換）。陳列対象 type の SSOT は `MARKETPLACE_BROWSE_TYPE_CODES`（`src/lib/marketplace/types.ts`）で管理し、`MARKETPLACE_TYPE_CODES`（全 5 type）とは責務を分離する。
- 結果として、チャレンジの取込導線（marketplace browse → import）は廃止され、チャレンジは §3.1 の 2 系統（自作 + auto）で完結する。

### 3.4 auto-challenge 生成アルゴリズム

auto-challenge は「苦手中心＋時々得意」の中期目標を、行動科学・教育心理学の確立知見に基づく **ヒューリスティック＋調整可能定数**で生成する。cron / ML / ルールエンジンは用いず、既存の活動ログ集計と `auto_challenges` テーブルで完結する。設計経緯・各定数の根拠・棄却した代替案は [rationale 12](../rationale/12-auto-challenge-generation-rationale.md) を参照。

#### 調整可能定数

| 定数 | 既定値 | 役割 |
|---|---|---|
| `TARGET_DELTA` | 1 | 実測週平均 + これ = base target |
| `MIN_TARGET` / `MAX_TARGET` | 2 / 7 | target の下限・上限（Fogg "make it tiny" で下限を 3→2） |
| `WEAK_BIAS_BASE` | 5 | 苦手カテゴリ優先の重み |
| `EVERY_N_WEEKS_STRONG` | 4 | 「得意深掘り週」を入れる周期 |
| `BUMP_NORMAL` / `BUMP_OVERSHOOT` | 1 / 2 | 達成時の昇圧（大幅超過なら +2） |
| `MISS_RESCUE_AFTER` | 2 | 連続未達でレスキュー（target 最小 + 得意週）に切替える閾値 |

#### 生成 → 達成 → 翌週適応

1. **カテゴリ選択（weighted interleaving）**: 直近2週の活動ログを5カテゴリ集計。記録3件未満は random（explore）。`consecutiveMissCount >= MISS_RESCUE_AFTER` なら最多カテゴリ（rescue-strength）。`weekIndex % EVERY_N_WEEKS_STRONG == 0` なら最多カテゴリ（strength = 時々得意）。それ以外は苦手バイアスの重み付き抽選（weakness）。直前週と同一カテゴリは原則回避。
2. **target 決定（ability ベース + Flow 適応）**: `base = clamp(round(週平均) + TARGET_DELTA, MIN, MAX)`。前週結果で3分岐 — 完了→`+BUMP`（boredom 回避）/ 半分以上達成→据置（折らない）/ 半分未満→`-1`（anxiety 脱出 = self-efficacy 保護）。rescue-strength は `MIN_TARGET` 固定で必ず達成体験を作る。
3. **達成判定**: 一致カテゴリの活動記録で +1、target 到達で completed（既存ロジック維持）。
4. **適応の非対称性**: target を上げるのは completed 時のみ・最大 +2・上限7。「未達でも下げて必ず達成させる」非対称設計は滞在時間延伸でなく達成体験の確保が目的（§2.3 / ADR-0012 整合、青天井で難度を煽らない）。

#### フィードバック改善（ユーザーカスタマイズ撤去後の唯一の改善入力）

カテゴリ別達成率 / 達成時の超過度 / 未達時の到達率 / 2連続未達発生率 / 得意週vs苦手週の達成率差を `auto_challenges` 集計で観測し、定数を調整する。skip/dismiss 率は UI 導線追加を伴うため別 Issue（Pre-PMF 後回し）。

#### 不採用（過剰実装の境界）

multi-armed bandit / Thompson sampling、BirdBrain 型 ML 難度調整、DDA、曜日×カテゴリ×年齢の多変量 rule table は **MAU・学習データ前提で Pre-PMF 未到達**のため不採用（[ADR-0010](../decisions/0010-pre-pmf-scope-judgment.md) / YAGNI）。「3分岐＋2連続未達特例」を超えたら設計を止めて PO に戻す。
