## 1. 設計背景

### 1.1 この設計がなかった場合に何が困るか

日々のコアループ（活動を記録 → ポイント獲得 → レベル・称号で可視化、[26-ゲーミフィケーション設計書.md](26-ゲーミフィケーション設計書.md)）だけでは、次の家庭課題が解けない:

1. **記録のマンネリ化** — 毎日同じ活動だけを記録し続けると、子供にとって「ただの作業」になり動機が薄れる。日々のコアループは「同じ行動の反復」を前提とするため、これ単体ではマンネリ化を是正できない。
2. **やる活動の偏り** — 子供は得意・好きなカテゴリ（例: 運動）ばかり記録し、苦手・後回しのカテゴリ（例: 勉強・生活）に手が伸びない。コアループはどのカテゴリを記録するかを誘導しないため、偏りが固定化する。
3. **兄弟間のきっかけ不足** — 同じテナント内に複数の子供がいても、各自が個別にコアループを回すだけでは、兄弟が互いを意識し協力する接点が生まれない。
4. **親の意図を期限つきで伝える手段がない** — 「今週はこれを頑張ってほしい」という親の中期的な期待を、子供に伝え・追跡する仕組みがコアループには無い。

チャレンジ機能は、これら 4 課題を「期間つきの中期目標」という 1 つの仕組みで解く。日々のコアループ（短期）と長期のゴール（卒業ジャーニー）の **中間にある "中期目標" レイヤ**を担う。

### 1.2 関連ユースケース

[03-ユースケース設計書.md](03-ユースケース設計書.md) の **UC-16「親子チャレンジ（親子で一緒に達成する目標）」**（価値: 家族の絆強化）の実装根拠が本設計書である。

## 2. 設計原則

### 2.1 顧客価値（core）

チャレンジ機能が提供する顧客価値は以下の 4 点に集約される（PO 言語化、2026-06-11 確定）:

| # | 顧客価値 | 解決する 1.1 の課題 |
|---|---|---|
| ① | **日々同じ活動の記録によるマンネリ化を是正する** | 1.1-1 |
| ② | **やる活動とやらない活動の偏りを是正する**（前週に未達成だったカテゴリを狙う） | 1.1-2 |
| ③ | **兄弟間のコミュニケーション・協力のきっかけを作る** | 1.1-3 |
| ④ | **親が期限つきで「今週はこれを頑張ってほしい」を設定する手段を提供する** | 1.1-4 |

**③ は競争ではなく協力である**（EPIC #2296 で競争タイプ UI を意図的に撤去）。「兄弟で勝ち負けを競わせる」のではなく「兄弟で力を合わせて 1 つの目標を達成する」体験に転換した。子供同士の優劣比較は自己肯定感を毀損するため不採用とし、協力タイプに固定する。

### 2.2 activity / checklist / reward との差別化

チャレンジは他の 3 リソースと役割が重複しないよう、以下 4 点で差別化する:

| 差別化軸 | チャレンジ | activity / checklist / reward |
|---|---|---|
| **期間スコープ** | 週・月単位の期間を持つ（始まりと終わりがある） | 期間概念を持たない（恒常的に存在） |
| **複数活動の集計** | 「合計 N 回」のように複数活動記録を集計して達成判定 | 1 件単位の記録 / 交換 |
| **兄弟連動** | 同一テナントの複数の子供で 1 目標を共同達成できる | 各 child に独立 |
| **手動請求型ポイント** | 期間内達成でボーナスポイントを付与（達成判定を伴う） | 記録ごと / 交換ごとの即時ポイント増減 |

### 2.3 Anti-engagement 整合（[ADR-0012](../decisions/0012-anti-engagement-principle.md)）

チャレンジは「滞在時間を延ばす」ための仕掛けではない。期間つき目標は「子供が自分から次に何をやろうか考える」きっかけ（自律性の育成）であり、連続演出・通知連打で介入時間を延伸する設計は採らない。達成演出は 1 件のみ表示し、連続演出を行わない。

## 3. 仕様

### 3.1 2 系統の実装

チャレンジは 2 つの系統で提供される。両者は補完関係にあり、使い分けは下表のとおり。

| 系統 | auto-challenge（週次自動生成） | child-challenge（親手動 + per-child instance） |
|---|---|---|
| **生成主体** | システム（毎週月曜に自動生成、親関与なし） | 親（`/admin/challenges` の作成フォーム） |
| **狙い** | 顧客価値 ②（偏り是正）— 前週の未達成カテゴリを優先して自動提案 | 顧客価値 ③④（親の意図 + 兄弟連動）— 親が期間・目標・配信先を指定 |
| **対象** | child ごとに 1 件 / 週 | 親が選んだ child（複数選択で兄弟連動の共同チャレンジ） |
| **データ** | `auto_challenges` テーブル（[08-DB §auto_challenges](08-データベース設計書.md)） | `child_challenges` テーブル（per-child instance、[ADR-0055](../decisions/0055-per-child-primary-data-model-pattern.md) / [08-DB §child_challenges](08-データベース設計書.md)） |
| **サービス** | `auto-challenge-service.ts` | `child-challenge-service.ts` |
| **詳細** | [26-ゲーミフィケーション設計書.md §9.2 ウィークリーチャレンジ](26-ゲーミフィケーション設計書.md) | [26-ゲーミフィケーション設計書.md §10.1 きょうだいチャレンジ](26-ゲーミフィケーション設計書.md) |

#### 使い分けの原則

- **auto-challenge は「親が何もしなくても偏りが是正される」最小努力の入口**。親が忙しくチャレンジを設定する余裕がなくても、システムが前週未達成カテゴリを狙ったチャレンジを毎週自動で用意する（顧客価値 ②）。
- **child-challenge は「親が明確な意図を期限つきで伝えたいとき」の手段**。「今週は読書を 5 回」「兄弟で合計 10 回お手伝い」など、家庭の方針に沿った中期目標を親が能動的に設定する（顧客価値 ③④）。兄弟を複数選択すると協力タイプの共同チャレンジになる。
- 両系統は排他ではなく**併存**する。auto-challenge が常時の地ならしを担い、child-challenge が親の意図を上乗せする。

### 3.2 marketplace 非陳列の決定（#2896）

**2026-06-11 PO 判断**: marketplace（みんなのテンプレート）の陳列対象を **活動 / ごほうび / チェックリストの 3 type に絞る**。チャレンジ（challenge-set）は marketplace の陳列対象から外す。

- **機能本体は `/admin/challenges` に保持**する。自作チャレンジ・auto-challenge・`child_challenges` は撤去しない（本設計書 §3.1 の 2 系統は継続）。
- marketplace 唯一の challenge-set preset（`japan-annual-events`、日本年間行事パック）は **地域限定の季節コンテンツ**であり、全ユーザーに恒常陳列する性質ではないため**廃止**する。
- challenge-set の **型 / schema / Registry 登録 / 直リンク経路は互換のため残置**する（型エラー連鎖の回避、`?import=` 受領経路の互換）。陳列対象 type の SSOT は `MARKETPLACE_BROWSE_TYPE_CODES`（`src/lib/marketplace/types.ts`）で管理し、`MARKETPLACE_TYPE_CODES`（全 5 type）とは責務を分離する。
- 結果として、チャレンジの取込導線（marketplace browse → import）は廃止され、チャレンジは §3.1 の 2 系統（自作 + auto）で完結する。
