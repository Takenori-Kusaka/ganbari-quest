## 1. 設計方針

### デュアルバックエンド構成

本アプリケーションは2つのデプロイ環境に対応するため、DB バックエンドを抽象化している:

| 環境 | DB | 用途 |
|------|-----|------|
| AWS Lambda（本番 SaaS） | DynamoDB | シングルテーブル設計、マルチテナント対応 |
| NUC サーバー（ローカル） | SQLite | 単一ファイルDB、単一テナント |

### アーキテクチャ

```
src/lib/server/db/
  ├── interfaces/          # リポジトリインターフェース（17ファイル）
  ├── dynamodb/            # DynamoDB 実装
  │   ├── keys.ts          # PK/SK 生成ヘルパー
  │   ├── counter.ts       # ID 自動採番
  │   └── *-repo.ts        # エンティティ別リポジトリ
  ├── sqlite/              # SQLite 実装（Drizzle ORM）
  │   └── *-repo.ts
  ├── factory.ts           # 環境変数で実装を切り替え
  ├── schema.ts            # SQLite スキーマ定義
  └── *-repo.ts            # ファサード（factory からの re-export）
```

- `DB_BACKEND` 環境変数で `dynamodb` / `sqlite` を切り替え
- 全リポジトリに `tenantId: string` パラメータが必須
- SQLite 実装では tenantId を受け取るが内部では無視（単一テナント）

### SQLite 固有の方針

- **Drizzle ORM**: TypeScript型安全なスキーマ定義
- **WAL モード**: 読み書き並行性を向上
- **バックアップ**: 日次で git push（DBファイルのコピー）

### Schema 変更時の lazy migration 必須 SSOT (#2508 / 3 dimension SSOT)

**背景**: PR #2480 (#2362 PR-5 Phase 1) で `checklist_templates` を per-child instance
(`child_id NOT NULL`) から family master (`tenant_id NOT NULL`) に flip した際、
`schema.ts` / `create-tables.ts` は更新したが、既存 production DB を新 schema に上げる
**shadow-table recreation migration** が `client.ts` startup から呼ばれていなかったため、
NUC startup が `no such column: tenant_id` (SQL_CREATE_TABLES 内の `CREATE INDEX`) で完全に
block された (2026-05-27 NUC down)。

**SSOT**: schema 変更 PR では **3 file の同期更新を必須** とする。

| # | ファイル | 役割 |
|---|---|---|
| 1 | `src/lib/server/db/schema.ts` | drizzle table 定義 (TypeScript 型 SSOT) |
| 2 | `src/lib/server/db/create-tables.ts` | `CREATE TABLE/INDEX IF NOT EXISTS` 群 (新規 DB / dev / CI 用) |
| 3 | **`src/lib/server/db/migration/lazy-startup-migrations.ts`** | 既存 production DB を新 schema にアップグレードする shadow-table recreation / DROP COLUMN / FK target switch |

(1) と (2) を更新しただけでは既存 production DB は壊れる。本 file の更新漏れは
**NUC startup blocking** という形で爆発するため、以下に該当する schema 変更 PR は
必ず (3) へ migration を追加すること:

- `ALTER TABLE DROP COLUMN` (SQLite 3.35+ で限定対応、shadow recreation 推奨)
- FK target 変更 (SQLite 直接サポートなし、shadow recreation 必須)
- NOT NULL 制約変更 (同上)
- 既存 row への DEFAULT 値 backfill 含むカラム追加

なお、**個別カラム追加 (`ALTER TABLE ADD COLUMN ... DEFAULT ...`) は `schema-validator.ts`
の `validateAndMigrate` が drizzle schema 差分から自動適用する**ため、shadow recreation
対象外。

**startup 実行順序** (`src/lib/server/db/client.ts`):

```
1. applyLazyStartupMigrations(sqlite)  ← 旧 schema を新 schema 互換に揃える
2. sqlite.exec(SQL_CREATE_TABLES)      ← 新規 table / index を idempotent に作成
3. validateAndMigrate(sqlite)          ← drizzle 差分検出 + 個別 ALTER ADD COLUMN
```

**冪等性**: 各 migration block は `tableExists` / `hasColumn` / `hasFkToActivities`
guard で「既に新形式なら skip」を保証。startup ごとに毎回呼ばれても no-op。

**再発防止 CI gate**: `scripts/check-schema-migration-completeness.mjs` が `schema.ts` 差分から
NUC 起動 block 級の変更を検出し、既存 DB upgrade 経路の同期漏れを blocker 化する
(CI `ci.yml` job `schema-migration-completeness-check`)。検出対象は (1) DROP COLUMN /
(2) FK target 変更 / (3) NOT NULL 反転 / (4) cross-table flip / (5) 新規 table 追加
(create-tables.ts または lazy-startup-migrations.ts への同期がなければ blocker) /
(6) 新規 NOT NULL かつ DEFAULT なし列追加 (`validateAndMigrate` が自動 ALTER 不能で exit 1 する境界)。
非破壊の ADD COLUMN (nullable / DEFAULT 付き) は `validateAndMigrate` が自動適用するため対象外。

### per-child repo 共通設計原則 (#2845)

per-child instance を扱う repo (message / stamp-card / reward-redemption / special-reward /
child-activity / child-challenge / battle / certificate 等) に共通する 3 原則。
現状実装は AWS 公式 single-table design (pooled multi-tenant partitioning / item collection +
composite SK / adjacency list / overloaded GSI、`keys.ts` = key builder SSOT) に準拠する。
OSS 抽象化層 (ElectroDB / dynamodb-toolbox / onetable / TypeDORM) は「SQLite = schema SSOT、
DynamoDB = parity 実装」の主従関係と二重 SSOT 化で衝突するため**手書き維持 + パターン借用**
(ElectroDB は PMF 後の repo 全面 rewrite 第一候補)。比較詳細は research
(`tmp/research-single-table-patterns-2026-06-11.md`、#2845) 参照。

#### 原則 1: 所有権検証責務 — repo 入口で full composite-key addressing

- **per-child instance への mutation は `(childId, id, tenantId)` の複合キーを必須とする**
  (id-only mutation 禁止)。single-table では item の identity は PK+SK 全体であり、
  SQL surrogate id 単独は identity ではない (relational leaky abstraction / CWE-639 形状の温床)。
- DynamoDB: PK を `childPK(childId, tenantId)` で直接構成し GetItem / Query (KeyCondition) /
  UpdateItem する。SK に id 以外の要素 (timestamp / weekStart) を含む entity は
  child partition Query + id filter で解決する (tenant Scan 禁止)。直接 UpdateItem する場合は
  `ConditionExpression: 'attribute_exists(PK)'` で phantom 書込みを防ぎ、
  `ConditionalCheckFailedException` → undefined / affected 0 に正規化する。
- SQLite: `WHERE id = ? AND child_id = ?` で同一セマンティクス (不一致 = affected 0)。
- **Limit + FilterExpression の併用禁止** (#2842): DynamoDB の `Limit` は filter 適用「前」の
  評価件数上限。filter 付き走査は `ExclusiveStartKey` で全ページ走査し一致で早期 return する。
- 適用済: `markMessageShown` / `updateCardStatus*` / `updateRedemptionRequestStatus` /
  `markRedemptionResultShown` / `markRewardShown` / `updateSpecialReward` / `deleteSpecialReward` /
  `deleteTemplateItem` (checklist、templateId partition Query) / `deleteOverride` (checklist、
  child partition Query) / `markMissionCompleted` (daily-mission、exact Key) /
  `incrementDownloadCount` (cloud-export、exact Key)。
  B2 (Limit:1 + Filter 併用) 根治済: `findUnshownReward` / `findMissionBonusRecord` (全ページ走査化)。
- **tenant 無束縛 Scan の残党 (棚卸 2026-06-12 / #3044 修正後)**: write 解決 = trial-history
  `scanKeyById(historyId)` (Stripe webhook 経由 conversion 更新、id-only で tenant 文脈なし。
  低頻度 + paging 済。当該 repo を触る PR で tenantId 伝搬 or sparse GSI 化を判断) の 1 箇所のみ。
  activity-repo の read 系 (`hasActivityLogs` / `findActivityLogById`) と write/aggregate 系
  (`markActivityLogCancelled` / `getActivityLogCounts` / `deleteDailyMissionsByActivity`) は #3044 で
  全て tenant 束縛済 (下記「tenant 束縛済だが childId を interface が受けない tenant Scan 解決の
  同型残党」へ移行。activity-repo の tenant 無束縛 Scan は残ゼロ)。
- tenant 束縛済だが childId を interface が受けない **tenant Scan 解決の同型残党** (cross-tenant
  には到達しないが原則 1 の full composite-key addressing 未適用): activity facade
  (`updateActivity` / `deleteActivity` / `setActivityVisibility`) / activity-repo の read + write/aggregate
  (`hasActivityLogs` / `findActivityLogById` / `markActivityLogCancelled` / `getActivityLogCounts` /
  `deleteDailyMissionsByActivity`、#3044 で `begins_with(PK, T#<tenant>#CHILD#)` を FilterExpression に
  追加し全テーブル Scan を tenant child partition に閉じた。LOG# / MISSION# とも child partition
  配下 (`activityLogKey` / `dailyMissionKey`) のため同一 PK 規約。read 系も write 系と同じく
  FilterExpression での tenant 束縛が原則。childId 未受領のため exact Key 化は不可) /
  child-challenge (`resolveKeyById` 系 mutation) / battle (`completeBattle`) / certificate
  (`findCertificateById`、read) / sibling-cheer (`markShown`)。新規 repo は本原則を必須とし、残党は
  当該 repo を触る PR で本原則に揃える。SQLite/demo backend は単一家族 DB (activity_logs に
  tenant_id 列なし) で構造的に tenant 分離されるため WHERE 追加不要 (interface の tenantId は未使用)。
- **意図的例外 (tenant 文脈が構造的に存在しない経路、Scan + paging で維持)**: cloud-export
  `findByPin` (公開 PIN download、no-tenant read) / `deleteExpired` (tenant 横断 cron) /
  trial-history の active-trial cron Scan / cancellation-reason (tenant 横断 ops read)。
  いずれも read または cron で、原則 2 の閾値超過時に GSI 移行を起票する。

#### 原則 2: tenant 横断 Scan → GSI 移行の閾値

tenant 横断 listing (`findRedemptionRequestsByTenant` / `expireOldRedemptions` /
`hasPendingByReward` / `findAllByTenant` 等) は低頻度のため Pre-PMF では Scan + 属性 filter で
許容する (ADR-0010)。以下の**いずれか**を満たした時点で sparse GSI (未使用の GSI1) への移行を
起票する (AWS 公式に数値基準はなく本プロダクトの提案値):

| 移行トリガ | 閾値 |
|---|---|
| 対象 entity の tenant 内 item 数 | > 1,000 (1KB item 想定で Scan が 1MB = 複数ページ化する目安) |
| 当該経路の p95 latency | > 500ms |
| `ScannedCount / ReturnedCount` 比 | > 100 が常態化 |

検知手段: CloudWatch `ConsumedReadCapacityUnits` (table-level) + アプリ logger で Scan 実行時に
`ScannedCount` / `ReturnedCount` を構造化ログ出力 (Pre-PMF は専用 metric より log で十分)。

#### 原則 3: timestamp 規約

- **文字列 timestamp の正規形 = ISO 8601 'T' 区切り UTC** (`new Date().toISOString()`)。
  固定長・辞書順 = 時系列順で SK 埋め込みにも安全。
- **TTL / 数値演算用途は epoch 秒 (number)** (DynamoDB TTL 仕様)。二本立てが正。
- **形式跨ぎの文字列比較・sort 禁止**: 空白区切り (`CURRENT_TIMESTAMP`) と 'T' 区切りは
  `' '(0x20) < 'T'(0x54)` のため同一属性内に混在すると文字列 sort が壊れる。
  SQLite ↔ DynamoDB の文字列 timestamp を越境比較する場合は parse して epoch 比較する。
- SQLite の `CURRENT_TIMESTAMP` default 列 (空白区切り、例: `parent_messages.sent_at`) は
  同一列内の形式混在 (= `ORDER BY sent_at DESC` の sort 崩れ) を避けるため現行 default を維持する。
  ISO-T への統一は既存行の backfill migration とセットで別判断 (片側だけの書込み形式変更は禁止)。

### child_activities (per-child instance、#2362 PR-3、ADR-0055)

#### 設計背景

旧 `activities` (family-wide master + `age_min/max` で per-child visibility) を、
**aggregate root が child に閉じた per-child instance** に refactor (ADR-0055)。
SSOT: `docs/design/data-model-resource-scope.md` §4.1。

User 直接判断 (`tmp/user-question/2026-05-23-customer-use-case-data-model-qa.md` §1):
「親管理目線でもどこまでも子供主体であるべき」「年齢フィルタは marketplace 側のみで、
親管理画面では子供別管理に統一」。

#### 設計原則

- `child_id NOT NULL` で cross-child invariant を構造的に排除 (DDD Aggregate Root)
- `age_min / age_max` は marketplace 側 (`activity-pack` Strategy) の表示 filter にのみ残す
- `gradeLevel / subcategory / description` は使用 0 件の cleanup
- index は `(child_id, is_archived)` + `(child_id, sort_order)` の 2 系統
- 取込時の child binding は AdminApp 側で `ChildSelectionDialog` 経由
  (`ctx.childIds: readonly number[]`、CWE-598 privacy 排除)

#### DynamoDB キー設計（single-table、#2824 / ADR-0055）

本番 cognito Lambda は `DATA_SOURCE=dynamodb` で稼働するため、SQLite と機能等価の
DynamoDB 実装 (`src/lib/server/db/dynamodb/child-activity-repo.ts`) を持つ。
per-child instance は child partition 配下に置き、`findActivitiesByChild` を**追加 GSI なし**の
単一 partition Query で完結させる（cross-child access を構造的に防ぐ、ADR-0055 §3.1）。

| エンティティ | PK | SK | 補足 |
|---|---|---|---|
| child_activity | `T#<tid>#CHILD#<childId>` | `CHILDACT#<paddedId>` | id は 8 桁 0 埋め。activity_logs / point_ledger と同 partition に同居 |

| アクセスパターン | 操作 | 条件 |
|---|---|---|
| `findActivitiesByChild(childId)` | Query | `PK = T#<tid>#CHILD#<childId> AND begins_with(SK, 'CHILDACT#')`。archive / visibleOnly filter + sort_order 昇順は in-memory（SQLite SSOT と一致） |
| `findActivityById(id, childId)` | GetItem | `PK = T#<tid>#CHILD#<childId>, SK = CHILDACT#<paddedId>` |
| `insertActivity` | counter 採番 → PutItem | SQLite schema default（sort_order=0 / is_visible=1 / source='seed' / daily_limit=null 等）を明示的に埋め、`.returning()` 等価の row を返す |
| `insertActivitiesBulk` | 直列 PutItem | 取込時 per-child 配信の核心経路。返り値（作成 row）を import service が persist honesty 検証に使う（#2824） |
| `updateActivity` / `setActivityVisibility` | UpdateItem（`ConditionExpression: attribute_exists(PK)`、`ALL_NEW`） | 別 child / 不在は `ConditionalCheckFailedException` → undefined |
| `deleteActivity` | DeleteItem（`ALL_OLD`） | 削除前 row を返す（SQLite `.returning()` 等価） |
| `archiveActivities(ids, reason)` / `restoreArchivedActivities(reason)` | Scan（PK/SK projection）→ UpdateItem | child partition に分散するため id / reason 横断は Scan。低頻度（プラン降格時のみ）のため GSI 不要（Pre-PMF / ADR-0010） |

CDK（`infra/lib/storage-stack.ts`）の `MainTable` は単一テーブル（PK + SK + GSI1 + GSI2）を既に有しており、
新規 SK prefix `CHILDACT#` は既存テーブルにそのまま乗るため **CDK 変更不要**（追加 GSI なし）。

> **#2824 経緯**: 本 repo は PR #2455 で stub 化され、#2263 hotfix で「read=空 / write=throw」化された。
> 「write 経路は本番に到達しない」仮定は marketplace の per-child 取込（`importActivities` →
> `insertActivitiesBulk`）で破れ、本番で取込が永続せず UI が「N 件登録しました」と偽る CRITICAL の
> 一因となった。本実装で根治。helper 経由 stub（`return notImplemented(...)` 等）の再混入は
> `scripts/check-dynamodb-stub-parity.mjs`（baseline ratchet）が CI で検出する。

#### DynamoDB キー設計（activity_log / point_ledger / family-master facade、#2824 Wave 4A）

`src/lib/server/db/dynamodb/activity-repo.ts` の write 8 method を本実装化（コアループ
「活動記録 → ポイント付与」の本経路。本番 main Lambda は `DATA_SOURCE=dynamodb`）。log/ledger は
child partition 配下に置き、既存 read（`findDailyLog` / `findStreakLogs` / `findActivityLogs` /
集計）が読む key 形式・非正規化属性に整合させる。family-master CRUD は SQLite (#2458-A1) と同じく
per-child `child_activities`（`child-activity-repo.ts`）へ委譲し、旧 `activities` partition
（SK=MASTER）への write は 0 件を維持する。

| エンティティ | PK | SK | 補足 |
|---|---|---|---|
| activity_log | `T#<tid>#CHILD#<childId>` | `LOG#<recordedDate>#<paddedId>` | id 8 桁 0 埋め。read の JOIN 代替に `activityName` / `activityIcon` / `categoryId` を insert 時に child_activities lookup して非正規化保存 |
| point_ledger | `T#<tid>#CHILD#<childId>` | `POINT#<createdAt>#<paddedId>` | createdAt は ISO 文字列（`YYYY-MM-DD…`）で `countPointLedgerEntriesByTypeAndDate` の `begins_with(createdAt, :date)` に整合 |

| アクセスパターン | 操作 | 条件 |
|---|---|---|
| `insertActivityLog` | counter 採番 → child_activities lookup（非正規化）→ PutItem | `findActivityLogs`（ActivityLogSummary）/ `getCategoryCountsByDate` / `findTodayLogsWithCategory` 等が LOG item の denormalized 属性を直接読むため、SQLite の innerJoin（activity_logs ⋈ child_activities）相当を insert 時に解決して埋める |
| `insertPointLedger` | counter 採番 → PutItem | `countPointLedgerEntriesByType` / `getComboPointsGranted` 等が POINT item の type / amount / description / createdAt を読む形式で書く |
| `insertActivity` | findFirstChild（Scan）→ child_activities へ委譲 | facade signature 不変。tenant の最初の child に bind（SQLite #2458-A1 と同型） |
| `updateActivity` / `setActivityVisibility` / `deleteActivity` | id → childId 逆引き（Scan）→ child_activities へ委譲 | facade が childId を受けないため CHILDACT# を Scan で逆引き。低頻度（admin 操作）のため GSI 不要（Pre-PMF / ADR-0010） |
| `archiveActivities` / `restoreArchivedActivities` | child_activities へ委譲 | 旧 activities partition でなく per-child instance を archive/restore（downgrade / resource-archive サービスの barrel 経由呼び出しに整合） |

> **#2824 Wave 4A 経緯**: 旧コメント「DynamoDB backend は production 未使用（main Lambda は sqlite）」は
> stale な誤記だった（`infra/lib/compute-stack.ts` の `ganbari-quest-app` は `DATA_SOURCE=dynamodb` +
> `AUTH_MODE=cognito`）。write 8 method が `NotImplementedError` throw のままだと本番で活動記録・
> ポイント付与が throw する CRITICAL gap だったため本実装で根治。「SK=MASTER への write 0 件」不変条件は
> `tests/unit/services/activity-legacy-table-write-zero-dynamodb.test.ts`（write 成功かつ MASTER 不可触を assert）で回帰固定。

#### 設計背景（#1755 / #1709-A: 「今日のおやくそく」優先度）

checklist_templates.kind 列削除（持ち物純化）と引き換えに、活動側へ「保護者が
今日中にやってほしい」フラグ立てを移管する。子供 UI の冒頭セクション・達成率
ボーナス（preschool/elementary +5、junior/senior +3、baby 0）は本カラムを起点に算出する。

#### 設計原則

- 既存レコードは backfill で `priority='optional'` に揃える（NULL 許容しない）

- 集計 API（`getMustActivitiesToday` / `computeMustCompletionBonus`）は service 層
  （`activity-service.ts`）に集約し、複数 routes / hook で同一ロジックを再実装しない

#### #1689 DynamoDB キー設計（push_subscription / notification_log 単一テーブル）

##### 設計背景

`push_subscriptions` / `notification_logs` テーブルは SQLite (NUC) で先行実装され、本番 AWS 側 (`src/lib/server/db/dynamodb/push-subscription-repo.ts`) は #1666 PR merge 時点では `Not implemented` stub のままだった (#1021 段階的リリース禁止に抵触)。本 Issue (#1689) で SQLite 機能等価の DynamoDB 本実装を起こし stub を解消する。

##### 設計原則

1. **Pre-PMF 規模での GSI 不要原則** (ADR-0010): 1 家族あたり <10 デバイス・<100 通知/日 の規模では PK 単一 Query で十分。追加 GSI を作ると DynamoDB ストレージコストが線形に増える上、新規 IAM 権限・運用上の障害点も増える。
2. **migration script (#1666) との SK prefix 互換**: 既存の `scripts/migrate-dynamodb-push-subscriber-role.mjs` が `begins_with(SK, 'PUSH_SUB#')` で対象を Scan する設計のため、SK 規約は `PUSH_SUB#<...>` を厳守する。
3. **endpoint 単位の高速 lookup**: `findByEndpoint(endpoint, tenantId)` は `(endpoint, tenantId)` の組で呼ばれるため、endpoint の SHA-256 ハッシュ (先頭 16 hex) を SK に埋め込み、PK + SK で GetItem 1 回で取得する。Scan 不要。
4. **subscriberRole 必須属性** (#1593 ADR-0023 I6): PutItem 時に `'parent' | 'owner'` を必ず付与。不正値 (旧 `'child'` / NULL / 空文字列) は送信側 (`notification-service.ts`) で skip される二重防御。

##### キー設計表

| エンティティ | PK | SK | 補足 |
|------------|----|----|------|
| push_subscription | `T#<tenantId>#PUSH_SUB` | `PUSH_SUB#<endpointHash>` | endpointHash = SHA-256(endpoint) 先頭 16 hex (64 bit、衝突確率 < 10^-9 / 同一テナント内) |
| notification_log | `T#<tenantId>#NOTIF_LOG` | `NOTIF#<sentAt>#<paddedId>` | sentAt = ISO 8601 (`YYYY-MM-DDTHH:mm:ss.sssZ`)、paddedId = `nextId('notificationLog', tenantId)` を 8 桁 0 埋め |

##### アクセスパターン

| 関数 | DynamoDB 操作 | キー条件 |
|------|--------------|---------|
| `findByTenant(tenantId)` | Query | `PK = T#<tid>#PUSH_SUB AND begins_with(SK, 'PUSH_SUB#')` |
| `findByEndpoint(endpoint, tenantId)` | GetItem | `PK = T#<tid>#PUSH_SUB, SK = PUSH_SUB#<endpointHash>` (取得後に endpoint 完全一致で hash 衝突を二重チェック) |
| `insert(input)` | PutItem | counter で id 採番 → 上記キーで PutItem |
| `deleteByEndpoint(endpoint, tenantId)` | DeleteItem | `PK = T#<tid>#PUSH_SUB, SK = PUSH_SUB#<endpointHash>` |
| `insertLog(input)` | PutItem | counter で id 採番 → `PK = T#<tid>#NOTIF_LOG, SK = NOTIF#<sentAt>#<paddedId>` |
| `countTodayLogs(tenantId, today)` | Query (`Select=COUNT`) | `PK = T#<tid>#NOTIF_LOG AND SK BETWEEN 'NOTIF#<today>T00:00:00' AND 'NOTIF#<today>T99:99:99'` |
| `findRecentLogs(tenantId, limit)` | Query (`ScanIndexForward=false`) | `PK = T#<tid>#NOTIF_LOG AND begins_with(SK, 'NOTIF#') LIMIT <limit>` |

##### CDK 変更有無

CDK (`infra/lib/storage-stack.ts`) の `MainTable` は単一テーブル設計 (PK + SK + GSI1 + GSI2) を既に有しており、本 Issue は **追加 GSI なしで実現できるため CDK 変更不要**。新規 PK prefix (`T#<tid>#PUSH_SUB` / `T#<tid>#NOTIF_LOG`) は既存テーブルにそのまま乗る。

### stripe_webhook_events (#2641 / Phase 5 子 3 / Phase 7 PR-1)

Stripe Webhook の冪等性 dedup ログ。同一 `event.id` (`evt_*`) の重複処理を防ぎ、二重課金 / 二重ライセンス発行を回避する。

Stripe at-least-once delivery 仕様 + CLI `stripe events resend` で同一 event.id 再送が正規動作 (公式: <https://docs.stripe.com/webhooks#handle-duplicate-events>) のため、`event.id` を PK にした「処理済み event ログ」を SSOT として持つ。

| カラム | 型 | NULL | デフォルト | 説明 |
|--------|------|------|-----------|------|
| event_id | TEXT | NO | PK | Stripe `evt_*` (immutable、Stripe 側 SSOT) |
| event_type | TEXT | NO | - | event.type (`checkout.session.completed` / `invoice.paid` / `subscription_schedule.aborted` 等) |
| processed_at | TEXT | NO | CURRENT_TIMESTAMP | handler 実行完了時刻 (ISO 8601)、retention cutoff の基準 |
| handler_result | TEXT | NO | - | `'success'` / `'error'` / `'skipped'` (未購読 event 型は skipped) |
| error_message | TEXT | YES | NULL | handler 例外時の Stripe.Error.message (最大 500 文字 truncate、PII strip 済) |
| retry_count | INTEGER | NO | 0 | 同一 event.id の再到達回数 (初回 = 0、replay/resend で increment) |
| tenant_id | TEXT | YES | NULL | 関連 tenant_id (handler が解決できた場合のみ、analytics 用) |

**インデックス:**
- `idx_stripe_webhook_events_processed_at` (processed_at) — 30 日 retention cron 用
- `idx_stripe_webhook_events_type_result` (event_type, handler_result) — analytics 集計用

**意図的に除外したカラム:**
- `payload` (Stripe event 本体 JSON) — Stripe Events API (30 日保持) が SSOT。自社 DB に payload 全文を保存すると PII (customer email / billing details) が DB 流入する OWASP 違反 + DB サイズ膨張のため意図的除外 (Pre-PMF 過剰防衛回避、ADR-0010)
- `webhook_endpoint_id` — Pre-PMF では endpoint 1 個 (`/api/stripe/webhook`)、複数 endpoint 運用時に拡張

**DynamoDB 対応:**
- PK = `'STRIPE_WEBHOOK_EVENT'` (固定)、SK = `<event.id>` (例: `evt_1ABCxyz`)
- single-partition global (write rate Pre-PMF <100/日、PMF 後でも <10k/日 → DynamoDB 1k WCU/sec/partition 物理限界未満)
- TTL native 機能で 30 日後に自動削除 (`storage-stack.ts:29 timeToLiveAttribute='ttl'` 既設定、CDK 変更不要)
- `cancellation_reasons` / `graduation_consent` / `analytics_aggregate` と同 single-partition パターン

**Retention (ADR-0049 整合):**
- Stripe Events API 保持期間 30 日と同期し、自社 DB も 30 日で物理削除
- SQLite は `lazy-startup-migrations.ts` で日次 cron が `processed_at < (now - 30d)` を delete (Phase 7 PR-5 連動)
- DynamoDB は item-level TTL で AWS が自動削除 (cron 不要)

**Handler 統合 (Phase 7 PR-4a 連動):**
- `src/lib/server/services/stripe-service.ts` の `handleWebhookEvent` dispatcher 入口 (L221) で `findByEventId` dedup check → 既存時 skip、不在時に handler 実行後 `insert` (Stripe 公式 best practice)
- 全 8 event 型 (現行 5 + 子 1 #2641 新規 `subscription_schedule.*` 3 種) を一律処理し、reason 別 dedup ロジックを散在させない
- HTTP status は常に 200 (重複検知時も)。4xx/5xx 返却は Stripe 側 retry をトリガし重複到達を増やすため (Stripe 公式 `Acknowledge events immediately`)

### reward_redemption_requests

ごほうびショップ交換申請（#1337）。子供が交換申請し、親が承認/却下する親子連携フロー。
設計詳細: `docs/design/06-UI設計書.md §15`

| カラム | 型 | NULL | デフォルト | 説明 |
|--------|------|------|-----------|------|
| id | INTEGER | NO | autoincrement | 主キー |
| child_id | INTEGER | NO | - | FK → children.id（CASCADE DELETE） |
| reward_id | INTEGER | NO | - | FK → special_rewards.id（RESTRICT: 申請中は削除禁止） |
| requested_at | INTEGER | NO | - | 申請日時（Unix epoch 秒） |
| status | TEXT | NO | 'pending_parent_approval' | 申請状態: `pending_parent_approval` / `approved` / `rejected` / `expired` |
| parent_note | TEXT | YES | NULL | 却下時の親メモ（最大 100 文字）。承認時は NULL |
| resolved_at | INTEGER | YES | NULL | 承認/却下日時（Unix epoch 秒） |
| resolved_by_parent_id | TEXT | YES | NULL | 承認/却下した保護者の認証 userId (cognito sub 等)。監査証跡。local 実行モードや userId 不明時は NULL。**#3320 で INTEGER→TEXT**: 整数の親内部 ID は存在せず identity.userId は文字列 (cognito sub) のため。旧実装は常に 0 (解決者不明 sentinel) を記録していた |
| shown_to_child_at | INTEGER | YES | NULL | 子供への完了/却下通知を表示した日時（NULL = 未表示） |
| reward_title | TEXT | YES | NULL | **申請時点 snapshot**（#2832）。申請作成時の reward タイトル。NULL = snapshot 列導入前の旧行（live JOIN 値に fallback） |
| reward_points | INTEGER | YES | NULL | **申請時点 snapshot**（#2832）。申請作成時の reward ポイント。承認時の控除はこの値を使う |
| reward_icon | TEXT | YES | NULL | **申請時点 snapshot**（#2832）。申請作成時の reward アイコン |

**インデックス:**
- `(child_id, status)` — 子供の申請一覧クエリ用
- `(reward_id, status)` — 削除時の pending チェック用

**snapshot 仕様（申請時点値、#2832）:**
- 交換申請の表示（親の申請一覧 / 子供への結果通知）と承認時のポイント控除は、**申請時点の reward 内容（名前 / ポイント / アイコン）**を正とする
- SQLite: `insertRedemptionRequest` が `reward_title` / `reward_points` / `reward_icon` に snapshot を保存し、読み出しは `COALESCE(snapshot, live JOIN 値)`（NULL の旧行は live 値に fallback）
- DynamoDB: redemption item への非正規化保存（`childName` / `rewardTitle` / `rewardIcon` / `rewardPoints`）が同じ snapshot を担う
- 帰結: 親が reward を編集（名前 / ポイント変更）しても、処理待ちの交換申請は申請時点の内容のまま処理される（`/admin/rewards` 編集 dialog が note で明示）

**制約:**
- `UNIQUE (child_id, reward_id)` WHERE `status = 'pending_parent_approval'` — 同一報酬への二重申請防止（アプリ層でチェック）
- `reward_id` は RESTRICT 参照。**pending 申請が存在する `special_rewards` の削除は service 層 `deleteReward`（`special-reward-service.ts`）が `hasPendingByReward` ガードで拒否する**（#2832、`/admin/rewards` `?/delete` action → 409 + 「申請を承認/却下してから削除」ガイダンス）
- pending が無い reward の削除時は、当該 reward の**解決済（approved/rejected/expired）申請履歴行も同時に削除**される（FK 整合、SQLite はトランザクション / DynamoDB は等価の item 削除）
- `deleteByTenantId`（アカウント削除 / tenant cleanup 経路）は **pending ガード対象外**（pending ごと全削除が正、#2832）

**status 遷移:**
```
pending_parent_approval
  → approved     （親が承認）
  → rejected     （親が却下）
  → expired      （30 日経過後、日次 cron が自動移行）
```

**ポイント減算タイミング:** `approved` への遷移時のみ `point_ledger` に記録（type: `'reward_redemption'`）。  
`requested_at` 時点ではポイントを保留しない（シンプルさ優先）。

#### DynamoDB キー設計（single-table、#2824 Phase 2A / ADR-0055）

本番 cognito Lambda は `DATA_SOURCE=dynamodb` で稼働するため、SQLite と機能等価の
DynamoDB 実装 (`src/lib/server/db/dynamodb/reward-redemption-repo.ts`) を持つ。per-child
instance を child partition 配下に置き、`findRedemptionRequestsByChild` を**追加 GSI なし**の
単一 partition Query で完結させる（ADR-0055 §3.1）。

| エンティティ | PK | SK | 補足 |
|---|---|---|---|
| reward_redemption_request | `T#<tid>#CHILD#<childId>` | `REDEMPT#<paddedId>` | id は 8 桁 0 埋め。special_rewards / activity_logs と同 partition に同居。JOIN 代替に `childName` / `rewardTitle` / `rewardIcon` / `rewardPoints` を insert 時に非正規化保存 |

| アクセスパターン | 操作 | 条件 |
|---|---|---|
| `findRedemptionRequestsByChild(childId)` | Query | `PK = T#<tid>#CHILD#<childId> AND begins_with(SK, 'REDEMPT#')`。requested_at 降順は in-memory（SQLite SSOT と一致） |
| `insertRedemptionRequest` | counter 採番 → child / reward 解決 → PutItem | status default = `pending_parent_approval`（SQLite schema 一致）。childName / reward fields を非正規化保存し tenant 横断 read の N+1 を回避 |
| `findRedemptionRequestsByTenant` / `expireOldRedemptions` / `hasPendingByReward` | Scan（`begins_with(PK, T#<tid>#CHILD#)` + `begins_with(SK, REDEMPT#)`） | child partition に分散するため tenant 横断は Scan。親確認 / cron / 削除（低頻度）のみで GSI 不要（Pre-PMF / ADR-0010）。status / childId / limit は in-memory filter |
| `updateRedemptionRequestStatus(childId, id)` / `markRedemptionResultShown(childId, id)` | composite key で UpdateItem 直接実行（`ALL_NEW` + `attribute_exists(PK)`） | #2845 課題①: childId + id で PK/SK を直接構成（Scan 不使用）。不一致は ConditionalCheckFailed → undefined。undefined はスキップ / null は明示 SET（SQLite `.set({...})` 等価） |
| `findPendingByChildAndReward` / `findUnshownResultByChild` | Query（child partition）+ in-memory filter | 二重申請 / 未表示通知。後者は reward 結合フィールドを露出 |
| `deleteByTenantId` | Scan → BatchWrite | `CHILD#*` 配下の `REDEMPT#` を削除（`deleteItemsByPkPrefix`） |

**残高整合:** ポイント減算は service 層（`reward-redemption-service.approveRedemption`）が
`insertPointEntry` で別途行う（SQLite と同じ責務分割）。本 repo は status 更新のみ担い残高に
触れないため TransactWriteItems は不要。新規 SK prefix `REDEMPT#` は既存テーブルにそのまま
乗るため **CDK 変更不要**（追加 GSI なし）。

### checklist_templates

#### 設計背景（#2362 PR-5 / ADR-0055: family master 化）

旧 schema (#1755 / #1709-A 時点) は `child_id NOT NULL` の per-child instance だったため、
家族で 1 つの持ち物リスト (例: 「学校バッグの中身」) を運用したい場合、各お子さまごとに
重複コピーを作る必要があった。`data-model-resource-scope.md §4.2` で「family master template
+ per-child progress」モデルに整理し、PR-5 Phase 1 で schema flip を実施した。

設計原則:

- **template は family scope** (`tenant_id` 列で family 一意化)

- 旧ルーチン枠レコードは migration で drop（#1709-D で marketplace の該当プリセットを
  activity 由来に移管予定）

family master 持ち物チェックリスト定義。family scope (tenant_id) で一意化。

本番 cognito Lambda は `DATA_SOURCE=dynamodb` で稼働するため、SQLite と機能等価の
DynamoDB 実装 (`src/lib/server/db/dynamodb/stamp-card-repo.ts`) を持つ。card は per-child

| エンティティ | PK | SK | 補足 |
|---|---|---|---|
| stamp_cards | `T#<tid>#CHILD#<childId>` | `STMPCARD#<weekStart>` | weekStart 一意 = SQLite uniqueIndex(child_id, week_start) と等価。special_rewards / activity_logs と同 partition に同居。`id` は属性として保持 |

|---|---|---|
| `findEnabledStampMasters` | （DB アクセスなし） | `DEFAULT_STAMP_MASTERS_DATA` SSOT 16 件を `isEnabled === 1` で filter して返す（#2845 課題④: SQLite `WHERE is_enabled=1` parity。tenant 別カスタマイズなし Pre-PMF、空配列 → service の NO_STAMPS_AVAILABLE 回避） |
| `findCardByChildAndWeek(childId, weekStart)` | GetItem | `PK = T#<tid>#CHILD#<childId>, SK = STMPCARD#<weekStart>`。core loop で頻出（home load 毎）のため単一 GetItem |
| `insertCard` | counter 採番 → PutItem | status default = `collecting`（SQLite schema 一致）。redeemedPoints / redeemedAt = null |
| `findEntriesWithMasterByCardId(cardId)` | Query | `PK = T#<tid>#STMPCARD#<cardId> AND begins_with(SK, 'STMPENT#')`。slot 昇順 + master JOIN(name/emoji/rarity) は in-memory（LEFT JOIN miss は null、SQLite SSOT と一致） |
| `insertEntry` | PutItem（`attribute_not_exists(PK)`） | SQLite onConflictDoNothing 等価（同 (cardId, slot) 既存なら no-op） |
| `updateCardStatus(childId, id)` / `updateCardStatusIfCollecting(childId, id)` | child partition Query（id filter、全ページ走査）で PK/SK 解決 → UpdateItem | #2845 課題①: tenant + child 境界を KeyCondition で構造的に担保（tenant Scan 撤去）。後者は `ConditionExpression: #status = collecting` で冪等ガード（成功=1 / ConditionalCheckFailed=0、SQLite `result.changes` 等価） |
| `deleteByTenantId` | Scan → BatchWrite（2 経路） | `CHILD#*` 配下の `STMPCARD#`（cards）と `STMPCARD#*` 配下の `STMPENT#`（entries）を削除（`deleteItemsByPkPrefix`） |

新規 SK prefix `STMPCARD#` / `STMPENT#` は既存テーブルにそのまま乗るため **CDK 変更不要**
（追加 GSI なし）。schema 自体に変更はなく DynamoDB 実装の追加のみ。

#### DynamoDB キー設計 (#2824 Wave 3A、ADR-0055)

本番 cognito Lambda は `DATA_SOURCE=dynamodb` で稼働するため、SQLite と機能等価の
DynamoDB 実装 (`src/lib/server/db/dynamodb/message-repo.ts`) を持つ。おうえんメッセージは
child partition 配下に置き、`findMessages` / `findUnshownMessage` / `countUnshownMessages` を
**追加 GSI なし**の単一 partition Query で完結させる（ADR-0055 §3.1）。

| エンティティ | PK | SK | 補足 |
|---|---|---|---|
| parent_message | `T<tid>#CHILD#<childId>` | `MSG#<paddedId>` | id は 8 桁 0 埋め。activity_logs / point_ledger と同 partition に同居。read は全て child 軸のため非正規化不要 |

| アクセスパターン | 操作 | 条件 |
|---|---|---|
| `insertMessage` | counter 採番 → PutItem | icon default = `💌`（SQLite schema 一致）。sentAt は ISO 文字列で保存 |
| `findMessages(childId, limit)` | Query | `PK = T<tid>#CHILD#<childId> AND begins_with(SK, 'MSG#')`。sent_at 降順（同値は id 降順 tiebreaker）+ limit は in-memory（SQLite SSOT と一致） |
| `findUnshownMessage(childId)` / `countUnshownMessages(childId)` | Query（child partition）+ in-memory filter | `shown_at IS NULL` を filter。前者は最新 1 件、後者は件数 |
| `markMessageShown(childId, messageId)` | composite key（`parentMessageKey`）で UpdateItem 直接実行（`ALL_NEW` + `attribute_exists(PK)`） | #2845 課題①: childId + messageId で PK/SK を直接構成（Scan 不使用）。不一致は ConditionalCheckFailed → undefined |
| `deleteByTenantId` | Scan → BatchWrite | `CHILD#*` 配下の `MSG#` を削除（`deleteItemsByPkPrefix`） |

新規 SK prefix `MSG#` は既存テーブルにそのまま乗るため **CDK 変更不要**（追加 GSI なし）。
おうえんメッセージは LP (`feature-cheer-message`) が訴求する機能であり、本 repo は #2263 hotfix で

| 項目 | 値 |
|------|----|
| PK | `T#<tenantId>#TRIAL` （tenant 単位 partition） |
| SK | `HIST#<paddedId>` （8 桁 0 埋め、counter 採番 = SQLite auto-increment 互換、辞書順 = id 昇順） |

- `findLatestByTenant(tenantId)`: tenant partition を `ScanIndexForward=false` で Query `Limit 1` → 最大 id (=最新) を 1 read（GSI 不要）。

実装: `src/lib/server/db/dynamodb/trial-history-repo.ts`（挙動 SSOT は `sqlite/trial-history-repo.ts`）。

## 6.5 保持期間ポリシー（プラン別 retention） — #717, #729, #745

本プロジェクトの履歴保持ポリシーの **SSOT（Single Source of Truth）**。
現行の正仕様は [ADR-0028: 物理削除導入（archive）](../decisions/archive/0028-retention-physical-delete.md)。
旧仕様「表示フィルタのみ」の ADR-0027 は #1262 sub-B で削除（git 履歴参照）。

### 6.5.1 プラン別保持期間

| プラン | `historyRetentionDays` | 意味 | 物理削除 |
|--------|----------------------|------|---------|
| free | `90` | 90 日より前のレコードは非表示 + 日次バッチで削除 | **あり（90 日）** |
| standard | `365` | 1 年より前のレコードは非表示 + 日次バッチで削除 | **あり（365 日）** |
| family | `null` | 制限なし（全期間表示） | なし |

値の正は `src/lib/server/services/plan-limit-service.ts` の `PLAN_LIMITS[tier].historyRetentionDays`。設計書とコードが乖離した場合はコードを正として設計書側を修正すること。

### 6.5.2 retention の実装：表示フィルタ + 物理削除（二層構造）

**retention は「読み取り時の表示フィルタ」と「日次物理削除バッチ」の二層で実装する。**

pricing page が約束する「90 日 / 1 年 / 無制限」の履行責務を実データ削除で果たしつつ、
バッチ失敗時のフォールバックとして表示フィルタを残す（多層防御）。

#### 第 1 層：表示フィルタ（`applyRetentionFilter`）

`src/lib/server/services/plan-limit-service.ts`:

| 関数 | 責務 | 呼び出し側 |
|------|------|-----------|
| `getHistoryCutoffDate(tier)` | カットオフ日（YYYY-MM-DD）を返す。`null` なら制限なし | 内部ヘルパ（表示フィルタ / 削除バッチ両方から使用） |
| `applyRetentionFilter(tier, { from, to })` | `from` が cutoff より前なら cutoff で上書きした日付範囲を返す | 履歴一覧 API の呼び出し側 |
| `hasArchivedData(tenantId, childId, tier)` | cutoff より前にデータがあるか判定（アップグレード誘導バナー用） | admin ページ |

**呼び出し側:**

| ファイル | 用途 |
|---------|------|
| `src/routes/(child)/[uiMode=uiMode]/(character)/history/+page.server.ts` | 子供の履歴画面で `applyRetentionFilter` 適用後に `getActivityLogs` を呼ぶ |
| `src/routes/(parent)/admin/children/+page.server.ts` | 管理画面の子供ページで `applyRetentionFilter` + `hasArchivedData` を使用 |
| `src/lib/features/admin/components/ArchiveBanner.svelte` | `hasArchivedData === true` のときアップグレード誘導を表示 |

#### 第 2 層：日次物理削除バッチ（`retention-cleanup-service`）

`src/lib/server/services/retention-cleanup-service.ts`:

| 関数 | 責務 |
|------|------|
| `cleanupExpiredData({ dryRun })` | 全テナントを走査 → プランティア解決 → cutoff 算出 → 各 child の 3 テーブルから `recorded_date < cutoffDate` を物理削除 |

**処理フロー:**

1. `getRepos().auth.listAllTenants()` で全テナントを取得
2. 各テナントについて try/catch で独立処理（1 テナントの失敗が他に波及しない）
3. `deriveLicenseStatus(tenant)` で `licenseStatus` を決定
   - `stripeSubscriptionId` あり + `status=active|grace_period` → `'active'`
   - `stripeSubscriptionId` あり + その他 → `'suspended'`
   - `stripeSubscriptionId` なし → `'none'`
4. `resolveFullPlanTier(tenantId, licenseStatus, planId)` でティアを解決（トライアル考慮、[ADR-0024（archive）](../decisions/archive/0024-plan-tier-resolution-pattern.md)）
5. `getHistoryCutoffDate(tier)` で cutoff 日を算出。`null`（family）ならスキップ
6. `findAllChildren(tenantId)` で子を列挙
7. 各子について以下 3 テーブルから物理削除:
   - `activity_logs` → `repos.activity.deleteActivityLogsBeforeDate(childId, cutoffDate, tenantId)`
   - `point_ledger` → `repos.point.deletePointLedgerBeforeDate(childId, cutoffDate, tenantId)`
   - `login_bonuses` → `repos.loginBonus.deleteLoginBonusesBeforeDate(childId, cutoffDate, tenantId)`
8. 結果 `{tenantsProcessed, tenantsSkipped, childrenProcessed, activityLogsDeleted, pointLedgerDeleted, loginBonusesDeleted, errors[]}` を構造化ログに出力

**エンドポイント:**

`src/routes/api/cron/retention-cleanup/+server.ts`:

| メソッド | 動作 | 認証 |
|---------|------|------|
| `POST` | `{dryRun?: boolean}` — 本番実行 or dry-run | `Authorization: Bearer <CRON_SECRET>` (移行期は `OPS_SECRET_KEY` も後方互換で可 / ADR-0033) |
| `GET` | dry-run 固定（ヘルスチェック用途） | 同上 |

- `CRON_SECRET` / `OPS_SECRET_KEY` のいずれも未設定時は 404 を返してエンドポイントの存在を秘匿
- 認証失敗は 401
- サービス層エラーは 500 + JSON エラーメッセージ

**スケジュール（現状 → 後続チケット）:**

- **現状**: OPS ダッシュボード経由で手動呼び出し（日次目安）
- **後続**: CDK EventBridge Rule で日次自動化（Lambda Web Adapter 経由の変換層が必要）

#### SQLite / DynamoDB 両バックエンドの実装

| バックエンド | activity_logs 削除 | point_ledger 削除 | login_bonuses 削除 |
|------------|------------------|-----------------|------------------|
| SQLite | `db.delete(activityLogs).where(and(eq(childId), lt(recordedDate, cutoff)))` | `lt(createdAt, cutoffDate)`（ISO タイムスタンプは YYYY-MM-DD との辞書順比較で機能） | `lt(loginDate, cutoffDate)` |
| DynamoDB | `Query: PK=childPK AND SK BETWEEN 'LOG#' AND 'LOG#<cutoffDate>'` + BatchWrite（25件ずつ） | 同形式で `POINT#` prefix | 同形式で `LOGIN#` prefix + client-side strict-less-than フィルタ |

DynamoDB の SK 辞書順比較:
- `LOG#<cutoffDate>#<id>` > `LOG#<cutoffDate>` （`#` が追加されるため）→ cutoff 当日のレコードは除外される
- login_bonuses は SK = `LOGIN#<date>` で cutoff 当日が exact match しうるため、client-side で `loginDate < cutoffDate` を追加フィルタ

### 6.5.3 削除対象と非削除対象

| 物理削除する | 理由 |
|-------------|------|
| `activity_logs` | 活動ログ本体。pricing の約束の核心 |
| `point_ledger` エントリ | ポイント履歴の明細 |
| `login_bonuses` | ログインボーナス履歴 |

| 物理削除しない | 理由 |
|---------------|------|
| `point_ledger` の `BALANCE` 集計（DynamoDB）/ 現在残高 | **ポイントは消えず、過去明細だけが消える**。子供の UX 破壊を避けるための #729 の設計判断 |
| `report_daily_summaries` 等の集計テーブル | 月次レポート機能は全期間の集計を前提に動作する。"がんばりの証" は恒久保持 |
| `children` / `activities` / `settings` 等のマスタ | データの意味を破壊する |
| `child_achievements` / `special_rewards` / `stamp_cards` / `stamp_entries` | 現時点では retention 範囲外（将来再検討） |

**ポイント残高の非削除設計（#729 の核心）**:

- `point_ledger` は過去の明細、`BALANCE` は現時点の総残高（`ADD` 演算で更新）
- retention 削除で明細が消えても残高は変わらない
- 結果: 「90 日前にもらったポイントの明細」は消えるが、**そのポイントは今も使える**
- pricing の「90 日履歴保持」の解釈として一貫: 保持されるのは "履歴" であって "ポイントそのもの" ではない

### 6.5.4 集計値は保持期間の影響を受けない

`report_daily_summaries` テーブル（§3 参照）および将来追加される他の集計系テーブルは、プランに関係なく**全期間の集計値を保持**する。

- `applyRetentionFilter` は当該テーブルには**適用しない**
- `retention-cleanup-service` も当該テーブルには**一切触れない**
- 月次レポート機能（有料）は全期間の集計を前提に動作する
- カラム例: `activity_count`, `total_points`, `streak_days`, `new_achievements`

**理由**: 集計値は "がんばりの証" であり、プラン downgrade で失われてはならない。かつ日次 1 行 × 家族数しかなく物理サイズが小さい。

### 6.5.5 プラン downgrade 時の扱い

- downgrade 直後は DB にデータが残っている
- 次回リクエストから `applyRetentionFilter` が cutoff より前のデータを隠す（表示フィルタ層）
- 次の日次バッチ実行時に cutoff より前のデータが物理削除される（削除バッチ層）
- 再 upgrade しても**物理削除されたデータは戻らない**（ADR-0027 からの後退点）
- downgrade 時点の UI 警告（"過去のデータが近日中に削除されます" モーダル）は UI 層の責務
- 削除予告メール（14 日前通知）は後続チケットで実装予定

### 6.5.6 retention と他の物理削除の使い分け

物理削除には以下の種類がある:

| トリガ | 対象 | grace period | 実装 | 備考 |
|-------|------|-------------|------|------|
| **retention cleanup（日次バッチ）** | 各 child の `activity_logs` / `point_ledger` / `login_bonuses` で cutoff より前のレコード | プラン別（90/365 日） | `retention-cleanup-service.cleanupExpiredData()` | 本節の主題。集計・マスタ・残高は触らない |
| 「家族のデータを全部消す」（admin/settings） | children + テナントスコープ全般（認証は残す） | なし | `data-service.clearAllFamilyData()` | テナント・メンバーシップ・招待・認証は残るため、続けて使い続けられる |
| ユーザーによるアカウント削除 | 当該テナントの**全レコード**（認証・テナント本体を含む） | 30 日（[ADR-0022（archive）](../decisions/archive/0022-billing-data-lifecycle-consistency.md)） | `account-deletion-service.deleteOwnerOnlyAccount()` 他 | Stripe を先にキャンセル |
| 違反アカウント強制削除 | 当該テナントの全レコード | なし | OPS 画面からの手動操作 | |

retention（日次小範囲削除）と アカウント削除（全レコード削除）は **別軸** であり、削除範囲を混同しないこと。

### 6.5.7 データ削除スコープマトリクス (#739)

「家族のデータを全部消す」と「アカウント削除」は従来ロジックが分離しており、
削除範囲のズレ（例: `trial_history` が残る / `special_rewards` テンプレートが残る）
がバグの温床だった。共通下位関数を `tenant-cleanup-service.ts` に切り出し、
両方の呼び出し側から同じヘルパを呼ぶ構造に統一する。

**共通下位関数**:

| ヘルパ | 対象 | 実装 |
|-------|------|------|
| `deleteAllChildrenData(tenantId)` | children + child_files + カスケード配下（activity_logs / point_ledger / statuses / status_history / stamp_cards / stamp_entries / child_achievements / login_bonuses / character_images / evaluations / special_rewards / checklist_logs / checklist_overrides） | `tenant-cleanup-service.ts` |
| `deleteTenantScopedData(tenantId)` | activities / viewer_tokens / cloud_exports / push_subscriptions / voice / settings / checklists (templates+items) / daily_missions / evaluations (rest_days) / points / stamp_cards / statuses / login_bonuses / special_rewards (templates) / activity_prefs / activity_mastery / messages / **trial_history** / sibling_cheers / report_daily_summaries / images (#2458 で sibling_challenges + #2295 で season_events / tenant_events + #3213 で auto_challenges 撤去済。child_challenges は child cascade で削除) | `tenant-cleanup-service.ts` |

**呼び出し側のスコープ**:

| 呼び出し側 | `deleteAllChildrenData` | `deleteTenantScopedData` | テナント/認証レコード |
|------------|:---:|:---:|:---:|
| `clearAllFamilyData` (admin/settings → 家族データ全クリア) | ✅ | ✅ (#739 追加) | ❌（残す） |
| `fullTenantDeletion` (Pattern 1: Owner のみ削除) | ✅ | ✅ | ✅ (削除) |
| `deleteOwnerFullDelete` (Pattern 2b: Owner 全削除) | ✅ | ✅ | ✅ (削除) |
| `transferOwnershipAndLeave` (Pattern 2a: 権限移譲) | ❌ | ❌ | ❌（Owner のみ削除） |
| `deleteChildAccount` (Pattern 3) | 対象子のみ | ❌ | child 行のみ |
| `deleteMemberAccount` (Pattern 4: Viewer/親離脱) | ❌ | ❌ | membership のみ |

**#739 の主要な差分**: `clearAllFamilyData` が従来は children のみを削除していたため、
`trial_history` が残ってトライアル再申込ができない / `special_rewards` テンプレートが
残る等の症状があった。スコープマトリクスの ✅ 欄に `deleteTenantScopedData` を追加
することで 2 箇所のロジックが同じ下位関数を呼ぶ構造になった。

### 6.5.8 アップグレード誘導バナー

`hasArchivedData()` が true を返す家族に対して、`ArchiveBanner.svelte` が
「過去のデータが隠れています。ファミリープランにアップグレードすれば全期間閲覧できます。
ただし削除バッチが走ると物理削除され、その後の再閲覧はできなくなります」という
アップグレード誘導を表示する。

文言は ADR-0028 の物理削除方針に合わせて「削除される前に救済される」ニュアンスに統一すること。

### 6.5.9 禁止事項

- **集計テーブルへの retention 適用禁止**: `report_daily_summaries` 等の集計系には `applyRetentionFilter` / 物理削除バッチを**通さない**
- **retention 対象テーブルの無条件追加禁止**: `retention-cleanup-service` に新しい child 配下テーブルを追加する場合は本 SSOT と ADR-0028 を同時に更新し、削除可能性を再評価すること（特に集計・残高・マスタ）
- **dry-run を省略した本番実行禁止**: 新しい環境・新しいデータ投入の直後は必ず `POST { "dryRun": true }` で削除予定件数を確認してから本実行すること

## 8. スキーマバージョニングと遅延マイグレーション

### 8.1 概要

レコードごとにスキーマバージョン (`_sv`) を付与し、読み取り時に旧バージョンを検出・変換・書き戻す **Lazy Migration** パターンを採用。

| パターン | 役割 |
|---------|------|
| Schema Versioning | レコードごとに `_sv` フィールドでバージョン管理 |
| Lazy Migration (On-Read Upgrade) | 読み取り時に旧バージョンを検出・変換 |
| Write-Back (Optimistic Locking) | 変換結果をDBに書き戻し（DynamoDB: ConditionExpression） |
| Migration Pipeline | V1→V2→V3→...の順序保証された変換チェーン |

### 8.2 `_sv` カラム

対象テーブル: `children`, `statuses`（今後拡大予定）

- SQLite: `_sv INTEGER` カラム（nullable）。既存レコードは `NULL` = V1 扱い
- DynamoDB: `_sv` 属性。既存アイテムは属性なし = V1 扱い
- 新規レコードは挿入時に `_sv: LATEST` を自動付与

### 8.3 ディレクトリ構成

```
src/lib/server/db/migration/
  index.ts           — Public API: hydrate(), withVersion(), getPipeline()
  pipeline.ts        — MigrationPipeline クラス
  registry.ts        — SchemaRegistry（エンティティ別バージョン管理）
  types.ts           — SchemaTransformer インターフェース
  writeback.ts       — DynamoDB Write-Back ヘルパー
  transformers/
    child.ts         — Child V1→V2 (displayConfig, birthdayBonusMultiplier 等のデフォルト保証)
    status.ts        — Status V1→V2 (peakXp のデフォルト保証)
```

### 8.4 エンティティバージョン定義

| エンティティ | LATEST | MIN_SUPPORTED | 変更内容 |
|-------------|--------|---------------|---------|
| child | 2 | 1 | V2: displayConfig, birthdayBonusMultiplier, lastBirthdayBonusYear のデフォルト保証 |
| status | 2 | 1 | V2: peakXp のデフォルト保証（0 かつ totalXp > 0 の場合コピー） |

### 8.5 リポジトリ統合

- **DynamoDB**: 読み取り関数で `hydrate()` → `didMigrate` なら `writeBackDynamoDB()` で Optimistic Locking 書き戻し
- **SQLite**: 読み取り関数で `_sv` チェック → null/旧版なら `UPDATE SET _sv = LATEST`
- **挿入時**: `withVersion()` で `_sv: LATEST` を自動付与

### 8.6 Schema 変更時の SSOT 4 dimension (#2510 / 2026-05-27 NUC incident 教訓)

§8.1〜§8.5 は **row-level (record-level) lazy migration** (`_sv` カラムで record ごとに version 管理し、読み取り時に変換) を扱う。一方、**table-level / column-level の schema 破壊変更** (column 追加 / FK target 切替 / DROP COLUMN / cross-table semantic flip 等) は別の仕組みで対応する必要があり、本節で扱う。

NUC startup blocking (Issue #2508 / PR #2509) + NUC activities data 完全消失 (Issue #2510) 連続発生の教訓として、**schema 変更 PR は以下 4 dimension の SSOT を必ず同期更新する**:

| dim | file | 責務 | 違反時の影響 |
|---|---|---|---|
| 1 | `src/lib/server/db/schema.ts` | drizzle table 型 SSOT (TypeScript 型) | type error / build fail |
| 2 | `src/lib/server/db/create-tables.ts` | 新規 DB / dev / CI 用の `CREATE TABLE IF NOT EXISTS` + INDEX | fresh DB で `no such table/column` |
| 3 | `src/lib/server/db/migration/lazy-startup-migrations.ts` (structural) | shadow-table recreation / DROP COLUMN / FK target switch / NOT NULL 変更 | 既存 production DB 起動 block (例: PR #2480 → NUC down ~6 分) |
| 4 | `src/lib/server/db/migration/lazy-startup-migrations.ts` (data copy) | cross-table semantic flip 時の row 移動 (例: per-table → per-child) | 既存 data orphan 化、UI 全件表示不可 (例: PR #2487 → NUC user history 完全消失) |

#### dim 3 と dim 4 の責務分離

両者は同 file 内に並んで実装されるが、責務は明確に異なる:

- **dim 3 (structural)**: schema 形状を旧→新に合わせる (列 / FK / 制約の整合)。例: `migrateActivityFkSwitchover` (`activities` → `child_activities` への FK target 切替)
- **dim 4 (data copy)**: データ自体を旧 table 群から新 table 群へ意味的に意味のある形で再配置する。例: `migrateActivitiesLegacyDataCopy` (旧 `activities` 行を per-child `child_activities` に copy + 4 table の FK remap)。**PR #2513 で `lazy-startup-migrations.ts` に実装済** (startup 時 `migrateActivityFkSwitchover` の直後に自動実行、新規 NUC / dev / CI でも適用)。手動緊急復旧版は `scripts/recover-activities-data.mjs` に同 logic を維持 (NUC container 単発実行用、SQL logic を同期)

#### 事例集

| PR | dim 充足 | 結果 |
|---|---|---|
| PR #2480 (checklist_templates per-child → family master flip) | 1, 2, 3, 4 (PR #2509 で `migrateChecklistTemplatesFamilyFlip` 内に data copy も含めて完遂) | per-child template → assignments 1 row 移行込で完全復旧 |
| PR #2487 (activities → child_activities flip) | 1, 2, 3 のみ | dim 4 漏れ → NUC user data 完全消失 (history 全件 orphan、UI 表示 0)。Issue #2510 で緊急 recovery (`scripts/recover-activities-data.mjs`、PR #2512) → **PR #2513 で `migrateActivitiesLegacyDataCopy` を lazy-startup-migrations.ts に組込み恒久 fix 完遂** (dim 4 をコードで補完) |
| PR #2491 (#2458-A2 demo + dynamodb 同期) | 別 backend | sqlite 影響なし (DynamoDB 側は別途 backfill 必要) |

### 8.7 data 保全 gate — runtime orphan 自動 gate + backup restore 検証 (#2519 / #2518 KPT)

§8.6 の 4 dimension SSOT は「schema 破壊変更 PR が同期更新すべき file」を定める **予防策**。一方、予防が漏れた場合 (PR #2487 = dim 4 漏れ) を **実行時に検出する 2 つの gate** を #2519 で追加した。「検証を増やす」のではなく、**既存 gate の盲点 2 点を埋める最小十分対応** (ADR-0010 Bucket A: data 消失 = 親離脱の最重要 failure class)。

#### gate 1: runtime data orphan 自動 gate

| 項目 | 内容 |
|---|---|
| 実体 | `src/lib/server/db/migration/data-integrity-guards.ts` `assertNoDataOrphans(sqlite)` |
| 呼出点 | `client.ts` の startup sequence、`applyLazyStartupMigrations()` + `SQL_CREATE_TABLES` + `validateAndMigrate()` の **後** (= migration 完了後) |
| 監視対象 | core 4 table (`activity_logs` / `daily_missions` / `activity_mastery` / `child_activity_preferences`) の `activity_id → child_activities(id)` orphan。全 table 全 FK の毎起動チェックは過剰のため不採用 (cross-table flip 爆心地に限定) |
| 検出時の挙動 | 常に `console.error` + (本番/dev) Discord alert (`emitOrphanAlert` DI 経由、fire-and-forget) + (test/CI) throw で fail-loud。**起動は止めない** (復旧 script は app 経由で実行するため、起動 block すると復旧不能になる) |
| 盲点埋め | 既存 `scripts/check-orphan-tables.mjs` は static schema dead table を見るが runtime data orphan (FK 切れ row) を見ない = #2510 を捕捉できなかった構造的理由 |
| 回帰テスト | `tests/integration/db/data-orphan-gate.test.ts` (legacy schema 自動修復で orphan=0 / 直接 orphan を throw で検出 / 健全 DB は throw せず / onOrphan callback) |

#### gate 2: backup restore + integrity 検証

| 項目 | 内容 |
|---|---|
| 実体 | `scripts/verify-backup-restore.cjs` |
| 検証内容 | 最新 backup を一時 file に restore → `PRAGMA integrity_check`=ok + `PRAGMA foreign_key_check`=空 + 主要 table (`children` / `child_activities` / `activity_logs` / `point_ledger`) row count > 0 を assert。全 PASS で exit 0、1 つでも fail で exit 1 |
| cron 組込 | `docker-compose.yml` の `backup` profile が daily 03:00 JST に `backup-db.cjs` → `verify-backup-restore.cjs` を chain 実行。verify 失敗時は `DISCORD_ALERT_WEBHOOK_URL` 宛に Discord alert (cron は `.cjs` のため SvelteKit module を import せず、最小 fetch を script 内に持つ) |
| backup-db.cjs 格上げ | `backup-db.cjs` の backup 直後 integrity チェックを「table 数 > 0」から `integrity_check`=ok + `foreign_key_check`=空 に格上げ。WAL-safe な `db.backup()` は維持 |
| 盲点埋め | 従来の `backup-db.cjs` は「table 数 > 0」しか確認せず「backup の存在 ≠ restore 可能」(GitLab 2017 postmortem 教訓) を満たさなかった。NUC は単一 SQLite で冗長性ゼロのため backup の restore 可能性が最後の砦 |

#### scope 外 (ADR-0010 過剰防衛却下)

- Litestream フル / 多重 backup 冗長 → 単一家庭 NUC では cron + `.backup()` + restore 検証で十分
- candidate 1 (`migrateActivitiesLegacyDataCopy` data copy migration) は **PR #2513 で実装済** (§8.6 実装ステータス参照)、#2519 対象外
