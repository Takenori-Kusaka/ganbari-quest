## 1. 設計背景

### 課題・経緯

V2MOM はSalesforce が広めた組織のアライメントフレームワーク（Vision / Values / Methods / Obstacles / Measures）。個人開発体制のがんばりクエストに導入した背景は以下の通り:

| 背景 | 詳細 |
|------|------|
| PO 1 人体制のバイアスリスク | 課金・管理画面偏重 / Growth・Marketing 欠落という AI エージェント活用特有のバイアスが発生した（ADR-0010 制定の契機） |
| 優先度の軸がない | Issue が増えるにつれて「何のために何をやっているか」の軸が失われつつあった |
| 四半期ごとの方針確認 | Pre-PMF → PMF の節目で方針転換の可能性があり、明示的なレビューサイクルが必要 |

2026-04-09 に Q2 V2MOM を策定し、四半期単位で方針を確認・更新するプロセスを確立した。

**関連**: 34-V2MOM.md の Vision・Values は `01-企画書.md` §2 の設計原則と同期を維持する必要がある。変更する場合は両ファイルを同時更新すること。

### 関連ペルソナ

| ペルソナ | V2MOM との関係 |
|---------|--------------|
| P1 田中ゆかり（親・メイン） | V2MOM の M3（ユーザー獲得）・Measures（サインアップ数）の主な対象 |
| P2 鈴木大介（親・サブ） | OSS 版の継続的な品質向上が M2（SaaS 基盤安定化）に間接的に影響 |

## 2. 設計原則

1. **四半期単位の更新** — V2MOM は四半期ごとに見直す。前期の Measures の達成状況を評価してから次期を策定する
2. **ビジョンと Values は長期固定** — Vision と Values は容易に変更しない。変更する場合は `01-企画書.md` §2 の設計原則と同期させる
3. **Methods は実装可能なものだけ** — Pre-PMF 段階では抱負ではなく実行可能な施策のみを Methods に記載する（ADR-0010 準拠）
4. **Measures は定量指標のみ** — 「UX を改善する」ではなく「デモ完走率 30%」のように計測可能な指標を設定する
5. **Obstacles の正直な記述** — 改善できていない障害もそのまま残す。次期 V2MOM への引き継ぎ情報として機能する

## Vision（ビジョン）

**子供が自分から がんばりたくなる — 日本一やさしい 子育てゲーミフィケーションアプリ**

---

## Values（価値観）

1. **子供ファースト** — 全ての機能は子供の成長と楽しさを最優先に設計する
2. **親の負担ゼロ** — アプリの管理に時間を取らせない。設定は最小限、自動化を最大化
3. **ポジティブのみ** — 罰則・減点は一切なし。がんばりを認め、褒め、励ます
4. **活動の動機づけ以外は有害** — アプリ内滞在時間の最大化ではなく、現実世界の活動を促す
5. **シンプルに保つ** — 機能を増やすより、コア体験を磨く

---

## Methods（方法）

### M1: コアループの完成度向上
- 活動記録 → スタンプガチャ → レベルアップの体験を磨く
- おみくじ演出・紙吹雪エフェクトの品質向上
- 画像アセット（シール16種）の生成

### M2: SaaS 基盤の安定化
- AWS Lambda 本番環境の安定運用
- CI/CD パイプラインの信頼性向上
- 設計書と実装の同期（#609）

### M3: ユーザー獲得の基盤構築
- LP の改善（Before/After、スクリーンショット）
- デモモードの充実
- フリープランでの初期ユーザー獲得

### M4: 法務・コンプライアンス対応
- COPPA 対応方針の確立
- 利用規約・プライバシーポリシーの法務レビュー
- Cookie ポリシーの整備

---

## Obstacles（障害）

1. **個人開発のリソース制約** — 開発・デザイン・マーケティングを1人で行う
2. **画像アセットの品質** — 子供向けの魅力的なビジュアルが必要だが、デザイナーがいない
3. **ユーザー獲得コスト** — 有料広告予算がゼロ
4. **5年齢モードの並行実装** — 同じ機能を5ルートに実装する保守コスト（#567 で統合予定）
5. **AWS コスト** — 無料枠内での運用が前提
