## 1. デザイン原則

### 最優先原則: 4歳児が使える

- **文字に頼らない**: アイコン・色・形でナビゲーション
- **大きなタップターゲット**: 最低 64x64px（推奨 80x80px）
- **即座のフィードバック**: タップしたら即反応（色変化、スケールアニメーション）
- **シンプルな遷移**: 最大2タップで目的達成
- **明確な視覚的区別**: 子供別のテーマで「自分のページ」を実感

### 4.8 スタンプカード（ログインボーナス）

**正仕様（ADR-0004 参照）:**

#### スタンプカード構造
- **一週間単位**のスタンプカード（月曜〜日曜）
- スタンプは一週間に**最大5個**まで押印可能（**曜日に依存しない**）
- 一日一回ログインボーナス表示時にスタンプカードへ押印アニメーションとともに押印

#### スタンプデザイン
- `static/assets/stamps/` の**イラスト画像アセット**（6種）を使用
- おみくじランク: 大大吉(UR) / 大吉(SR) / 中吉(R) / 小吉(R) / 吉(N) / 末吉(N)
- レア度確率: N=60%, R=25%, SR=12%, UR=3%

#### スタンプカードUI
```
┌──────────── スタンプカード ─── 4/7〜4/13 ─┐
│                                             │
│   ┌─ ─ ─┐  ┌─ ─ ─┐  ┌─ ─ ─┐            │  ← 上段3スロット
│   │ 🎋  │  │ 🌟  │  │  3  │            │     (画像 or 番号)
│   └─ ─ ─┘  └─ ─ ─┘  └─ ─ ─┘            │
│        ┌─ ─ ─┐  ┌─ ─ ─┐                 │  ← 下段2スロット
│        │  4  │  │  5  │                 │     (staggered)
│        └─ ─ ─┘  └─ ─ ─┘                 │
│                                             │
│   ████████████░░░░░░░░  2/5              │  ← 進捗バー
│                                             │
│   ✨ あと3回でコンプリート！                 │
└─────────────────────────────────────────────┘
```

**レイアウト:** 3+2のスタガード配置（リアルなスタンプカード風）

#### ポイント計算

| タイミング | 計算方法 | 例 |
|-----------|---------|-----|
| **押印時（即時）** | `基本5pt × レア度倍率` | N:5pt, R:10pt, SR:20pt, UR:40pt |
| **翌週redeem** | `スタンプ数 × 10pt + コンプリートボーナス` | 5枠コンプリート: 50+50=100pt |

- レア度倍率: N=1, R=2, SR=4, UR=8
- コンプリートボーナス: 50pt（5/5達成時のみ）
- redeem時はレア度**無関係**（スタンプ個数のみ）

#### 押印オーバーレイ演出（StampPressOverlay）
```
┌─────────────────────────────────┐
│                                 │
│     今週 3回目！                │
│                                 │
│  [🎋] [🌟] [   ] [   ] [   ]  │  ← 5スロットミニ表示
│                                 │
│       ┌──────────────┐         │
│       │   🌟 大吉    │         │  ← 当選画像（大きく表示）
│       └──────────────┘         │
│                                 │
│         +20pt                   │  ← 即時ポイント
│     3にちれんぞく！              │
│     あと2回でコンプリート！       │
│                                 │
│     [ やったね！ ]               │
│                                 │

```

| フェーズ | 演出 | 時間 |
|---------|------|------|
| card | スタンプカードミニ表示、今日のスロットがパルス | 0-0.4s |
| press | スタンプ画像が大きくバウンス登場、SR/URはパーティクル | 0.4-1.2s |
| points | ポイント表示、連続日数、コンプリート状況 | 1.2s〜 |
| weekly | 前週の自動redeem結果（月曜のみ） | ユーザー操作後 |

### 4.9 おみくじスタンプ演出（OmikujiStampOverlay）

ログインボーナス受取時に表示されるおみくじ風スタンプ演出オーバーレイ。

**Props:**
- `open: boolean` — 表示状態
- `rank: string` — おみくじランク（大大吉/大吉/中吉/小吉/吉/末吉）
- `totalPoints: number` — 獲得ポイント
- `multiplier: number` — 倍率（ロイヤルティ階層による）
- `consecutiveDays: number` — 連続ログイン日数

**演出フェーズ:**
1. `shake` — おみくじ箱の振動アニメーション
2. `result` — ランク表示（大吉以上はパーティクル＋グロー）
3. `stamp` — スタンプカードへの押印アニメーション

**ランク別演出:**
| ランク | 色 | エフェクト |
|--------|------|----------|
| 大大吉(UR) | ゴールド/パープル | パーティクル+グロー+虹 |
| 大吉(SR) | ゴールド | パーティクル+グロー |
| 中吉(R) | インディゴ | なし |
| 小吉(R) | ティール | なし |
| 吉(N) | エメラルド | なし |
| 末吉(N) | アンバー | なし |

**コンポーネント:** `src/lib/ui/components/OmikujiStampOverlay.svelte`

**プラン別の送信可否 (#772):**

| 種別 | free | standard | family |
|-----|------|----------|--------|
| スタンプ送信 | o | o | o |
| ひとことメッセージ（自由テキスト） | - | - | o |

### 4.x ダイアログ状態管理（DialogFSM）

子供ホーム画面では、ページロード時に複数のダイアログ（ログインボーナス、コンボボーナス、スタンプ押印、特別報酬、親からのメッセージ等）が同時にトリガーされうる。これらを安全に制御するため、単一の FSM で管理する。

**ファイル**: `src/lib/features/child-home/dialog-state-machine.ts`

| 概念 | 説明 |
|------|------|
| `current` | 現在表示中のダイアログ型（'idle' で非表示） |
| `queue` | 待機中ダイアログの FIFO キュー |
| `processed` | 既に処理済みのダイアログキー（`${type}:${id}` 形式） |
| `PRIORITY_ORDER` | 自動トリガー時の優先度: adventure > stampPress > specialReward > parentMessage > birthday > monthlyReward > siblingCheer > celebration |

**動作フロー**:
1. ページロード → `$effect` → `fsm.onDataLoad(triggers)` で自動トリガー対象を優先度順にエンキュー
2. 最初のダイアログを即座に `current` に設定、残りは `queue` へ
3. ダイアログ閉じ → `fsm.close()` → `queue` の先頭を `current` に
4. `processed` set で同一ダイアログの再トリガーを防止
5. birthday は自動トリガー対象外（バナークリックでのみ `fsm.transition('birthday', ...)` で開く）

**参照**: ADR-0002（ダイアログキュー必須）

### 9.x プラン情報の parent ↔ child データフロー (#789)

child UI にもプラン依存機能（きょうだいランキング、家族向け特別報酬表示、自由記述メッセージ等）が存在するため、`planTier` / `planLimits` / `isPremium` を child UI 配下で参照可能にする。

**解決の所在（単一責任）**:

- `src/routes/(child)/+layout.server.ts` の `load` で以下を **1 回だけ** 解決する:
  - `planTier` ← `resolveFullPlanTier(tenantId, licenseStatus, plan)`
  - `planLimits` ← `getPlanLimits(planTier)`
  - `isPremium` ← `isPaidTier(planTier)`
- 各 `+page.server.ts` / `+page.svelte` は SvelteKit の `parent()` / `$props().data` を通じて上記 3 値を参照する
- **子ページで `resolveFullPlanTier` / `getPlanLimits` を再度呼ぶのは禁止**（重複 DB 参照 + 契約のドリフト源）
  - 例外: フォーム送信 action（`+page.server.ts` の `actions`）は `parent()` にアクセスできないため、action 内で再度解決する

**データフロー**:

```
hooks.server.ts
  └─ event.locals.context { tenantId, licenseStatus, plan }
       │
       ↓
(child)/+layout.server.ts  ← プラン解決の単一責任
  │  resolveFullPlanTier → planTier
  │  getPlanLimits(planTier) → planLimits
  │  isPaidTier(planTier)   → isPremium
  │
  └─ return { child, balance, planTier, planLimits, isPremium, ... }
       │
       ↓ parent() / $props().data
(child)/[uiMode]/home/+page.server.ts
  │  const parentData = await parent();
  │  if (parentData.planLimits.canSiblingRanking) { ... }
  │
  └─ return { siblingRanking, ... }
       │
       ↓ $props().data
(child)/[uiMode]/home/+page.svelte  ← data.planLimits で表示制御
```

**child UI 側でのプラン依存機能の表示制御**:

- きょうだいランキング: `data.planLimits.canSiblingRanking` が `false` の場合は非表示
- 自由記述おうえんメッセージ: `data.planLimits.canFreeTextMessage` が `false` の場合は非表示
- カスタム特別報酬: `data.planLimits.canCustomReward` が `false` の場合は非表示
- プレミアムテーマ等: `data.isPremium` で分岐

## 10. プラン/トライアル UI パターン (#743)

有料機能の disabled 表示・トライアル文脈・アップグレード導線を、画面ごとのアドホック実装ではなく **共通コンポーネントと固定パターン** で実現する。新しい有料機能を追加するときは、本セクションの 1 / 2 / 3 のいずれかに沿って実装すること。

### 10.1 プラン状態 × コンポーネント対応表

| プラン状態 | license plan | trial 状態 | resolvePlanTier 結果 | 主要表示 | 主要 CTA |
|-----------|--------------|-----------|---------------------|---------|---------|
| 完全 free（未使用） | `free` | `not-started` | `free` | TrialBanner（not-started） | 「7日間 無料で試す」 |
| トライアル中 | `free` | `active` | `standard` or `family` | TrialBanner（active） + PlanStatusCard（trial-badge 付き） | 「本契約する」 |
| トライアル終了 free | `free` | `expired` | `free` | TrialBanner（expired） + PlanStatusCard（free） | 「⭐ アップグレード」 |
| standard 契約 | `standard` | n/a | `standard` | PlanStatusCard（standard） + 初回 PremiumWelcome | 「⭐⭐ ファミリーへ」 / 「プランの詳細」 |
| family 契約 | `family` | n/a | `family` | PlanStatusCard（family） + 初回 PremiumWelcome | 表示なし（最上位） |

> **注**: トライアル中は `resolvePlanTier` が `standard`/`family` を返すため、機能制限ロジック（`getPlanLimits`）は契約済みと同じ挙動になる。一方で license plan は `free` のままなので、UI 側は **PlanStatusCard の `trialStatus` プロップ** で「今使えている理由」を明示する必要がある（#730）。

### 10.2 disabled 表示パターン（共通原則）

free/standard ユーザーから有料機能に触れたとき、**機能の存在は示し、実行は不可、アップセル CTA を表示する** のが原則。隠す（hidden）のも、403 を踏ませるのも禁止。

| パターン | コンポーネント | 用途 | 例 |
|---------|--------------|------|---|
| **A. ボタン disabled + Badge** | `FeatureGate display="inline" buttonLabel="..."` | リスト内・カード末尾の単発操作 | 「クラウドエクスポート」ボタン（free） |
| **B. セクションオーバーレイ + Badge** | `FeatureGate display="section"` | パネル・カード全体（中身をぼかし表示） | 「AI提案」パネル（free） |
| **C. カード全体置き換え** | 個別実装（PremiumBadge + CTA） | データ管理カード等の重要セクション | 設定画面のクラウド共有カード（#773） |

実装ルール:

- 既存有料機能を追加・改修するときは **必ず `FeatureGate` を経由する**。`{#if planTier !== 'free'}` の生分岐をコンポーネント内に書かない。
- バックエンドでは別途 `requirePaidTier()` 等で 403 を返す。フロントの `FeatureGate` は **二重防御** であって唯一の砦ではない。
- A/B どちらを選ぶかは「ユーザーが押せる単位」で決める。1 ボタン = A、1 機能セット = B。

### 10.3 TrialBanner 仕様

- 配置: admin ホーム最上部（`AdminHome.svelte`）
- 状態: `not-started` / `active` / `urgent`（残り 1 日以下） / `expired` の 4 つ
- 表示条件:

| 状態 | isTrialActive | trialUsed | planTier | 表示 |
|------|--------------|-----------|----------|------|
| not-started | false | false | free | 🎁「7日間 無料で試す」 |
| active | true | true | free→standard/family | ⭐ または ⏰（urgent）「無料体験中」 |
| urgent | true | true | free→standard/family | ⏰「無料体験は明日で終了します」 |
| expired | false | true | free | 📦「無料体験が終了しました」 |
| 非表示 | - | - | standard / family（契約済み） | （TrialBanner は表示しない） |

CTA 文言（変更時は本表＋テストを同時更新）:

- not-started → `7日間 無料で試す` / 送信中は `開始中...`
- active / urgent → `プランを見る`（リンク先 `/admin/license`）
- expired → `⭐ アップグレード`（リンク先 `/admin/license`）

`data-testid` 規約: `trial-banner-not-started` / `trial-banner-start-button` / `trial-banner-active-cta` / `trial-banner-expired-cta`。

### 10.4 PlanStatusCard 仕様

- 配置: admin ホーム上部、TrialBanner の下
- 表示要素:
  - プラン名 + アイコン（free=なし / standard=⭐ / family=⭐⭐）
  - トライアル中バッジ（`isOnTrial` のとき）
  - 統計: カスタム活動 `n/max`、こども `n/max`、データ保持 `n日間 or 無制限`
- CTA:

| 状態 | CTA |
|------|-----|
| free | `⭐ スタンダードにアップグレード`（単一） |
| standard | `プランの詳細` + `⭐⭐ ファミリーへ`（2 列） |
| family | CTA なし |
| 任意プラン × トライアル中 | `本契約する` + 「{tier}の全機能を体験中です。…」の説明 |

`data-testid` 規約: `plan-status-trial-badge` / `plan-status-trial-note` / `plan-status-trial-cta` / `plan-status-free-cta`。

### 10.5 PremiumWelcome 仕様

- 種別: 中央モーダル（オーバーレイ）
- 表示条件: `isPaidTier(planTier) === true` かつ settings の `premium_welcome_shown !== 'true'`
- 表示内容: アイコン（standard=⭐ / family=⭐⭐）、解放された機能リスト（`getUnlockedFeatures(planTier)`）、CTA 「さっそく始める →」
- dismiss 動作: CTA 押下で `dismissPremiumWelcome` action（`POST ?/dismissPremiumWelcome`）→ `premium_welcome_shown` settings に `'true'` を保存 → モーダルを閉じる
- 二重表示防止: settings は tenant スコープ。一度 dismiss すれば同 tenant のすべての parent には再表示されない

### 10.6 アップグレード/ダウングレードフロー

詳細な状態遷移は [`docs/design/diagrams/plan-trial-flow.drawio`](diagrams/plan-trial-flow.drawio) を参照。

**アップグレードの主な経路**:

1. TrialBanner（not-started）→ 「7日間 無料で試す」 → `?/startTrial` action → トライアル開始 → 即座に有料機能解放（ただし license plan は free のまま）
2. TrialBanner / PlanStatusCard / FeatureGate の各 CTA → `/admin/license` → Stripe Checkout → webhook で license plan 更新 → 次回 admin ロード時に PremiumWelcome 表示

**ダウングレード**: 詳細フローと超過リソース処理の確認画面は #738 / #754 で別途設計（現状は `/admin/license` から Stripe Customer Portal を経由）。

**画面遷移概略**:

```
admin/+layout.server.ts (load)
  ├─ resolveFullPlanTier → planTier
  ├─ getTrialStatus → trialStatus
  │
admin/+page.server.ts (load) — parent() で planTier を継承
  ├─ isPaidTier(planTier) → isPaid
  └─ if isPaid && premium_welcome_shown !== 'true' → showPremiumWelcome=true
       │
       ↓
AdminHome.svelte
  ├─ <TrialBanner ...trialStatus />            … free 系のみ
  ├─ <PlanStatusCard ...trialStatus />         … 全プラン
  ├─ ... 通常コンテンツ ...
  └─ {#if welcomeVisible}<PremiumWelcome />    … 初回 paid のみ
        └─ onDismiss → fetch ?/dismissPremiumWelcome → welcomeVisible=false
```

## 12. デモモード（実行モード）仕様 (#1180 / ADR-0039)

### 12.1 目的と位置づけ

デモモードは**本番ルート上で駆動する実行モード**である。`src/routes/demo/**` という別ルートツリーは ADR-0039 で廃止された。#296 / #1129 / #1147 / #1180 の 4 回連続して発生した「デモと本番のデザイン乖離」を構造的に解消することが目的。

- デモと本番は**同一コードパス**を通る（同期漏れが発生不可能）
- DB は触らず in-memory シードで動作（Linear demo workspace 方式）
- reload で fresh start

### 12.2 判定フロー

| 段階 | 入力 | 出力 |
|------|------|------|
| 入口 | URL に `?mode=demo` クエリ | cookie `gq_demo=1`（有効期限 4h、httpOnly、SameSite=Lax）を発行 |
| 各リクエスト | cookie `gq_demo=1` の存在 | `event.locals.isDemo = true` |
| 退出 | `/demo/exit` 到達 | cookie 削除 → 本番コンテキストに復帰 |

`event.locals.isDemo` が server-side の Single Source of Truth。`+page.server.ts` / `+layout.server.ts` は `locals.isDemo` を `data.isDemo` として client に配布し、クライアントは `$page.data.isDemo` を参照する。

### 12.3 書き込みガード

`hooks.server.ts` で一元処理する。

- `locals.isDemo === true` かつ HTTP メソッドが `POST / PUT / PATCH / DELETE` のとき、`200 { ok: true, demo: true }` を返して no-op 化
- UI 側コード（form actions / fetch）は無改変のまま「成功扱い」で進む（Stripe test-mode と同じ）
- 例外許可リスト: `/api/feedback`, `/api/demo/exit`, `/api/health`, `/api/demo-analytics`

**403 ではなく 200 を返す理由**: エラーダイアログ表示は子供向け UX を損なう。見た目は「やった感」を維持し、リロードで消えるので副作用もない。

### 12.4 デモコンテキスト（in-memory）

`$lib/server/demo/build-demo-context.ts` がリクエスト毎に合成する。

| フィールド | 値 |
|-----------|-----|
| `tenantId` | `'demo'` |
| `role` | `'owner'` |
| `licenseStatus` | `ACTIVE` |
| `plan` | `?plan=` クエリ優先、既定 `family`（cookie 永続化は既存ロジック踏襲） |
| seed データ | `src/lib/server/demo/demo-data.ts` の 5 年齢モードプリセットを流用 |

サービス層 (`$lib/server/services/*`) は `isDemo` を一切判定しない。サービスが DB 呼び出しをする前に hooks が no-op で返すため、サービス側の分岐は構造的に不要になる。

### 12.5 オーバーレイ（DemoBanner）

上部固定バナー + 下部の `DemoGuideBar`（既存）の 2 段構成。

- **DemoBanner（新規、上部）**: Primitives `Alert variant="trial"` を流用した sticky header。`$page.data.isDemo === true` のとき表示。退出ボタン（`/demo/exit`）と「ほんとうに始める」CTA（`/auth/signup`）を常設
- **DemoGuideBar（既存、下部）**: 7 ステップの誘導。既存のまま `+layout.svelte` から条件付き mount
- 年齢モード対応: `fontScale` に追従。baby / preschool はひらがな併記

**禁忌**:
- hex 直書き禁止（§2）
- 文言は `labels.ts` の `DEMO_LABELS`（新設）経由で参照（ADR-0009 準拠）
- `?screenshot=1` / `hideDemoOverlays` 方式は #1181 で導入されたが本 ADR で撤去

### 12.6 LP スクリーンショット撮影

`scripts/take-lp-screenshots.mjs` は `?mode=demo` 付きの本番 URL を撮影する。

- 旧撮影パス `site/screenshots/*.webp` は**保持**（履歴参照用）
- 新撮影パス `site/screenshots/v2/*.webp` に新たに 29 枚配置
- `site/index.html` / `site/pamphlet.html` の `<img>` src を新パスに一括更新
- Playwright は cookie でデモ状態を起動し、localStorage フラグでバナー非表示化（専用ヘルパー `tests/e2e/helpers/demo-screenshot.ts`）

### 12.7 廃止する資産

| 廃止対象 | 移行先 |
|---------|-------|
| `src/routes/demo/**` 全 4,722 行 | 本番ルート `(child)/[uiMode=uiMode]/**` + `(parent)/**` に統合 |
| `src/lib/server/demo/demo-service.ts` のサービス DI 切替 | hooks の no-op ガードに置換 |
| `?screenshot=1` / `hideDemoOverlays` (#1181) | E2E 専用 localStorage ヘルパー |
| CLAUDE.md 並行実装チェックリスト「デモ版 → 本番版の同期」 | 項目ごと削除可能 |

### 12.8 CI 禁則

`scripts/check-no-demo-route-duplication.mjs` を追加。`src/routes/demo/` 配下にファイルが再作成されたら CI を fail。`.github/workflows/ci.yml` に組み込む。

### 12.9 旧 URL 対応

`src/lib/server/routing/legacy-url-map.ts` に以下エントリを追加（308）。

| from | to |
|------|-----|
| `/demo` | `/?mode=demo` |
| `/demo/signup` | `/auth/signup?mode=demo` |
| `/demo/admin` | `/admin?mode=demo` |
| `/demo/admin/*` | `/admin/*?mode=demo` |
| `/demo/[uiMode]/home` 等 | `/[uiMode]/home?mode=demo` |

既存の `/demo/kinder` → `/demo/preschool` エントリは削除（統合後は `/preschool` 側に委譲）。

## §13 baby モード — 親向け準備モード UI（#1300 / ADR-0011）

### 概要

## §14 ごほうびショップ陳列原則（#1336）

### 14.1 概要

ごほうびショップ（L3 経済層）の陳列方針を定義する。詳細なゲーミフィケーション設計は [`26-ゲーミフィケーション設計書.md §12`](26-ゲーミフィケーション設計書.md#12-ごほうびショップl3-経済層) を参照。

### 14.2 陳列原則

- 親が **プリセット一覧から選んで追加** + **カスタム追加**（ゼロから作成）の 2 経路
- プリセットは **3 系統** に視覚的に分類される（14.3 参照）
- ポイント単価（X pt 交換）は **親が自由設定**（プリセットは目安値を提示するのみ）
- ショップは「親が許可した欲望」と「子の本物の欲望」の交点（`26-ゲーミフィケーション設計書.md §2.2` 参照）

### 14.3 3 系統の定義と意義

| 系統 | `shopCategory` | 説明 | 差別化意義 |
|------|--------------|------|-----------|
| 実物 | `physical` | お菓子・文房具・おもちゃ等の現物報酬 | 直接的な物質報酬。子供が「欲しい」と感じる最も直感的な動機源 |
| お金 | `money` | お小遣い変換（金額相当のポイント交換） | 経済リテラシー教育への橋渡し。「努力→お金」の実感を年齢に応じて自然に伝える |
| 特権 | `privilege` | 夜更かし・ゲーム時間延長・好きなメニューリクエスト等 | BusyKid/Greenlight 等の海外サービスとの差別化の核。日本家庭の「がんばり報酬」文化に合致し、モノではなく「体験・自由」への欲求を刺激する |

**「特権」系統が差別化の核である理由**:
- 日本の子育て家庭では、モノより「許可」が価値を持つ場面が多い（夜更かし・ゲーム等）
- 親が「OK を出すことそのもの」をごほうびにできる点が、単純なポイント型サービスとの差異
- 子供は「ポイントを貯めれば自分の要望が通る」という自己効力感を得る

### 14.4 プリセット構成と系統マッピング

プリセットは `src/lib/data/preset-rewards.ts` で管理。各プリセットに `shopCategory` フィールドが付与されている。

| グループ名 | `shopCategory` | 主な内容 |
|-----------|--------------|---------|
| もの | `physical` | シール・おかし・文房具・本・おもちゃ・マンガ |
| たいけん | `privilege` | おでかけ・がいしょく・えいが・ゲーム時間・YouTube時間 |
| おこづかい | `money` | おこづかい 100円〜1000円 |
| とくべつ | `privilege` / `money` | 夜更かし券・朝寝坊券・家族でゲーム・メニューリクエスト・旅行貯金 |

> **注**: UI 上での 3 系統のフィルタリング・タブ分類は将来の実装課題。現時点では `shopCategory` はデータ定義のみ（Committed）。

## §15 ごほうびショップ 親子連携フロー（#1337）

> ゲーミフィケーション設計書 §12（ごほうびショップ L3 経済層）の UI 実装仕様。
> DB スキーマ: `docs/design/08-データベース設計書.md` §3 `reward_redemption_requests`  
> API: `docs/design/07-API設計書.md` §3.26

### §15.1 設計背景・目的

子供が「欲しいごほうびを親に申請する」体験を中心に、親が承認/却下するワークフローを提供する。
現行の `special_rewards` テーブルは保護者が子に**直接付与**するフローのみで、子主体の申請機能がない。
本フローで「子が選んで親が承認する」L3 経済層の完成形を実現する。

**Anti-engagement 整合確認 (ADR-0012)**:
- 申請中バッジは最小表示（「受け取り待ち」1 行のみ）— 複数申請を煽る導線なし
- 子供側専用タブは設けない — ホーム画面の申請済みアイテムに badge を乗せるのみ
- 「早く申請しよう」「もっと交換しよう」等の能動的 CTA は一切使用しない
- 完了通知は次回アクセス時に 1 回のみ表示（繰り返しループなし）

---

### §15.2 フロー全体図

```
[子供] ショップ画面でアイテム閲覧
    ↓ ポイント充足で「こうかんする」ボタン活性化
[子供] 「こうかんする」タップ → 確認 Dialog
    ↓ 「はい」
[システム] 申請レコード作成 (status: pending_parent_approval)
           ポイントはこの時点では減算しない
    ↓
[子供ホーム] 該当アイテムに「受け取り待ち」バッジ表示
    ↓ 非同期（親が管理画面を確認するまで待機）
[親] 管理画面 > ごほうび > 申請一覧タブで確認
    ↓
[親] 承認 or 却下 を選択
    ↓（承認）                        ↓（却下）
[システム]                        [親] 却下理由メモ入力（任意）
  ポイント減算                         ↓
  point_ledger 記録               [システム] 申請 status → rejected
  申請 status → approved               ↓
    ↓                            [子供] 次回アクセス時に却下通知
[子供] 次回アクセス時に              「理由: ○○」メモ表示
  「交換完了！」overlay 表示         ポイント減算なし
  コンテンツ: タイトル + アイコン
```

---

### §15.3 子供側 UI

#### §15.3.1 ショップ陳列画面（既存）

既存の `SpecialRewardOverlay.svelte` / ショップ画面に「こうかんする」ボタンを追加する。

| 要素 | 仕様 |
|------|------|
| 交換ボタン活性条件 | `child.points >= reward.requiredPoints && request.status !== 'pending_parent_approval'` |
| 二重申請防止 | 同一 reward_id で `pending_parent_approval` が存在する場合はボタンを「申請中」バッジに置換 |
| 確認 Dialog | 「○○ と こうかんする？（X ポイント）」+ 「はい」/「やめる」の 2 択 |
| ポイント不足時 | ボタン disabled + 「あと X ポイント」プログレスバー（既存仕様を維持） |

#### §15.3.2 申請中バッジ

ショップ陳列アイテムに状態バッジを重ねる。

| status | バッジ表示 |
|--------|-----------|
| `pending_parent_approval` | 「受け取り待ち」（amber / muted） |
| `approved` | 「こうかん済み」（success / subdued） |
| `rejected` | 「却下された」（muted） |

- バッジは 1 行テキスト + `Badge.svelte` primitive — 自作 badge 禁止
- バッジ表示は最大 1 件のみ（最新申請の状態を表示）

#### §15.3.3 完了 / 却下 通知 overlay

次回アクセス時（ホーム画面ロード時）に `shown_to_child_at` が NULL の承認/却下申請があれば表示。

| 通知種別 | 表示内容 |
|---------|---------|
| 承認 | 「{タイトル} もらったよ！」+ アイコン（アニメ付き） |
| 却下 | 「{タイトル} は ちょっとまってね」+ 親メモ（存在する場合のみ） |

- コンポーネント: `RedemptionResultOverlay.svelte`（新設）
- 表示後即座に `shown_to_child_at` を更新（1 回のみ）
- `Dialog.svelte` primitive を使用

---

### §15.4 親側 UI

#### §15.4.1 申請一覧タブ

`/admin/rewards` ページに「申請」タブを追加する（既存「ごほうび一覧」タブと並列）。

| 要素 | 仕様 |
|------|------|
| 表示対象 | `status = 'pending_parent_approval'` の申請、子供名 + アイテム名 + 申請日 |
| 空状態 | 「申請はありません」（Alert.svelte info variant） |
| 承認ボタン | 「承認して渡した」— タップ後に確認不要（ポイント減算は不可逆のため 1 タップ確認は有用だが UX を優先してシンプルに） |
| 却下ボタン | 「却下する」→ 却下理由入力フォーム（任意 textarea、最大 100 文字）→ 「確定」 |
| 既決申請 | 「承認済み」「却下済み」タブに履歴表示（`status = 'approved' OR 'rejected'`、最新 30 件） |

#### §15.4.2 承認 / 却下 操作 UI

- 承認: `Button.svelte` (success variant) — `「承認して渡した」`
- 却下: `Button.svelte` (danger variant) — `「却下する」`
- 却下理由: `FormField.svelte` type="textarea" rows=3 （任意）
- 操作後は即時リスト更新（optimistic update ではなく画面再ロード）

---

### §15.5 エッジケース処理方針

| ケース | 処理方針 |
|--------|---------|
| **30 日 pending 期限切れ** | 日次 cron (`/api/cron/expire-redemptions`) が `requested_at < NOW() - 30d` かつ `status = 'pending_parent_approval'` を `expired` に更新。子供への通知はしない（静かな失効）。cron 実装パターン: `verifyCronAuth` 使用（`07-API設計書.md §5` 参照） |
| **並列申請（数量制限なし）** | 複数の異なる reward_id への申請は許可。同一 reward_id への複数申請は禁止（API 側で `UNIQUE (child_id, reward_id, status='pending_parent_approval')` 相当のチェック） |
| **親がごほうびを削除** | `special_rewards` 削除 API で `reward_redemption_requests.status = 'pending_parent_approval'` が存在する場合は `409 Conflict` を返し削除を禁止。親 UI には「申請中のため削除できません。先に却下してください」メッセージを表示 |
| **子供が削除される** | `children` 削除時に CASCADE DELETE で申請レコードも削除 |
| **ポイント不足後の申請（レース）** | API 側でポイント残高を再確認。不足の場合は `400 INSUFFICIENT_POINTS` を返す |

## §16 自動スリープ & 使用時間可視化（#1292）

### 16.1 設計背景

Anti-engagement 原則（ADR-0012）に基づき、子供が際限なくアプリを使い続けることを防ぐため、15分連続使用後に自動スリープを行う仕組みを実装。保護者が使用状況を把握できるよう、親ダッシュボードに「本日の使用時間」を表示する。

### 16.2 Phase 1: 15分自動スリープ（実装済み）

#### 対象画面

- 子供ホームレイアウト `src/routes/(child)/+layout.svelte`
- 対象年齢モード: preschool / elementary / junior / senior（**baby モード除外**）

#### 動作仕様

| 項目 | 仕様 |
|------|------|
| 連続使用上限 | 15分（900秒） |
| `/battle` パスのグレース期間 | +2分（合計17分） |
| 非アクティブ判定 | 1分間操作なし → タイマーリセット |
| スリープ時の遷移先 | `/switch`（子供切替画面） |
| baby モード | 自動スリープ対象外（baby = 親向け準備モード） |

> **注**: Issue #1292 AC には遷移先 `/select` と記載されているが、`/select` は廃止済みのため正しい遷移先は `/switch`（子供切替画面）を使用している。

#### タイマーリセット条件

- `visibilitychange`（タブ非表示→表示）でリセット
- `pointermove` / `pointerdown` / `keydown` などユーザーインタラクション発生時にリセット

#### ロジック実装

- `src/lib/features/auto-sleep.ts` に `createAutoSleepTimer()` ファクトリ関数として分離
- `getHidden` 依存注入により Vitest 単体テスト可能な設計

### 16.3 Phase 1: 親ダッシュボード「本日の使用時間」表示（実装済み）

#### 表示場所

- `src/lib/features/admin/components/AdminHome.svelte`
- `data-testid="today-usage-section"` で識別可能

#### 表示条件

- `usage_logs` テーブルに当日のログが存在する場合のみ表示
- デモモードでは非表示（`!isDemo` 条件）

#### 表示仕様

| 要素 | 仕様 |
|------|------|
| セクションタイトル | 「⏱️ 本日の使用時間」 |
| 子供ごとのカード | `Card.svelte` プリミティブ使用 |
| プログレスバー | `role="progressbar"` + `aria-valuenow/min/max` 付与 |
| 最大表示時間 | 15分（900秒）= 100% |
| カード accessibility | `role="group"` + `aria-label="{childName}の本日使用時間"` |
| インラインスタイル | `style:width` 動的値のみ（DESIGN.md §9 許容範囲） |

#### データ取得

- サービス関数: `findTodayUsageLogs()` — 全子供の当日ログを1クエリで一括取得（N+1なし）
- DBインデックス: `idx_usage_logs_child_date`（子供×日付の複合インデックス）

### 16.4 Phase 2: 週次使用時間 bar chart（別 Issue で対応予定）

`getWeeklyUsageSummary()` サービス関数は Phase 1 で実装済みだが、週次グラフ UI は **別 Issue** で実装予定。

- 実装済み: `src/lib/server/services/usage-log-service.ts` の `getWeeklyUsageSummary()`
- 未実装: 週次累計 bar chart UI コンポーネント（管理画面への組み込み）
- 対応 Issue: Phase 2 実装 Issue（`06-UI設計書.md` §16.4 参照）にて追記予定
