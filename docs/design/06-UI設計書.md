## 1. デザイン原則

### 最優先原則: 4歳児が使える

- **文字に頼らない**: アイコン・色・形でナビゲーション
- **大きなタップターゲット**: 最低 64x64px（推奨 80x80px）
- **即座のフィードバック**: タップしたら即反応（色変化、スケールアニメーション）
- **シンプルな遷移**: 最大2タップで目的達成
- **明確な視覚的区別**: 子供別のテーマで「自分のページ」を実感

### 4.6 おやカギコード認証画面 (S-07) + PIN gate modal (EPIC #2310 / #2353 / #2992)

**初回は作る・既存は入る** (#2992、EPIC #2990): PIN 未設定テナントには gate modal を login でなく**新規作成 (入力→確認の 2 段) フロー**で表示する (Apple Screen Time / Google Family Link 同型)。parent-gate の verify 経路では既定 PIN 照合を行わない (`verifyPin` は未設定時 `PIN_NOT_SET`)。`DEFAULT_PIN` (`5086`、「がんばり」語呂合わせ) は legacy local 経路 (`/login` の `login()` / `changePin()` の現コード照合、#1360) のみで有効。

#### 4.6.1 PIN gate modal (S-07a) — `/switch` inline

EPIC #2310 で `/admin/*` middleware が未認証時 `/switch?pinRequired=1&next=…` redirect → `/switch` page で auto-open する Dialog 内 PinInput。`isPinConfigured(tenantId)` (load で `pinConfigured` として供給) で**作成モード / 入力モードに分岐**する。

| 項目 | 仕様 |
|---|---|
| trigger 1 | `pinRequired=1` query (middleware redirect) で初期 `pinModalOpen=true` |
| trigger 2 | `/switch` の「🔒 保護者の見守り画面」link click (#2353 漢字化、AC4) |
| モード分岐 (#2992) | `pinConfigured === false` → **作成モード** (`data-testid="parent-gate-create"`、`data-step="enter"→"confirm"`): 1 段目入力 → 確認入力 → 一致で `POST /api/v1/parent-gate/setup` → session cookie 発行 → next へ。不一致は `gateCreateMismatch` エラー + 1 段目からやり直し。`true` → 従来の入力モード |
| 入力 UI | `PinInput` primitive (4 桁、mask: true) |
| 送信 (入力モード) | `POST /api/v1/parent-gate/verify { pin }` → `gq_parent_session` 署名 cookie 発行。未設定 tenant への verify は `PIN_NOT_SET` (作成モードへの race guard) |
| 送信 (作成モード) | `POST /api/v1/parent-gate/setup { pin }` — **未設定 tenant のみ許可** (設定済みへは 403 `ALREADY_CONFIGURED` = 子供による上書き穴の防止。変更は現 PIN 確認付き changePin、忘れは reset 経路)。成功で verify と同じ session cookie 発行。rate limit 20/15min per IP |
| 失敗時 | `pinError` 表示 + `pinInputKey += 1` で入力欄リセット |
| ロックアウト時 (#2991) | 連続失敗でロックされた際、エラーに**ロック解除の絶対時刻**「HH:MM まで待ってから」を表示 (`OYAKAGI_LABELS.gateLockedUntilNotice`)。サーバが返す `lockedUntil` を `toLocaleTimeString('ja-JP',{hour,minute})` で整形。時刻不明時のみ `OYAKAGI_LABELS.lockedError` (時刻なし) を fallback。秒カウントダウンは不採用 (NIST SP 800-63B / iOS は残り時間明示、NN/g #1 visibility) |
| 初期 PIN ヒント | **非表示** (#2353 設計欠陥 5、AC6: 子供が見て即入力できる脆弱性)。#2992 以降は未設定者が自分で作成するため既定 PIN ヒント自体が不要。setup 完了画面 / onboarding dialog の案内も「初めて入るときに作成」型 (`SETUP_COMPLETE_LABELS.pinHintSuffix` / `PIN_GATE_ONBOARDING_LABELS.dialogPinHint`) |
| 「PINを忘れた方」link (#2993) | **入力モード + cognito identity のみ** modal 下部に表示 (作成モードは PIN 未存在 / local の救済は operator reset #2994 のため非表示)。click → `/auth/reset-pin` 遷移 |

#### 4.6.2 PIN reset 画面 (S-07b) — `/auth/reset-pin` (#2993、cognito 専用)

PIN 忘れ救済はアカウントパスワード再入力方式 (Apple Screen Time 同型)。email はセッション既知 (`locals.identity.email`) のため手入力なし・メール往復なし。

- 1 画面構成: ログイン中アカウント表示 (read-only) + アカウントパスワード (`FormField type="password"`) + 新 PIN (`PinInput` 4 桁 mask) → 送信
- 送信: `POST /api/v1/parent-gate/reset-verified { password, newPin }` → パスワード re-auth (Cognito InitiateAuth / dev は DEV_USERS 照合) → `setupPin` (上書き) + parent session cookie 発行
- success 時: 「再設定が完了しました」 + 「ご家族の見守り画面へ」CTA (`/admin` 遷移、session 発行済のため再入力不要)
- cognito identity 以外 (local / anonymous) は load で `/auth/login` redirect (local の救済は operator reset #2994)
- 旧 `/auth/forgot-pin` は `LEGACY_URL_MAP` で本画面へ 308 redirect

#### 4.13.1 ページ別オンデマンドガイド（v2 PageGuideOverlay、ヘッダー `?` ボタン）

管理画面の各ページ固有のヘルプ。AdminLayout ヘッダーの `?`（❓）ボタンを押すと、その時開いているページの `PageGuide` 定義（`what` / `how` / `goal` の三部構成ステップ）を `PageGuideOverlay` でスポットライト表示する。ユーザーマニュアルを別途用意せずに、画面内で機能説明を完結させる手段（#2375 で v1 PageHelpButton から v2 へ統一）。

**positioning は driver.js（MIT）に委譲（#2926）:** バブルの配置・viewport 自動調整・scroll-into-view・spotlight backdrop cutout は driver.js が担う。`PageGuideOverlay.svelte` は store の active 状態に追従して driver.js tour を起動するオーケストレータに徹し、手動 positioning（targetRect 計測 / `bottom`/`top`/`left`/`right` 固定配置 / 垂直クランプ / 自前 SVG spotlight）は持たない。3 タブ UI（`PageGuideBubble`）は driver.js popover の `onPopoverRender` で wrapper に `mount()` して流用する。`selector` を省略した step は driver.js が画面中央 modal で表示する。

**構成:**
- `PageGuideOverlay.svelte` — driver.js オーケストレータ（store → driver.js tour 起動 / popover wrapper への PageGuideBubble mount / spotlight ring CSS）
- `PageGuideBubble.svelte` — 吹き出し本体（3 タブ UI、driver.js popover wrapper 内に mount。wrapper が `role="dialog"` + `aria-modal` + `aria-labelledby="page-guide-title"` を持つ）
- `page-guide-store.svelte.ts` — ガイド active / 完了状態管理（Svelte 5 runes、localStorage 永続）。step ナビゲーションは driver.js が SSOT
- `page-guide-registry.ts` — **パス → `_guide.ts` のマッピング SSOT**
- 各ページの `src/routes/(parent)/admin/<page>/_guide.ts` — `PageGuide` 定義（`pageId` / `title` / `icon` / `steps[]`）

**close 操作:** Escape キー・「とじる」ボタン・最終 step の「かんりょう！」で閉じる。backdrop（暗幕）click では閉じない（`overlayClickBehavior` を no-op にし、❓ 連打や誤タップでの不意の dismiss を防ぐ）。

**`?` ボタンの表示条件:** AdminLayout は現在のパスを `getPageGuide(path)` で解決し、ガイドが登録されているページでのみ `?` ボタンを表示する（未登録ページでは `?` 自体が出ない）。したがって **新規 admin ページを追加したら `_guide.ts` を作成し `page-guide-registry.ts` に登録しないと `?` が脱落する**（#2905 = #2294 EPIC で新設された checklists / challenges + status が未登録のまま `?` が見えなくなっていた regression の復旧）。

**登録済みページ（`page-guide-registry.ts` SSOT）:** `/admin` / `/admin/children` / `/admin/activities` / `/admin/rewards` / `/admin/checklists` / `/admin/challenges` / `/admin/status` / `/admin/points` / `/admin/reports` / `/admin/cheer` / `/admin/settings`。

**presence 不変条件（導線インベントリテスト、#2905）:** 全登録ページで「`?` トリガが存在し、click でガイド overlay が開き、Escape で閉じる」を `tests/e2e/admin-page-guide-presence.spec.ts` がループ検証する。新ページの登録漏れ / 登録解除はこの spec が必ず fail させる（M6 構造 invariant）。

**narrative 標準（3 部構成、ADR-0012）:** 各ページの guide は **「①概要 → ②画面の見方 → ③最頻操作」の 3 部構成**に統一する。1 ページの step は **最大 5**（情報過多回避）。`①概要` は `selector` を省略し画面中央 modal でページ全体像を説明する（特定 UI に紐付かない俯瞰のため）。`②画面の見方` / `③最頻操作` は **具体的な `data-tutorial` アンカー**（例: `[data-tutorial="rewards-child-tabs"]` / `[data-tutorial="points-first-balance"]` — いずれも常設の小〜中要素）を target にし、`points-section` のような一覧 section 全体 = 巨大コンテナは target にしない（spotlight cutout が viewport 大半を覆い、バブル配置の余地が物理的に消えるため）。各 step は `what` / `how` / `goal` の 3 フィールドを深く記述する（`page-guide-types.ts`）。

**selector 省略 step の中央 modal 再 positioning:** `①概要` のような `selector` 省略 step は driver.js が画面中央 modal で表示する。driver.js は mount 前の空 wrapper 幅で center 計算するため、Svelte bubble mount 後に wrapper が最終幅（max 360px）へ成長すると左端がずれる。`PageGuideOverlay.svelte` は `selector` 省略 step に限り `requestAnimationFrame` 後に `driver.refresh()` を呼び、最終幅で再 center する（element 紐付き step は driver.js の collision 回避が target 基準で働くため refresh しない）。

**layout 不変条件（#2926）:** 全登録ページ × デスクトップ + モバイル viewport × 各 step で「(a) バブルが対象要素を覆い隠さない（非重複）/ (b) バブルが viewport 内に完全収容（見切れない）/ (c) spotlight が表示中」を `tests/e2e/page-guide-layout-invariant.spec.ts` が機械検証する。手動 positioning 撤去後にレンダリング不具合（重複 / 見切れ / spotlight 不全）が再発するとこの spec が fail させる。(a) 非重複検証の exempt は、**target の上下にも左右にもバブル全体を clear して置く余地が幾何学的に存在しない（横/縦クリアランス物理不在）ケースのみ**（radar / 残高グリッド / viewport 下端付近の薄い要素など、driver.js positioning でも回避できない正当な幾何制約）。`①概要` の中央 modal step は active target が無いため検証対象外。**「巨大 target を常時 exempt にする」分岐は存在しない** — 上記 narrative 3 部構成統一で巨大コンテナ target を撤廃したため、本 exempt が発動するのは正当な幾何制約ケースに限られる。

**コンポーネント:** `src/lib/ui/components/PageGuideOverlay.svelte`, `src/lib/features/admin/components/AdminLayout.svelte`（ヘッダー `?` ボタン配線）

### 4.x ダイアログ状態管理（DialogFSM）

### 4.22 「今日のおやくそく」カード演出統合（ActivityCard `isMust`、#2146）

子供ホーム画面で priority='must' 活動を識別する演出。**#2146 で旧 `MustProgressBar` 専用セクション（#1709-C / #1757）を撤廃し、ActivityCard 自身に統合**した。Anti-engagement 原則（ADR-0012）に従い、モーダル離脱阻害なし・連続演出なし・追加アニメーションなしで完結する設計。

**ファイル**: `src/lib/ui/components/ActivityCard.svelte`（`isMust` prop）
**呼び出し元**: `src/lib/features/child-home/components/ProdDashboardSections.svelte`（本番 + demo 共通 SSOT、demo は #2097 PR-B2 / #2187 で本番 routes に統合済）
**サーバ load**: `tryGrantMustCompletionBonus(childId, today, uiMode, tenantId)` を `+page.server.ts` で引き続き呼び出し、達成状況集計とボーナス冪等付与を担う（API 後方互換維持）。全達成 bonus 通知は `+page.svelte` の `mustToastShown` `$effect` から Toast で行う（旧バー内 pulse 演出に代わる演出経路）。

#### §1 設計背景

#1709-C / #1757 で導入した `MustProgressBar` は専用セクションとしてホーム上部に常駐し、9 件の must 活動が存在するときに「あと 9 こ」表示で画面エリアを大きく占有する問題が #2146 で顕在化（demo.ganbari-quest.com で B-7 kinder-starter pack を import した 902 ひなちゃんで再現）。
PO 既出方針として「must 表現はカード演出に統合し、専用セクションは廃止」が確定していたため、#2146 で実装を完遂する。

#### §2 設計原則

1. **滞在時間最短化**: 専用セクション撤廃で活動カード群の表示エリアを確保。ribbon badge は flow inline で card 上に重畳のみ
2. **冪等性**: 全達成ボーナス機構は server 側（`tryGrantMustCompletionBonus`）で従来通り維持。`point_ledger.type='must_completion_bonus'` を 1 子供 × 1 日に 1 行のみ。同日 2 回目以降の load では `granted=false`
3. **連続演出禁止**: `granted=true` のときに `+page.svelte` の Toast を 1 回だけ演出（モーダルなし）
4. **age tier 共通仕様**: 4 年齢コアモード（preschool/elementary/junior/senior）で同一の ribbon 文言「⭐ おやくそく」を表示。baby は ADR-0011 で親準備モードのため non-baby 共通の ActivityCard に到達しない
5. **labels SSOT 準拠（ADR-0045）**: `UI_COMPONENTS_LABELS.activityCardMustBadge`（'⭐ おやくそく'）/ `activityCardMust`（aria-label サフィックス）。文字列直書き禁止

#### §3 仕様

##### 表示条件（4 年齢モード）

| uiMode | カード badge | 文言 | 全達成 bonus |
|--------|---------|------|------------|
| `baby` | **非表示** | — | 0pt（baby は親準備モードのため BabyHomePage に分岐） |
| `preschool` | priority='must' && !completed で表示 | 「⭐ おやくそく」 | +5pt（server 集計） |
| `elementary` | 同上 | 「⭐ おやくそく」 | +5pt |
| `junior` | 同上 | 「⭐ おやくそく」 | +3pt |
| `senior` | 同上 | 「⭐ おやくそく」 | +3pt |

`M = 0`（must 活動が 1 件も設定されていない）の場合は単純にどの ActivityCard にも ribbon が付かないため UI 上「おやくそく」概念が出現しない（旧 MustProgressBar の mount 判定不要）。

##### Card UI 構成

```
┌──────────────┐
│ ⭐おやくそく  │  ← ribbon (左上、completed 時は非表示)
│   🦷        │
│  はみがき    │  ← gold border (priority='must' のとき)
│              │
└──────────────┘
```

完了時:

```
┌──────────────┐
│              │  ← ribbon 撤去
│   💮         │  ← 既存の completed 演出
│ はみがき     │
│              │
└──────────────┘
```

##### サーバ側集計（tryGrantMustCompletionBonus、API 後方互換維持）

1. `getMustActivitiesToday(childId, today, tenantId)` → `{ logged, total, activities }` を取得
2. `total === 0` または `logged < total` → 付与せず `granted=false` を返却
3. `logged === total && total > 0`:
   - `computeMustCompletionBonus(uiMode, true)` で +5pt or +3pt or 0pt（baby）を算出
   - `countPointLedgerEntriesByTypeAndDate(childId, 'must_completion_bonus', today, tenantId)` が 0 のときのみ `insertPointLedger` で 1 行追加
   - 既に付与済みなら `granted=false` を返す（冪等）

##### Anti-engagement 適合チェック（ADR-0012）

### 8.3 折りたたみ動作

> **#2148 (γ 採用、2026-05-17)**: 子供画面 (`ProdDashboardSections.svelte`) では
> `<CategorySection collapsible={false}>` を強制し、カテゴリヘッダー誤タップで活動グリッドが
> 全消失する不具合を構造的に排除した。`CategorySection.svelte` 自体も `collapsible=false` の
> 場合はヘッダーを `<button>` ではなく `<div>` で描画し、`expanded` を常に true 固定とする
> 二重防御を実装している。
>
> 業界 prior art 7/7 全件 (Khan Academy Kids / Duolingo ABC / ABCmouse 等) で子供画面の
> カテゴリヘッダー誤タップによる全消失は存在しないことを確認済 (Code.org は教師管理画面のみで
> 折りたたみ機能を提供する C2 パターン)。詳細:
> [docs/reference/07-research-child-collapsible-prior-art.md](../reference/07-research-child-collapsible-prior-art.md)
>
> `display-config.ts` の `getDefaultDisplayConfig()` は現状子供画面唯一の利用箇所のため、
> 全年齢で `collapsible: false` を返すよう変更済。親管理画面で将来 CategorySection を利用する
> 場合は、親画面の発火経路で `collapsible={true}` を明示的に渡せば旧仕様の折りたたみが復活する
> （prop と「もっと見る」ボタン経路は温存）。
>
> 以下は親管理画面で `collapsible=true` で使う場合の動作仕様 (子供画面では適用されない):

- `collapsible=true` かつ `itemsPerCategory > 0` かつ実際のアイテム数が上限を超える場合に折りたたみ表示
- `max-height` + `overflow: hidden` で折りたたみ領域を制御
- 下部にグラデーションフェードオーバーレイ表示
- 「▼ もっとみる（のこり N こ）」ボタンで展開/折りたたみ切替

### 9.x プラン情報の parent ↔ child データフロー (#789)

child UI にもプラン依存機能（きょうだいランキング、家族向け特別報酬表示、自由記述メッセージ等）が存在するため、`planTier` / `planLimits` / `isPremium` を child UI 配下で参照可能にする。

**解決の所在（単一責任）**:

- `src/routes/(child)/+layout.server.ts` の `load` で以下を **1 回だけ** 解決する:
  - `planTier` ← `resolveFullPlanTier(tenantId, licenseStatus, plan)`
  - `planLimits` ← `getPlanLimits(planTier)`
  - `isPremium` ← `isPaidTier(planTier)`
- 各 `+page.server.ts` / `+page.svelte` は SvelteKit の `parent()` / `$props().data` を通じて上記 3 値を参照する
- **子ページで `resolveFullPlanTier` / `getPlanLimits` を再度呼ぶのは禁止**（重複 DB 参照 + 契約のドリフト源）
  - 例外: フォーム送信 action（`+page.server.ts` の `actions`）は `parent()` にアクセスできないため、action 内で再度解決する

**データフロー**:

```
hooks.server.ts
  └─ event.locals.context { tenantId, licenseStatus, plan }
       │
       ↓
(child)/+layout.server.ts  ← プラン解決の単一責任
  │  resolveFullPlanTier → planTier
  │  getPlanLimits(planTier) → planLimits
  │  isPaidTier(planTier)   → isPremium
  │
  └─ return { child, balance, planTier, planLimits, isPremium, ... }
       │
       ↓ parent() / $props().data
(child)/[uiMode]/home/+page.server.ts
  │  const parentData = await parent();
  │  if (parentData.planLimits.canSiblingRanking) { ... }
  │  // #2471 / ADR-0055: 子供 home は必ず per-child API を使う
  │  getChildActivities(child.id, tenantId, { childAge: child.age })  ← per-child instance のみ
  │  // 旧 getActivities(tenantId) は tenant 全 child を aggregate するため使用禁止
  │  // (5 children 環境では同名 activity が 5 倍に重複 render される bug、PR #2455 Round 6 で発覚)
  │
  └─ return { siblingRanking, ... }
       │
       ↓ $props().data
(child)/[uiMode]/home/+page.svelte  ← data.planLimits で表示制御
```

**child UI 側でのプラン依存機能の表示制御**:

- きょうだいランキング: `data.planLimits.canSiblingRanking` が `false` の場合は非表示
- 自由記述おうえんメッセージ: `data.planLimits.canFreeTextMessage` が `false` の場合は非表示
- カスタム特別報酬: `data.planLimits.canCustomReward` が `false` の場合は非表示
- プレミアムテーマ等: `data.isPremium` で分岐

## 10. プラン/トライアル UI パターン (#743)

有料機能の disabled 表示・トライアル文脈・アップグレード導線を、画面ごとのアドホック実装ではなく **共通コンポーネントと固定パターン** で実現する。新しい有料機能を追加するときは、本セクションの 1 / 2 / 3 のいずれかに沿って実装すること。

### 10.1 プラン状態 × コンポーネント対応表

| プラン状態 | license plan | trial 状態 | resolvePlanTier 結果 | 主要表示 | 主要 CTA |
|-----------|--------------|-----------|---------------------|---------|---------|
| 完全 free（未使用） | `free` | `not-started` | `free` | TrialBanner（not-started） | 「7日間 無料で試す」 |
| トライアル中 | `free` | `active` | `standard` or `family` | TrialBanner（active） + PlanStatusCard（trial-badge 付き） | 「本契約する」 |
| トライアル終了 free | `free` | `expired` | `free` | TrialBanner（expired） + PlanStatusCard（free） | 「⭐ アップグレード」 |
| standard 契約 | `standard` | n/a | `standard` | PlanStatusCard（standard） + 初回 PremiumWelcome | 「⭐⭐ ファミリーへ」 / 「プランの詳細」 |
| family 契約 | `family` | n/a | `family` | PlanStatusCard（family） + 初回 PremiumWelcome | 表示なし（最上位） |

> **注**: トライアル中は `resolvePlanTier` が `standard`/`family` を返すため、機能制限ロジック（`getPlanLimits`）は契約済みと同じ挙動になる。一方で license plan は `free` のままなので、UI 側は **PlanStatusCard の `trialStatus` プロップ** で「今使えている理由」を明示する必要がある（#730）。

### 10.2 disabled 表示パターン（共通原則）

free/standard ユーザーから有料機能に触れたとき、**機能の存在は示し、実行は不可、アップセル CTA を表示する** のが原則。隠す（hidden）のも、403 を踏ませるのも禁止。

| パターン | コンポーネント | 用途 | 例 |
|---------|--------------|------|---|
| **A. ボタン disabled + Badge** | `FeatureGate display="inline" buttonLabel="..."` | リスト内・カード末尾の単発操作 | 「クラウドエクスポート」ボタン（free） |
| **B. セクションオーバーレイ + Badge** | `FeatureGate display="section"` | パネル・カード全体（中身をぼかし表示） | 「AI提案」パネル（free） |
| **C. カード全体置き換え** | 個別実装（PremiumBadge + CTA） | データ管理カード等の重要セクション | 設定画面のクラウド共有カード（#773） |

実装ルール:

- 既存有料機能を追加・改修するときは **必ず `FeatureGate` を経由する**。`{#if planTier !== 'free'}` の生分岐をコンポーネント内に書かない。
- バックエンドでは別途 `requirePaidTier()` 等で 403 を返す。フロントの `FeatureGate` は **二重防御** であって唯一の砦ではない。
- A/B どちらを選ぶかは「ユーザーが押せる単位」で決める。1 ボタン = A、1 機能セット = B。

### 10.3 TrialBanner 仕様

- 配置: admin ホーム最上部（`AdminHome.svelte`）
- 状態: `not-started` / `active` / `urgent`（残り 1 日以下） / `expired` の 4 つ
- 表示条件:

| 状態 | isTrialActive | trialUsed | planTier | 表示 |
|------|--------------|-----------|----------|------|
| not-started | false | false | free | 🎁「7日間 無料で試す」+ 制限機能の列挙（contextual paywall、#2901） |
| active | true | true | free→standard/family | ⭐ または ⏰（urgent）「無料体験中」 |
| urgent | true | true | free→standard/family | ⏰「無料体験は明日で終了します」 |
| expired | false | true | free | 📦「無料体験が終了しました」 |
| 非表示 | - | - | standard / family（契約済み） | （TrialBanner は表示しない） |

#### contextual paywall（not-started、#2901 AC2）

generic な「全機能無料」訴求だけでは「どの機能が無料版で使えないのか」をユーザーが recognition できない（NN/G #6 recognition rather than recall）ため、not-started バナーに **free 版で制限される主要機能を列挙**する（`TRIAL_LABELS.bannerGatedHeading` + `bannerGatedFeatures`）。機能名は `FEATURE_LABELS` / `CHILD_TERMS` SSOT 由来（ADR-0045、直書き禁止）。ダークパターン（偽の緊急性・解約妨害）は含めない（ADR-0012 anti-engagement）。

CTA 文言（変更時は本表＋テストを同時更新）:

- not-started → `7日間 無料で試す` / 送信中は `開始中...`
- active / urgent → `プランを見る`（リンク先 `/admin/license`）
- expired → `⭐ アップグレード`（リンク先 `/admin/license`）

`data-testid` 規約: `trial-banner-not-started` / `trial-banner-start-button` / `trial-banner-active-cta` / `trial-banner-expired-cta` / `trial-banner-gated-features`（#2901）。

### 10.4 PlanStatusCard 仕様

- 配置: admin ホーム上部、TrialBanner の下
- 表示要素:
  - プラン名 + アイコン（free=なし / standard=⭐ / family=⭐⭐）
  - トライアル中バッジ（`isOnTrial` のとき）
  - 統計: カスタム活動 `n/max`、こども `n/max`、データ保持 `n日間 or 無制限`
- CTA:

| 状態 | CTA |
|------|-----|
| free | `⭐ スタンダードにアップグレード`（単一） |
| standard | `プランの詳細` + `⭐⭐ ファミリーへ`（2 列） |
| family | CTA なし |
| 任意プラン × トライアル中 | `本契約する` + 「{tier}の全機能を体験中です。…」の説明 |

`data-testid` 規約: `plan-status-trial-badge` / `plan-status-trial-note` / `plan-status-trial-cta` / `plan-status-free-cta`。

### 10.5 PremiumWelcome 仕様

- 種別: 中央モーダル（オーバーレイ）
- 表示条件: `isPaidTier(planTier) === true` かつ settings の `premium_welcome_shown !== 'true'`
- 表示内容: アイコン（standard=⭐ / family=⭐⭐）、解放された機能リスト（`getUnlockedFeatures(planTier)`）、CTA 「さっそく始める →」
- dismiss 動作: CTA 押下で `dismissPremiumWelcome` action（`POST ?/dismissPremiumWelcome`）→ `premium_welcome_shown` settings に `'true'` を保存 → モーダルを閉じる
- 二重表示防止: settings は tenant スコープ。一度 dismiss すれば同 tenant のすべての parent には再表示されない

### 10.6 アップグレード/ダウングレードフロー

詳細な状態遷移は [`docs/design/diagrams/plan-trial-flow.drawio`](diagrams/plan-trial-flow.drawio) を参照。

**アップグレードの主な経路**:

1. TrialBanner（not-started）→ 「7日間 無料で試す」 → `?/startTrial` action → トライアル開始 → 即座に有料機能解放（ただし license plan は free のまま）
2. TrialBanner / PlanStatusCard / FeatureGate の各 CTA → `/admin/license` → Stripe Checkout → webhook で license plan 更新 → 次回 admin ロード時に PremiumWelcome 表示

**ダウングレード**: 詳細フローと超過リソース処理の確認画面は #738 / #754 で別途設計（現状は `/admin/license` から Stripe Customer Portal を経由）。

#### 10.6.1 NUC vs SaaS 実行モード分岐 (EPIC #2327 / ADR-0051)

`/admin/license` は **`data.runtimeMode === 'nuc-prod'`** で 2 panel に分岐する (薄ラッパー実装):

| runtimeMode | 表示 panel | セクション数 | 主要訴求 |
|---|---|---|---|
| `nuc-prod` | `NucLicensePanel` (116 行) | 3 (Edition badge + 利用状況 + サポート link) | セルフホスト版、全機能利用可能、無制限 |
| `aws-prod` / `local-debug` / `demo` / `build` | `SaasLicensePanel` (580 行) | 7 (trial banner + 現在のプラン + プラン利用状況 + ライセンスキー + プラン管理 + 支払い履歴 + Stripe Portal 遷移) | プラン契約 / 解約 / ライセンスキー |

NUC で削除する 5 セクション (詳細根拠は ADR-0051 §2.3 / `docs/design/nuc-saas-runtime-bifurcation.md` §2.3):

- ライセンスキー適用 (NUC = self-host = キー入力意味なし)
- 「現在のプラン」(free/standard/family) (NUC = 自動格上げ、本文表示は混乱の元)
- プラン管理 placeholder ("決済機能は現在準備中です") (NUC = 決済機能不要)

#### サインアップ画面の入力項目（`/auth/signup`、#2810 Phase 7 PR-L1）

サインアップ画面 (`src/routes/auth/signup/+page.svelte`) の入力項目は **メールアドレス / パスワード / パスワード確認 + 3 種の同意チェック（利用規約 / プライバシーポリシー / 米国域外移転 #1638）** のみ。Google OAuth でのサインアップも併設する。メール登録時は確認コード入力 step（`SIGNUP_CODE_EXPIRY_MINUTES`）を経て登録が完了する。

- **NUC セルフホスト版は対象外**: NUC は「全機能無制限・課金なし」（`NUC_EDITION_TERMS`）の信頼ベース運用で、ライセンスキー判定を行わない。SaaS サインアップ画面の入力経路撤去は、NUC のライセンスキー仕様（`/admin/license` の `NucLicensePanel` / `site/help/license-key.html`）には影響しない。

#### `/admin/cheer` — 応援機能仕様 (#2267 EPIC #2266 / #2300 EPIC #2294 ⑥)

##### §1. 設計背景

旧 `/admin/messages` はおうえんスタンプ・ひとことメッセージのみで P 付与なし、子供画面に並ぶ stamp ピンが「動機付けの根拠（運動会一位等の具体的事象）」と連動しない構造上の弱点があった。PO 定義 (2026-05-19) で「**応援 = 任意の理由で直接子供にポイント付与**、スタンプ/メッセージは P 付与に付随する理由表現」と再整理し、`/admin/messages` を **#2275** で 308 redirect 統合した。

`/admin/cheer` は親が現実イベント直後に「reason」「ポイント」「カテゴリ」「アイコン」を 1 form で素早く入力して送る前提の UI。長い滞在を狙わない (ADR-0012 anti-engagement)。完了後に「もう 1 回応援」誘導 CTA は意図的に置かない (#2267 AC4)。

##### §2. 設計原則

1. **入力は 7 step 単一 form**（子供選択 → 理由 → ポイント → カテゴリ → アイコン → 付随スタンプ/メッセージ → 確認）。step 間 page 遷移なしで親が現実イベント後に 30 秒以内で送信完了できる
2. **labels SSOT** (ADR-0045): `CHEER_LABELS` を `$lib/domain/labels.ts` に集約。reason / icon / category / hint / button label すべて compound 経由参照、`*.svelte` 直書きしない
3. **Anti-engagement** (ADR-0012): 完了画面に連続誘導 CTA 禁止 (#2267 AC4)。シーズン期間中の自動配信 push 禁止 (#2300 採用判断 — 「親が能動的に承認する 1 タップ」のみ)
4. **5 年齢モード共通**: 親管理画面のため `uiMode` 非依存、5 全モードで同 UI

##### §3. 仕様

| Step | 入力 | バリデーション |
|------|------|---------------|
| 1 | 子供選択 (`selectedChildId`) | 1 件以上必須。children 0 件なら「まずこどもを登録してください」誘導表示で form 非表示 |
| 2 | 理由 (`reason`、`textarea`) | 1〜`CHEER_REASON_MAX_LENGTH` (100) 文字必須。`<n>/100（あと N 文字）` カウンタ表示 |
| 3 | ポイント (`points`、`number`) | `CHEER_POINTS_MIN`〜`CHEER_POINTS_MAX` (1〜10000) 必須 |
| 4 | カテゴリ (`category`、`NativeSelect`) | `CHEER_CATEGORIES` の 5 種から選択必須（うんどう / べんきょう / せいかつ / こうりゅう / そうぞう） |
| 5 | アイコン (`icon`、`text`) | 絵文字（maxlength 4）。デフォルト 🎉 |
| 6 | 付随スタンプ + ひとことメッセージ（任意） | `STAMP_PRESETS` から 0〜1 件選択 + 120 文字以内 textarea |

#### `/admin/activities/[id]/edit` — 活動編集 / must トグル仕様 (#1755 / #1756 #1709-A/B)

##### §1. 設計背景

旧来の admin 設計には「ルーティン」と「持ち物」の 2 種を扱う `/admin/checklists` の kind タブと、活動マスタを編集する `/admin/activities` インライン編集が並存していた。子供のフロー上、生活習慣（旧ルーティン）は「活動マスタの 1 属性として記録できる」のが自然であり、**チェックリストは持ち物に純化**しつつ、活動マスタに **「今日のおやくそく」(`priority='must'`) 属性** を持たせる方針に統合した（ADR-0009 SSOT・ADR-0010 Pre-PMF: 機構統合で複雑度削減）。

旧インライン編集は同じ画面内に複数行が同時展開されると DOM 構造の入れ子が深くなり、誤操作（別 row の保存）や form action のバインドミスが発生していたため、**独立 URL `/admin/activities/[id]/edit` に分離**して must トグルとあわせ単一 form で扱う。

##### §2. 設計原則

1. **must は活動マスタの 1 属性**：チェックリスト kind の「ルーティン」を切り出す代わりに `activities.priority` 列で表現し、子供 UI（1709-C）は「今日のおやくそく」セクションとして強調表示する
2. **編集は独立 URL**：一覧画面に複数の編集 form を同時開閉しない。一覧 → 編集 → 一覧 redirect の 3 ステップで完結
3. **labels.ts SSOT**：トグルラベル / 説明文 / 一覧 Badge ラベルは `ACTIVITY_PRIORITY_LABELS` および `ACTIVITY_PRIORITY_FORM_LABELS`（労働の must / 説明 / Badge / 編集ボタン文言を集約）から参照する
4. **service 経由**：form action は `$lib/server/services/activity-service.ts` の `updateActivity()` 経由で priority を含めて更新する。`+page.server.ts` から ORM 直接呼び出しは禁止

##### §3. 仕様

| 項目 | 値 / 動作 |
|------|----------|
| URL | `/admin/activities/[id]/edit`（活動 ID パスパラメータ） |

| load | `getActivityById(id, tenantId)` + `CATEGORY_DEFS` を返す |

##### マーケットプレイス一覧 tag filter 既定 8 件 + expansion (#2765 / Round 18 Cluster I)

`/marketplace` 左サイドバー filter (desktop column + mobile drawer) の tag chip は、5 type 全体で **50+ 件並列表示** されており、Hick's Law (DESIGN.md §10) 違反 + ADR-0012 Anti-engagement の「最短経路」原則違反 + 認知負荷過多で「うちの家族にはどれが合うのか」即時判断不能となっていた UX defect (Round 18 評価 Round 3、Issues #11 / #15 / #19)。

| 状態 | tag 表示数 | 追加 UI | 用途 |
|------|----------|---------|------|
| default | 人気上位 8 件 | 「もっと見る (残 N 件)」link button | 8 割の保護者が頻出 tag で即決できる経路 |
| expanded | 全 tag 表示 (50+) | 「タグをたたむ」link button | long-tail tag (祖父母世帯、特殊ニーズ等) を探したい上級者向け救済経路 |

**popularity sort SSOT (`getAllTags()`)**:
- 旧実装は `[...tags].sort()` (Set + alphabetical) のため preset 出現頻度 0 件の tag も上位表示されていた
- 新実装は `frequency desc + localeCompare('ja') asc tiebreaker` で **頻度順を data 層 SSOT 化** (`src/lib/data/marketplace/index.ts:148-170`)
- caller は `+page.server.ts:137` (`tags: getAllTags()` で 1 箇所) のみ。`+page.svelte` は `data.tags` 経由間接利用、admin import / detail page は `item.tags` 直接参照のため `getAllTags()` 順序変更の影響範囲外
- silent breaking change ではあるが、(a) tail tag は expansion UI で引き続き到達可能 (b) URL share 経路 (`?tag=<value>` filter) の意味論変化なし (c) caller が server load 1 箇所のため副次影響なし、で受容可能と判断

**expansion state 保持原則**:
- desktop column / mobile drawer は変数独立 (`tagsExpandedDesktop` / `tagsExpandedMobile`)、両者間で expansion state を共有しない (variant ごと独立 UX)
- page reload 時に state reset (URL 永続化なし)。URL 共有時に意図しない展開を強制せず、初回訪問者は default 8 件 fold で迎える整合性を優先 (Anti-engagement 整合)
- `aria-expanded` でスクリーンリーダ整合

**設計理由 (DESIGN.md §10 Hick's Law + ADR-0012 整合)**:
- 「記録する → 数秒で閉じる」最短経路を filter 段階で先に発揮させる (50 件 chip 全て読まなくても 8 件で即決できる経路を default 化)
- expansion は「user 意図的操作のみで起動」する明示的展開 (auto-expand / hover-expand / 自動 scroll などの侵襲的演出は不採用、Anti-engagement)
- AI tag 推薦 / 年齢別 re-rank 等は Round 18 再評価次第で別 Issue (YAGNI、本 PR は popularity sort + fold only)

**実装**:
- `src/routes/marketplace/+page.svelte` (desktop / mobile drawer 同 snippet を `variant: 'desktop' | 'mobile'` で再利用、DRY)
- `src/lib/data/marketplace/index.ts` `getAllTags()` (frequency desc + ja-locale tiebreaker)

- 検証: `tests/unit/domain/marketplace-items.test.ts` (popularity sort 5 ケース、TS strict undefined narrowing 含む) + `tests/e2e/marketplace-filter.spec.ts` (Pixel 7 + desktop で chip count ≤ 8 + expansion toggle)

## 12. デモモード（実行モード）仕様 (#1180 / ADR-0039)

### 12.1 目的と位置づけ

デモモードは**本番ルート上で駆動する実行モード**である。`src/routes/demo/**` という別ルートツリーは ADR-0039 で廃止された。#296 / #1129 / #1147 / #1180 の 4 回連続して発生した「デモと本番のデザイン乖離」を構造的に解消することが目的。

- デモと本番は**同一コードパス**を通る（同期漏れが発生不可能）
- DB は触らず in-memory シードで動作（Linear demo workspace 方式）
- reload で fresh start

### 12.2 判定フロー

| 段階 | 入力 | 出力 |
|------|------|------|
| 入口 | URL に `?mode=demo` クエリ | cookie `gq_demo=1`（有効期限 4h、httpOnly、SameSite=Lax）を発行 |
| 各リクエスト | cookie `gq_demo=1` の存在 | `event.locals.isDemo = true` |
| 退出 | `/demo/exit` 到達 | cookie 削除 → 本番コンテキストに復帰 |

`event.locals.isDemo` が server-side の Single Source of Truth。`+page.server.ts` / `+layout.server.ts` は `locals.isDemo` を `data.isDemo` として client に配布し、クライアントは `$page.data.isDemo` を参照する。

### 12.3 書き込みガード

`hooks.server.ts` で一元処理する。

- `locals.isDemo === true` かつ HTTP メソッドが `POST / PUT / PATCH / DELETE` のとき、`200 { ok: true, demo: true }` を返して no-op 化
- UI 側コード（form actions / fetch）は無改変のまま「成功扱い」で進む（Stripe test-mode と同じ）
- 例外許可リスト: `/api/feedback`, `/api/demo/exit`, `/api/health`, `/api/demo-analytics`

**403 ではなく 200 を返す理由**: エラーダイアログ表示は子供向け UX を損なう。見た目は「やった感」を維持し、リロードで消えるので副作用もない。

### 12.4 デモコンテキスト（in-memory）

`$lib/server/demo/build-demo-context.ts` がリクエスト毎に合成する。

| フィールド | 値 |
|-----------|-----|
| `tenantId` | `'demo'` |
| `role` | `'owner'` |
| `licenseStatus` | `ACTIVE` |
| `plan` | `?plan=` クエリ優先、既定 `family`（cookie 永続化は既存ロジック踏襲） |
| seed データ | `src/lib/server/demo/demo-data.ts` の 5 年齢モードプリセットを流用 |

サービス層 (`$lib/server/services/*`) は `isDemo` を一切判定しない。サービスが DB 呼び出しをする前に hooks が no-op で返すため、サービス側の分岐は構造的に不要になる。

### 12.5 オーバーレイ（DemoBanner）

上部固定バナー + 下部の `DemoGuideBar`（既存）の 2 段構成。

- **DemoBanner（新規、上部）**: Primitives `Alert variant="trial"` を流用した sticky header。`$page.data.isDemo === true` のとき表示。退出ボタン（`https://ganbari-quest.com/`、本番 LP に戻る）と「本当に始める」CTA（`https://ganbari-quest.com/auth/signup`、本番 Cognito signup）を常設。**#2097 Phase B (2026-05-17)**: ① ctaStart は大人 (保護者) 向けバナーのため漢字「本当に始める」(旧: ひらがな「ほんとうに始める」)、② signupHref / exitHref は demo Lambda (`demo.ganbari-quest.com`) で叩くと 404 / Cognito 未注入 signup 画面が表示される問題への構造的対策として本番 ganbari-quest.com への absolute URL に固定する。SSOT は `src/lib/domain/labels.ts` `DEMO_LABELS`
- **DemoGuideBar（既存、下部）**: 7 ステップの誘導。既存のまま `+layout.svelte` から条件付き mount
- 年齢モード対応: `fontScale` に追従。baby / preschool はひらがな併記

**禁忌**:
- hex 直書き禁止（§2）
- 文言は `labels.ts` の `DEMO_LABELS`（新設）経由で参照（ADR-0009 準拠）
- `?screenshot=1` / `hideDemoOverlays` 方式は #1181 で導入されたが本 ADR で撤去

### 12.6 LP スクリーンショット撮影

`scripts/take-lp-screenshots.mjs` は `?mode=demo` 付きの本番 URL を撮影する。

- 旧撮影パス `site/screenshots/*.webp` は**保持**（履歴参照用）

## §14 ごほうびショップ陳列原則（#1336）

### 14.1 概要

ごほうびショップ（L3 経済層）の陳列方針を定義する。詳細なゲーミフィケーション設計は [`26-ゲーミフィケーション設計書.md §12`](26-ゲーミフィケーション設計書.md#12-ごほうびショップl3-経済層) を参照。

### 14.2 陳列原則

- 親が **プリセット一覧から選んで追加** + **カスタム追加**（ゼロから作成）の 2 経路
- プリセットは **3 系統** に視覚的に分類される（14.3 参照）
- ポイント単価（X pt 交換）は **親が自由設定**（プリセットは目安値を提示するのみ）
- ショップは「親が許可した欲望」と「子の本物の欲望」の交点（`26-ゲーミフィケーション設計書.md §2.2` 参照）

### 14.3 3 系統の定義と意義

| 系統 | `shopCategory` | 説明 | 差別化意義 |
|------|--------------|------|-----------|
| 実物 | `physical` | お菓子・文房具・おもちゃ等の現物報酬 | 直接的な物質報酬。子供が「欲しい」と感じる最も直感的な動機源 |
| お金 | `money` | お小遣い変換（金額相当のポイント交換） | 経済リテラシー教育への橋渡し。「努力→お金」の実感を年齢に応じて自然に伝える |
| 特権 | `privilege` | 夜更かし・ゲーム時間延長・好きなメニューリクエスト等 | BusyKid/Greenlight 等の海外サービスとの差別化の核。日本家庭の「がんばり報酬」文化に合致し、モノではなく「体験・自由」への欲求を刺激する |

**「特権」系統が差別化の核である理由**:
- 日本の子育て家庭では、モノより「許可」が価値を持つ場面が多い（夜更かし・ゲーム等）
- 親が「OK を出すことそのもの」をごほうびにできる点が、単純なポイント型サービスとの差異
- 子供は「ポイントを貯めれば自分の要望が通る」という自己効力感を得る

### 14.4 プリセット構成と系統マッピング

プリセットは `src/lib/data/preset-rewards.ts` で管理。各プリセットに `shopCategory` フィールドが付与されている。

| グループ名 | `shopCategory` | 主な内容 |
|-----------|--------------|---------|
| もの | `physical` | シール・おかし・文房具・本・おもちゃ・マンガ |
| たいけん | `privilege` | おでかけ・がいしょく・えいが・ゲーム時間・YouTube時間 |
| おこづかい | `money` | おこづかい 100円〜1000円 |
| とくべつ | `privilege` / `money` | 夜更かし券・朝寝坊券・家族でゲーム・メニューリクエスト・旅行貯金 |

### 14.5 3 系統タブ UI 実装（#2157）

子供側ショップ画面 (`/(child)/[uiMode]/shop`) に 4 タブ (すべて / もの / おこづかい / とくべつ) を Ark UI Tabs primitive (`$lib/ui/primitives/Tabs.svelte`) で実装。デフォルトは「すべて」、ABCmouse multi-venue prior art (一次確認) 整合。

| タブ key (`shopCategory`) | 表示ラベル (子供向け hiragana) | 内訳 |
|---|---|---|
| `all` | すべて | 全 reward |
| `physical` | もの | 現物報酬 (シール・おかし・本・おもちゃ等) |
| `money` | おこづかい | 金額相当のポイント交換 |
| `privilege` | とくべつ | 特権 / 体験 / 時間延長 / メニューリクエスト等 |

### 14.6 shopCategory 派生機構（#2157、暫定実装）

- `money` 判定: 「おこづかい」「貯金」「○○円」 / 通貨絵文字 (🪙 💴 💵 💰 💸)
- `privilege` 判定: 「時間」「○○けん」「リクエスト」「おでかけ」「えいが」 / 体験絵文字 (🌙 😴 🎮 📺 🚗 🍽️ 🎬 等)
- `physical` 判定: 上記以外 (現物デフォルト)

preset (`preset-rewards.ts`) との一致率は `tests/unit/domain/shop-category.test.ts` で 80% 以上を担保。将来 (PMF 後) DB schema 拡張で `shop_category` 列を追加した際は本ヒューリスティックを deprecate する。

### 14.7 ポイント範囲 + 交換可能フィルタ（#2160）

## §15 ごほうびショップ 親子連携フロー（#1337）

> ゲーミフィケーション設計書 §12（ごほうびショップ L3 経済層）の UI 実装仕様。
> DB スキーマ: `docs/design/08-データベース設計書.md` §3 `reward_redemption_requests`  
> API: `docs/design/07-API設計書.md` §3.26

### §15.1 設計背景・目的

子供が「欲しいごほうびを親に申請する」体験を中心に、親が承認/却下するワークフローを提供する。
現行の `special_rewards` テーブルは保護者が子に**直接付与**するフローのみで、子主体の申請機能がない。
本フローで「子が選んで親が承認する」L3 経済層の完成形を実現する。

**Anti-engagement 整合確認 (ADR-0012)**:
- 申請中バッジは最小表示（「受け取り待ち」1 行のみ）— 複数申請を煽る導線なし
- 子供側専用タブは設けない — ホーム画面の申請済みアイテムに badge を乗せるのみ
- 「早く申請しよう」「もっと交換しよう」等の能動的 CTA は一切使用しない
- 完了通知は次回アクセス時に 1 回のみ表示（繰り返しループなし）

#### §15.4.3 ごほうび編集 / 削除と pending redemption ガード（#2832）

`/admin/rewards` の選択中 child のごほうび一覧（`rewards-per-child-list`）から各 reward を編集 / 削除できる。
処理待ち（pending）の交換申請との整合は以下で担保する。

| 操作 | 仕様 |
|------|------|
| 一覧表示 | reward 行に icon / タイトル / ポイント + `編集` / `削除` ボタン。pending 申請がある reward には「交換申請 処理待ち」バッジ（`reward-pending-badge-<id>`） |
| **削除（ガード）** | `?/delete` action → `deleteReward` service が `hasPendingByReward` で pending を検査。**pending あり → 409 で削除拒否**し、「交換申請が処理待ちのため削除できません。申請を承認または却下してから削除してください」（`ADMIN_REWARDS_PAGE_LABELS.deletePendingBlocked`、labels SSOT）を Toast + in-page banner の 2 層 feedback（DESIGN.md §5）で表示。削除確認 dialog にも同警告を先出しする |
| 削除（成功時） | 確認 dialog（不可逆 note「この操作は取り消せません。このごほうびの交換履歴も削除されます。」+ danger Button）→ 当該 reward と解決済の交換申請履歴行を削除 → 成功 Toast + banner + 一覧から除去。audit log（`logger.info`）を記録 |
| **編集（案 b、PO 判断）** | pending 中も編集を**許容**。編集 dialog（タイトル / ポイント / アイコン）に pending 申請がある場合のみ「申請済みの交換は申請時点の内容（名前・ポイント）で処理されます」note（`editPendingNote`）を表示し、snapshot 仕様（08-DB設計書 §reward_redemption_requests）を明示する。プランゲートは追加と同じ（編集での実質新規作成 bypass を防ぐ）。削除はプランゲートなし（downgrade 後もデータ整理可能） |

### §15.5 エッジケース処理方針

| ケース | 処理方針 |
|--------|---------|
| **30 日 pending 期限切れ** | 日次 cron (`/api/cron/expire-redemptions`) が `requested_at < NOW() - 30d` かつ `status = 'pending_parent_approval'` を `expired` に更新。子供への通知はしない（静かな失効）。cron 実装パターン: `verifyCronAuth` 使用（`07-API設計書.md §5` 参照） |
| **並列申請（数量制限なし）** | 複数の異なる reward_id への申請は許可。同一 reward_id への複数申請は禁止（API 側で `UNIQUE (child_id, reward_id, status='pending_parent_approval')` 相当のチェック） |
| **親がごほうびを削除** | `?/delete` action（`deleteReward` service の `hasPendingByReward` ガード、#2832）が pending 申請存在時に `409 Conflict` を返し削除を禁止。親 UI には「交換申請が処理待ちのため削除できません。申請を承認または却下してから削除してください」を 2 層 feedback で表示（§15.4.3） |
| **親がごほうびを編集（pending 中）** | 編集は許容（案 b、#2832）。処理待ち申請は申請時点 snapshot（reward_title / reward_points）で表示・控除されるため編集は波及しない。編集 dialog に snapshot 仕様 note を表示（§15.4.3） |
| **子供が削除される** | `children` 削除時に CASCADE DELETE で申請レコードも削除 |
| **ポイント不足後の申請（レース）** | API 側でポイント残高を再確認。不足の場合は `400 INSUFFICIENT_POINTS` を返す |

## §18 初月価値プレビュー体験（#1600 / ADR-0023 I9）

### 18.1 目的

ADR-0023 §5 I9 / C-Q13 「ちゃんと使う前にやめる、価値実感が 1 ヶ月後だが学習コストしか見えない」に対し、
**初月の「価値プレビュー」体験**を 1 週間で見せることで初動離脱を防ぐ。

親 P1（契約直後 1-2 週間の最も離脱しやすい時期のユーザー）に対し:

- 「あと N 日続ければこんな成果が見える」というプレビュー（マイルストーン的演出）
- 親レポートで「子供の頑張りグラフ」が早期に意味を持つ
- 「先が見える」感覚で継続モチベーション

### 18.2 構成要素

| コンポーネント | 場所 | 役割 |
|----|----|----|
| `value-preview-service.ts` | `src/lib/server/services/` | `getTenantValuePreview(tenantId)` で集計を返す。新規 DB スキーマ追加なし（既存 `activity_logs` + `children.created_at` を流用、Pre-PMF YAGNI） |

### 18.3 マイルストーン定義（6 種）

`MILESTONES` 配列で管理（`value-preview-service.ts`）。

| id | kind | threshold | UI ラベル（`MILESTONE_LABELS`） |
|----|----|----|----|
| `first_record` | count | 1 | 「はじめての記録」 |

- **count 系**: 該当 child の有効な `activity_logs` 件数 ≥ threshold で達成

### 18.9 ChallengeBanner vs MilestoneBanner vs MilestoneBellButton — SSOT 用途整合 (#2172)

過去に「Banner suffix が 2 つあり用途が混乱する」という SSOT 違反疑義が PO レビューで挙がったため、3 component の **目的 / データソース / 配置場所 / 命名根拠 / 寿命** を本節で SSOT として確定する。命名 rename は行わず、用途別の独立 component として維持する（PO 確認後）。

|----|----|----|----|----|----|
| `ChallengeBanner.svelte` | `src/lib/ui/components/` | `child-challenge-service` (`getActiveChildChallengesWithSiblings`) (#2458-B per-child 化) | `(child)/[uiMode]/home/+page.svelte` のホーム本体 | **きょうだいチャレンジ** (週次共闘 challenge) の per-child 進捗 + 報酬受取 form + 兄弟連動 progress bar 一覧。子供がアクティブに操作する **コアゲームプレイ UI** | アクティブな challenge がある限り常時表示、completed → reward claim までインタラクティブ |
| `MilestoneBanner.svelte` | `src/lib/features/value-preview/` | `value-preview-service` (`getTenantValuePreview` の `milestones`) | (#2168 で全 mount 撤去、`@deprecated`) | (legacy) **個人の活動マイルストーン達成** の 1 件単位通知。`#2168` 以前は child layout の `<main>` 内に絶対配置 banner | `#2168` 以降は **mount されない** (撤去済、LP screenshot `?screenshot=all` 撮影系のフォールバック用途で source 残置) |
| `MilestoneBellButton.svelte` | `src/lib/features/value-preview/` | 同上 (`value-preview-service.milestones`) | `(child)/+layout.svelte` Header の `notificationSlot` | **マイルストーン未閲覧件数の bell + dot badge**。click で history `?kind=milestones` (#2170) 遷移 + 全件 localStorage で seen 記録 | header に常駐、未閲覧件数が 0 件で badge 非表示 |

**命名整合性の根拠 (rename しない理由)**:
- `ChallengeBanner` = **インタラクティブ form を含むワイド banner** (mount 上 banner-shaped UI)。command-oriented UI で「banner」suffix は形状を示す
- `MilestoneBanner` = (deprecated) かつての banner-shaped UI で同名は許容範囲 (mount されないため UI 競合なし、source は将来の `?screenshot=all` 互換のため保持)
- `MilestoneBellButton` = `#2168` 以降の正規 entry point。**Button** suffix が示す通り bell icon button (banner ではない)
- 用途別に分離されており、`MilestoneBanner` を rename しても import path 影響のみで mount 階層は変わらない (mount 0 件)。Pre-PMF コスト > rename ベネフィット で **rename 不要** (PO #2172 確定)

**MN-1 (#2168) 完了後の依存関係 (AC3)**:
```
ChallengeBanner ← (child)/[uiMode]/home/+page.svelte         # 独立 mount、依存なし
MilestoneBellButton ← (child)/+layout.svelte (notificationSlot) # MilestoneBanner と独立
MilestoneBanner ← (none)                                       # @deprecated、mount 0 件
```
両者は **完全に独立** しており、`MilestoneBanner` を将来削除しても `ChallengeBanner` には影響しない。

**MILESTONE_LABELS の年齢別 variant (#2169 / ADR-0015)**:
- `getMilestoneLabel(id, ctx={ageTier})` で preschool = ひらがな / elementary 以上 = 漢字 を分岐
- `getMilestoneBannerTitle(ctx)` で bannerTitle (「やったね！」/「達成しました」) を分岐
- `MilestoneBanner` / `MonthlyValuePreview` / history `?kind=milestones` (#2170) は全て同 helper 経由
- カタカナ「マイルストーン」訴求は子供向けに違和感あり、`bannerTitle: 'やったね！'` (preschool) / `'達成しました'` (kanji) に置換

### 18.10 history 画面 4 タブ階層 (#2170)

`(child)/[uiMode]/(character)/history/+page.svelte` は時系列管理データの **統合 view** として、4 種のデータタブを提供する。

#### タブ構造

| outer タブ (kind) | データソース | inner タブ | URL 例 |
|----|----|----|----|
| 活動 (activities) | `getActivityLogs` | 期間 (today/week/month) | `?kind=activities&period=week` |
| 達成 (achievements) | `getActiveChildChallengesWithSiblings` (#2458-B per-child 化、`ChallengeBanner` と同じ source) | なし | `?kind=achievements` |
| 交換 (purchases) | `getRedemptionRequestsForChild` (ごほうびショップ申請履歴 #1337) | なし | `?kind=purchases` |
| 記念 (milestones) | `getTenantValuePreview` の `child.milestones` (#1600 と同じ source、#2169 の年齢別 variant 経由表示) | なし | `?kind=milestones` |

#### 設計原則

- **2 階層タブ (kind → period)**: 期間タブは活動データのみ意味があるため、活動タブ内側でのみ表示
- **並列 fetch (Promise.allSettled)**: 4 種データを並列取得し、失敗したタブのみ空配列 fallback (history 全体は守る)
- **read-only**: 編集 / 削除機能なし (Pre-PMF YAGNI、ADR-0010)
- **bell click 遷移先**: `MilestoneBellButton` (`#2168`) の click 先は `?kind=milestones`
- **年齢別ラベル**: kind タブラベルは `historyKindActivities` / `historyKindAchievements` / `historyKindPurchases` / `historyKindMilestones` で variants 化 (preschool = かつどう/できたこと/こうかん/やったね、elementary 以上 = 活動/達成/交換/記念)
- **purchase status badge**: approved / rejected / pending / expired を `data-status-tone` 経由でセマンティックトークン (feedback-success / -error / -warning) 配色

## §19 LP / Legal 文言の SSOT 経路 (#1683 umbrella 完遂, #1704)

### 19.1 設計背景

LP (`site/index.html` 等) と Legal HTML (`site/{privacy,terms,sla,tokushoho}.html`) には合計 693 件の日本語ハードコード文言が並行実装されており、用語変更時の同期漏れが頻発していた。ADR-0009 §例外節では当初「法的文書は法務要件で固定」「SEO 対象 meta タグはクローラー JS 未実行」を理由に SSOT 化対象外として運用していたが、PO 方針「SSOT 漏れのコンテンツは存在してはいけない」と矛盾し、i18n 対応の前提整備も不可能だった。#1683 umbrella で 693 件全件 SSOT 化を達成（2026-04-30）。

### 19.2 設計原則

1. **LP / Legal 全 HTML は `data-lp-key` 属性ベース注入経路に統一**: `<element data-lp-key="namespace.key">` 形式で SSOT 値を `src/lib/domain/labels.ts` の `LP_*_LABELS` namespace から注入する
2. **innerHTML + DOMPurify サニタイズ**: 法的文書の構造タグ（`<h1>` / `<p>` / `<ul>` / `<ol>` / `<table>` / `<dl>` 等）保持のため `textContent` 直代入から `innerHTML = DOMPurify.sanitize(...)` 方式へ刷新（ADR-0025 accepted）
3. **印刷タイミング保証**: pamphlet.html は `window.print()` 起動前に `beforeprint` イベントで `reapplyLpKeys()` を再実行
4. **CDN 配信**: site/ は GitHub Pages 上の static アセットで npm bundle 経路を持たないため、DOMPurify は `<script src="https://cdn.jsdelivr.net/npm/dompurify@3/dist/purify.min.js" defer>` で CDN 配信
5. **例外**: SEO 対象の `<title>` / `<meta>` タグはクローラ JS 未実行のため SSOT 注入不可、`labels.ts` 値と完全一致するよう人間が手動同期（ADR-0009 §例外）

### 19.3 仕様

| 項目 | 内容 |
|------|------|
| **対象ファイル** | site/ 全 10 HTML (index / pricing / faq / pamphlet / selfhost / help/license-key / privacy / terms / sla / tokushoho) |
| **SSOT 経路** | `src/lib/domain/labels.ts` の `LP_*_LABELS` namespace → `scripts/generate-lp-labels.mjs` で `site/shared-labels.js` を生成 → DOMContentLoaded で `applyLpKeys()` が `data-lp-key` 属性付き要素に innerHTML + DOMPurify sanitize で注入 |
| **LEGAL namespace** | `LP_LEGAL_PRIVACY_LABELS` (17 keys) / `LP_LEGAL_TERMS_LABELS` (23 keys) / `LP_LEGAL_SLA_LABELS` (11 keys) / `LP_LEGAL_TOKUSHOHO_LABELS` (3 keys) = 計 54 keys |
| **CI gate** | `scripts/check-lp-ssot.mjs` で `lp-ssot-baseline.json` `count: 0` を維持（ratchet）。LEGAL namespace の各 key が対応 HTML の `data-lp-key` で参照されているかを逆方向検証 |
| **DOMPurify config** | `ALLOWED_TAGS`: 既存 inline 装飾 + 構造タグ（h1-h4, p, ul, ol, li, div）+ table 系 + 図版（figure/figcaption）+ semantic（header/section/dl）/ `ALLOWED_ATTR`: href/target/rel/class/aria-*/id/data-contact-context / `ALLOW_DATA_ATTR: false` |
| **Table row 例外**: 親要素が `<table>` / `<thead>` / `<tbody>` / `<tr>` で値が `<tr>` 系から始まる場合、XHTML パーサ (`PARSER_MEDIA_TYPE: 'application/xhtml+xml'`) で sanitize して row 構造を保持 |
| **Print E2E**: `tests/e2e/pamphlet-print-ssot.spec.ts` で `emulateMedia({ media: 'print' })` + `beforeprint` イベント発火後の DOM 状態を検証 |

### 19.4 関連 ADR / Issue

- ADR-0009（labels SSOT 原則） — §例外「法的文書」は #1703 / ADR-0025 で正式 supersede
- ADR-0025（LP SSOT 注入機構の innerHTML 化 + XSS 設計） — accepted 2026-04-30 / #1683 完遂
- ADR-0014（labels / i18n 機構選定） — 本仕様により i18n 拡張の前提整備完了

## §20 NotificationPermissionBanner (#2115 / #2116, EPIC #2114)

### 20.1 設計背景

過去 #293（基盤）/ #1593（anti-engagement 適合）/ #1666 / #1689（DynamoDB）で server side の構造防御は完了済みだが、UI 補完が抜けており、PO 報告（2026-05-14）で「通知を受け取る」クリック後に状態変化なし（loading なし / try-catch なし / permission 再評価なし / Toast なし）の重大 UX 不整合が発覚。同時に「informed consent」観点で「どんな通知がいつ・どこに来るか」を許可前に把握できない構造的欠陥も指摘された。

実装ファイル: `src/lib/features/admin/components/NotificationPermissionBanner.svelte`（AdminHome 内で `{#if welcomeVisible}` の後段に配置）。

### 20.2 設計原則

1. **Silent 失敗禁止**: subscribe フロー (`subscribeToPush()`) は必ず try/catch + `.ok` 判定。例外も拒否も generic も全て UI feedback (Toast / inline error fallback) を返す
2. **Informed consent UI 化（2 段階開示）**: descCompact (第 1 段階) で 4 要素（頻度 / 内容 / 親端末限定 / quiet hours）を一目把握可能にし、`<details>` (第 2 段階) で 3 系統 (Reminder / Streak-warning / Achievement) の例文を開示
3. **SSOT 整合**: 例文は `reminder/+server.ts` の `sendPushNotification(..., 'きょうも がんばろう！', ...)` と一致（用語変更時 grep で同期保証）
4. **Pre-PMF 過剰防衛回避** (ADR-0010): 通知種別 ON/OFF 細分化 UI は採用しない / リトライボタン追加なし / `/admin/settings` への誘導 link のみ

### 20.3 仕様

| 項目 | 内容 |
|------|------|
| **trigger 条件** | `isPushSupported() && permission === 'default' && !dismissed && !subscribed` (visible `$derived`) |
| **第 1 段階 (descCompact)** | 「毎日 1 回まで、お子さまのがんばりリマインダーを親端末にお届けします（21:00-07:00 はお休み）」(`FEATURES_LABELS.notificationBanner.descCompact`) |
| **第 2 段階 (disclosure)** | `<details>` で開閉。reminderTitle / streakWarningTitle / achievementTitle の `dl` + 親端末限定 + quiet hours + 「設定画面でいつでも OFF にできます」+ `/admin/settings` リンク |
| **state machine** | `loading: boolean` (二重クリック防止 + ボタン disable + `aria-busy="true"` + 「設定中…」表記) / `errorState: 'denied' \| 'generic' \| null` (inline fallback UI) / `subscribed: boolean` (true で `visible=false`) / `permission: NotificationPermission` (`$effect` 初期評価 + subscribe 完了後再評価) |
| **success path** | `subscribeToPush()` が subscription を返す → `showToast('通知を有効化しました', ..., 'success')` + `subscribed = true` → バナー消滅 |
| **failure path (返値 null)** | permission 再評価 → denied なら errorDescDenied、それ以外は errorDescGeneric を inline 表示 + `/admin/settings` link |
| **failure path (throw)** | `console.error` + 上記同様の inline error fallback UI |
| **POST 失敗時の整合性** | `push-subscription.ts` の subscribe POST が `!res.ok` → `subscription.unsubscribe()` でブラウザ側も巻き戻し + throw（サーバ DB と subscription の整合性維持） |
| **labels SSOT** | `FEATURES_LABELS.notificationBanner.{descCompact, loadingLabel, toastSuccessTitle, toastSuccessDesc, errorTitle, errorDescDenied, errorDescGeneric, errorSettingsLinkLabel, disclosureLabel, disclosureContent.{reminderTitle, reminderExample, streakWarningTitle, streakWarningExample, achievementTitle, achievementExample}, disclosureParentOnly, disclosureQuietHours, disclosureOffNote, disclosureSettingsLinkLabel}` |
| **testid** | `notification-permission-banner` (root) / `notification-banner-cta` / `notification-banner-dismiss` / `notification-banner-desc-compact` / `notification-banner-error` / `notification-banner-settings-link` / `notification-banner-disclosure` |
| **Unit test** | `tests/unit/components/notification-permission-banner.test.ts` (#2115 AC8 / #2116 AC1-4 をカバー) |
| **E2E test** | `tests/e2e/notification-permission-banner.spec.ts` (#2115 AC7 / #2116 disclosure 描画 smoke) |

## §21 Phase Rewards-Cheer-Shop EPIC (#2266 / 2026-05-19)

### 21.1 設計背景

PO 報告 (2026-05-19) 「親管理画面で命名 × 実装 × 期待の三重ズレ + 応援 P 付与機能の完全欠落 + メッセージ単独機能の存在意義なし」を構造的に解消するため、ごほうび / 応援 / 公開テンプレート / 子供 shop の **4 系統責務再整理**を実施。

実装 PR: #2293 (Wave 16-A cheer 中核 / #2267 / #2270 / #2274 / #2275 / messages 廃止) + #2292 (Wave 16-B rewards CRUD / #2268 / #2269 / #2273) + 本 PR (Wave 16-C 用語 SSOT + 設計書同期 + retention 調査)。

### 21.4 用語 SSOT (#2276)

ADR-0045 §3.3 atom / compound 責務分離に従い、以下 atom を `src/lib/domain/terms.ts` に追加:

- `CHEER_TERMS` = { canonical: '応援', action: '応援する', reasonField: 'できごと' }
- `REWARD_TERMS` = { menu: 'ごほうび管理', shop: 'ごほうびショップ', preset: 'プリセット', canonical: 'ごほうび' }
- `TEMPLATE_TERMS` = { userFacing: 'みんなのテンプレート', short: 'テンプレート', browse: 'みんなのテンプレートを見る' }

「マーケットプレイス」リテラルは UI 露出禁止 (ADR-0041 確定移行)、`NAV_ITEM_LABELS` / `MARKETPLACE_LABELS` / `ADMIN_CHECKLISTS_PAGE_LABELS` / `ADMIN_RULES_PAGE_LABELS` / `REWARDS_LABELS` 等を atom 経由参照に整理。

#### marketplace type 名 + 概念アイコン SSOT (#2899)

marketplace top の type カード名 (`MARKETPLACE_TYPE_LABELS`、`src/lib/domain/marketplace-item.ts`) と `UnifiedImportHub` タブ名 (registry `displayLabel`、`src/lib/marketplace/`) は、ページタイトル「みんなのテンプレート」と重複させず兄弟 type と同型の単独名詞で統一する (命名規則 SSOT は DESIGN.md §6「marketplace type 命名規則」)。

| type | 表示名 (`MARKETPLACE_TYPE_LABELS` / `displayLabel`) | 概念アイコン (`MARKETPLACE_TYPE_ICONS` = `CONCEPT_ICONS`) |
|------|------|------|
| `activity-pack` | 活動セット (旧「みんなのテンプレート（活動）」を是正) | 📝 (`CONCEPT_ICONS.activity`、旧 📦 を是正。📋 は checklist と衝突するため不採用) |
| `reward-set` | ごほうびセット | 🎁 (`CONCEPT_ICONS.reward`) |
| `checklist` | チェックリスト | 📋 (`CONCEPT_ICONS.checklist` = `icons.ts` `ICON_CHECKLIST` 同値。✅ は activity 混同回避のため不採用) |
| `rule-preset` | とくべつルール | 📜 (`CONCEPT_ICONS.rule`) |
| `challenge-set` | チャレンジ集 | 🎯 (`CONCEPT_ICONS.challenge`) |

システム概念に紐づくアイコン (marketplace / overflow menu / admin 取込導線の「みんなのテンプレート」🏪 等) は `CONCEPT_ICONS` atom (`src/lib/domain/terms.ts`) を SSOT とし、各 admin 一覧の browse link は `{CONCEPT_ICONS.template} {TEMPLATE_TERMS.browse}` の形で参照する (直書きは `check-no-plan-literals.mjs` kind=concept-icon で検出)。ユーザーがカスタマイズする `item.icon` / `COMMON_ICONS` picker は対象外 (DESIGN.md §7 整合)。

### 21.5 DB schema 拡張

`parent_messages` テーブルに以下 2 列追加 (実装 PR #2293 / Wave 16-A):

- `bonus_points` INTEGER NULL — 応援 P 付与額（NULL = 旧来のメッセージのみ通知）
- `reward_category` TEXT NULL — 応援カテゴリ（運動 / 学習 / 生活 / 創作 / 社会 等）

詳細は `docs/design/08-データベース設計書.md` 参照。
