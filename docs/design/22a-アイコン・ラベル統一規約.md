# アイコン・ラベル統一規約

## 概要

本ドキュメントは、がんばりクエスト内で使用するアイコン（絵文字）とラベル（テキスト）の統一規約を定める。
#0289 の総点検で発見された不整合の再発を防止するための恒久的なルール。

## アイコン定数

全てのナビゲーション・セクションアイコンは `src/lib/domain/icons.ts` に一元定義されている。
新しいアイコンを追加する際は**必ずこのファイルに定数を追加**し、コンポーネントから参照すること。

### ナビゲーションアイコン

| 定数名 | 値 | 用途 |
|--------|-----|------|
| `ICON_HOME` | 🏠 | ホーム画面 |

### 使用禁止のアイコン

| アイコン | 理由 |
|---------|------|
| 🛡️ | 「防御」の意味。ステータスには不適切 |
| 👤 | 「プロフィール」の意味。「切替」には不適切 |
| 📊 | 「データ分析」の意味。子供向けUIには不適切（親管理画面のみ許容） |
| 🏅 | 「メダル」の意味。「称号」とは微妙にズレ |

## 本番・デモの整合性

- ボトムナビの項目数・構成は本番とデモで**必ず一致**させること
- デモは「本番の模擬体験」であり、構造的な差異は許容しない
- アイコン・ラベルの変更は両方に同時適用すること

#### CI gate 運用ルール（継続）

- **ratchet rule**: `scripts/hardcoded-strings-baseline.json` の `count` を上げる（増加方向）変更は禁止
- 新規追加コードでハードコード JP を増やすと CI red (`scripts/check-hardcoded-strings.mjs`)
- LP 側の同等 gate: `scripts/check-lp-ssot.mjs` (`scripts/lp-ssot-baseline.json`) も baseline 0 維持

#### 残されたフォローアップ

- TypeScript ファイル（非 Svelte / 非 HTML）の hardcoded JP は scope 外。`scripts/check-hardcoded-strings.mjs` は `src/**/*.svelte` のみを検査対象とする
- ESLint `local/no-hardcoded-jp-text` の `<style>` ブロック除外: 別 Issue で対応推奨（今回は実コメントの英語化で実質回避）

#### i18n 対応への展望 (#1683 / V2MOM Q3+)

baseline 0 達成により、`labels.ts` を `i18n` 対応辞書化する基盤が整った。英語版 LP / 多言語化が必要となった段階で、`labels.ts` の構造を `{ ja: {...}, en: {...} }` 形式に拡張することで対応可能。

### LP / Legal SSOT 100% 達成 (#1683 umbrella 完遂, 2026-04-30)

#### §1 設計背景

PR #1680 で Svelte 側の `scripts/check-hardcoded-strings.mjs` が baseline 0 を達成した時点でも、LP 側 (`site/**/*.html`) には依然として 693 件の日本語ハードコード（LP 339 件 + 法的文書 354 件）が残存していた。これは ADR-0009 §例外節で「法的文書は法務要件で固定」「SEO 対象 meta タグはクローラー JS 未実行」を理由に SSOT 化対象外として運用されていたため。一方、PO 方針「SSOT 漏れのコンテンツは存在してはいけない」と直接矛盾し、i18n 対応の前提整備が完成しない構造的問題があった。

#### §2 設計原則

1. **LP / Legal を含めた全ユーザー露出文言の SSOT 化**: SEO 静的 meta タグ (`<title>` 含む) は ADR-0009 §例外で残すが、それ以外の HTML 内日本語文言は全て `labels.ts` の `LP_*_LABELS` namespace 経由で `data-lp-key` 属性ベースに注入する
2. **innerHTML 注入 + DOMPurify サニタイズ**: 法的文書の `<h1>` / `<p>` / `<ul>` / `<ol>` / `<table>` / `<dl>` 等の構造タグを保ったまま SSOT 化するため、`textContent` 直接代入から `innerHTML = DOMPurify.sanitize(...)` 方式へ刷新（ADR-0025）
3. **印刷タイミング保証**: pamphlet.html は `window.print()` 起動前に `beforeprint` イベントで `reapplyLpKeys()` を再実行し、SSOT 注入が完了した状態で印刷タイミングに突入することを保証する
4. **逆方向 LEGAL coverage check**: `scripts/check-lp-ssot.mjs` は LP_LEGAL_*_LABELS の各 key が対応 HTML の `data-lp-key` で参照されているかを CI で検証

#### §3 仕様

| 項目 | 内容 |
|------|------|
| **対象ファイル** | site/index.html / pricing.html / faq.html / pamphlet.html / selfhost.html / help/license-key.html / privacy.html / terms.html / sla.html / tokushoho.html (全 10 件) |
| **削減数** | LP 339 件 (PR #1718 / #1683-B) + Legal 354 件 (PR #1717 / #1683-C) = **693 件 → 0 件** |
| **LEGAL namespace** | `LP_LEGAL_PRIVACY_LABELS` (17 keys) / `LP_LEGAL_TERMS_LABELS` (23 keys) / `LP_LEGAL_SLA_LABELS` (11 keys) / `LP_LEGAL_TOKUSHOHO_LABELS` (3 keys) = 計 54 keys |
| **CI gate** | `scripts/lp-ssot-baseline.json` `count: 0` 維持。`EXCLUDED_LEGAL_FILES` は撤廃 |
| **DOMPurify 配信** | `<script src="https://cdn.jsdelivr.net/npm/dompurify@3/dist/purify.min.js">` を全 10 ファイルの `<head>` に CDN 注入 |
| **印刷 E2E** | `tests/e2e/pamphlet-print-ssot.spec.ts` (#1704) |
| **ADR 状態** | ADR-0025 が `proposed (amended)` → **accepted** 昇格 / ADR-0009 §例外節「法的文書」を正式 supersede 表記に更新 |
