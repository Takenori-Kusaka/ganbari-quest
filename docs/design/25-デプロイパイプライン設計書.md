## 2. CI パイプライン

### 2.1 トリガー

| イベント | ワークフロー | 実行内容 |
|---------|------------|---------|
| Pull Request (open/sync) | `ci.yml` | lint + test + docker validate |
| Push to main | `deploy.yml` | test + build + deploy (AWS) |
| Dependabot PR | `dependabot-auto-merge.yml` | auto-merge (patch/minor) |
| スケジュール (毎週月曜) | `codeql.yml` | セキュリティスキャン |
| スケジュール (月初) | `cost-audit.yml` | AWS コスト監査 |

### 2.2 CI ジョブ構成（`ci.yml`）

```
Pull Request
    ├── [parallel] lint-and-test
    │   ├── biome check .
    │   ├── npx svelte-check
    │   └── npx vitest run
    ├── [parallel] e2e-test
    │   └── npx playwright test (Chromium)
    ├── [parallel] docker-build
    │   └── Docker build validation
    └── [parallel] CodeQL
        └── Security analysis (JavaScript/TypeScript)
```

全ジョブが PASS しなければ PR はマージ不可（GitHub ブランチ保護ルール）。

### 2.3 必須チェック

| チェック名 | 内容 | 必須 |
|-----------|------|------|
| `lint-and-test` | Biome lint + svelte-check 型検査 + Vitest ユニットテスト | ✅ |
| `e2e-test` | Playwright E2E テスト（Chromium） | ✅ |
| `docker-build` | Docker ビルド検証 | ✅ |
| `CodeQL` | セキュリティ脆弱性スキャン | ✅ |
| `dependabot` | Dependabot 互換性チェック | SKIP可 |
| `changes` | 変更ファイル検出（条件付き実行用） | ✅ |
| `label` | 自動ラベル付与 | ✅ |

## 3. AWS Lambda デプロイパイプライン（SaaS本番）

### 3.1 フロー

```
git push main
    → GitHub Actions (deploy.yml)
    → Step 1: Test (lint + unit + e2e)
    → Step 2: Storage CDK Deploy (DynamoDB + S3)
    → Step 3: Docker Build (ARM64 / Graviton)
    → Step 4: ECR Push (Amazon ECR)
    → Step 5: CDK Deploy All (CloudFront + Lambda + API Gateway + Cognito + SES)
    → Step 6: Lambda Function Update (新イメージ適用)
```

### 3.2 Docker イメージ

| 項目 | 値 |
|------|-----|
| ベースイメージ | `node:22-slim` |
| アーキテクチャ | ARM64（AWS Graviton） |
| ランタイム | Lambda Web Adapter (aws-lambda-web-adapter) |
| ビルド | Multi-stage（build → production） |
| ECR リポジトリ | `ganbari-quest` (ap-northeast-1) |

### 3.3 CDK スタック順序

```
1. GanbariQuestStorage (DynamoDB, S3, ECR)  ← 依存なし
2. GanbariQuestAuth (Cognito, SSM)           ← 依存なし
3. GanbariQuestCompute (Lambda, Function URL) ← Storage, Auth に依存
4. GanbariQuestNetwork (CloudFront, Route53)  ← Compute に依存
5. GanbariQuestOps (CloudWatch, SNS, Budgets) ← Compute, Storage, Network に依存
6. GanbariQuestSes (SES, 受信パイプライン)     ← 依存なし
```

### 3.4 環境変数・シークレット

| シークレット | 格納先 | 用途 |
|------------|--------|------|
| `AWS_ACCESS_KEY_ID` | GitHub Secrets | AWS認証 |
| `AWS_SECRET_ACCESS_KEY` | GitHub Secrets | AWS認証 |
| `AWS_REGION` | GitHub Secrets | ap-northeast-1 |
| SSM Parameters | AWS SSM | アプリ設定（Cognito ID, Stripe Key 等） |

### 3.5 ロールバック手順

1. `gh run list` で直近の成功デプロイを特定
2. 該当コミットに `git revert` して main に push → 自動デプロイ
3. 緊急時: AWS Console から Lambda の前バージョンに手動切替

## 4. NUC ローカルサーバーデプロイ（LAN内）

### 4.1 デプロイ手順

**順序厳守**: stop → migrate → build → up（#0099 WAL 破損障害の教訓）

```bash
# 1. コンテナを安全に停止（graceful shutdown で WAL flush）
ssh kusaka-server@192.168.68.79 "cd C:\Docker\ganbari-quest && docker compose stop app"

# 2. DB マイグレーション（コンテナ停止中 = 競合なし）
ssh kusaka-server@192.168.68.79 "cd C:\Docker\ganbari-quest && node scripts/add-xxx.cjs data/ganbari-quest.db"

# 3. pull → ビルド → 起動
ssh kusaka-server@192.168.68.79 "cd C:\Docker\ganbari-quest && git pull && docker compose build && docker compose up -d"

# 4. 動作確認
ssh kusaka-server@192.168.68.79 "curl -s http://localhost:3000/api/health"
```

### 4.2 NUC サーバー構成

| 項目 | 値 |
|------|-----|
| ホスト | `192.168.68.79` (LAN内) |
| OS | Windows |
| Docker パス | `C:\Docker\ganbari-quest` |
| DB パス | `C:\Docker\ganbari-quest\data\ganbari-quest.db` |
| DB モード | SQLite WAL |
| 認証 | PIN コードのみ（LAN内限定） |
| アーキテクチャ | x86_64 |

### 4.3 NUC デプロイの禁止事項

- `docker compose up -d` だけで済ませない（WAL 不整合リスク）
- コンテナ実行中に DB ファイルを直接操作しない
- マイグレーションスクリプトをコンテナ内で実行しない（ホスト側で実行）

## 5. ブランチ戦略

### 5.1 ブランチ運用

| ブランチ | 用途 | 保護 |
|---------|------|------|
| `main` | 本番（AWS 自動デプロイ対象） | ✅ 直 push 禁止 |
| `feat/*`, `fix/*`, `infra/*` | 作業ブランチ | - |
| `design/*` | 設計・UI 変更 | - |
| `dependabot/*` | 依存更新（自動） | - |

### 5.2 PR ワークフロー

```
1. ブランチ作成 (feat/xxx, fix/xxx 等)
2. 実装 + テスト
3. gh pr create --draft
4. CI 全通過確認
5. gh pr ready <番号>
6. レビュー → マージ（レビューチーム）
7. main push → AWS 自動デプロイ
```

### 5.3 ブランチ保護ルール

- main への直接 push 禁止
- Draft PR はマージ不可
- 全 CI チェック必須
- Force push 禁止
