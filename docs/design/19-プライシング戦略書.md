## 1. プライシング方針

### 1.1 基本原則

1. **ワンコイン以下の心理的閾値**: スタンダードプランは月額500円を超えない
2. **チューニングは無料、創造は有料**: 既存機能の調整は制限せず、新しいコンテンツの追加で課金
3. **データ人質にしない**: 無料→有料は機能制限で動機づけ、データ削除ではなくアーカイブ
4. **セルフホスト版は全機能無料**: OSSとしての信頼性を維持
5. **個人開発者として持続可能**: AWS原価＋最小限の固定費を有料ユーザー3名で回収可能

### 1.2 ターゲット層の支払い感度

| 区分 | 月額相場 | 出典 |
|------|---------|------|
| 子供向け教育サブスク | ¥500〜2,000 | スマイルゼミ、チャレンジタッチ |
| 子供向けアプリ課金（中央値） | ¥300〜1,000 | App Annie / data.ai 2025 |
| 習慣化アプリ | ¥300〜800 | みんチャレ、Habitica |
| 育児記録アプリ | ¥0〜400 | ぴよログ、ぴよログプレミアム |

**判断**: 月額500円はこの市場の「安い」〜「妥当」の間に位置し、導入障壁が最小。

## 2. 料金プラン（確定版）

### 2.1 プラン一覧

|  | フリー | スタンダード | ファミリー |

| **年額** | ¥0 | ¥5,000（17%OFF） | ¥7,800（17%OFF） |
| **対象** | お試し・ライトユーザー | 子供1〜2人の家庭 | 3人以上 or 全機能利用 |
| **Stripe Price ID** | — | `STRIPE_PRICE_STANDARD_MONTHLY` | `STRIPE_PRICE_FAMILY_MONTHLY` |
| **年額 Price ID** | — | `STRIPE_PRICE_STANDARD_YEARLY` | `STRIPE_PRICE_FAMILY_YEARLY` |

### 2.2 価格設定根拠

**スタンダード ¥500/月**:
- 「ワンコイン」の分かりやすさ
- 競合最安値帯（ぴよログ ¥400、みんチャレ ¥500）と同水準
- Stripe手数料後の手取り: ¥482/月

**ファミリー ¥780/月**:
- スタンダードの1.56倍 — 「もうちょっと出せば全部」の心理
- 子供3人以上の家庭は教育費意識が高く、¥1,000以下なら抵抗感が低い
- 競合上位プラン（Habitica $5≒¥750、OurHome $4.99≒¥749）と同水準

**年額 17%OFF**:
- 2ヶ月分無料に相当（分かりやすい訴求）
- 業界データ: 年額プラン利用者の継続率は月額の2.5倍（RevenueCat 2025）
- スタンダード年額: ¥5,000（月額換算 ¥417）
- ファミリー年額: ¥7,800（月額換算 ¥650）

## 4. 無料→有料の転換戦略

### 4.1 トライアル設計（#314 改定）

| 要素 | 仕様 |
|------|------|
| 期間 | 7日間（デフォルト。admin_grant/campaign で変更可能） |
| 機能範囲 | スタンダード相当（デフォルト。tier パラメータで family も可能） |
| 開始方式 | ユーザーがライセンス管理画面から明示的に開始（自動開始廃止） |
| 乱用防止 | `trial_history` テーブルで履歴管理。`user_initiated` は1テナント1回 |
| 再付与 | `campaign` / `admin_grant` ソースは再付与可能 |
| Stripe連携 | Stripe側 `trial_period_days` 廃止、アプリ側で一元管理 |
| トライアル終了後 | フリープランに戻る（Stripe自動課金なし） |

### 4.2 自然な導線（ナッジポイント）

| シーン | トリガー | 表示 |
|--------|---------|------|
| 子供3人目追加 | `checkChildLimit()` | 「スタンダードプランで家族みんな使えます」 |
| カスタム活動4個目 | `checkActivityLimit()` | 「もっとオリジナル活動を作りたい？」 |
| 履歴91日目到達 | 履歴アーカイブ処理 | 「過去の記録をいつでも見返せるようにしましょう」 |
| 実績ロック表示 | 実績一覧画面 | 「スタンダードプランで全バッジが解除できます」|
| データエクスポート試行 | エクスポートボタン | 「お子さんの成長記録をJSON形式で保存」 |

### 4.3 リバーストライアル（段階的制限）

新規登録後の体験フロー:

```
Day 0-7   : 全機能解放（トライアル）
Day 8-97  : フリープラン適用（制限あり、データは保持）
Day 98+   : 90日超の履歴をアーカイブ（削除ではない）
            → 「アップグレードするとすべての履歴を復活できます」
```

**データアーカイブ方式**（#0269 で実装予定）:
- 90日超の履歴は非表示にするが、DBからは削除しない
- 有料プランにアップグレードすれば即座に全データ復活
- 「データ人質」ではなく「保管してあげている」というポジショニング

### 4A.1 価値軸: 「家族みんなで見守る」

## 5. 決済経路と手数料

### 5.1 現行（Phase 1: Web直販のみ）

| 決済経路 | 手数料率 | スタンダード手取り | ファミリー手取り |
|---------|---------|----------------|----------------|
| **Stripe（Web）** | 3.6% | ¥482/月 | ¥752/月 |

### 5.2 将来（Phase 2: アプリストア展開時）

| 決済経路 | 手数料率 | スタンダード手取り | ファミリー手取り |
|---------|---------|----------------|----------------|
| Google Play（$1M未満） | 15% | ¥425/月 | ¥663/月 |
| Apple App Store（$1M未満） | 15% | ¥425/月 | ¥663/月 |
| Google Play（$1M以上） | 30% | ¥350/月 | ¥546/月 |
| Apple App Store（$1M以上） | 30% | ¥350/月 | ¥546/月 |

**方針**: $1M未満である限り Small Business Program (15%) が適用される。個人開発で$1Mに到達する可能性は極めて低いため、15%前提で計画。

### 5.3 推奨: Web直販を最優先

- ストア手数料の差: 3.6% vs 15% → 差額は月額 ¥57/ユーザー
- 100有料ユーザーの場合: 年間 ¥68,400 の差
- Web版 PWA として提供すればアプリストア経由不要
- ストア展開は「リーチ拡大」目的であり「収益最大化」ではない

## 6. 損益分岐点分析

### 6.1 前提条件

```
固定費:
  AWS (最小構成)    : ¥342/月
  ドメイン          : ¥117/月
  バーチャルオフィス : ¥500/月（任意）
  ────────────────────────
  合計              : ¥459〜959/月

変動費:
  Stripe手数料: 売上 × 3.6%
```

### 6.2 損益分岐点

```
スタンダード1名あたり純収益 = ¥500 × (1 - 0.036) = ¥482/月
ファミリー1名あたり純収益 = ¥780 × (1 - 0.036) = ¥752/月

損益分岐点（固定費 ¥459）:
  スタンダードのみ: 459 / 482 ≈ 1名
  ファミリーのみ: 459 / 752 ≈ 1名

損益分岐点（固定費 ¥959, バーチャルオフィス含む）:
  スタンダードのみ: 959 / 482 ≈ 2名
  ファミリーのみ: 959 / 752 ≈ 2名
```

**有料ユーザー2名で黒字化**。これはSaaSとして極めて低い損益分岐点であり、個人開発の持続可能性を強く裏付ける。

## 7. 実装ロードマップ

### 7.0 License ↔ Stripe ライフサイクル SSOT

License Key と Stripe Subscription のライフサイクル（発行・更新・解約・失効・idempotency）の canonical 定義は **`docs/design/license-subscription-causality.md`** を参照。本書ではプラン料金・機能制限を定義し、状態遷移のロジックは因果関係マップを正とする。

- Stripe = source of truth（自アプリ DB は Stripe 状態のキャッシュ）
- `current_period_end` = license の `expiresAt`
- Dunning は Stripe に完全一任（`invoice.payment_failed` では何もしない）

### 7.5 解約タイミングとデータライフサイクル (#741, #784)

**決定**: 解約時は Stripe Subscription を**即時キャンセル**し、データは **30 日間の `grace_period`** で保持する。

#### 7.5.1 動機

- Stripe 側で課金が継続したまま DB だけ状態遷移すると、**チャージバック・Stripe アカウントリスク**が発生する（#784 の根本原因）。
- `cancel_at_period_end=true` では請求期間終了まで課金が続き、ユーザーが「解約したのに請求が来た」と感じる。
- データを即時削除するとアカウント削除の誤操作が致命的になるため、`grace_period` で 30 日間の撤回猶予を用意する。

#### 7.5.2 状態遷移

```
active ──[POST /admin/tenant/cancel]──▶ Stripe cancel (即時)
                                          │
                                          ▼
                                      grace_period (30 日)
                                          │
                 ┌────────────────────────┼───────────────────┐
                 │                        │                   │
         [reactivate]              [30 日経過]       [DELETE account]
                 │                        │                   │
                 ▼                        ▼                   ▼
          409 + /pricing              terminated          terminated
          (再 Checkout が必要)        (データ削除)        (#741 fix)
```

#### 7.5.3 実装上の必須ルール

| ルール | 理由 | 実装場所 |
|--------|------|---------|
| 解約時は Stripe cancel → DB 更新の順序 | Stripe 失敗時に DB だけ進むと課金継続クレームに直結 | `api/v1/admin/tenant/cancel/+server.ts` |
| Stripe 呼び出し失敗時は 500 を返し DB 未更新 | 整合性 > 可用性（同 PR #741 のアカウント削除と同じパターン） | 同上 |
| reactivate 時に `stripeSubscriptionId` が無ければ 409 + `redirectTo: /pricing` | 単に DB を active に戻しても課金は復活しないため、再購読必須の旨を明示 | `api/v1/admin/tenant/reactivate/+server.ts` |
| 解約後の再購読は新規 Checkout | `stripe.subscriptions.cancel` は復活不可（Stripe 仕様） | フロントエンドで `/pricing` へ遷移 |

## 7A. ライセンスキー運用

> **SSOT**: ライセンスキーのフォーマット・ライフサイクル・API・DB スキーマ・状態遷移は **[license-key-lifecycle.md](./license-key-lifecycle.md)** を参照。本章ではプライシング戦略との接続点のみ記載する。

### 7A.1 プライシングとの接続

- **発行トリガ**: Stripe Checkout 完了 (`checkout.session.completed` webhook) で自動発行
- **配布方法**: 決済完了メールに記載 + `/admin/license` から再取得可能
- **消費タイミング**: ユーザーが `/admin/license` 画面から consume API を叩くことで有料プラン昇格
- **プラン紐付け**: `LicenseRecord.plan` に `standard-monthly` / `standard-yearly` / `family-monthly` / `family-yearly` のいずれかを保存（Stripe Price ID から解決）
- **有効期限**: 発行から 90 日で自動失効 (`status='revoked'`, `revokedReason='expired'`)。期限前 7 日で通知メール送信
- **返金対応**: Stripe 返金時に `revokedReason='refund'` で失効 + テナントを `expired_free` に降格

### 7A.2 決済との因果関係

| プライシングイベント | License Key 状態 | 関連 |
|-------------------|----------------|------|
| 初回 Checkout 完了 | 新規発行 (`active`) | [license-subscription-causality.md](./license-subscription-causality.md) |
| ユーザー consume 操作 | `consumed` + テナント `paid` 昇格 | #824 |
| `invoice.paid` (継続課金) | — (License Key 自体は発行時に一度だけ使用) | — |
| `customer.subscription.deleted` | — (License Key は独立して consumed 状態維持) | — |
| 返金処理 | `revoked` (`reason='refund'`) | #824 |

> **重要**: ライセンスキーは「初回有料化のきっかけ」として設計されており、継続課金は Stripe subscription 自体で管理される。ライセンスキーが consume された後、テナントは Stripe subscription のライフサイクルに従って `active` → `grace_period` → `expired_retention` → `expired_free` と遷移する。

### 7A.3 競合との差別化

競合分析の詳細は [license-key-competitor-analysis.md](./license-key-competitor-analysis.md) (#811) 参照。本プロダクトの差別化ポイント:

- **HMAC-SHA256 署名付きキー**: 総当たり攻撃を 32^20 ≒ 10^30 まで困難化（Keygen.sh/JetBrains と同等）
- **90 日有効期限**: Gumroad/Lemon Squeezy の無期限キーに対し、漏洩リスクを時間軸で制限
- **完全監査ログ**: 全状態遷移を `LicenseEvent` に 7 年保持（会計書類保存期間対応）

## 8. KPI と見直し基準

### 8.1 追跡KPI

| KPI | 目標 | 測定方法 |
|-----|------|---------|
| 無料→有料転換率 | 5〜7% | 有料ユーザー数 / 全MAU |
| スタンダード:ファミリー比率 | 80:20 | Stripe ダッシュボード |
| 月額:年額比率 | 40:60 | Stripe ダッシュボード |
| 月次解約率 | <5% | Stripe Churn Analysis |
| ARPU | ¥550+ | 月間売上 / 有料ユーザー数 |
| トライアル→有料転換率 | 30%+ | Stripe Checkout 完了率 |

### 8.2 価格見直しトリガー

以下のいずれかに該当した場合、価格改定を検討する:

1. **転換率 < 3%** が3ヶ月連続 → 価格が高すぎる可能性、値下げ or 無料機能拡充を検討
2. **転換率 > 10%** が3ヶ月連続 → 価格が安すぎる可能性、新プラン追加を検討
3. **解約率 > 10%** が2ヶ月連続 → 価値が伝わっていない、機能差別化を見直し
4. **ファミリー比率 > 40%** → 需要があるため上位プラン（¥1,200程度）追加を検討
5. **AWS原価が月間売上の20%超** → インフラ最適化 or 価格引き上げ

## 10. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| 転換率が想定以下 | 収益不足 | 無料機能の絞り込み、ナッジポイント強化 |
| ストア手数料の負担増 | 利益率低下 | Web直販を優先誘導、PWA推進 |
| 競合の無料化 | 価格競争力低下 | OSS＋カスタマイズ性で差別化（価格以外の軸） |
| 為替変動（ストア展開時） | ドル建て手数料の増加 | JPY建て決済のみ当面維持 |
| 機能制限の不満 | 離脱増加 | 「チューニング無料」原則で基本体験は損なわない |

## 8. キャンペーン運用（#802 / #803）

### 8.1 キャンペーン配布の 2 経路

運営がユーザーに「無料でプランを解放する」手段は 2 経路ある。用途に応じて使い分ける。

| 観点 | 経路 A: Ops キャンペーンキー (#802) | 経路 B: Stripe 100% OFF プロモコード (#803) |
|------|-----------------------------------|---------------------------------------------|
| 発行元 | `/ops/license/issue` フォーム | Stripe Dashboard → Coupons / Promotion codes |
| 配布物 | `GQ-XXXX-...` のライセンスキー文字列（メール/CSV） | Checkout URL + プロモコード |
| ユーザー体験 | `/auth/signup` でキー入力 → 即プラン解放 | Stripe Checkout で CC 入力 → 本人確認後プラン解放 |
| 信用コスト | ゼロ（CC 不要） | Stripe の本人確認（fraud check）を通せる |
| 解約時の課金 | なし（一回限りの N 日解放） | Subscription が残る → `customer_portal` で解約必要 |
| 監査ログ | `ops_audit_log.license.issue` + `license_events.issued` | Stripe Dashboard + `invoice.paid` webhook log |
| 不正リスク | Ops の手動配布に依存 | プロモコードが Twitter 等に流出すると無限配布される |
| 向いている用途 | プレゼント / CS 補填 / インフルエンサー個別付与 | Black Friday 等の公開キャンペーン / 期間限定大量配布 |

### 8.2 推奨運用

**原則として経路 A (#802) を優先**する。以下の場合のみ経路 B を使う:

- 公開キャンペーン（SNS 等で URL/コードを配布する）で、Stripe の本人確認機構を通したい
- 大量 (100 件以上) に配る予定で、個別キー配布のメール運用が非現実的
- CC 登録済みユーザーに後続のアップセルを狙いたい（Subscription は残るので自動更新）

経路 B を選ぶ場合、プロモコードの流出対策として必ず以下を設定する:
- **Max redemptions**: 上限回数（例: 100 件）
- **Expires at**: 有効期限（例: 14 日）
- **First-time order only**: 既存顧客の再利用防止

### 8.3 経路 B (Stripe 100% OFF) の技術的注意点

`createCheckoutSession` は `allow_promotion_codes: true` を既に設定しているため、
Stripe Dashboard で Coupon + Promotion code を作成するだけで Checkout に適用できる。

100% OFF Checkout が完了したときの Webhook 動作:

| イベント | 動作 |
|---------|------|
| `checkout.session.completed` | `handleCheckoutCompleted` が tenant の `plan` / `status='active'` / `trialUsedAt` を更新。`issueLicenseKey` で `kind='purchase'` / `issuedBy='stripe:<sessionId>'` のキーを発行 (Customer の email に送信) |
| `invoice.paid` | amount_total=0 の場合も通常通り発火する。`handleInvoicePaid` は plan 更新・`planExpiresAt` 算出を実施 |
| `customer.subscription.updated` | 通常通り発火 |

ユーザー側は `/admin/license` で通常の有料プランと同じ表示になる（100% OFF の痕跡は Stripe Dashboard のみ）。

### 8.4 Stripe プロモコード作成手順（Ops 向け）

Stripe CLI ベースの手順:

```bash
# 1. Coupon を作成（100% OFF, 1 回限り）
stripe coupons create \
  --id "CAMPAIGN_2026_SPRING" \
  --percent-off 100 \
  --duration "once" \
  --max-redemptions 100 \
  --redeem-by $(date -d "+14 days" +%s)

# 2. Promotion code を発行（ユーザーに配布するのはこちら）
stripe promotion_codes create \
  --coupon "CAMPAIGN_2026_SPRING" \
  --code "GQ2026SPRING"
```

または Stripe Dashboard → Products → Coupons → Create coupon → Add promotion code。

発行後、ユーザーには以下の URL を配布:
- `https://ganbari-quest.com/admin/billing` から「プランを契約する」→ Checkout 画面で「プロモコードを追加」から `GQ2026SPRING` を入力

### 8.5 監査と振り返り

どちらの経路でも、キャンペーン実施後は以下を確認する:

- 経路 A: `/ops` のライセンスイベント一覧で `issued` ログを確認、`consumed` 率（実際にサインアップに至った割合）を集計
- 経路 B: Stripe Dashboard の Coupon 利用レポート（redemption count / 総割引額）

転換率が低い場合、次回は経路を入れ替える or 配布手段そのものを見直す。
