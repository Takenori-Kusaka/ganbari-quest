# セキュリティ設計書

## 1. 概要

がんばりクエストにおけるセキュリティ要件と実装を定義する。子供（未成年者）のデータを扱うアプリケーションとして、データ保護に特に注意を払う。

### 1.1 デプロイ環境

| 環境 | 認証モード | ネットワーク | 用途 |
|------|-----------|-------------|------|
| AWS Lambda（本番） | `cognito` | インターネット公開（CloudFront + HTTPS） | SaaS版 |
| NUC サーバー（ローカル） | `local` | LAN内限定（HTTP） | 自宅専用 |

セキュリティ要件は環境ごとに異なり、`AUTH_MODE` 環境変数で制御される。

---

## 2. 通信暗号化

### 2.1 本番環境（AWS）

```
[ユーザー] --HTTPS/TLS 1.2+--> [CloudFront] --HTTPS--> [Lambda Function URL]
```

- **TLS ポリシー**: CloudFront で `TLSv1.2_2021` を強制（`SecurityPolicyProtocol.TLS_V1_2_2021`）
- **証明書**: ACM（AWS Certificate Manager）による自動管理・自動更新
- **HSTS**: `Strict-Transport-Security: max-age=31536000; includeSubDomains`（cognito モードのみ hooks.server.ts で付与）
- **CloudFront → Lambda 間**: Lambda Function URL は HTTPS のみ

### 2.2 ローカル環境（NUC）

- LAN 内限定のため HTTP 通信を許容
- HSTS ヘッダは付与しない（cognito モード以外では省略）

## 3. 保存データ暗号化

### 3.1 DynamoDB（本番DB）

- **暗号化**: AWS managed key（デフォルト暗号化、DynamoDB 側で自動適用）
- **PITR**: 無効（コスト最適化）。代わりに AWS Backup で日次バックアップ（3日保持）

### 3.2 S3（アセットバケット）

- **暗号化**: SSE-S3（`BucketEncryption.S3_MANAGED`）
- **パブリックアクセス**: 全ブロック（`BlockPublicAccess.BLOCK_ALL`）
- **アクセス**: Lambda IAM ロール経由のみ
- **ライフサイクル**: `backups/` は30日で自動削除、`logs/` は1日後に Glacier へ

### 3.3 SQLite（NUCローカル）

- ファイルシステムレベルの保護に依存（LAN 内限定環境）
- WAL モード使用（`PRAGMA journal_mode=WAL`）

### 3.4 Cookie

| Cookie 名 | 用途 | 属性 |
|-----------|------|------|
| `identity_token` | Cognito ID Token（有効期限 1 時間） | HttpOnly, Secure, SameSite=Lax, Path=/ |
| `gq_refresh` | Cognito Refresh Token（有効期限 30 日、サイレントリフレッシュ用）(#1365) | HttpOnly, Secure, **SameSite=Strict**, Path=/ |
| `context_token` | 署名付きテナント情報 | HttpOnly, Secure, SameSite=Lax, Path=/ |
| `invite_code` | 招待コード一時保存 | HttpOnly, Secure, SameSite=Lax, Path=/ |

### 3.5 Stripe Webhook Events table (Phase 7 PR-1 / #2685)

Stripe webhook の at-least-once delivery + replay/resend に対する冪等性確保のため、`stripe_webhook_events` table で `event_id PRIMARY KEY` の dedup logic を構築する。本 table は **sensitive data 取扱を伴うため**、以下の方針で運用する。

#### 列構成と sensitive data 範囲

| 列 | 型 | sensitive 区分 | 保持期間 |
|----|----|---------------|---------|
| `event_id` | TEXT PK | low (Stripe 内部 ID) | 90 日 (ADR-0049 retention 整合) |
| `event_type` | TEXT | low (`customer.subscription.updated` 等) | 同上 |
| `processed_at` | TEXT (ISO 8601) | low | 同上 |
| `handler_result` | TEXT (`success` / `error` / `skipped`) | low | 同上 |
| `retry_count` | INTEGER | low | 同上 |
| `customer_id` | TEXT (`cus_xxx`) | mid (Stripe customer 識別子、内部結合用途) | 同上 |
| `error_message` | TEXT (nullable) | **high (Stripe.Error.param に email 等 PII 流入の可能性)** | 同上 |
| `payload` | TEXT JSON (nullable、Phase 7 PR-4a 判断) | **high (event payload 内に email / 名前等 PII 含む可能性)** | 同上 |

#### 中間期間 (expand → contract) の PII 露出 risk と緩和

Phase 7 は expand-then-contract pattern で PR-1 (本 PR、expand) → PR-4a (contract、webhook handler 統合) と段階配備する。**expand 段階 (本 PR) から contract 完了 (PR-4a merge) までの中間期間**には以下の risk が存在する。

| risk | 緩和策 | 実装 PR |
|------|-------|--------|
| `error_message` 列に `Stripe.Error.param` 由来の email / 名前等 PII 流入 | `error_message` 書込み時に PII strip logic (email regex / 名前 pattern 除去) を適用 | Phase 7 PR-4a (webhook handler 統合) |
| `payload` 列に Stripe event payload (customer.email 含む) 全文保存 | payload 保存方針を「event_type のみ + customer_id のみ」に限定、payload 全文保存しない | Phase 7 PR-4a 判断 |
| 中間期間に対象 table の存在自体が漏洩 risk となる scenario | DynamoDB は IAM Execution Role 物理分離 (ADR-0048) + tenant ID 必須、sqlite は NUC ローカル限定 | 既存 (本 PR 配備時点で有効) |

#### 暗号化 + 保持期間方針 (ADR-0049 retention 整合)

- **DynamoDB**: 既存 server-side encryption (AWS KMS managed key) + TTL 属性 (`ttl`) で 90 日後自動削除 (`storage-stack.ts:29 timeToLiveAttribute='ttl'`)
- **sqlite (NUC)**: ローカルディスク暗号化は user 環境依存。Phase 7 PR-4a で `lazy-startup-migrations.ts` に「90 日超過 row 削除」logic 追加判断
- **保持期間**: ADR-0049 §3 retention policy (free / trial / standard / family / nuc) に整合。webhook event log は subscription lifecycle audit 用途のため `standard / family / nuc` plan で 90 日 retention、free / trial は archive 後即削除

#### PII strip assertion (Phase 7 PR-4a Pre-Ready checklist)

Phase 7 PR-4a での webhook handler 統合時、以下の E2E assertion を追加する:

```
// Test mode で Stripe.Error.param に email 含む event を流す
// → stripe_webhook_events.error_message 内に email 不在を assert
expect(insertedRow.error_message).not.toMatch(/[a-z0-9._%+-]+@[a-z0-9.-]+/i);
expect(insertedRow.payload ?? '').not.toMatch(/[a-z0-9._%+-]+@[a-z0-9.-]+/i);
```

## 4. 認証

### 4.1 Cognito モード（本番 SaaS）

二層認証アーキテクチャを採用:

```
Layer 1: Identity（Cognito JWT）
  └ identity_token Cookie → JWT署名検証 → { userId, email }
      ↓ 期限切れ / 存在しない場合
      └ gq_refresh Cookie → Cognito /oauth2/token (refresh_token grant)
            ├── 成功 → 新 identity_token をセット → return identity
            └── 失敗 → gq_refresh を削除 → return null

Layer 2: Context（署名付きトークン）
  └ context_token Cookie → HMAC-SHA256検証 → { tenantId, role, licenseStatus }
```

**サイレントリフレッシュ (#1365)**: `resolveIdentity()` 内で ID Token が期限切れの場合に `gq_refresh` Cookie を用いて自動更新。ログアウト時は `/oauth2/revoke` エンドポイントで Refresh Token を失効させる。

**refresh cookie 保存経路 (#1365 / #3022)**: `gq_refresh` の保存箇所は ① OAuth callback (`src/routes/auth/callback/+server.ts`) ② signup 自動ログイン (`src/routes/auth/signup/+page.server.ts`) ③ email/password ログイン (`src/routes/auth/login/+page.server.ts` の `establishSession`、通常 / MFA / confirmCode の全成功経路) の 3 経路。新規ログイン経路を追加する際は `setRefreshCookie` (`src/lib/server/auth/providers/cognito-oauth.ts`) の配線を必須とする — identity_token のみ保存するとセッションが 1 時間で失効する。

**Cognito User Pool 設定:**

| 設定項目 | 値 |
|---------|-----|
| サインイン方法 | Email + Password |
| MFA | OPTIONAL（TOTP 対応、SMS 無効） |
| パスワードポリシー | 8文字以上、大小英字+数字必須、記号不要 |
| Email 検証 | 自動（確認コード方式） |
| Email 属性 | `required: true`, **`mutable: true`**（ADR-0017 / #1366: Google IdP 再認証時の更新拒否回避のため） |
| アカウント回復 | Email のみ |
| カスタム属性 | `tenantId`, `role` |
| アクセストークン有効期限 | 1時間 |
| ID トークン有効期限 | 1時間 |
| リフレッシュトークン有効期限 | 30日 |

> **注 (ADR-0017 / #1366)**: Email 属性は `mutable: true` に設定している。`mutable: false` の場合、Google OAuth の再認証時に Cognito が IdP から取得した email を「属性更新」として扱い `user.email: Attribute cannot be updated` エラーでログイン不能になるため。ユーザー自身が email を変更する UI は提供しない (Google 側で email が変わった場合のみ追従する)。

**JWT 検証:**

- `cognito-jwt.ts` で Cognito 公開鍵（JWKS）を取得・キャッシュ
- 署名アルゴリズム: RS256
- `iss`（発行者）と `aud`（クライアントID）を検証
- 有効期限（`exp`）を検証

**Context Token:**

- HMAC-SHA256 署名（`CONTEXT_TOKEN_SECRET` 環境変数）
- ペイロード: `{ tenantId, childId?, role, licenseStatus }`
- 有効期限: 24時間（Context 再取得間隔）

### 4.2 Local モード（NUC）

- 親の管理画面: おやカギコード認証（4桁、ブラウザ Cookie で保持）。legacy local 経路（`/login` の `login()` / `changePin()` の現コード照合）のみ `pin_hash` 未設定時に デフォルト値 `5086`（「がんばり」語呂合わせ）へフォールバック（#1360 互換）。parent-gate 経路は §4.3 のとおり未設定時は初回作成フロー（#2992、既定 PIN 照合なし）。脅威モデルは家庭内の軽い仕切り（子供が誤って管理画面に入らないためのロック）であり金融系 PIN 相当の強度は不要
- 子供画面: 認証なし（LAN 内限定のため）
- tenantId は固定値 `'local'`

> **PIN 機構の SSOT 位置付け (#2396 で再確認、2026-05-22)**: `src/lib/domain/validation/auth.ts` 冒頭の「後方互換: PIN 認証関連（#0134 で段階的に削除予定）」コメントは ADR-0050 受理 (2026-05-20) 以降前提が崩れ、PIN 機構は **§4.2 NUC ローカル認証 / §4.3 Parent-Gate / §4.4 PIN reset / Stripe portal 二段階確認 (#771)** の 4 機構で能動利用される現役 SSOT に転じている。段階削除は **実施しない**。詳細: [`docs/operations/pin-auth-legacy-migration-plan.md`](../operations/pin-auth-legacy-migration-plan.md)

### 4.3 親 PIN gate + 15 分 sliding session (EPIC #2310 / ADR-0050)

**脅威モデル**: 同端末（タブレット / ノート PC）を親と子供で共有する家庭において、子供が `/switch` の「🔒 ご家族の見守り画面」リンク経由で `/admin/*` に到達すること（PIN なしで親画面が無防備に晒される構造的問題）。Apple Screen Time / BusyKid 等の業界 prior art と整合させて 4 桁 PIN gate + httpOnly signed cookie + 15 分 inactivity sliding timeout を導入する。

#### 仕様

| 項目 | 値 | 根拠 |
|---|---|---|
| PIN 桁数 | 4 桁 (将来 6 桁拡張可) | NIST SP 800-63B-4 AAL1 + Apple Screen Time 同設計 |
| 認証方式 | 既存 `verifyPin` (auth-service.ts) + bcrypt | 流用、独自実装回避 (ADR-0014) |
| Lockout | 既存 `verifyPin` の throttling (連続失敗カウンタ + lockedUntil)。ロック時は modal に**解除の絶対時刻**「HH:MM まで待ってから」を表示 (#2991) | NIST AAL1 throttling 要件 + 残り時間明示 (SP 800-63B / iOS 同型、秒カウントダウン不採用) |
| Cookie 名 | `gq_parent_session` | SSOT: `parent-gate-session.ts` の `PARENT_SESSION_COOKIE_NAME` |
| 署名 | `cookie-signature` (HMAC-SHA256) | ADR-0050 で OSS 4 件比較 (cookie-signature / jose / iron-session / lucia-auth) |
| 暗号化 | なし | payload に PII 含まないため不要 (改ざん防止のみ) |
| Cookie 属性 | `httpOnly + sameSite=lax + secure(本番)` | OWASP Session Management Cheat Sheet |
| Inactivity timeout | 15 分 (sliding refresh) | NIST SP 800-63B-4 AAL1 推奨値 |
| Hard max | 24 時間 | cookie maxAge、家庭利用文脈で daily reset |
| 署名キー env | `PARENT_GATE_COOKIE_SECRET` (≥ 16 chars) | production 未設定で起動時 throw、dev は fallback + 警告 |

#### 有効化条件

| AUTH_MODE | COGNITO_DEV_MODE | PIN gate 動作 | 理由 |
|---|---|---|---|
| `anonymous` (demo Lambda、ADR-0048) | - | 無効 | demo 環境では PIN 機構自体が無意味 |
| `local` (dev / NUC) | - | 無効 | Pre-PMF Bucket B、認証なし開発者用途 |
| `cognito` + `COGNITO_DEV_MODE=true` | true | 無効 | E2E setup / DEV_USERS 経由の既存全 spec を破壊しないため |
| `cognito` (production) | (false) | **有効** | 同端末共有家庭の構造的 privacy 保護 |
| `PARENT_GATE_FORCE_ACTIVE=true` (任意 env) | - | **有効** | 手動動作確認 / SS 撮影用に override |

#### 主要 flow

1. **初回 PIN 作成 (#2992、「初回は作る・既存は入る」)**: PIN 未設定 tenant が gate に到達すると、modal は login でなく**新規作成 (入力→確認の 2 段)** を表示 → 一致で `POST /api/v1/parent-gate/setup` (JSON: `{pin}`) → setupPin + `gq_parent_session` cookie 発行 → `next` へ。`verifyPin` は未設定時 `PIN_NOT_SET` を返し既定 PIN 照合を行わない (gate 経路から `DEFAULT_PIN` 廃止。legacy local 経路 `login()` / `changePin()` のみ #1360 互換で維持)。setup は**未設定 tenant のみ許可** (設定済みへは 403 `ALREADY_CONFIGURED` = 子供による上書き穴の防止)
2. **PIN 入力**: 子供 / 親が `/switch` 「ご家族の見守り画面」リンク click → inline modal (Dialog + PinInput primitive、Ark UI) で 4 桁入力 → `POST /api/v1/parent-gate/verify` (JSON: `{pin}`) → 成功で `gq_parent_session` httpOnly signed cookie を発行 → `next` query で指定された `/admin/...` path に遷移
3. **Sliding refresh**: 各 `/admin/*` request 時に `+layout.server.ts` が cookie 検証 + `lastActiveAt` 更新 + 再 sign して cookie 再発行 (15 分 timeout 延長)
4. **未認証 / 失効時の redirect**: middleware が `/switch?pinRequired=1&next=<原 path>` に redirect → 入口で modal auto-open
5. **明示削除 (EPIC 構造的核心 / 子#2314)**: `/switch` の `select` action で `cookies.delete('gq_parent_session')` を実行。親が child mode に戻った瞬間に session 失効し、次の子の admin アクセスは再 PIN を要求される
6. **client inactivity redirect (放置時)**: 旧来 sliding/失効判定は server load (= ページ遷移時) のみだったため、admin 画面を放置すると失効後も画面が残り「操作だけ失敗 = 固まった」とユーザに見えた。`admin/+layout.svelte` の client タイマー (`startParentGateInactivityRedirect`、15 分 = `INACTIVITY_TIMEOUT_MS` と一致) が、PIN gate 有効時 (`pinGateActive`) のみアイドル 15 分で `POST /api/v1/parent-gate/logout` (parent session 破棄) → `/switch?timedOut=1` (子供選択画面) へ自動リダイレクトする。子供が放置画面を操作する穴を「固める」のでなく「子供選択画面に戻す」挙動で塞ぐ (ADR-0012 anti-engagement 整合)。dev/demo では起動しない

#### Anti-engagement 整合 (ADR-0012)

子供 UI 側に PIN 通知バナーは出さない (engagement 寄与しない / 子供への余計な学習回避)。modal は親操作起点 (`/switch` のリンク click) のみで開く。

#### 初期 PIN ヒント表示ポリシー (Issue #2353 Phase A、2026-05-21)

業界 PIN 採用 4 サービス調査 (Apple Screen Time / Nintendo / Roblox / BusyKid) で「初期 PIN ヒントは setup 時のみ、認証 modal では非表示」が prior art 100%。これに整合し、本プロダクトも以下を遵守する:

| 表示文脈 | 初期 PIN ヒント `OYAKAGI_LABELS.defaultValueHint` (= `'初期値は 5086（がんばり）です'`) | 理由 |
|---|---|---|
| `/setup/complete/+page.svelte` (初期 setup 完了画面) | **非表示** (#2992 で「初めて入るときに作成します」案内に置換、`SETUP_COMPLETE_LABELS.pinHintSuffix`) | gate 経路は初回作成フローになり既定 PIN の事前伝達が不要化 |
| `/admin/settings/account/+page.svelte` (PIN 変更画面) | **表示** | legacy local 経路 (`changePin` の現コード照合) で未設定 tenant の現コード = `DEFAULT_PIN` のため、その文脈でのみ有効な案内 |
| `(child) onboarding dialog` (`PIN_GATE_ONBOARDING_LABELS.dialogPinHint`、Phase D) | **非表示** (#2992 で「初めて入るときに作成します」案内に置換) | 同上 (gate 経路の初回は作成フロー) |
| `/switch` PIN modal (`Dialog`) | **非表示** | 子供画面と同じ端末で表示される → 子供が見て即入れる脆弱性。#2992 以降は未設定者が自分で作るためヒント自体が不要 |

`OYAKAGI_LABELS.gateDefaultHint` (PIN modal 専用 atom) は #2353 Phase A で **削除**。再追加禁止 (子供脆弱性、業界 prior art ゼロ)。

#### banner 残存 bug 対策 (Issue #2353 Phase A、2026-05-21)

`/switch?pinRequired=1` 再アクセス時に banner だけ残って modal が出ない bug を構造的に修正:

- 根本原因: `$state` 初期化が初回 mount のみで、`data.pinRequired` query 再到達時に追従しない
- 修正: `src/routes/switch/+page.svelte` に `$effect(() => { if (data.pinRequired) { pinModalOpen = true; ... } })` で query 変化を watch
- 業界調査 (Wave 28-A Research): 8 サービス全てで「banner + modal は同時表示 or modal 単独」、「banner だけ + modal 出ない」設計は prior art ゼロ

### 4.3b ログイン手段マトリクス (#3025 — 認証/回復系設計の突合先 SSOT)

認証・本人確認・回復系の設計と AC は、**本マトリクスの全行で成立することを確認してから確定**する (federated 行の見落としで PIN reset が dead-end 化した #3025 の構造的再発防止)。

| ログイン手段 | Cognito パスワード | 本人確認に使える材料 | dev 再現 |
|---|---|---|---|
| email + パスワード (Cognito USER_PASSWORD_AUTH) | あり | パスワード再入力 (`authenticateWithCognito` re-auth) | `DEV_USERS` (owner 等) |
| Google OAuth (Cognito Hosted UI federation) | **なし** | **email-OTP (登録メールへ 6 桁の確認コード、#3070)**。Cognito は `prompt=login` を IdP に転送せず、recent-login (`auth_time`) は共有端末で silent SSO 無入力通過し得るため、子がアクセスできない email を確認材料にする | `DEV_USERS` の `google-owner@example.com` (`federated: true`、identities claim 付き JWT) |
| local (AUTH_MODE=local) | — (認証なし) | デプロイ制御面 (operator reset、§4.4) | `npm run dev` |

- federated 判定: id_token の `identities` claim の有無 (`identity.isFederated`)
- federated の本人確認は #3025 の requires-recent-login から #3070 で email-OTP に置換した。Cognito Hosted UI cookie 失効後の Google silent SSO で `auth_time` が refresh され、共有端末・親 session 生存時に子が無入力で recent-login を通過し得る穴があったため (実在の穴、ADR-0050 の speed bump 以下)。子はメールを読めないため email-OTP で確実に塞ぐ

### 4.4 PIN reset 機構 (#2993 / #3025 / #3070 / EPIC #2990)

PIN 忘れ救済は cognito モードで認証種別ごとに本人確認材料を分岐する:
- **password ユーザ**: アカウントパスワードの再入力 (Apple Screen Time 同型: 忘れた → アカウント資格情報で reset、#2993)
- **federated (Google) ユーザ**: 登録メールに送る 6 桁の確認コード = **email-OTP** (#3070)

**セキュリティ核心**: cognito 認証済みセッション ≠ 親 (家庭内共有端末で親のセッションのまま子供が操作する前提が parent-gate の存在意義)。セッションだけで reset を許すと子供が gate を突破できるため、子供が知らない材料 (アカウントパスワード / メール確認コード) を本人確認に使う。federated は Cognito パスワードを持たず、recent-login (#3025) は Cognito Hosted UI cookie 失効後の Google silent SSO で `auth_time` が refresh され共有端末で子が無入力通過し得る穴があったため、#3070 で email-OTP に置換した (子はメールを読めないため塞がる)。

| 項目 | 値 | 補足 |
|------|---|------|
| 本人確認 (password ユーザ) | Cognito InitiateAuth (USER_PASSWORD_AUTH) re-auth | `authenticateWithCognito` 流用、token は使わない (純粋なパスワード検証) |
| 本人確認 (federated ユーザ、#3070) | 登録メールへ 6 桁 OTP → 入力で照合。OTP は DB 非保存、code ハッシュ + 失効 (10 分) + tenantId + attempts を `cookie-signature` 署名 httpOnly cookie (`pin_reset_otp`) に格納する stateless 方式 (ADR-0050 署名機構流用、schema 変更なし) | 6 桁 / consume-once (成功で cookie clear) / 試行 5 回上限 (超過で cookie 失効) / enumeration 防止 (送信成否に依らず一定応答)。code 平文は cookie / ログに残さずハッシュのみ。timing-safe 比較 |
| email | `locals.identity.email` (セッション既知) | 手入力なし (enumeration 面も入力経路ごと消滅)。OTP もこの email にのみ送る |
| MFA 有効アカウント | `MFA_REQUIRED` 応答は success 扱い | パスワード正解後の第 2 要素要求のため、パスワード正解は確定している (password ユーザのみ) |
| dev mode | `authenticateDevUser` (DEV_USERS 照合、password ユーザ) / federated は `google-owner@example.com` | `isCognitoDevMode()` 分岐、実 AWS を呼ばない |
| rate limit | reset-verified: 5 req / 15 min per IP / reset-request-code: 5 req / 15 min per IP | brute force / OTP spam 防止 (OTP 試行上限と多重) |
| 成功時 | `setupPin` (上書き) + `gq_parent_session` cookie 発行 | verify 成功と同パラメータ |
| local モード | 400 `NOT_SUPPORTED` (page は `/auth/login` redirect) | local の救済は下記 operator reset |

#### operator-level reset (local / self-host、#2994)

self-host (docker-compose / PaaS / NUC) では「デプロイ環境の env を書ける = owner 本人」を実認証として、`PARENT_PIN_RESET=<未使用文字列>` env + 再起動で PIN を未設定状態 (= #2992 初回作成フロー誘導) に戻す。env 無しは完全 no-op / 同 token は `settings.pin_reset_applied` 照合で二度と適用しない (冪等) / 適用は `[AUDIT]` ログに記録。実装は `src/lib/server/services/pin-operator-reset.ts` (hooks.server.ts が初回リクエストで 1 回評価)。形態別フォールバック (sqlite3 直接 / PaaS one-off / DynamoDB update-item) と適用後 unset 手順は [runbooks/operator-pin-reset.md](../runbooks/operator-pin-reset.md) が SSOT。switch PIN modal は local では「サーバー管理者向けのリセット手順」案内を表示する (`gateOperatorResetNotice`)。reset 後〜owner が新 PIN を再設定するまでの間、legacy local 経路の `DEFAULT_PIN` 5086 fallback (`auth-service.ts`、#1360 互換) と新規作成フロー (#2992) の先着競合が残る。PIN は speed bump であり実認証でないという確定 policy (EPIC #2990) に基づく受容済み残余リスク (#3024 OBJ-3)。

#### フロー

1. `/switch` PIN modal の「PINを忘れた方」link (cognito identity のみ表示) → `/auth/reset-pin` 遷移
2. **password ユーザ**: アカウントパスワード + 新 PIN 入力 → POST `/api/v1/parent-gate/reset-verified` → re-auth 成功 → setupPin
3. **federated ユーザ (#3070)**: ①「確認コードを送る」→ POST `/api/v1/parent-gate/reset-request-code` (OTP 発行 + 署名 cookie set + メール送信) → ②メールの 6 桁コード + 新 PIN 入力 → POST `/api/v1/parent-gate/reset-verified` (OTP 検証) → 成功 → setupPin (consume-once で OTP cookie clear)
4. いずれも成功で setupPin + parent session cookie 発行 → 「再設定完了」 → `/admin` CTA (再入力不要)

#### 5.2.1 静的ファイル配信の tenant 一致検証（cross-tenant IDOR 防止、#3133）

`ROUTE_RULES` のロール検査は path 内の `tenantId` を知らないため、静的ファイル配信ルートは **ハンドラ層で tenant 一致を強制**する（二段防御）。`ROUTE_RULES` 未登録だと `findMatchingRule` が undefined → `authorizeCognito` の default-allow（`if (!rule) return { allowed: true }`）に落ち、認証済なら他テナントのファイルを GET できる cross-tenant IDOR になるため、両ルートを明示登録したうえでハンドラで以下を検証する:

| ルート | storage key 構造 | ハンドラ層 tenant 一致検証 |
|---|---|---|
| `/tenants/[...path]`（`src/routes/tenants/[...path]/+server.ts`） | `tenants/<tenantId>/<type>/<childId>/<file>`（`storage-keys.ts`、全 key が tenant-prefixed） | path 先頭セグメント `=== locals.context.tenantId`。不一致 / 未認証 → **404**（存在有無を漏らさない） |
| `/uploads/avatars/[filename]`（`src/routes/uploads/avatars/[filename]/+server.ts`） | legacy flat `uploads/avatars/avatar-<childId>-<...>`（旧形式のみ、tenant 非依存の flat 配置） | filename の `childId` を `getChildById(childId, locals.context.tenantId)` で自テナント存在確認したうえで、解決した child の `avatarUrl` が要求 filename の公開 URL（`storageKeyToPublicUrl('uploads/avatars/<filename>')` = `/uploads/avatars/<filename>`）と**一致する場合のみ**配信。一致しない / `avatarUrl` が null / 未認証 → **404** |

- **id-collision 対策（file ownership anchor、#3133 QM BLOCK 対応）**: child id は tenant-scoped に採番される（`nextId(child, tenantId)`、全テナントに id=1,2,... が存在し得る）一方、legacy flat key は tenant 非依存の flat 配置のため、`getChildById(childId, tenantId)` の成功は「自テナントに同 id の子供が居る」ことしか証明できず、その filename がその子供のアバターである保証にならない。child 存在確認だけだと、攻撃者が自テナントの child id=1 を用いて被害者テナントの `avatar-1-<victimSuffix>.png` を取得できる cross-tenant IDOR が残る。これを根治するため、**解決した child の `avatarUrl` が要求 filename を指すこと（file ownership）を必須条件**とし、指さない場合は fail-closed で 404 にする。
- **公開アセットの扱い**: favicon は `/favicon-32x32.png`（static）/ `/api/v1/images?type=favicon`（認証済）で配信され、`/tenants/**` `/uploads/**` を経由しない。`tenants/` 配下の storage key は構造上すべて tenant-prefixed のため、tenant 非依存の公開アセットは存在せず whitelist 不要。
- **モード別**: cross-tenant は cognito（マルチテナント SaaS）固有。local（NUC 単一家庭）/ anonymous（demo）provider は `authorize → allowed:true` だが、ハンドラ層 tenant 一致は全モードで効く（単一テナントでは自テナント一致で通過）。
- **回帰テスト**: `tests/integration/api/tenant-static-file-idor.test.ts`（cross-tenant → 404 / storage 非到達。`/uploads/avatars` は **id-collision ケース**＝自テナントに同 id child が居ても avatarUrl 不一致 / null なら 404 を含む）+ `tests/unit/auth/authorization.test.ts`（両ルートの認証必須化 = default-allow に落ちない）。
- **anchor robustness の不変条件（#3139）**: 本防御は「`child.avatarUrl` は server が tenant-scoped key からのみ生成し attacker が設定できない」ことに依存する。import/restore が attacker-influenced な値を `avatarUrl` に書けると anchor が silently 無効化されるため、`remapChildAvatarUrls` が外部入力由来の avatarUrl を verbatim 永続化しない（null か取込先 tenant prefix 配下の server key にのみ再構成）ことを `tests/unit/services/import-service-avatar-remap.test.ts` の invariant test で固定する（`insertChild` は構造的に avatarUrl 引数を取らない = 初期 null）。**URL デコード差異の不変条件（#3230）**: `isSafeRelativePath` / `remapChildAvatarUrls` は raw 文字列を扱い URL デコードしないため、`%2e%2e` / 二重エンコード / `..%2f` 等のエンコード traversal は literal な 1 セグメントとして扱われ tenant 境界を越えない。永続化 avatarUrl / `fileExists` probe key が取込先 tenant prefix（`tenants/<new>/`）内に閉じ別 tenant へ escape しないことを同 test の `#3230` invariant で固定する。
- **deny 理由の観測性 + log noise 抑制（#3139 / #3230）**: 全 deny path（`no-context` / `malformed-filename` / `no-child` / `ownership-mismatch` / `file-missing`）は存在秘匿のため一律 **404**（レスポンスは不変）だが、配信ハンドラ（`uploads/avatars/[filename]/+server.ts` の `denyAvatar`）が deny 理由を内部ログに記録し、攻撃 404（`ownership-mismatch` 等）と正常 404（default-icon fallback）の監視切り分けを可能にする。**log level は deny 種別で分ける（#3230）**: 未認証で自明に到達する pre-auth deny（`no-context` / `malformed-filename`）は攻撃シグナル価値が低く未認証連打で log 行が無制限増加し CloudWatch ingestion 課金 / NUC ディスク膨張を招くため **`logger.debug`**、攻撃シグナルとして意味のある post-auth deny（`no-child` / `ownership-mismatch` / `file-missing`）は **`logger.info`** で残す。response は全種別 404 で不変（情報は漏らさない）。

### 5.5 デモ Lambda IAM 分離（ADR-0048、#2126）

`demo.ganbari-quest.com` (デモ Lambda) と `app.ganbari-quest.com` (本番 Lambda) は **同 AWS Account 内で IAM Execution Role を物理的に分離** することで、デモ Lambda から本番リソースへの **アクセス自体を不可能** にする。これは設計の load-bearing security control であり、application-level filter (例: `if (isDemo) { ... }`) に依存しない物理的境界を保証する。

**設計根拠**:

- AWS Well-Architected SaaS Lens の **silo model** + SEC01-BP01 (account separation strongly recommended) に整合
- AWS Lambda Best Practices "Operating Lambda" blog の **Lambda 1:1 IAM role 原則** 整合
- 過去 8 回繰返した demo / production 並行実装の構造原因 (`if (isDemo)` 漏れ / cookie 誤判定 / auth bypass 混入) が **物理的に発生不可能** になる
- 個人開発で security incident 対応不能 (緊急 patch + 影響範囲調査 + 顧客通知) という現実制約に対する「IAM レベルの insurance」を ~$2.74/月で得る

#### 5.5.1 IAM Role 比較表

| リソース | `production-lambda-role` (本番) | `demo-lambda-role` (デモ) |
|---|---|---|
| **DynamoDB Table (production)** | full R/W (`grant: Table ARN`) | **アクセス権なし** (Role policy に Table ARN を含めない) |
| **DynamoDB Table (demo-*)** | アクセス権なし | grant: full R/W (`grant: demo-* prefixed Table ARN のみ`) |
| **Secrets Manager (production secrets)** | grant (Cognito UserPool ID / Stripe API Key 等) | **アクセス権なし** |
| **Cognito UserPool (production)** | grant: read | **アクセス権なし** |
| **S3 assets bucket** | grant: R/W | **grant: read-only** (画像のみ取得可、書き込み不可) |
| **CloudWatch Logs** | `/aws/lambda/ganbari-quest` (retention 30 日) | `/aws/lambda/ganbari-quest-demo` (retention 7 日) |
| **環境変数 `DATA_SOURCE`** | `dynamodb` | `demo` (in-memory fixture 経由) |
| **環境変数 `AUTH_MODE`** | `cognito` | `anonymous` (dummy user 固定) |

#### 5.5.2 物理境界が保証する Security Property

1. **本番 DynamoDB の漏洩リスク 0**: デモ Lambda が production Table ARN を IAM Policy に持たないため、コード bug / configuration error が混入しても `AccessDenied` で確実に拒否される
2. **本番 Secrets の漏洩リスク 0**: Stripe API Key / Cognito 設定値等の本番 Secrets を取得する API 呼び出し自体が IAM レベルで拒否される
3. **本番 Cognito 攻撃面 0**: デモ Lambda から本番 UserPool への JWT 検証 / user enumeration / brute-force 等の試行が IAM レベルで拒否される
4. **Blast Radius 構造保証**: デモ Lambda が侵害された場合でも、本番 DB / Secrets / UserPool への横展開が **物理的に不可能** (account 分離ほど extreme ではないが、Pre-PMF 個人開発の現実解として最適、ADR-0048 §選択肢 D 棄却根拠)

#### 5.5.3 サブドメイン分離

- **`demo.ganbari-quest.com`** (subdomain) を新規追加し、本番 origin (`app.ganbari-quest.com`) と独立 Lambda 配下で運用
- CloudFront: 既存 distribution に **alternate domain name `demo.ganbari-quest.com`** 追加、origin host header 駆動で demo Function URL に振り分け
- SSL cert は `*.ganbari-quest.com` wildcard 既存利用 (us-east-1 ACM)
- Route 53: A (Alias) record `demo.ganbari-quest.com` → CloudFront distribution
- IAM 分離 + subdomain 分離の二段構成で「同 origin で `?demo=true` 等の query parameter で切り替え」型の脆弱性 (auth bypass / tenant filter 漏れ) を構造的に排除

#### 5.5.4 デモ Lambda の Application 層

IAM 分離を補完する application 層の制約 (ADR-0048 §2 Application 設計):

- **Stateless**: Lambda module-level singleton で **user-specific mutable state を保持しない**。「demo で記録 → リロードで保持」は **client sessionStorage 限定** (tab 閉じで消失)
- **Anonymous Auth**: `AUTH_MODE=anonymous` で `AnonymousAuthProvider` 選択。dummy user (`anon-{requestId}`) + role='owner' + tenantId='demo' + licenseStatus=ACTIVE を返す
- **Test Double**: Repository Pattern + Abstract Factory で `DATA_SOURCE=demo` の場合 `DemoXxxRepo` を返す。**Fake (read) + Stub (write) hybrid**:
  - read API: in-memory fixture (`demo-data.ts`) から返却
  - write API: `{ ok: true, demo: true }` 200 no-op response (DB に永続化しない)

#### 5.5.5 IAM Physical Isolation Regression Test (Hard-Fail Gate)

IAM 分離は **CDK 単体テストで physical assertion** する。設計が破壊された (本番 Table ARN がデモ Role に混入する等) 場合に CI で hard-fail させる:

- **テスト**: `tests/unit/infra/multi-lambda-cdk.test.ts` C-1 (Multi-Lambda CDK regression)
- **検証内容**: synth した CloudFormation template から `demo-lambda-role` の Policy Document を抽出し、production Table ARN / production Secrets ARN / production UserPool ARN を一切含まないことを assert
- **失敗時の挙動**: CI hard-fail (PR merge ブロック)
- **設計意図**: IAM 分離は document policy ではなく **machine-enforced contract**。ADR-0006 (Safety Assertion Erosion Ban) と整合、assertion を弱める変更は CI で拒否

### 6.1 レートリミッター

- ルートハンドラで型チェック後、サービス層に委譲

### 6.3 エラーハンドリング

- `@sveltejs/kit` の `error()`, `json()` で一貫したエラーレスポンス
- `handleError` フックでサーバーエラーの詳細をログに記録
- クライアントには `{ message: 'Internal Server Error' }` のみ返却（内部情報非公開）

### 6.4 破壊操作 UI ガード — GitHub Danger Zone パターン (#2319 / #2321 / #2323)

不可逆な破壊的操作は **GitHub Danger Zone パターン** を共通適用し、誤操作リスクを構造的に低減する:

- **対象操作**: `accountDelete` (`/admin/settings/account`) / `clear-all` (`/admin/settings/data`)
- **視覚的隔離**: 赤枠 (`border: 2px solid var(--color-action-danger)`) + ヘッダー (`bg-feedback-error-bg` + `⚠️` アイコン) + ページ最下部配置

  1. 確認テキスト入力 (例: `削除` / `アカウントを削除します` を文字どおり入力)
  2. 同意 checkbox (「すべてのデータを削除することに同意します」等)
  3. 実行ボタン (Step 1 + Step 2 が両方満たされて初めて enabled)

`accountDelete` は owner (家族グループ全体に影響) / parent (非 owner) / child でメッセージを切替、owner かつ他メンバーがいる場合は権限移譲ダイアログを出して直接 full delete を避ける (#1781 削除グレースピリオド機構との併用)。

## 7. Web セキュリティ対策

### 7.1 セキュリティヘッダ

`hooks.server.ts` で全レスポンスに付与:

| ヘッダ | 値 | 目的 |
|--------|-----|------|
| `X-Frame-Options` | `DENY` | クリックジャッキング防止 |
| `X-Content-Type-Options` | `nosniff` | MIME スニッフィング防止 |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | リファラー情報の制限 |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` | 不要な API の無効化 |
| `Content-Security-Policy` | `default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: blob:; media-src 'self' blob:; font-src 'self'; connect-src 'self'; object-src 'none'; base-uri 'self'; frame-ancestors 'none'` | XSS/Polyglot 攻撃防止 |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | HTTPS 強制（cognito のみ） |
| `Cache-Control` | `no-cache, no-store, must-revalidate` | 認証ページのキャッシュ禁止 |

### 7.1.2 LP（静的サイト）の CSP（ADR-0029 / #1719）

`site/**` は GitHub Pages の静的配信であり、SvelteKit の `hooks.server.ts` を経由しない。
HTTP レスポンスヘッダーで CSP を付与できないため、`<meta http-equiv="Content-Security-Policy">` で同等の防御を行う。

**対象ファイル**（10 ページ全て）:

`site/index.html` / `pricing.html` / `faq.html` / `pamphlet.html` / `selfhost.html` / `privacy.html` / `terms.html` / `sla.html` / `tokushoho.html` / `help/license-key.html`

**CSP 内容**（全ページ共通）:

```html
<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net; style-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net; img-src 'self' data: blob:; font-src 'self'; connect-src 'self' https://cdn.jsdelivr.net; object-src 'none'; base-uri 'self'; form-action 'self'">
```

**指令の根拠**:

| 指令 | 値 | 根拠 |
|------|-----|------|
| `default-src` | `'self'` | LP は同一 origin で完結する fallback |
| `script-src` | `'self' 'unsafe-inline' https://cdn.jsdelivr.net` | DOMPurify (全 10 ページ) + splidejs (index.html) + budoux (index.html) を allowlist。`'unsafe-inline'` は inline `<script>` (initialization / Splide setup / mobile menu / contact form) と `<script type="application/ld+json">` 用 |
| `style-src` | `'self' 'unsafe-inline' https://cdn.jsdelivr.net` | splide-core CSS を allowlist。`'unsafe-inline'` は一部 inline `style="..."` 属性用 |
| `img-src` | `'self' data: blob:` | 同 origin 画像 + favicon + OGP + データ URI |
| `font-src` | `'self'` | 外部フォント未使用（system-ui） |
| `connect-src` | `'self' https://cdn.jsdelivr.net` | LP の動的 fetch / XHR は無いが、ブラウザ DevTools が jsDelivr 配信 JS の sourcemap (`.map`) を `connect-src` 経由で取得する (#2068)。`'self'` のみだと Chrome console に CSP violation 3 件 (splidejs / DOMPurify / budoux sourcemap) が常時表示される。`script-src` で既に許可済の origin と揃える |
| `object-src` | `'none'` | `<object>` / `<embed>` プラグイン禁止 |
| `base-uri` | `'self'` | `<base>` 注入による URL 書き換え攻撃防止 |
| `form-action` | `'self'` | フォーム送信先を同 origin に限定 |

**CDN ライブラリの SRI / pin マトリクス（ADR-0029 §決定 2）**:

| ライブラリ | 配信先 | pin 戦略 | SRI | CSP allowlist | 根拠 |
|-----------|--------|---------|-----|---------------|------|
| `@splidejs/splide@4.1.4` JS | `cdn.jsdelivr.net` | 完全 pin | **必須** (`sha384-...`) | 不要（補助） | bytes 固定で SRI が機能 |
| `@splidejs/splide-core@4.1.4` CSS | `cdn.jsdelivr.net` | 完全 pin | **必須** (`sha384-...`) | 不要（補助） | 同上 |
| `dompurify@3` | `cdn.jsdelivr.net` | major pin | **付与禁止** | **必須** | patch 自動取り込み優先（CVE 対応） |
| `budoux@latest` (index.html) | `cdn.jsdelivr.net` | range | **付与禁止** | **必須** | XSS 経路にならない補助ライブラリ |

**多層防御の構造**:

1. **第一層: DOMPurify サニタイズ** (ADR-0025) — `el.innerHTML = DOMPurify.sanitize(...)` で SSOT 注入時に XSS payload を除去
2. **第二層: CSP allowlist** (本節) — 攻撃者が任意 origin の script を注入する経路を構造的に塞ぐ
3. **第三層: SRI 付き完全 pin ライブラリ** (splidejs) — bytes が固定のライブラリは SRI で改ざん検知

**meta tag CSP の制約**:

- `frame-ancestors` / `report-uri` / `sandbox` は meta tag では効かない（HTTP ヘッダー専用）
- 本 LP は埋め込み禁止 (`X-Frame-Options: DENY`) も静的ホストでは制御不可で、Pre-PMF スコープ (ADR-0010) で許容済み
- 認証 / 動的 content / ユーザー固有データは LP に存在しないため脅威モデル上問題にならない

**E2E テスト**: `tests/e2e/lp-csp.spec.ts` で全 10 ページの CSP meta tag 存在検証 + ロード時 CSP violation ゼロ検証 + DOMPurify ロード成功 + SSOT 注入動作を担保。

### 7.2 XSS 対策

- SvelteKit はテンプレート出力を自動エスケープ（`{@html}` 以外）
- `{@html}` の使用は最小限に制限
- ユーザー入力はサーバー側でもバリデーション
- **user 由来静的ファイルの安全配信（#3105）**: user がアップロード / ZIP import 復元した静的ファイル（`/tenants/[...path]` / `/uploads/avatars/[filename]`）は、ラスタ画像（`image/jpeg` / `image/png` / `image/webp`）のみ `Content-Disposition: inline`、それ以外（**`image/svg+xml`** / `audio/*` / 不明 type）は `attachment` で配信する。SVG は XML 文書として `script` を含み得るため、CSP が `script-src 'self' 'unsafe-inline'` の本アプリでは inline 配信すると top-level navigation で stored XSS が成立する（#3105、ZIP import 復元路が avatar 防御を bypass していた）。判定は `file-sanitizer.ts` の `safeContentDisposition()` に集約し、全静的配信路で対称に適用する。`<img>` 等の subresource 読込は `Content-Disposition` を無視するため、正規の fallback SVG avatar 表示は維持される（img 経由 SVG では script も実行されない）。

### 7.3 CSRF 対策

- SvelteKit の Form Actions は組み込みの CSRF トークン検証を実施
- API エンドポイントは Cookie ベースの認証（SameSite=Lax）で CSRF リスクを軽減
- 状態変更 API は POST/PUT/DELETE メソッドを使用

### 7.4 インジェクション対策

- **SQL インジェクション**: Drizzle ORM のパラメータバインディングで防止（SQLite）
- **NoSQL インジェクション**: DynamoDB SDK のパラメータ化クエリで防止
- **コマンドインジェクション**: 外部コマンド実行なし

## 8. 子供データの特別保護方針

### 8.1 基本方針

がんばりクエストは子供（未成年者）のデータを扱うため、以下の原則を遵守する:

1. **最小限のデータ収集**: 必要最小限のデータのみ収集（ニックネーム、年齢、活動記録）
2. **個人識別情報の最小化**: 子供の本名・写真・位置情報は収集しない
3. **データの家族内限定**: テナント分離により他の家族のデータにアクセス不可
4. **保護者による管理**: 子供アカウントの作成・削除は保護者（owner/parent）のみ可能

### 8.2 収集するデータ

| データ種別 | 内容 | 保護レベル |
|-----------|------|-----------|
| プロフィール | ニックネーム、年齢、テーマカラー、UIモード | テナントスコープ |
| 活動記録 | 活動名、記録日時、ポイント | テナントスコープ |
| ステータス | レベル、経験値、連続日数 | テナントスコープ |
| 実績 | 獲得した実績一覧 | テナントスコープ |
| アバター画像 | AI 生成画像（S3 保存） | テナントスコープ |

### 8.3 収集しないデータ

- 本名、住所、電話番号
- 顔写真、動画
- 位置情報
- 他サービスとの連携データ
- 広告 ID、トラッキング情報

### 8.4 データ削除

- 子供プロフィール削除時: 関連する全データ（活動記録、ステータス、実績、アバター画像等）をトランザクションで一括削除
- テナント削除時: テナントに紐づく全データを削除

### 8.5 電気通信事業法 §27の12 対応（外部送信規律 公表）

電気通信事業法第27条の12（外部送信規律）に基づき、利用者の端末から外部第三者へ送信される情報を `site/privacy.html` 第8条で公表する。対応 Issue: #1590（ADR-0023 I1 派生）。

| 項目 | 内容 |
|------|------|
| 送信される情報 | ページ URL、リファラ、訪問時刻、画面解像度、ブラウザ言語、ユーザーエージェント |
| 送信先 | AWS（自社アカウント内 DynamoDB / Lambda / Cognito）、Stripe、AWS Bedrock、Google Gemini |
| 利用目的 | ウェブサイト機能提供および改善 / 課金処理 / コンテンツ生成 |
| 個人識別情報 | 利用者本人を直接識別する情報（氏名・住所・電話番号等）は取得しない |
| 利用者の選択肢 | ブラウザ Cookie ブロック等 |

**実装上の遵守事項**:
- 子供データ（活動記録 / プロフィール / アバター画像）は外部第三者へ送信しない（§8.7 と整合）
- AWS Bedrock / Google Gemini への生成 AI リクエストには家族内一意 ID を含めない
- 文言ドリフト防止: `LEGAL_LABELS` (`src/lib/domain/labels.ts`) のキー用語が `site/privacy.html` に出現することを `scripts/check-lp-ssot.mjs` で CI 検証

### 8.6 個人情報保護法 §28 対応（域外移転 + 本人同意取得）

個人情報保護法第28条に基づき、外国にある第三者への個人データ提供について、`site/privacy.html` 第10条で情報提供 + サインアップ時に本人同意を取得する。対応 Issue: #1638。

| 項目 | 内容 |
|------|------|
| 移転先国 | 米国（AWS バージニア北部リージョン: `us-east-1`） |
| 第三者の名称 | Amazon Web Services, Inc.（米国デラウェア州法人） |
| 法的根拠 | AWS DPA (Data Processing Addendum) + 標準契約条項 (SCC) |
| 移転される情報の範囲 | 利用者識別子（家族内一意 ID）、活動記録、課金関連情報 |
| 本人同意取得 UI | `src/routes/auth/signup/+page.svelte` の `agreedCrossBorder` チェックボックス（必須） |

**実装上の遵守事項**:
- `agreedCrossBorder` は `agreedTerms` / `agreedPrivacy` と同様に必須チェック（未チェックは submit ブロック）
- 日本国内サーバー（`ap-northeast-1` / 東京リージョン）の文言を新規追加しない（PR #1656 で削除済みの方針維持）
- 改正個情法（2026年成立予定）の本文確定後の差分対応は別 Issue で

### 8.7 未成年者の取扱い

`site/privacy.html` 第9条 + `site/terms.html` 第6条で明示する。対応 Issue: #1590。

| 項目 | 内容 |
|------|------|
| 親同意フレームワーク | 全年齢で運用。子供本人がアカウント作成不可（保護者のみ） |
| 利用者識別子 | 家族内一意 ID のみ（学校名・氏名・住所・電話番号等は取得しない） |
| 本人への直接接触 | アンケート・通知・メールマガジン等は一切行わない（連絡は保護者宛のみ） |
| データの域外送信 | ゼロ。自社 AWS アカウント内 DynamoDB のみで処理（生成 AI 等への送信なし） |
| 親による削除請求 | 優先処理（§5 利用者の権利・§6 データの削除） |

**実装上の遵守事項**:
- 子供アカウントは保護者（owner / parent ロール）のみ作成・削除可能
- 子供データを生成 AI（Bedrock / Gemini）に渡すリクエストには家族内一意 ID を含めない（§8.5 と整合）
- 「16歳未満」のしきい値は採用しない（全年齢で同一の親同意フレームワーク運用）

### 8.8 Web Push 通知の対象監査 (#1593, ADR-0023 I6)

子端末への直接マーケ通信は **永久 NG** とする（COPPA 改正 + ADR-0012 Anti-engagement の二重規制）。

#### 設計

`push_subscriptions` テーブルに `subscriber_role` カラム (`'parent' | 'owner'`、default `'parent'`) を持たせ、subscribe 時の role を必ず記録する。

| 層 | 防御 | 実装場所 |
|----|------|---------|
| API ガード | `POST /api/v1/notifications/subscribe` で `context.role === 'child'` の場合 403 + `code: 'CHILD_FORBIDDEN'` を返す | `src/routes/api/v1/notifications/subscribe/+server.ts` |
| 送信時ガード（二重防御） | `notification-service.sendPushNotification` が `findByTenant` 結果から `subscriberRole !== 'parent' && subscriberRole !== 'owner'` の subscription を skip + `logger.warn` | `src/lib/server/services/notification-service.ts` |
| スキーマ | `subscriber_role TEXT NOT NULL DEFAULT 'parent'` (ADR-0031 NULL 混在防止) | `src/lib/server/db/schema.ts` + `create-tables.ts` + `tests/e2e/global-setup.ts` + `tests/unit/helpers/test-db.ts` |
| 型 | `PushSubscriberRole = 'parent' \| 'owner'` (`'child'` は型に含めない) | `src/lib/server/db/types/index.ts` |
| ラベル SSOT | エラーメッセージは `PUSH_NOTIFICATION_LABELS.childSubscribeForbidden` を経由 | `src/lib/domain/labels.ts` |

#### テスト

- Unit: `tests/unit/routes/notifications-subscribe-api.test.ts` — child role の 403 拒否 / parent / owner の subscriberRole 記録 / 入力バリデーション
- Unit: `tests/unit/services/notification-service.test.ts` — 送信時の child / 不明 role skip 検証

#### 既存レコード backfill

本機能リリース前の既存 subscription は default `'parent'` で扱われる。これは `npm run dev:cognito` 環境では DEV_USERS の owner / parent / child の subscribe 試行が ADR-0023 §3.1 の通り「親管理画面動線のみ」に限定されてきた前提に依拠する。

**SQLite (NUC)**: `scripts/migrate-local.ts` で `ALTER TABLE ADD COLUMN ... DEFAULT 'parent'` + `UPDATE ... WHERE subscriber_role IS NULL OR ''` を実行済（#1593）。

**DynamoDB (本番 AWS) (#1666)**: `scripts/migrate-dynamodb-push-subscriber-role.mjs` で `entityType = 'PUSH_SUB'` または `SK begins_with 'PUSH_SUB#'` の item を Scan し、`subscriberRole` 不在 / NULL / 不正値を `'parent'` で UpdateItem する idempotent script を提供。実行手順は [`docs/runbooks/push-subscription-role-migration.md`](../runbooks/push-subscription-role-migration.md) 参照。実行タイミングは DynamoDB push-subscription-repo 本実装マージ後（実装本体は本 follow-up Issue とは別 Issue で扱う、ADR-0010 Pre-PMF / #1021 段階的対応禁止に整合）。検証は CloudWatch Logs `[notification] 非 parent/owner role の subscription への送信をスキップ` warn 件数 = 0。

### 8.9 Web Push subscribe endpoint の SSRF hardening (#3188 item3, CWE-918)

cron の server push (web-push) は `push_subscriptions` に保存済の `endpoint` URL へ POST する。`POST /api/v1/notifications/subscribe` が受け取った `endpoint` を scheme / host 検証なしで保存すると、internal host (`169.254.169.254` / `localhost` / RFC1918 等) を endpoint に指定した攻撃者経由で server-side request forgery (CWE-918) が成立し得る。これを保存前の入力検証で構造的に遮断する。

| 観点 | 仕様 | 実体 |
|---|---|---|
| 検証ロジック | `validatePushEndpoint(endpoint)` / `isValidPushKey(key, maxLen)` を独立モジュールに集約 | `src/lib/server/services/push-endpoint-validation.ts` |
| API ガード | subscribe で保存前に検証し、不正 endpoint は `400` + `code: 'INVALID_ENDPOINT'`、不正 key は `400` + `code: 'INVALID_KEY'` を返す (理由は `logger.warn`、ユーザーには汎用文言) | `src/routes/api/v1/notifications/subscribe/+server.ts` |
| scheme 制限 | `https:` のみ許可 (それ以外は拒否) | 同上 |
| host allowlist | 既知 Web Push service host のみ許可。`fcm.googleapis.com` / `android.googleapis.com` (FCM、Chrome/Edge) は exact 一致、`.push.services.mozilla.com` (Firefox) / `.push.apple.com` (Safari) / `.notify.windows.com` / `.wns.windows.com` (WNS) は dot-suffix 末尾一致。末尾一致のため `fcm.googleapis.com.attacker.com` 等の偽装ホストおよび internal host は弾かれる | 同上 |
| endpoint 長 | 2048 文字上限 | 同上 |
| key 形式 | `p256dh` / `auth` を base64 / base64url charset + 長さ上限 (`p256dh` 256 / `auth` 64) で検証 | 同上 |

allowlist 方式は新規ブラウザの push host を拒否し得るが、SSRF 面では「公開だが内部 proxy な host」も弾ける最も強い静的防御であり、主要ブラウザを網羅するため Pre-PMF では allowlist を採用する (ADR-0010)。検証は `tests/unit/services/push-endpoint-validation.test.ts` / `tests/unit/routes/notifications-subscribe-api.test.ts`。subscribe 側 hardening は #3188 item3 を対象とし、item1 (購読解除の状態不整合) / item2 (操作取消とエラーの区別 + aria-live) は同 issue で別途扱う。

#### 送信側 (cron) の再検証 — defense-in-depth (#3404)

subscribe API の保存前検証だけでは、過去レコード混入 / repo 直 insert / 将来の bulk import 経路で allowlist 外 endpoint が `push_subscriptions` に入った場合、実際に HTTP request を出す送信側 (`sendPushNotification` 内の `webpush.sendNotification`) が SSRF の実 sink (CWE-918) となる。これを塞ぐため、送信直前にも各 subscription の `endpoint` を `validatePushEndpoint` で再検証し、単一の検証経路に統一する。

| 観点 | 仕様 | 実体 |
|---|---|---|
| 送信前再検証 | `sendPushNotification` が送信対象 subscription を `validatePushEndpoint(endpoint).ok` で filter し、allowlist 外は `logger.warn` を残して当該 subscription への送信を skip する (有効 subscription が 0 件なら送信処理全体を skip) | `src/lib/server/services/notification-service.ts` |
| 検証 SSOT | subscribe / send で同一の `validatePushEndpoint` を共有し、host allowlist / scheme / 長さ判定を 1 箇所に集約する (二重実装しない) | `src/lib/server/services/push-endpoint-validation.ts` |
| 内部例外の非露出 | subscribe API の catch は内部例外を client に返さず、汎用 message + `logger.error` (server log のみ) とする (ADR-0062) | `src/routes/api/v1/notifications/subscribe/+server.ts` |
| 検証 | 送信側 SSRF skip の単体テスト (internal host を skip / 正規 endpoint は送信) | `tests/unit/services/notification-service.test.ts` |

#3404 のうち item2 (`url.href` 正規化) / item3 (`INVALID_ENDPOINT` metric / alarm) は runtime bypass が無く observability の範疇のため、本対応では扱わず Pre-PMF scope 外とする (ADR-0010)。

## 9. 監査ログ

### 9.1 ログ設計

`src/lib/server/logger.ts` による構造化ログ:

## 11. インフラセキュリティ

### 11.1 IAM

- Lambda 実行ロールに最小権限を付与:
  - DynamoDB: テーブルへの CRUD のみ
  - S3: アセットバケットへの Read/Write のみ
  - Cognito: UserPool への読み取りのみ
  - SSM: パラメータの読み取りのみ

### 11.2 ネットワーク

- CloudFront → Lambda Function URL（HTTPS のみ）
- Lambda はパブリック VPC 内で実行（DB アクセスは DynamoDB エンドポイント経由）
- S3 バケットはパブリックアクセス完全ブロック

### 11.3 シークレット管理

| シークレット | 保存場所 | 用途 |
|------------|---------|------|
| Cognito User Pool ID | SSM Parameter | JWT 検証 |
| Cognito Client ID | SSM Parameter | 認証フロー |
| Context Token Secret | 環境変数 | Context Token 署名 |
| Stripe API Key | SSM Parameter | 決済処理 |
| Stripe Webhook Secret | SSM Parameter | Webhook 署名検証 |

### 11.4 CI/CD セキュリティ

- **GitHub Actions**: main ブランチへの push で自動デプロイ
- **CodeQL**: プッシュごとにセキュリティスキャン実行
- **Dependabot**: 依存ライブラリの脆弱性を自動検出・PR作成
- **コミット前チェック**: Biome lint + svelte-check + Vitest + Playwright E2E
- **高権限 Actions の SHA pin (#3298 / #3318)**: secret を消費する / provenance を発行する / cache を書く / 本番 AWS role を assume する高権限 Actions（`actions/create-github-app-token` / `actions/attest` / `actions/cache` / `aws-actions/configure-aws-credentials` / `aws-actions/amazon-ecr-login`）は mutable な floating tag（`@v3` 等）ではなく full-length commit SHA + `# vX.Y.Z` コメントで pin する（tag 再付け替えによる supply-chain 注入の防止、GitHub 公式 hardening guide 整合）。`aws-actions/*` 2 件は `id-token: write` で本番 AWS role を assume し ECR push + CDK Lambda deploy 権限に直結する（blast radius が cache poisoning より大きい）ため高権限 subset に含める。floating tag への regression は `scripts/check-action-sha-pin.mjs`（CI `deps-supply-chain-check` job、SSOT = `HIGH_PRIVILEGE_ACTIONS`）が hard-fail する。repo 全体の SHA pin 化は churn / 運用コストが大きく Pre-PMF（ADR-0010）で PO 判断待ち（高権限 subset のみ強制）。dependabot は SHA pin + コメント方式でも更新追従する

## 12. 既知の制限事項

1. **レートリミッターがインスタンス間で共有されない**: Lambda のインスタンスごとに独立した Map で管理。大量のリクエストが複数インスタンスに分散した場合、制限が緩くなる可能性がある
2. **PITR 無効**: DynamoDB の Point-in-Time Recovery はコスト最適化のため無効。AWS Backup（日次3日保持）で代替
