# セキュリティ設計書

## 1. 概要

がんばりクエストにおけるセキュリティ要件と実装を定義する。子供（未成年者）のデータを扱うアプリケーションとして、データ保護に特に注意を払う。

### 1.1 デプロイ環境

| 環境 | 認証モード | ネットワーク | 用途 |
|------|-----------|-------------|------|
| AWS Lambda（本番） | `cognito` | インターネット公開（CloudFront + HTTPS） | SaaS版 |
| NUC サーバー（ローカル） | `local` | LAN内限定（HTTP） | 自宅専用 |

セキュリティ要件は環境ごとに異なり、`AUTH_MODE` 環境変数で制御される。

---

## 2. 通信暗号化

### 2.1 本番環境（AWS）

```
[ユーザー] --HTTPS/TLS 1.2+--> [CloudFront] --HTTPS--> [Lambda Function URL]
```

- **TLS ポリシー**: CloudFront で `TLSv1.2_2021` を強制（`SecurityPolicyProtocol.TLS_V1_2_2021`）
- **証明書**: ACM（AWS Certificate Manager）による自動管理・自動更新
- **HSTS**: `Strict-Transport-Security: max-age=31536000; includeSubDomains`（cognito モードのみ hooks.server.ts で付与）
- **CloudFront → Lambda 間**: Lambda Function URL は HTTPS のみ

### 2.2 ローカル環境（NUC）

- LAN 内限定のため HTTP 通信を許容
- HSTS ヘッダは付与しない（cognito モード以外では省略）

---

## 3. 保存データ暗号化

### 3.1 DynamoDB（本番DB）

- **暗号化**: AWS managed key（デフォルト暗号化、DynamoDB 側で自動適用）
- **PITR**: 無効（コスト最適化）。代わりに AWS Backup で日次バックアップ（3日保持）

### 3.2 S3（アセットバケット）

- **暗号化**: SSE-S3（`BucketEncryption.S3_MANAGED`）
- **パブリックアクセス**: 全ブロック（`BlockPublicAccess.BLOCK_ALL`）
- **アクセス**: Lambda IAM ロール経由のみ
- **ライフサイクル**: `backups/` は30日で自動削除、`logs/` は1日後に Glacier へ

### 3.3 SQLite（NUCローカル）

- ファイルシステムレベルの保護に依存（LAN 内限定環境）
- WAL モード使用（`PRAGMA journal_mode=WAL`）

### 3.4 Cookie

| Cookie 名 | 用途 | 属性 |
|-----------|------|------|
| `identity_token` | Cognito ID Token（有効期限 1 時間） | HttpOnly, Secure, SameSite=Lax, Path=/ |
| `gq_refresh` | Cognito Refresh Token（有効期限 30 日、サイレントリフレッシュ用）(#1365) | HttpOnly, Secure, **SameSite=Strict**, Path=/ |
| `context_token` | 署名付きテナント情報 | HttpOnly, Secure, SameSite=Lax, Path=/ |
| `invite_code` | 招待コード一時保存 | HttpOnly, Secure, SameSite=Lax, Path=/ |

## 4. 認証

### 4.1 Cognito モード（本番 SaaS）

二層認証アーキテクチャを採用:

```
Layer 1: Identity（Cognito JWT）
  └ identity_token Cookie → JWT署名検証 → { userId, email }
      ↓ 期限切れ / 存在しない場合
      └ gq_refresh Cookie → Cognito /oauth2/token (refresh_token grant)
            ├── 成功 → 新 identity_token をセット → return identity
            └── 失敗 → gq_refresh を削除 → return null

Layer 2: Context（署名付きトークン）
  └ context_token Cookie → HMAC-SHA256検証 → { tenantId, role, licenseStatus }
```

**サイレントリフレッシュ (#1365)**: `resolveIdentity()` 内で ID Token が期限切れの場合に `gq_refresh` Cookie を用いて自動更新。ログアウト時は `/oauth2/revoke` エンドポイントで Refresh Token を失効させる。

**Cognito User Pool 設定:**

| 設定項目 | 値 |
|---------|-----|
| サインイン方法 | Email + Password |
| MFA | OPTIONAL（TOTP 対応、SMS 無効） |
| パスワードポリシー | 8文字以上、大小英字+数字必須、記号不要 |
| Email 検証 | 自動（確認コード方式） |
| Email 属性 | `required: true`, **`mutable: true`**（ADR-0017 / #1366: Google IdP 再認証時の更新拒否回避のため） |
| アカウント回復 | Email のみ |
| カスタム属性 | `tenantId`, `role` |
| アクセストークン有効期限 | 1時間 |
| ID トークン有効期限 | 1時間 |
| リフレッシュトークン有効期限 | 30日 |

> **注 (ADR-0017 / #1366)**: Email 属性は `mutable: true` に設定している。`mutable: false` の場合、Google OAuth の再認証時に Cognito が IdP から取得した email を「属性更新」として扱い `user.email: Attribute cannot be updated` エラーでログイン不能になるため。ユーザー自身が email を変更する UI は提供しない (Google 側で email が変わった場合のみ追従する)。

**JWT 検証:**

- `cognito-jwt.ts` で Cognito 公開鍵（JWKS）を取得・キャッシュ
- 署名アルゴリズム: RS256
- `iss`（発行者）と `aud`（クライアントID）を検証
- 有効期限（`exp`）を検証

**Context Token:**

- HMAC-SHA256 署名（`CONTEXT_TOKEN_SECRET` 環境変数）
- ペイロード: `{ tenantId, childId?, role, licenseStatus }`
- 有効期限: 24時間（Context 再取得間隔）

### 4.2 Local モード（NUC）

- 親の管理画面: おやカギコード認証（4桁、ブラウザ Cookie で保持）。`pin_hash` 未設定時は デフォルト値 `5086`（「がんばり」語呂合わせ）にフォールバック（#1360）。脅威モデルは家庭内の軽い仕切り（子供が誤って管理画面に入らないためのロック）であり金融系 PIN 相当の強度は不要
- 子供画面: 認証なし（LAN 内限定のため）
- tenantId は固定値 `'local'`

### 4.3 親 PIN gate + 15 分 sliding session (EPIC #2310 / ADR-0050)

**脅威モデル**: 同端末（タブレット / ノート PC）を親と子供で共有する家庭において、子供が `/switch` の「🔒 ご家族の見守り画面」リンク経由で `/admin/*` に到達すること（PIN なしで親画面が無防備に晒される構造的問題）。Apple Screen Time / BusyKid 等の業界 prior art と整合させて 4 桁 PIN gate + httpOnly signed cookie + 15 分 inactivity sliding timeout を導入する。

#### 仕様

| 項目 | 値 | 根拠 |
|---|---|---|
| PIN 桁数 | 4 桁 (将来 6 桁拡張可) | NIST SP 800-63B-4 AAL1 + Apple Screen Time 同設計 |
| 認証方式 | 既存 `verifyPin` (auth-service.ts) + bcrypt | 流用、独自実装回避 (ADR-0014) |
| Lockout | 既存 `verifyPin` の throttling (連続失敗カウンタ + lockedUntil) | NIST AAL1 throttling 要件 |
| Cookie 名 | `gq_parent_session` | SSOT: `parent-gate-session.ts` の `PARENT_SESSION_COOKIE_NAME` |
| 署名 | `cookie-signature` (HMAC-SHA256) | ADR-0050 で OSS 4 件比較 (cookie-signature / jose / iron-session / lucia-auth) |
| 暗号化 | なし | payload に PII 含まないため不要 (改ざん防止のみ) |
| Cookie 属性 | `httpOnly + sameSite=lax + secure(本番)` | OWASP Session Management Cheat Sheet |
| Inactivity timeout | 15 分 (sliding refresh) | NIST SP 800-63B-4 AAL1 推奨値 |
| Hard max | 24 時間 | cookie maxAge、家庭利用文脈で daily reset |
| 署名キー env | `PARENT_GATE_COOKIE_SECRET` (≥ 16 chars) | production 未設定で起動時 throw、dev は fallback + 警告 |

#### 有効化条件

| AUTH_MODE | COGNITO_DEV_MODE | PIN gate 動作 | 理由 |
|---|---|---|---|
| `anonymous` (demo Lambda、ADR-0048) | - | 無効 | demo 環境では PIN 機構自体が無意味 |
| `local` (dev / NUC) | - | 無効 | Pre-PMF Bucket B、認証なし開発者用途 |
| `cognito` + `COGNITO_DEV_MODE=true` | true | 無効 | E2E setup / DEV_USERS 経由の既存全 spec を破壊しないため |
| `cognito` (production) | (false) | **有効** | 同端末共有家庭の構造的 privacy 保護 |
| `PARENT_GATE_FORCE_ACTIVE=true` (任意 env) | - | **有効** | 手動動作確認 / SS 撮影用に override |

#### 主要 flow

1. **PIN 入力**: 子供 / 親が `/switch` 「ご家族の見守り画面」リンク click → inline modal (Dialog + PinInput primitive、Ark UI) で 4 桁入力 → `POST /api/v1/parent-gate/verify` (JSON: `{pin}`) → 成功で `gq_parent_session` httpOnly signed cookie を発行 → `next` query で指定された `/admin/...` path に遷移
2. **Sliding refresh**: 各 `/admin/*` request 時に `+layout.server.ts` が cookie 検証 + `lastActiveAt` 更新 + 再 sign して cookie 再発行 (15 分 timeout 延長)
3. **未認証 / 失効時の redirect**: middleware が `/switch?pinRequired=1&next=<原 path>` に redirect → 入口で modal auto-open
4. **明示削除 (EPIC 構造的核心 / 子#2314)**: `/switch` の `select` action で `cookies.delete('gq_parent_session')` を実行。親が child mode に戻った瞬間に session 失効し、次の子の admin アクセスは再 PIN を要求される

#### 関連実装

- `src/lib/server/services/parent-gate-session.ts` (cookie 発行 / 検証 / sliding refresh)
- `src/routes/api/v1/parent-gate/verify/+server.ts` (POST endpoint、既存 verifyPin 流用)
- `src/routes/api/v1/parent-gate/logout/+server.ts` (POST endpoint、cookie 削除)
- `src/routes/(parent)/admin/+layout.server.ts` (middleware)
- `src/routes/switch/+page.svelte` (PIN modal UI、PinInput primitive 初実用化)
- `src/routes/switch/+page.server.ts` `select` action (cookie 明示削除)

#### Anti-engagement 整合 (ADR-0012)

子供 UI 側に PIN 通知バナーは出さない (engagement 寄与しない / 子供への余計な学習回避)。modal は親操作起点 (`/switch` のリンク click) のみで開く。

#### 初期 PIN ヒント表示ポリシー (Issue #2353 Phase A、2026-05-21)

業界 PIN 採用 4 サービス調査 (Apple Screen Time / Nintendo / Roblox / BusyKid) で「初期 PIN ヒントは setup 時のみ、認証 modal では非表示」が prior art 100%。これに整合し、本プロダクトも以下を遵守する:

| 表示文脈 | 初期 PIN ヒント `OYAKAGI_LABELS.defaultValueHint` (= `'初期値は 5086（がんばり）です'`) | 理由 |
|---|---|---|
| `/setup/complete/+page.svelte` (初期 setup 完了画面) | **表示** | 親が初期 setup を完了した直後の文脈、適切 |
| `/admin/settings/account/+page.svelte` (PIN 変更画面) | **表示** | 親が自ら PIN 変更を試みる文脈、適切 |
| `(child) onboarding dialog` (`PIN_GATE_ONBOARDING_LABELS.dialogPinHint`、Phase D) | **表示 (短縮版)** | 親が初回のみ読む onboarding dialog、「以降表示しない」で消える |
| `/switch` PIN modal (`Dialog`) | **非表示** | 子供画面と同じ端末で表示される → 子供が見て即入れる脆弱性 |

`OYAKAGI_LABELS.gateDefaultHint` (PIN modal 専用 atom) は #2353 Phase A で **削除**。再追加禁止 (子供脆弱性、業界 prior art ゼロ)。

#### banner 残存 bug 対策 (Issue #2353 Phase A、2026-05-21)

`/switch?pinRequired=1` 再アクセス時に banner だけ残って modal が出ない bug を構造的に修正:

- 根本原因: `$state` 初期化が初回 mount のみで、`data.pinRequired` query 再到達時に追従しない
- 修正: `src/routes/switch/+page.svelte` に `$effect(() => { if (data.pinRequired) { pinModalOpen = true; ... } })` で query 変化を watch
- 業界調査 (Wave 28-A Research): 8 サービス全てで「banner + modal は同時表示 or modal 単独」、「banner だけ + modal 出ない」設計は prior art ゼロ

### 4.4 PIN reset 機構 (#2353 設計欠陥 4 / ADR-0050 §4 補論)

PR #2325 マージ後、PIN 忘れ救済導線が無く本番 user が permanent lockout 状態に陥る設計欠陥が PO 指摘 (#2353)。業界標準 (Auth0 / Cognito password reset / NextAuth Email provider / WP password reset) の **SES magic link + signed token + DB consume** パターンで補強。

| 項目 | 値 | 補足 |
|------|---|------|
| token 機構 | jose `SignJWT` (HS256) | 既存 dependency 流用 (bundle 増 0) |
| TTL | 30 分 | 業界標準 (Auth0 1h / Cognito 1h / GitHub 15-30min) |
| consume 機構 | settings table `pin_reset_jti_consumed:<jti>` = 'true' | 1 回限り消費、tenant scope |
| 署名 secret | `PARENT_GATE_COOKIE_SECRET` (流用) | aud 分離で cookie session token と混用防止 |
| 配信 | SES (既存基盤再利用) | sendPinResetEmail / wrapTemplate 整合 |
| rate limit | 5 req / 15 min per IP (request) + 20 req / 15 min per IP (verify) | enumeration 防止 |

#### フロー

1. `/switch` PIN modal の「PINを忘れた方」link → `/auth/forgot-pin` 遷移
2. email 入力 → POST `/api/v1/parent-gate/reset/request` → 全所属 tenant に token 発行 + SES 送信
3. ユーザがメール magic link click → `/auth/reset-pin/<token>` 遷移
4. 新 PIN 入力 → POST `/api/v1/parent-gate/reset/verify` → setupPin + JTI consume
5. 「再設定完了」success state → `/switch` への CTA で gate 通過試行

#### enumeration 攻撃防止

email 未登録 / SES 失敗時も `/api/v1/parent-gate/reset/request` は 200 を返す (1Password / Bitwarden / Apple 同一パターン)。logger.info で内部観測のみ。

#### 関連実装

- `src/lib/server/services/pin-reset-service.ts` (token 発行 / 検証 / consume)
- `src/lib/server/services/email-service.ts` `sendPinResetEmail` (SES 配信)
- `src/routes/api/v1/parent-gate/reset/request/+server.ts` (POST endpoint)
- `src/routes/api/v1/parent-gate/reset/verify/+server.ts` (POST endpoint)
- `src/routes/auth/forgot-pin/+page.svelte` (Step 1 email 入力)
- `src/routes/auth/reset-pin/[token]/+page.svelte` (Step 2 新 PIN 設定、PinInput primitive 再利用)

## 5. 認可

### 5.1 ロール体系

| ロール | 対象 | 権限 |
|--------|------|------|
| `owner` | テナント作成者 | 全操作 + メンバー管理 |
| `parent` | 招待された保護者 | 子供管理 + 設定変更 |
| `child` | 子供 | 子供画面のみ |

### 5.2 ルート保護マトリクス

| ルートパターン | 許可ロール | 未認証時リダイレクト |
|--------------|-----------|-------------------|
| `/admin/license` | owner, parent | `/auth/login` |
| `/admin/members` | owner, parent | `/auth/login` |
| `/admin/**` | owner, parent | `/auth/login` |
| `/child/**` | owner, parent, child | `/auth/login` |
| `/api/v1/admin/**` | owner, parent | 401 |
| `/api/v1/**` | owner, parent, child | 401 |
| `/`, `/auth/**`, `/switch`, `/legal/**` | 全員（認証不要） | — |
| `/api/health` | 全員（認証不要） | — |
| `/api/stripe/webhook` | 全員（認証不要、Stripe署名で検証） | — |

認可チェックは `hooks.server.ts` → `authorization.ts` の `authorizeCognito()` で一元実行。ルート × ロール × ライセンス状態の三軸で判定する。

### 5.3 ライセンス状態による制限

| ライセンス状態 | アクセス制限 |
|--------------|------------|
| `active` | 制限なし |
| `none` | 制限なし（Free tier） |
| `expired` | ライセンス管理・決済ページのみアクセス可。他はリダイレクト |
| `suspended` | 読み取り専用（GET 許可、POST/PUT/DELETE は API レイヤで制御） |

### 5.4 テナント分離

- DynamoDB の PK に `T#<tenantId>#` プレフィックスを付与
- 全リポジトリメソッドに `tenantId: string` を必須パラメータとして要求
- `hooks.server.ts` で `event.locals.context.tenantId` をセットし、全ルートからサービス層に伝搬
- テナントID は Context Token から取得（改竄不可、HMAC署名済み）

### 5.5 デモ Lambda IAM 分離（ADR-0048、#2126）

`demo.ganbari-quest.com` (デモ Lambda) と `app.ganbari-quest.com` (本番 Lambda) は **同 AWS Account 内で IAM Execution Role を物理的に分離** することで、デモ Lambda から本番リソースへの **アクセス自体を不可能** にする。これは設計の load-bearing security control であり、application-level filter (例: `if (isDemo) { ... }`) に依存しない物理的境界を保証する。

**設計根拠**:

- AWS Well-Architected SaaS Lens の **silo model** + SEC01-BP01 (account separation strongly recommended) に整合
- AWS Lambda Best Practices "Operating Lambda" blog の **Lambda 1:1 IAM role 原則** 整合
- 過去 8 回繰返した demo / production 並行実装の構造原因 (`if (isDemo)` 漏れ / cookie 誤判定 / auth bypass 混入) が **物理的に発生不可能** になる
- 個人開発で security incident 対応不能 (緊急 patch + 影響範囲調査 + 顧客通知) という現実制約に対する「IAM レベルの insurance」を ~$2.74/月で得る

#### 5.5.1 IAM Role 比較表

| リソース | `production-lambda-role` (本番) | `demo-lambda-role` (デモ) |
|---|---|---|
| **DynamoDB Table (production)** | full R/W (`grant: Table ARN`) | **アクセス権なし** (Role policy に Table ARN を含めない) |
| **DynamoDB Table (demo-*)** | アクセス権なし | grant: full R/W (`grant: demo-* prefixed Table ARN のみ`) |
| **Secrets Manager (production secrets)** | grant (Cognito UserPool ID / Stripe API Key 等) | **アクセス権なし** |
| **Cognito UserPool (production)** | grant: read | **アクセス権なし** |
| **S3 assets bucket** | grant: R/W | **grant: read-only** (画像のみ取得可、書き込み不可) |
| **CloudWatch Logs** | `/aws/lambda/ganbari-quest` (retention 30 日) | `/aws/lambda/ganbari-quest-demo` (retention 7 日) |
| **環境変数 `DATA_SOURCE`** | `dynamodb` | `demo` (in-memory fixture 経由) |
| **環境変数 `AUTH_MODE`** | `cognito` | `anonymous` (dummy user 固定) |

#### 5.5.2 物理境界が保証する Security Property

1. **本番 DynamoDB の漏洩リスク 0**: デモ Lambda が production Table ARN を IAM Policy に持たないため、コード bug / configuration error が混入しても `AccessDenied` で確実に拒否される
2. **本番 Secrets の漏洩リスク 0**: Stripe API Key / Cognito 設定値等の本番 Secrets を取得する API 呼び出し自体が IAM レベルで拒否される
3. **本番 Cognito 攻撃面 0**: デモ Lambda から本番 UserPool への JWT 検証 / user enumeration / brute-force 等の試行が IAM レベルで拒否される
4. **Blast Radius 構造保証**: デモ Lambda が侵害された場合でも、本番 DB / Secrets / UserPool への横展開が **物理的に不可能** (account 分離ほど extreme ではないが、Pre-PMF 個人開発の現実解として最適、ADR-0048 §選択肢 D 棄却根拠)

#### 5.5.3 サブドメイン分離

- **`demo.ganbari-quest.com`** (subdomain) を新規追加し、本番 origin (`app.ganbari-quest.com`) と独立 Lambda 配下で運用
- CloudFront: 既存 distribution に **alternate domain name `demo.ganbari-quest.com`** 追加、origin host header 駆動で demo Function URL に振り分け
- SSL cert は `*.ganbari-quest.com` wildcard 既存利用 (us-east-1 ACM)
- Route 53: A (Alias) record `demo.ganbari-quest.com` → CloudFront distribution
- IAM 分離 + subdomain 分離の二段構成で「同 origin で `?demo=true` 等の query parameter で切り替え」型の脆弱性 (auth bypass / tenant filter 漏れ) を構造的に排除

#### 5.5.4 デモ Lambda の Application 層

IAM 分離を補完する application 層の制約 (ADR-0048 §2 Application 設計):

- **Stateless**: Lambda module-level singleton で **user-specific mutable state を保持しない**。「demo で記録 → リロードで保持」は **client sessionStorage 限定** (tab 閉じで消失)
- **Anonymous Auth**: `AUTH_MODE=anonymous` で `AnonymousAuthProvider` 選択。dummy user (`anon-{requestId}`) + role='owner' + tenantId='demo' + licenseStatus=ACTIVE を返す
- **Test Double**: Repository Pattern + Abstract Factory で `DATA_SOURCE=demo` の場合 `DemoXxxRepo` を返す。**Fake (read) + Stub (write) hybrid**:
  - read API: in-memory fixture (`demo-data.ts`) から返却
  - write API: `{ ok: true, demo: true }` 200 no-op response (DB に永続化しない)

#### 5.5.5 IAM Physical Isolation Regression Test (Hard-Fail Gate)

IAM 分離は **CDK 単体テストで physical assertion** する。設計が破壊された (本番 Table ARN がデモ Role に混入する等) 場合に CI で hard-fail させる:

- **テスト**: `tests/unit/infra/multi-lambda-cdk.test.ts` C-1 (Multi-Lambda CDK regression)
- **検証内容**: synth した CloudFormation template から `demo-lambda-role` の Policy Document を抽出し、production Table ARN / production Secrets ARN / production UserPool ARN を一切含まないことを assert
- **失敗時の挙動**: CI hard-fail (PR merge ブロック)
- **設計意図**: IAM 分離は document policy ではなく **machine-enforced contract**。ADR-0006 (Safety Assertion Erosion Ban) と整合、assertion を弱める変更は CI で拒否

## 6. API セキュリティ

### 6.1 レートリミッター

インメモリ方式のレートリミッター（`src/lib/server/security/rate-limiter.ts`）:

| 対象 | 上限 | ウィンドウ | 適用条件 |
|------|------|----------|---------|
| API 全般 | 100 req/min per IP | 1分 | cognito モードのみ |

- Lambda 単一プロセス向け（インスタンス間共有なし）
- **ライセンスキー専用レートリミッター** (`rate-limit-service.ts`): IP と email の二次元チェック。超過時に Discord incident チャネルへ自動通知（10 分間の重複抑制付き）。汎用 `rate-limiter.ts` とは分離し、ドメイン固有のビジネスロジックを担当

### 6.2 入力バリデーション

- API エンドポイントではリクエストボディの型チェック・値域チェックを実施
- `$lib/domain/validation/` に共通バリデーションルール定義
- ルートハンドラで型チェック後、サービス層に委譲

### 6.3 エラーハンドリング

- `@sveltejs/kit` の `error()`, `json()` で一貫したエラーレスポンス
- `handleError` フックでサーバーエラーの詳細をログに記録
- クライアントには `{ message: 'Internal Server Error' }` のみ返却（内部情報非公開）

### 6.4 破壊操作 UI ガード — GitHub Danger Zone パターン (#2319 / #2321 / #2323)

不可逆な破壊的操作は **GitHub Danger Zone パターン** を共通適用し、誤操作リスクを構造的に低減する:

- **対象操作**: `accountDelete` (`/admin/settings/account`) / `clear-all` (`/admin/settings/data`)
- **視覚的隔離**: 赤枠 (`border: 2px solid var(--color-action-danger)`) + ヘッダー (`bg-feedback-error-bg` + `⚠️` アイコン) + ページ最下部配置
- **3-step 確認**:
  1. 確認テキスト入力 (例: `削除` / `アカウントを削除します` を文字どおり入力)
  2. 同意 checkbox (「すべてのデータを削除することに同意します」等)
  3. 実行ボタン (Step 1 + Step 2 が両方満たされて初めて enabled)

`accountDelete` は owner (家族グループ全体に影響) / parent (非 owner) / child でメッセージを切替、owner かつ他メンバーがいる場合は権限移譲ダイアログを出して直接 full delete を避ける (#1781 削除グレースピリオド機構との併用)。

## 7. Web セキュリティ対策

### 7.1 セキュリティヘッダ

`hooks.server.ts` で全レスポンスに付与:

| ヘッダ | 値 | 目的 |
|--------|-----|------|
| `X-Frame-Options` | `DENY` | クリックジャッキング防止 |
| `X-Content-Type-Options` | `nosniff` | MIME スニッフィング防止 |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | リファラー情報の制限 |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` | 不要な API の無効化 |
| `Content-Security-Policy` | `default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: blob:; media-src 'self' blob:; font-src 'self'; connect-src 'self'; object-src 'none'; base-uri 'self'; frame-ancestors 'none'` | XSS/Polyglot 攻撃防止 |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | HTTPS 強制（cognito のみ） |
| `Cache-Control` | `no-cache, no-store, must-revalidate` | 認証ページのキャッシュ禁止 |

### 7.1.2 LP（静的サイト）の CSP（ADR-0029 / #1719）

`site/**` は GitHub Pages の静的配信であり、SvelteKit の `hooks.server.ts` を経由しない。
HTTP レスポンスヘッダーで CSP を付与できないため、`<meta http-equiv="Content-Security-Policy">` で同等の防御を行う。

**対象ファイル**（10 ページ全て）:

`site/index.html` / `pricing.html` / `faq.html` / `pamphlet.html` / `selfhost.html` / `privacy.html` / `terms.html` / `sla.html` / `tokushoho.html` / `help/license-key.html`

**CSP 内容**（全ページ共通）:

```html
<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net; style-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net; img-src 'self' data: blob:; font-src 'self'; connect-src 'self' https://cdn.jsdelivr.net; object-src 'none'; base-uri 'self'; form-action 'self'">
```

**指令の根拠**:

| 指令 | 値 | 根拠 |
|------|-----|------|
| `default-src` | `'self'` | LP は同一 origin で完結する fallback |
| `script-src` | `'self' 'unsafe-inline' https://cdn.jsdelivr.net` | DOMPurify (全 10 ページ) + splidejs (index.html) + budoux (index.html) を allowlist。`'unsafe-inline'` は inline `<script>` (initialization / Splide setup / mobile menu / contact form) と `<script type="application/ld+json">` 用 |
| `style-src` | `'self' 'unsafe-inline' https://cdn.jsdelivr.net` | splide-core CSS を allowlist。`'unsafe-inline'` は一部 inline `style="..."` 属性用 |
| `img-src` | `'self' data: blob:` | 同 origin 画像 + favicon + OGP + データ URI |
| `font-src` | `'self'` | 外部フォント未使用（system-ui） |
| `connect-src` | `'self' https://cdn.jsdelivr.net` | LP の動的 fetch / XHR は無いが、ブラウザ DevTools が jsDelivr 配信 JS の sourcemap (`.map`) を `connect-src` 経由で取得する (#2068)。`'self'` のみだと Chrome console に CSP violation 3 件 (splidejs / DOMPurify / budoux sourcemap) が常時表示される。`script-src` で既に許可済の origin と揃える |
| `object-src` | `'none'` | `<object>` / `<embed>` プラグイン禁止 |
| `base-uri` | `'self'` | `<base>` 注入による URL 書き換え攻撃防止 |
| `form-action` | `'self'` | フォーム送信先を同 origin に限定 |

**CDN ライブラリの SRI / pin マトリクス（ADR-0029 §決定 5）**:

| ライブラリ | 配信先 | pin 戦略 | SRI | CSP allowlist | 根拠 |
|-----------|--------|---------|-----|---------------|------|
| `@splidejs/splide@4.1.4` JS | `cdn.jsdelivr.net` | 完全 pin | **必須** (`sha384-...`) | 不要（補助） | bytes 固定で SRI が機能 |
| `@splidejs/splide-core@4.1.4` CSS | `cdn.jsdelivr.net` | 完全 pin | **必須** (`sha384-...`) | 不要（補助） | 同上 |
| `dompurify@3` | `cdn.jsdelivr.net` | major pin | **付与禁止** | **必須** | patch 自動取り込み優先（CVE 対応） |
| `budoux@latest` (index.html) | `cdn.jsdelivr.net` | range | **付与禁止** | **必須** | XSS 経路にならない補助ライブラリ |

**多層防御の構造**:

1. **第一層: DOMPurify サニタイズ** (ADR-0025) — `el.innerHTML = DOMPurify.sanitize(...)` で SSOT 注入時に XSS payload を除去
2. **第二層: CSP allowlist** (本節) — 攻撃者が任意 origin の script を注入する経路を構造的に塞ぐ
3. **第三層: SRI 付き完全 pin ライブラリ** (splidejs) — bytes が固定のライブラリは SRI で改ざん検知

**meta tag CSP の制約**:

- `frame-ancestors` / `report-uri` / `sandbox` は meta tag では効かない（HTTP ヘッダー専用）
- 本 LP は埋め込み禁止 (`X-Frame-Options: DENY`) も静的ホストでは制御不可で、Pre-PMF スコープ (ADR-0010) で許容済み
- 認証 / 動的 content / ユーザー固有データは LP に存在しないため脅威モデル上問題にならない

**E2E テスト**: `tests/e2e/lp-csp.spec.ts` で全 10 ページの CSP meta tag 存在検証 + ロード時 CSP violation ゼロ検証 + DOMPurify ロード成功 + SSOT 注入動作を担保。

### 7.2 XSS 対策

- SvelteKit はテンプレート出力を自動エスケープ（`{@html}` 以外）
- `{@html}` の使用は最小限に制限
- ユーザー入力はサーバー側でもバリデーション

### 7.3 CSRF 対策

- SvelteKit の Form Actions は組み込みの CSRF トークン検証を実施
- API エンドポイントは Cookie ベースの認証（SameSite=Lax）で CSRF リスクを軽減
- 状態変更 API は POST/PUT/DELETE メソッドを使用

### 7.4 インジェクション対策

- **SQL インジェクション**: Drizzle ORM のパラメータバインディングで防止（SQLite）
- **NoSQL インジェクション**: DynamoDB SDK のパラメータ化クエリで防止
- **コマンドインジェクション**: 外部コマンド実行なし

## 8. 子供データの特別保護方針

### 8.1 基本方針

がんばりクエストは子供（未成年者）のデータを扱うため、以下の原則を遵守する:

1. **最小限のデータ収集**: 必要最小限のデータのみ収集（ニックネーム、年齢、活動記録）
2. **個人識別情報の最小化**: 子供の本名・写真・位置情報は収集しない
3. **データの家族内限定**: テナント分離により他の家族のデータにアクセス不可
4. **保護者による管理**: 子供アカウントの作成・削除は保護者（owner/parent）のみ可能

### 8.2 収集するデータ

| データ種別 | 内容 | 保護レベル |
|-----------|------|-----------|
| プロフィール | ニックネーム、年齢、テーマカラー、UIモード | テナントスコープ |
| 活動記録 | 活動名、記録日時、ポイント | テナントスコープ |
| ステータス | レベル、経験値、連続日数 | テナントスコープ |
| 実績 | 獲得した実績一覧 | テナントスコープ |
| アバター画像 | AI 生成画像（S3 保存） | テナントスコープ |

### 8.3 収集しないデータ

- 本名、住所、電話番号
- 顔写真、動画
- 位置情報
- 他サービスとの連携データ
- 広告 ID、トラッキング情報

### 8.4 データ削除

- 子供プロフィール削除時: 関連する全データ（活動記録、ステータス、実績、アバター画像等）をトランザクションで一括削除
- テナント削除時: テナントに紐づく全データを削除

### 8.5 電気通信事業法 §27の12 対応（外部送信規律 公表）

電気通信事業法第27条の12（外部送信規律）に基づき、利用者の端末から外部第三者へ送信される情報を `site/privacy.html` 第8条で公表する。対応 Issue: #1590（ADR-0023 I1 派生）。

| 項目 | 内容 |
|------|------|
| 送信される情報 | ページ URL、リファラ、訪問時刻、画面解像度、ブラウザ言語、ユーザーエージェント |
| 送信先 | AWS（自社アカウント内 DynamoDB / Lambda / Cognito）、Stripe、AWS Bedrock、Google Gemini |
| 利用目的 | ウェブサイト機能提供および改善 / 課金処理 / コンテンツ生成 |
| 個人識別情報 | 利用者本人を直接識別する情報（氏名・住所・電話番号等）は取得しない |
| 利用者の選択肢 | ブラウザ Cookie ブロック等 |

**実装上の遵守事項**:
- 子供データ（活動記録 / プロフィール / アバター画像）は外部第三者へ送信しない（§8.7 と整合）
- AWS Bedrock / Google Gemini への生成 AI リクエストには家族内一意 ID を含めない
- 文言ドリフト防止: `LEGAL_LABELS` (`src/lib/domain/labels.ts`) のキー用語が `site/privacy.html` に出現することを `scripts/check-lp-ssot.mjs` で CI 検証

### 8.6 個人情報保護法 §28 対応（域外移転 + 本人同意取得）

個人情報保護法第28条に基づき、外国にある第三者への個人データ提供について、`site/privacy.html` 第10条で情報提供 + サインアップ時に本人同意を取得する。対応 Issue: #1638。

| 項目 | 内容 |
|------|------|
| 移転先国 | 米国（AWS バージニア北部リージョン: `us-east-1`） |
| 第三者の名称 | Amazon Web Services, Inc.（米国デラウェア州法人） |
| 法的根拠 | AWS DPA (Data Processing Addendum) + 標準契約条項 (SCC) |
| 移転される情報の範囲 | 利用者識別子（家族内一意 ID）、活動記録、課金関連情報 |
| 本人同意取得 UI | `src/routes/auth/signup/+page.svelte` の `agreedCrossBorder` チェックボックス（必須） |

**実装上の遵守事項**:
- `agreedCrossBorder` は `agreedTerms` / `agreedPrivacy` と同様に必須チェック（未チェックは submit ブロック）
- 日本国内サーバー（`ap-northeast-1` / 東京リージョン）の文言を新規追加しない（PR #1656 で削除済みの方針維持）
- 改正個情法（2026年成立予定）の本文確定後の差分対応は別 Issue で

### 8.7 未成年者の取扱い

`site/privacy.html` 第9条 + `site/terms.html` 第6条で明示する。対応 Issue: #1590。

| 項目 | 内容 |
|------|------|
| 親同意フレームワーク | 全年齢で運用。子供本人がアカウント作成不可（保護者のみ） |
| 利用者識別子 | 家族内一意 ID のみ（学校名・氏名・住所・電話番号等は取得しない） |
| 本人への直接接触 | アンケート・通知・メールマガジン等は一切行わない（連絡は保護者宛のみ） |
| データの域外送信 | ゼロ。自社 AWS アカウント内 DynamoDB のみで処理（生成 AI 等への送信なし） |
| 親による削除請求 | 優先処理（§5 利用者の権利・§6 データの削除） |

**実装上の遵守事項**:
- 子供アカウントは保護者（owner / parent ロール）のみ作成・削除可能
- 子供データを生成 AI（Bedrock / Gemini）に渡すリクエストには家族内一意 ID を含めない（§8.5 と整合）
- 「16歳未満」のしきい値は採用しない（全年齢で同一の親同意フレームワーク運用）

### 8.8 Web Push 通知の対象監査 (#1593, ADR-0023 I6)

子端末への直接マーケ通信は **永久 NG** とする（COPPA 改正 + ADR-0012 Anti-engagement の二重規制）。

#### 設計

`push_subscriptions` テーブルに `subscriber_role` カラム (`'parent' | 'owner'`、default `'parent'`) を持たせ、subscribe 時の role を必ず記録する。

| 層 | 防御 | 実装場所 |
|----|------|---------|
| API ガード | `POST /api/v1/notifications/subscribe` で `context.role === 'child'` の場合 403 + `code: 'CHILD_FORBIDDEN'` を返す | `src/routes/api/v1/notifications/subscribe/+server.ts` |
| 送信時ガード（二重防御） | `notification-service.sendPushNotification` が `findByTenant` 結果から `subscriberRole !== 'parent' && subscriberRole !== 'owner'` の subscription を skip + `logger.warn` | `src/lib/server/services/notification-service.ts` |
| スキーマ | `subscriber_role TEXT NOT NULL DEFAULT 'parent'` (ADR-0031 NULL 混在防止) | `src/lib/server/db/schema.ts` + `create-tables.ts` + `tests/e2e/global-setup.ts` + `tests/unit/helpers/test-db.ts` |
| 型 | `PushSubscriberRole = 'parent' \| 'owner'` (`'child'` は型に含めない) | `src/lib/server/db/types/index.ts` |
| ラベル SSOT | エラーメッセージは `PUSH_NOTIFICATION_LABELS.childSubscribeForbidden` を経由 | `src/lib/domain/labels.ts` |

#### テスト

- Unit: `tests/unit/routes/notifications-subscribe-api.test.ts` — child role の 403 拒否 / parent / owner の subscriberRole 記録 / 入力バリデーション
- Unit: `tests/unit/services/notification-service.test.ts` — 送信時の child / 不明 role skip 検証

#### 既存レコード backfill

本機能リリース前の既存 subscription は default `'parent'` で扱われる。これは `npm run dev:cognito` 環境では DEV_USERS の owner / parent / child の subscribe 試行が ADR-0023 §3.1 の通り「親管理画面動線のみ」に限定されてきた前提に依拠する。

**SQLite (NUC)**: `scripts/migrate-local.ts` で `ALTER TABLE ADD COLUMN ... DEFAULT 'parent'` + `UPDATE ... WHERE subscriber_role IS NULL OR ''` を実行済（#1593）。

**DynamoDB (本番 AWS) (#1666)**: `scripts/migrate-dynamodb-push-subscriber-role.mjs` で `entityType = 'PUSH_SUB'` または `SK begins_with 'PUSH_SUB#'` の item を Scan し、`subscriberRole` 不在 / NULL / 不正値を `'parent'` で UpdateItem する idempotent script を提供。実行手順は [`docs/runbooks/push-subscription-role-migration.md`](../runbooks/push-subscription-role-migration.md) 参照。実行タイミングは DynamoDB push-subscription-repo 本実装マージ後（実装本体は本 follow-up Issue とは別 Issue で扱う、ADR-0010 Pre-PMF / #1021 段階的対応禁止に整合）。検証は CloudWatch Logs `[notification] 非 parent/owner role の subscription への送信をスキップ` warn 件数 = 0。

## 9. 監査ログ

### 9.1 ログ設計

`src/lib/server/logger.ts` による構造化ログ:

| ログレベル | 用途 | 例 |
|-----------|------|-----|
| `INFO` | 正常処理 | リクエストログ（メソッド、パス、ステータス、レイテンシ） |
| `WARN` | 注意事項 | レートリミット超過、認証失敗、404 |
| `ERROR` | エラー | サーバーエラー（500）、外部API障害 |

### 9.2 ログ対象

- **リクエストログ**: 全 HTTP リクエスト（静的ファイル除外）のメソッド・パス・ステータスコード・処理時間
- **認証イベント**: ログイン成功/失敗、JWT検証失敗
- **レートリミット**: 超過イベント（IP、パス）
- **エラー**: サーバーエラーの詳細（スタックトレース含む）

### 9.3 ログの保存

- **Lambda**: CloudWatch Logs に自動出力
- **NUC**: コンソール出力（Docker logs で確認）
- S3 `logs/` プレフィックスに保存されたログは1日後に Glacier へアーカイブ

## 11. インフラセキュリティ

### 11.1 IAM

- Lambda 実行ロールに最小権限を付与:
  - DynamoDB: テーブルへの CRUD のみ
  - S3: アセットバケットへの Read/Write のみ
  - Cognito: UserPool への読み取りのみ
  - SSM: パラメータの読み取りのみ

### 11.2 ネットワーク

- CloudFront → Lambda Function URL（HTTPS のみ）
- Lambda はパブリック VPC 内で実行（DB アクセスは DynamoDB エンドポイント経由）
- S3 バケットはパブリックアクセス完全ブロック

### 11.3 シークレット管理

| シークレット | 保存場所 | 用途 |
|------------|---------|------|
| Cognito User Pool ID | SSM Parameter | JWT 検証 |
| Cognito Client ID | SSM Parameter | 認証フロー |
| Context Token Secret | 環境変数 | Context Token 署名 |
| Stripe API Key | SSM Parameter | 決済処理 |
| Stripe Webhook Secret | SSM Parameter | Webhook 署名検証 |

### 11.4 CI/CD セキュリティ

- **GitHub Actions**: main ブランチへの push で自動デプロイ
- **CodeQL**: プッシュごとにセキュリティスキャン実行
- **Dependabot**: 依存ライブラリの脆弱性を自動検出・PR作成
- **コミット前チェック**: Biome lint + svelte-check + Vitest + Playwright E2E

## 12. 既知の制限事項

1. **レートリミッターがインスタンス間で共有されない**: Lambda のインスタンスごとに独立した Map で管理。大量のリクエストが複数インスタンスに分散した場合、制限が緩くなる可能性がある
2. **PITR 無効**: DynamoDB の Point-in-Time Recovery はコスト最適化のため無効。AWS Backup（日次3日保持）で代替
