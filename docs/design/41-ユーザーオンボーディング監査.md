## 1. 概要

本レポートは、がんばりクエストの **Activation フロー**（LP 到達 → サインアップ → 初回活動記録 → シール獲得）を P1 ペルソナの視点で通し操作し、UX 上の摩擦ポイント・改善案をまとめたものである。

### Activation 4 ステップ定義

| Step | ゴール | 画面 |
|------|--------|------|
| 1 | LP 到達 → サインアップ完了 | `site/index.html` → `/auth/signup` |
| 2 | /admin 初回到達 → チュートリアル開始 | `/admin` |
| 3 | 最初の子供登録 | `/admin/children` |
| 4 | 最初の活動記録 → シール獲得 | `/(child)/[uiMode]/home` |

## 2. Step 1: LP → サインアップ

### 2.1 LP 構成の現状

**CTA ヒエラルキー（ヒーローセクション）:**

```
[🎮 まずはデモで体験する]  [無料ではじめる]
```

- デモボタンが左（優先位置）、サインアップが右
- バッジ: ✅基本無料 / 🔒データは家族だけのもの / 📱スマホ対応 / 🌐セルフホスト可
- カルーセルで子供画面・ステータス画面・管理画面のスクリーンショットを表示
- フローティング CTA バー（モバイル固定）: 「7日間、全機能を無料で体験 — クレジットカード不要ですぐ始められます」

**P1 視点の気づき:**

| # | 気づき | 深刻度 |
|---|--------|--------|
| 1-1 | 「デモ体験」と「無料ではじめる」の2つのボタンが同じ重みで並んでおり、**どちらをクリックすべきか迷う**。P1 は「まず試したい」と思うのでデモを選ぶが、デモ体験後にサインアップへの導線が弱い | 中 |
| 1-2 | ヒーロー下のバッジ「セルフホスト可」は P1 には意味不明。P2（ITエンジニア）向けの訴求が P1 の情報密度を上げている | 低 |
| 1-3 | 料金表（3プラン）がページ下部にあるが、**フリープランの制限（子供1人・活動5個）が小さい文字**で書かれており、登録後に「えっ、制限あるの？」となりやすい | 中 |
| 1-4 | LP のカルーセルスクリーンショットは静的画像。**アニメーション GIF やショートムービーで動きを見せた方が RPG 感が伝わる** | 低 |

### 2.2 サインアップフォーム

**フォーム構成（2 ステップ）:**

1. **Step 1 — アカウント作成**
   - メールアドレス入力
   - パスワード（8文字以上・大文字小文字数字必須）
   - パスワード確認
   - Google OAuth ボタン（代替パス）
   - ライセンスキー入力（折りたたみ・オプション）
   - 利用規約 / プライバシーポリシー同意チェック

2. **Step 2 — メール確認**
   - 6桁確認コード入力
   - 再送ボタン（60秒クールダウン）
   - 確認成功後、自動ログイン → `/admin` へリダイレクト

**P1 視点の気づき:**

| # | 気づき | 深刻度 |
|---|--------|--------|
| 2-1 | **利用規約チェックボックスが、リンクをクリックするまで disabled**（#588）。P1 は「チェックしようとしたらグレーアウトで動かない…なぜ？」と混乱する。ヒント文「先に利用規約をお読みください」はあるが、直感的でない | 高 |
| 2-2 | パスワード要件（大文字小文字数字）が入力時にリアルタイムでフィードバックされるが、**要件一覧が初期表示されていない**。P1 は最初に「password123」と入力して弾かれる体験をする | 中 |
| 2-3 | メール確認コードの有効期限が明示されていない。P1 が「子供がぐずって中断」→ 30分後に戻った時、期限切れかどうか分からない | 中 |
| 2-4 | Google OAuth パスが用意されているのは良い。しかし**ボタンの位置がフォームの下部**にあり、見落とされやすい | 中 |
| 2-5 | ライセンスキーセクションは折りたたみだが、展開すると4フィールド + 同意チェック + 確認モーダルが増え、**オプション機能にしてはフォーム密度が高い** | 低 |

### 2.3 デスクトップ vs モバイル差分

| 項目 | デスクトップ | モバイル |
|------|------------|---------|
| LP フローティング CTA | 非表示 | 画面下部に常時固定表示 |
| サインアップフォーム幅 | 中央寄せ・適切 | 全幅・入力しやすい |
| Google OAuth ボタン | フォーム下部 | 同左（見落としやすい位置は同じ） |
| パスワードバリデーション | フォーム右横にツールチップ | フォーム下部にインライン |

## 3. Step 2: /admin 初回到達

### 3.1 チュートリアルオーバーレイ

初回ログイン時に**ステップ式チュートリアル**が自動起動する。

**構成: 9 チャプター / 最大 23 ステップ**

| # | チャプター | ステップ数 | 対象ページ | プラン制限 |
|---|-----------|-----------|-----------|-----------|
| 1 | はじめに | 4 | /admin | なし |
| 2 | こどもの登録 | 3 | /admin/children | なし |
| 3 | 活動の管理 | 3 | /admin/activities | Standard+ (1 step) |
| 4 | 報酬とポイント | 2 | /admin/rewards, /admin/points | なし |
| 5 | レポート | 2 | /admin/reports | なし |
| 6 | メッセージ送信 | 2 | /admin/messages | なし |
| 7 | データ管理 | 1 | /admin/settings | Standard+ |
| 8 | 設定と日常 | 4 | /admin | なし |
| 9 | アップグレード | 1 | /admin | なし |

**P1 視点の気づき:**

| # | 気づき | 深刻度 |
|---|--------|--------|
| 3-1 | **23 ステップは多すぎる**。P1 は 17:30 に帰宅して夕食準備の合間に試している。5ステップ目あたりで「後でいいや」とスキップする可能性が高い | 高 |
| 3-2 | チュートリアルとは別に**オンボーディングチェックリスト**（6項目）も存在する。**2つのオンボーディング機構が併存**しており、メンタルモデルが分裂する | 高 |
| 3-3 | チュートリアルが「説明を読む」形式で、**実際の操作を促す形式ではない**。「ここをタップしましょう」→ 実際にタップ → 成功フィードバック、のようなインタラクティブガイドの方が記憶に残る | 中 |
| 3-4 | Free プランユーザーのチュートリアルに Standard+ 限定ステップが混在しており、「やりたいのに使えない」体験が生まれる | 中 |

### 3.2 オンボーディングチェックリスト（必須4項目 + 任意1項目）

#1361 で「必須/任意の分離 + 折りたたみ UI」を実装。4必須 + 2任意の構成（`required` フィールドで制御）。

| # | 項目 | required | 完了条件 | 遷移先 |
|---|------|----------|---------|--------|
| 1 | 子供を登録する | true | children.length > 0 | /children |
| 2 | 活動パックを選ぶ | true | activities.length > 0 | /activities |
| 3 | ごほうびプリセットを選ぶ | true | reward_templates 1 件以上 | /rewards |
| 4 | チェックリストを作る | true | checklist テンプレート存在 | /checklists |
| 5 | 子供の画面を確認する | true | childScreenVisited フラグ | /switch |
| 6 | おやカギコードを変更する | false | pinHash 設定済み | /settings |

**UI 仕様（#1361）:**
- 必須4項目は常時表示
- 任意項目は `<details>/<summary>` で折りたたみ表示（初期状態: 閉じる）
- 必須4項目が全て完了した時点で「✅ はじめてのセットアップ完了!」を表示（任意が未完でも祝福）
- `allCompleted` は `required: true` のアイテムのみで計算（`onboarding-service.ts` line 103）
- `nextRecommendation` は未完の required アイテムを優先し、全必須完了後は未完の optional を返す

#1359 で追加: 項目 3「ごほうびプリセットを選ぶ」。コアループ L1（子供が活動記録 → ポイント → ごほうびで交換）を成立させるには、サインアップ直後にごほうびが存在する状態が必要。

**P1 視点の気づき:**

| # | 気づき | 深刻度 |
|---|--------|--------|
| 3-5 | チェックリスト 6 項目は適切な粒度。**しかしチュートリアル 23 ステップと並行表示されるため、「どっちに従えばいいの？」問題**が発生する | 高 |
| 3-6 | 「おやカギコードを変更する」は `required:false`（任意）に変更済み（#1360）。P1 は後から設定可能で、初回体験の流れを止めない。デフォルト値 5086 で管理画面アクセス可 | 解消 |
| 3-7 | 折りたたみ UI (#1361) により任意項目が非表示になり、必須4項目に集中できる。「さらに便利にする設定（任意・1 項目）」として展開可能。 | 解消 |

## 4. Step 3: 最初の子供登録

### 4.1 子供登録フォーム

**フィールド:**
- ニックネーム（必須）— プレースホルダー: 「例: たろうくん」
- 生年月日（任意）— 入力すると年齢が自動計算
- 年齢（0-18歳・必須）
- テーマカラー（必須）— pink / blue / green / orange / purple

**P1 視点の気づき:**

| # | 気づき | 深刻度 |
|---|--------|--------|
| 4-1 | フォームは最小限で適切。ただし**テーマカラー選択が「必須」であり、P1 は「何のための色？」と迷う可能性**がある。説明が不足 | 低 |
| 4-2 | P1 は子供2人（ゆい 7歳 + たけし 5歳）を登録したいが、**Free プランは子供2人まで**という制限がこの段階で初めて顕在化する（LP では小さい文字だった）。3人目の登録でブロックされた時に初めて理解する | 中 |
| 4-3 | 1人ずつ登録する UI のため、**2人の子供を連続で登録する場合の導線が弱い**。1人目登録後に一覧に戻り、再度「追加」を押す必要がある | 低 |

## 5. Step 4: 最初の活動記録 → シール獲得

### 5.1 子供画面への到達

**経路:** /admin → 子供一覧 → 子供選択 → /switch → 子供のホーム画面

**P1 視点の気づき:**

| # | 気づき | 深刻度 |
|---|--------|--------|
| 5-1 | 管理画面から子供画面への切り替えが **3クリック以上**かかる（子供選択 → /switch → 子供ホーム）。P1 は「子供に見せたいのに、たどり着くまでが長い」と感じる | 中 |
| 5-2 | 子供画面に遷移する時に **おやカギコード入力を求められる場合がある**。デフォルト値 5086 を知らない P1 は戸惑う可能性があるが、設定画面の「初期値は 5086 (がんばり) です」という説明文が誘導（#1360）| 低 |

### 5.2 活動記録フロー

**子供ホーム画面:**
- カテゴリ別（うんどう / べんきょう / せいかつ / ひと / アート）に活動カードが並ぶ
- 活動カードをタップ → 詳細 → 「きろく！」ボタン
- 記録完了 → ポイント獲得演出 → コンボ・デイリーミッション・スタンプカード

**子供チュートリアル（4チャプター / 9ステップ）:**
- 簡潔でひらがなのみ。年齢に適した言葉遣い
- 「このカードをタップしてみよう！」→ 実際にタップ → フィードバック

**P1 視点の気づき:**

| # | 気づき | 深刻度 |
|---|--------|--------|
| 5-3 | 子供チュートリアルは親チュートリアル (23 ステップ) と比べて**適切な分量 (9 ステップ)**で、子供が自分で操作しながら進められる | なし（良好） |
| 5-4 | 活動パックが未選択の場合（オンボーディングチェックリスト項目2未完了）、**子供画面に活動カードが表示されず空の状態になる**。P1 は「何も出てこない…壊れてる？」と不安になる | 高 |
| 5-5 | 初回のポイント獲得演出は**ゲーミフィケーションのコア体験**であり、ここでの感動が継続利用を決める。現在の演出（ポイント加算 + コンボ表示）は機能的だが、**レベルアップのような「初回特別演出」がない** | 中 |

## 6. デモフロー分析

### 6.1 デモ画面の構成

**エントリーポイント:** `/demo`
- 5人のデモ子供が事前作成済み（がんばり家のこどもたち）
- ガイド付きデモ（6ステップ）or 自由探索を選択可能
- ヘッダーに常時表示: 「これはデモです。データは保存されません。」
- プラン切替タブ: Free / Standard / Family

**スクリーンショット確認結果:**

| 画面 | 状態 |

### 6.2 デモの効果と課題

**P1 視点の気づき:**

| # | 気づき | 深刻度 |
|---|--------|--------|
| 6-1 | デモは P1 の「まず安心して試したい」ニーズに応えており、**サインアップ前の最大のハードルを下げている** | なし（良好） |

## 8. Activation フローの時間見積もり（P1 視点）

| Step | 現状の所要時間 | 理想の所要時間 | ボトルネック |
|------|-------------|-------------|------------|
| LP 閲覧 → CTA クリック | 2-5分 | 1-2分 | CTA の選択肢迷い |
| サインアップフォーム入力 | 3-5分 | 1-2分 | 利用規約リンク必須制約 + パスワード要件 |
| メール確認 | 1-3分 | 1分 | メール受信待ち |
| /admin 到達 → チュートリアル | ~~5-15分~~ → 2-3分（QW-3 #955 / PR #961 で quickMode 導入済み。初回はチャプター1 のみ 4 ステップ） | 2-3分 | ~~23ステップのチュートリアル~~ → 解消済み |
| 子供登録 | 2-3分 | 1-2分 | 適切 |
| 子供画面到達 → 初回記録 | 3-5分 | 1-2分 | 活動パック選択 + 画面切り替え |
| **合計** | **16-36分** | **7-12分** | |

P1 の可処分時間は 21:00 の子供就寝後。**30分以上かかると「今日はもういいや」と離脱する可能性が高い。**

## 8.1 QW-3 実装サマリ（PR #961 / #955）

Issue #955 および PR #961 で「チュートリアルのクイックモード」を導入。#961 QA ラウンドで親チャプター限定ガードを追加。実装の要点:

- **初回起動はクイックモード (`quickMode=true`)**: `startTutorial()` を引数なしで呼ぶと、チャプター 1（4 ステップ）のみを表示。チャプター 1 最終ステップ完了時に「使い始める」/「もっと詳しく見る」の 2 択ダイアログを表示し、Activation Time を従来の 5-15 分から 2-3 分へ短縮。
- **2 択完了ダイアログ**: 「使い始める」で `finishQuickTutorial()` が走りサーバーに `action: 'complete'` を送信。「もっと詳しく見る」で `continueFullTutorial()` がチャプター 2 以降を順次再生。ラベルは `src/lib/domain/labels.ts` の `TUTORIAL_LABELS` に集約（SSOT）。
- **AdminHome から全ガイドへの導線**: 管理ホームに "くわしいガイドを最初から見る" カード (`data-testid="admin-view-full-guide"`) を配置。明示チャプター指定 (`startTutorial(1)`) で `quickMode=false` となり、クイック完了ダイアログを挟まず全チャプターを通し再生する。
- **親チャプター限定ガード (#961 QA)**: `TutorialState.isParentChapters` を導入。`setChapters()` は `chapters === TUTORIAL_CHAPTERS` のときのみ true。子チュートリアル（`(child)/+layout.svelte` で `setChapters()` 呼び出し）では `quickMode` が強制的に無効化され、保護者向けの完了ダイアログやチャプター 2 継続プロンプトが子供画面に漏れない。

**関連ファイル**:
- `src/lib/ui/tutorial/tutorial-store.svelte.ts` — 状態機械本体
- `src/lib/features/admin/components/AdminHome.svelte` — 全ガイド導線
- `src/lib/domain/labels.ts` — `TUTORIAL_LABELS`（用語 SSOT）
- `tests/unit/tutorial/tutorial-store.test.ts` — 状態遷移の単体テスト
- `tests/e2e/tutorial-verification.spec.ts` — Case A/B の E2E 検証

### スクリーンショットの制限事項

- **認証保護ルート（LP・サインアップ・/admin・子供画面）**: SvelteKit の認証ミドルウェアにより、未認証状態ではキャプチャ不可。本監査では認証なし環境で実施したため、これらのルートは 500 エラーとなりキャプチャを断念した。
- **モバイルビューポート**: 今回の監査ではモバイル用スクリーンショットを取得していない。
- **デモルート（`/demo`）**: 認証不要のため正常にキャプチャできた（4枚）。
- **今後の対応**: cognito-dev E2E 環境が利用可能になった時点で、認証保護ルートおよびモバイルビューポートのスクリーンショットを補足予定。

## 10. 総合評価

### 良い点
1. **デモ体験が充実**しており、サインアップ前のリスク認知を下げている
2. **子供チュートリアル（9ステップ）は適切な分量**で、年齢に合った言葉遣い
3. **オンボーディングチェックリスト（6項目）のゴール設計は的確** — 子供登録 → 活動選択 → ごほうびプリセット → おやカギコード（任意） → チェックリスト → 子供画面確認（#1359 で「ごほうびプリセット」追加、#1360 で「おやカギコード」任意化）
4. **年齢別 UI モード**（baby/kinder/lower/upper/teen）が各年齢に最適化されている
5. **Google OAuth** によるワンクリック登録パスが用意されている

### 要改善点
1. **23ステップのチュートリアルが長すぎる**。初回体験を阻害している
2. **チュートリアルとチェックリストの二重構造**が認知負荷を増大
3. **利用規約リンク必須制約**がサインアップの摩擦になっている
4. **デモが Family プラン相当**のため、Free プランとのギャップが大きい
5. **活動パック未選択時の空画面**が初回体験を台無しにするリスク
