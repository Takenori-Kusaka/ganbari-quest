## 2. データ処理の概要

### 2.1 処理の目的

子供の日常活動（お手伝い・学習・運動等）をポイント化し、ゲーミフィケーション手法で動機付けを行う家庭向け Web アプリケーション。

### 2.2 処理の法的根拠

- **GDPR Art. 6(1)(a)**: 保護者の明示的同意（サインアップ時に利用規約・プライバシーポリシーへの同意を取得）
- **GDPR Art. 8**: 子供のデータ処理に対する保護者同意（本サービスは保護者が管理者として利用し、子供のプロフィールを作成する）

### 2.3 処理するデータ

| カテゴリ | データ | 感度 | 保存先 |
|---------|--------|------|--------|
| 子供プロフィール | ニックネーム、年齢、UIモード | 低〜中 | DynamoDB |
| 活動記録 | 活動名、記録日時、獲得ポイント | 中 | DynamoDB |
| ステータス | レベル、経験値、連続日数 | 低 | DynamoDB |
| 実績 | 獲得実績・称号 | 低 | DynamoDB |
| アバター画像 | AI生成画像 or アップロード画像 | 中 | S3 |
| 保護者メールアドレス | 認証用 | 高 | Cognito User Pool |
| 決済情報 | クレジットカード情報 | 極高 | Stripe（外部、自社保存なし） |

### 2.4 収集しないデータ

- 子供の本名・住所・電話番号
- 顔写真・動画
- 位置情報・GPS データ
- 他サービスとの連携データ
- 広告 ID・トラッキング情報

### 2.5 データ保存期間（Storage Limitation, GDPR Art. 5(1)(e)）

本サービスはプランに応じた履歴保持期間を設定し、期限を超過したデータを**日次バッチで物理削除**する。
これは GDPR Art. 5(1)(e)「保存制限の原則」および pricing page の公開約束を履行するための仕組み。

| プラン | 活動ログ・ポイント履歴・ログインボーナスの保持期間 | 対象テーブル |
|--------|------------------------------------------|------------|
| free | **90 日** | `activity_logs`, `point_ledger`, `login_bonuses` |
| standard | **365 日** | 同上 |
| family | 無制限（削除なし） | 同上 |

**削除フロー:**

1. 日次バッチ `retention-cleanup-service.cleanupExpiredData()` が全テナントを走査
2. 各テナントのプランティアを `resolveFullPlanTier()` で解決（トライアル考慮）
3. プランの cutoff 日（free=90 日前 / standard=365 日前）を算出
4. 各子の 3 テーブルから `recorded_date < cutoffDate` を物理削除（復元不能）
5. family プラン（トライアル含む）はスキップ

**削除されない（恒久保持）データ:**

- `children` / `activities` / `settings` 等のマスタ（データの意味を破壊するため）
- `report_daily_summaries` 等の集計テーブル（月次レポートは全期間の集計を前提）
- **ポイント残高**（BALANCE）— 明細は消えるが残高は維持（子供の UX 破壊を避けるための #729 の設計判断）
- `child_achievements` / `special_rewards` / `stamp_cards` 等（現時点では retention 範囲外、将来再検討）

**多層防御:**

バッチが失敗した場合のフォールバックとして、`applyRetentionFilter()` による読み取り時の表示フィルタも並行して動作する。これにより cutoff を超えたデータが UI に露出することはない。

**実装の詳細**: [08-データベース設計書.md §6.5](08-データベース設計書.md) / [ADR-0028](../decisions/0028-retention-physical-delete.md)

### 2.6 アカウント削除時のライフサイクル

- ユーザーによるアカウント削除: **30 日猶予期間**後に認証・テナント・全レコードを物理削除（ADR-0022）
- 違反アカウント強制削除: OPS 画面から即時削除
- 削除対象: DynamoDB 全レコード + S3 アセット + Cognito ユーザー

## 3. リスク評価

### 3.1 リスクマトリクス

| リスク | 影響度 | 発生可能性 | 対策後リスク | 緩和策 |
|--------|--------|-----------|------------|--------|
| 不正アクセスによるデータ漏洩 | 高 | 低 | **低** | Cognito JWT認証、テナント分離、TLS 1.2+、IAM最小権限 |
| クロステナントデータ混入 | 高 | 極低 | **極低** | DynamoDB PK にテナントプレフィックス、全層でtenantId必須 |
| 子供データの目的外利用 | 高 | 極低 | **極低** | データ最小化方針、広告/トラッキング非使用、第三者提供なし |
| アバター画像経由の攻撃 | 中 | 低 | **極低** | Sharp re-encode、マジックバイト検証、CSP、Content-Type ホワイトリスト |
| ブルートフォース攻撃 | 中 | 中 | **低** | IP ベースレートリミッター、Cognito パスワードポリシー |
| セッションハイジャック | 高 | 低 | **極低** | HttpOnly/Secure/SameSite Cookie、HSTS、短い有効期限 |
| データ削除の不完全（アカウント削除） | 中 | 低 | **低** | 30日猶予期間後の自動完全削除（DB + S3 + Cognito） |
| 保存制限原則違反（GDPR Art. 5(1)(e)） | 中 | 中→**低** | **低** | プラン別 retention cleanup 日次バッチ（free 90 日 / standard 365 日）＋ 表示フィルタの二層防御（ADR-0028） |
| バックアップ内の残存データ | 低 | 中 | **低** | S3 バックアップ30日自動削除、AWS Backup 3日保持 |
| サブプロセッサーでのデータ漏洩 | 高 | 極低 | **低** | AWS/Stripe は GDPR 準拠済み、DPA 締結確認済み |
| 内部関係者による不正アクセス | 中 | 極低 | **極低** | IAM ロール分離、CloudWatch ログ監視 |

### 3.2 リスク評価基準

- **影響度**: 低（不便）/ 中（精神的苦痛）/ 高（重大な権利侵害）
- **発生可能性**: 極低（理論上のみ）/ 低（技術的に困難）/ 中（一般的な攻撃で可能）/ 高（容易）

## 4. 実施済みの保護措置

### 4.1 技術的措置

| 措置 | 実装 | 参照 |
|------|------|------|
| 通信暗号化 | TLS 1.2+（CloudFront）、HSTS | セキュリティ設計書 §2 |
| 保存データ暗号化 | DynamoDB AWS managed key、S3 SSE-S3 | セキュリティ設計書 §3 |
| 認証 | Cognito JWT + HMAC Context Token | セキュリティ設計書 §4 |
| 認可 | ロールベースアクセス制御、テナント分離 | セキュリティ設計書 §5 |
| レートリミッター | IP ベース、認証/API 別制限 | セキュリティ設計書 §6 |
| セキュリティヘッダ | X-Frame-Options, CSP, HSTS, nosniff 等 | セキュリティ設計書 §7 |
| ファイルサニタイズ | Sharp re-encode、マジックバイト検証 | file-sanitizer.ts |
| 入力バリデーション | 型チェック、値域チェック | セキュリティ設計書 §6.2 |
| S3 パブリックアクセスブロック | BLOCK_ALL | storage-stack.ts |
| CI セキュリティスキャン | CodeQL、Dependabot、npm audit | GitHub Actions |

### 4.2 組織的措置

| 措置 | 実装 |
|------|------|
| 同意管理 | サインアップ時の利用規約・プライバシーポリシー同意（バージョン管理付き） |
| 同意再取得 | ポリシーバージョン更新時の再同意フロー |
| 同意記録 | IP、UA、タイムスタンプ、バージョンを DynamoDB に記録 |
| データ主体の権利 | アクセス権（UI閲覧）、ポータビリティ（JSON/ZIPエクスポート）、削除権（アカウント削除） |
| データ最小化 | 本名・住所・位置情報等を収集しない方針 |
| アカウント削除 | 30日猶予期間付き完全削除（DB + S3 + Cognito） |
| プライバシーポリシー | www.ganbari-quest.com/privacy.html で公開 |

### 4.3 データ主体の権利保障

| 権利 | GDPR 条項 | 対応状況 |
|------|----------|---------|
| アクセス権 | Art. 15 | ✅ UI で子供データを閲覧可能 |
| 訂正権 | Art. 16 | ✅ ニックネーム・年齢等の編集機能 |
| 削除権 | Art. 17 | ✅ アカウント完全削除（30日猶予 → 自動消去） |
| データポータビリティ | Art. 20 | ✅ JSON/ZIP エクスポート機能 |
| 処理の制限 | Art. 18 | ✅ 猶予期間中は読み取り専用モード |
| 同意の撤回 | Art. 7(3) | ✅ アカウント削除 = 同意撤回と同義 |

## 7. 評価結論

### 7.1 総合評価

本サービスは子供の個人情報を処理するため GDPR Art. 35 に基づく DPIA が必要であるが、以下の理由により**リスクは適切に緩和されている**と評価する:

1. **データ最小化**: 子供の本名・住所・位置情報等の高感度データを収集しない
2. **テナント分離**: DynamoDB PK プレフィックスにより家族間のデータ分離を保証
3. **暗号化**: 通信（TLS 1.2+）および保存データ（DynamoDB/S3 暗号化）の両方で保護
4. **権利保障**: アクセス・訂正・削除・ポータビリティの全権利に対応済み
5. **同意管理**: バージョン管理付きの同意取得・再同意フローを実装済み
6. **削除の完全性**: 30日猶予期間後にDB・S3・Cognitoから完全削除

### 7.2 残留リスクと受容

| 残留リスク | レベル | 受容理由 |
|-----------|--------|---------|
| バックアップ内の残存データ（最大30日） | 低 | S3ライフサイクルで自動削除。30日は合理的な保持期間 |
| Lambda インスタンス間のレートリミッター非共有 | 低 | 現在のトラフィック規模では実質的な影響なし |
| us-east-1 リージョンでのデータ保存 | 低 | AWS DPA + SCC で法的根拠を確保。EU展開時に eu-west-1 への移行を検討 |

### 7.3 今後のアクション

| アクション | 優先度 | 時期 |
|-----------|--------|------|
| DPA 締結状況の定期確認（年1回） | 中 | 年次 |
| EU ユーザー増加時の eu-west-1 リージョン移行検討 | 低 | EU展開時 |
| CloudTrail 有効化（管理者操作の監査） | 中 | 運用フェーズ |
