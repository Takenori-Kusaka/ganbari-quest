## 1. 設計背景

### なぜこの設計書が必要か

本設計書は、「がんばりクエスト」ゲーミフィケーション機構の **Single Source of Truth** として機能する。

設計書が必要になった背景は、2026-04-21 の B6 ブレスト（Issue #1328）で以下の構造的問題が顕在化したことにある:

| 問題 | 詳細 |
|------|------|
| 並列フィードバック系の独立成長 | おみくじ・レベル・称号・実績・バトルが明示的な設計意図なく独立成長し、ポイント経済が歪む状態が続いていた |
| 設計意図の不在による drift | 明示的な設計原則がなかったため、LP に「シールガチャ」文言が混入（#1307 B9）するなど、実装と説明が乖離した |
| ポイント経済の未定義 | L1/L2/L3 3 層モデルが未定義のため、各機構が「ポイント供給か表現層か」の判断基準がなかった |

本設計書は以上の問題を解消するため、設計原則（why）と仕様（what）を 3 部構成で明文化する。

### 企画書 01 との関係

本設計書は `docs/design/01-企画書.md` §2 の設計原則 7〜10 を **ゲーミフィケーション専門設計書として展開** したものである。

|-------------|--------------|
| 原則 7: 親子ポイント経済モデル | §2.2 親子ポイント経済モデル要約 |

## 2. 設計原則

### 2.1 基本原則

1. **活動記録の動機づけ以外は有害** — コレクション閲覧・着せ替え等のアプリ内滞在目的機能は禁止（ADR-0012）
2. **親が褒める仕組み** — アプリはきっかけを作るだけ。最終的な報酬は親からの言葉。アプリが主役になってはならない
3. **ポジティブトーンのみ** — 罰則・減点・ネガティブフィードバックは一切設けない
4. **年齢に応じた体験** — 5 つの年齢モード（baby / preschool / elementary / junior / senior）で UI とメッセージを最適化する（ADR-0015）
5. **L2/L3 は L1 を補強するのみ** — おみくじ・バトル等は「活動記録した結果として発生する体験」。記録なしに得られるゲーム体験は設けない
6. **Variable-ratio 適正活用** — おみくじスタンプの確率変動はログイン連動（日 1 回 cap）に限定する。連続ガチャ・無制限スタンプは Anti-engagement 原則違反

> **good/bad 例**:
>
> bad: 「スタンプ獲得回数を無制限にして子供がもっとアプリを開くようにする」（L2 が目的化しており Anti-engagement 原則違反）
>
> good: 「ログイン（= 活動記録開始）に紐づいた日 1 回のおみくじスタンプで、記録行動のきっかけを作る」

### 2.2 親子ポイント経済モデル要約

ポイント経済は **親（供給側）と子（需要側）の二者間市場** として設計する。

| 側面 | 親の役割（供給側） | 子の役割（需要側） |
|------|----------------|----------------|
| 活動 | 「やってほしい活動」をプリセット登録（活動→ポイント レート設定） | プリセットから選んで実行（能動的選択） |
| 評価 | 定量化された努力量を見る | 自分の蓄積を見て達成感 |
| ごほうび | 「渡してもよいごほうび」をショップに陳列（実物・お小遣い・特権） | 欲しいごほうびを「あと X pt」と自分の動機源にする |

**差別化核心**: 親は褒める必要がない（定量化が代替）、子は言うことを聞く必要がない（自分の欲しい報酬が待つ）。ショップが**親の「許可した欲望」と子の「本物の欲望」の交点**。

詳細は `docs/design/01-企画書.md` §2.7 を参照。

### 2.3 コアループ 3 層モデル

| 層 | 名称 | 強化対象 | 代表機能 | 出力 |
|----|------|---------|---------|------|
| **L1** | 活動層 | 個別活動完了という日常行動 | 活動記録（2タップ） | ポイント付与 |
| **L2** | メタ習慣層 | 「毎日継続して入力する」習慣 | スタンプカード（rarity 変動） | ポイント付与（変動） |
| **L3** | 経済層 | 累積ポイント → 実ごほうびへの交換 | ごほうびショップ | 実物/お小遣い/特権 |

各層は独立しており、L2/L3 は L1 の記録行動を補強する役割のみを持つ（L2/L3 自体が目的化することは Anti-engagement 原則違反）。

### 2.4 ポイント統一原則

- **唯一の通貨**: ポイントのみ（独自通貨・スタンプコイン等は設けない）
- **唯一の出口**: ごほうびショップのみ（他の消費経路を設けない）
- **レベル・称号・実績 = 表現のみ**: ポイント累積の視覚化であり、独立した報酬を付与しない
- **スタンプカード redeem の出力もポイント**: 例外なし

### 2.5 Anti-engagement 境界（ADR-0012 参照）

| 許容 | 禁止 |
|------|------|
| 日 1 回 cap のおみくじスタンプ | 連続ガチャ・無制限スタンプ |
| 活動記録後のバトル 1 回 | バトル連打・自動再生バトル |
| L2 によるメタ習慣形成 | L2 の目的化（スタンプ集め自体が目的になる設計） |
| レベルアップ演出（5秒・自動閉じ） | インフィニットスクロール・通知連打 |

### 2.6 Anti-habituation 原則

スタンプカードのレア度分散（N/R/SR/UR）は **anti-habituation（習慣化への飽き対策）** として設計される。

**設計意図**:
- 同じレアリティのスタンプが連続すると子供が飽きる（habituation）
- レア度の変動によって毎回のスタンプが「どのレアリティか」という小さなサプライズを生み出す
- このサプライズは **variable-ratio reinforcement** として機能し、習慣形成を助ける

**射幸心訴求との違い（retrofit 禁止）**:
- 本機構の目的は「アプリ滞在時間の延長」ではなく「継続記録習慣の形成支援」
- UR（3%）が出なかったことによる不満・再挑戦欲求を意図的に生む設計は禁止
- 1 日 1 回の cap が本原則のガード機構として機能する

詳細は `docs/rationale/02-core-loop-rationale.md` §採用案とその理由 を参照。

## 4. 並列フィードバック系の位置づけ

本章は §2.4 ポイント統一原則の具体的な適用表。新機能追加時は本表に位置づけを必ず明記すること。

| 機構 | 分類 | 役割 | 出力 |
|---|---|---|---|
| 活動記録 | L1 | ポイント取得主軸 | ポイント |
| スタンプカード | L2 | メタ習慣強化 + anti-habituation | ポイント (rarity 変動) |
| おみくじ | L1 副次 | デイリー動機付け | ポイント |
| コンボ・ストリーク | L1 修飾 | レート倍率 | ポイント倍率 |
| バトル | L1 副次 | RPG 演出 + 副次ポイント | ポイント (10-80pt) |
| レベル | 表現 | ポイント累積の視覚化 | 称号変化 (独立報酬なし) |
| 称号・実績 | 表現 | ポイント履歴の装飾 | 表示のみ (独立報酬なし) |
| ごほうびショップ | L3 | 唯一の出口 | 実物/お金/特権 |

**運用ルール**:
- 「表現」に分類される機構に独立したポイント供給を追加することは §2.4 ポイント統一原則違反
- 新規機構は本表に追加してから実装に着手すること（ADR-0003 準拠）

## 5. ポイントシステム

### 5.1 ポイント獲得

| 項目 | 仕様 |
|------|------|
| 基本ポイント | 活動定義ごとに設定（1〜100） |
| ストリークボーナス | 連続記録日数に応じた倍率（下記参照） |
| コンボボーナス | 同日複数カテゴリ記録で倍率加算 |
| ログインボーナス | 日次初回アクセス時に付与 |

### 5.2 ストリーク倍率

| 連続日数 | 倍率 |
|---------|------|
| 1-2日 | ×1.0 |
| 3-6日 | ×1.2 |
| 7-13日 | ×1.5 |
| 14-29日 | ×1.8 |
| 30日以上 | ×2.0 |

### 5.3 コンボシステム

同日中に異なるカテゴリの活動を記録するとコンボ倍率が加算される。

| コンボ数 | 追加倍率 |
|---------|---------|
| 2カテゴリ | +0.1 |
| 3カテゴリ | +0.2 |
| 4カテゴリ | +0.3 |
| 5カテゴリ（全制覇） | +0.5 |

### 5.4 ポイント減衰

長期間記録がない場合にポイントが緩やかに減衰する（設定で ON/OFF 可能）。

| 未記録期間 | 減衰率 |
|-----------|--------|
| 1-6日 | なし |
| 7-13日 | -1%/日 |
| 14日以上 | -2%/日（下限: 最大ポイントの50%） |

## 7. スタンプカードシステム

スタンプカードは **L2 メタ習慣層** の実装機構（§2.3 / §4 参照）。

### 7.1 概要

活動記録のたびにスタンプを1枚獲得し、カードに貼る。
カードが完成するとポイントが付与され、新しいカードが発行される。

### 7.2 ログインおみくじスタンプ（L2 メタ習慣層）

**ADR-0012 Anti-engagement 原則 / ADR-0013 LP truth 準拠**: ログイン連動の 1 日 1 回おみくじ。  
「ガチャ」という呼称は射幸性連想があるため廃止し、**ログインおみくじスタンプ** に統一する（#1311, #1313）。

#### 仕組み

- ログイン（日 1 回 cap）で `loginStamp` action が呼ばれ、ランダムにレアリティ・スタンプが選択される
- Variable-ratio reinforcement によって **飽きにくく**、anti-habituation 原則に基づく（§2.6 参照）
- 1 回の interaction は数秒で完了（Anti-engagement 原則を遵守）
- 実装: `src/lib/server/services/stamp-card-service.ts` → `stampToday()` (日 1 回 cap: `ALREADY_STAMPED` error)

| レアリティ | 出現率 | スロットの色 |
|-----------|--------|------------|
| N (Normal) | 60% | シルバー |
| R (Rare) | 25% | ゴールド |
| SR (Super Rare) | 12% | レインボー |
| UR (Ultra Rare) | 3% | ホログラム |

```typescript
// src/lib/server/services/stamp-card-service.ts
// loginStamp action から呼ばれる (1 日 1 回 cap)
async function stampToday(tenantId, childId): Promise<StampResult | 'ALREADY_STAMPED'>
```

## 12. ごほうびショップ（L3 経済層）

## 15. 年齢・誕生日設計

> **#1379 Sub B-1**: 実装から仕様を復元 (ADR-0001 準拠)。Committed（実装済み）と Aspirational（未実装）を明確に区別する (ADR-0013)。

### 15.1 データモデル

| カラム | 型 | 役割 |
|--------|-----|------|
| `child.birthDate` | `TEXT` (YYYY-MM-DD), nullable | **誕生日 SSOT**。入力された場合は年齢の唯一の真実の源 |
| `child.age` | `INTEGER` | 年齢キャッシュ。`birthDate` がある場合は `calculateAge(birthDate)` で計算。ない場合はユーザー直接入力値 |
| `child.uiMode` | `TEXT` ('baby'\|'preschool'\|'elementary'\|'junior'\|'senior') | 年齢 UI モード。`getDefaultUiMode(age)` から導出。誕生日ボーナス claim 時に自動更新 |
| `child.uiModeManuallySet` | `INTEGER` (0 or 1), default 0 | **Committed (#1382, PR #1463 実装済み)**。親が uiMode を明示的に選択したフラグ。`1` の場合、age 変更時の `uiMode` 自動再計算をスキップする |

### 15.2 年齢から uiMode へのマッピング

`getDefaultUiMode(age: number)` (`src/lib/domain/validation/age-tier.ts`):

| 年齢 | uiMode | 日本語名 |
|------|--------|---------|
| 0 〜 2 | `baby` | 乳幼児 |
| 3 〜 5 | `preschool` | 幼児 |
| 6 〜 12 | `elementary` | 小学生 |
| 13 〜 15 | `junior` | 中学生 |
| 16 以上 | `senior` | 高校生 |

### 15.3 状態遷移（Committed / 現行実装）

#### 子供登録時 (`/admin/children` POST ?/create)

```
birthDate 入力あり:
  child.birthDate = birthDate
  child.age       = calculateAge(birthDate)   ← (0 〜 18 クランプ)
  child.uiMode    = getDefaultUiMode(age)

birthDate 入力なし (年齢のみ):
  child.birthDate = null
  child.age       = Number(ageStr)            ← 0 〜 18 の範囲でバリデーション
  child.uiMode    = getDefaultUiMode(age)
```

#### 子供更新時 (`/admin/children` POST ?/update を経由した `child-service.editChild()`)

```
uiMode が明示指定された場合:
  child.uiMode            = 指定値
  child.uiModeManuallySet = 1          ← 手動設定フラグを立てる

age が変更された場合 (uiMode 未指定):
  child.age = Number(ageStr)
  child.uiMode = recalcUiMode(existing, newAge):
    uiModeManuallySet = 1 → 既存 uiMode を維持 (保護者 overrides 尊重)
    uiModeManuallySet = 0 → getDefaultUiMode(newAge) で自動再計算

birthDate が変更された場合:
  child.birthDate = birthDate (または null)
  birthDate あり → child.age = calculateAge(birthDate) → uiMode も recalcUiMode で更新
  birthDate なし → child.age = Number(ageStr) (入力があれば) → uiMode も recalcUiMode で更新
```

#### 誕生日ボーナス claim 時 (`birthday-bonus-service.claimBirthdayBonus()`)

```
claimBirthdayBonus():
  child.age                   = (更新しない — age-recalc cron (#1381) に移譲, #1522)
  child.uiMode                = getDefaultUiMode(newAge)      ← uiModeManuallySet に関わらず常に上書き
                                                                 (ポリシー: 誕生日は節目、保護者が必要なら再設定)
  child.lastBirthdayBonusYear = currentYear                   ← 重複 claim 防止
  PointEntry 追加             = age × 100 × birthdayBonusMultiplier
```

### 15.4 SSOT 優先度表

| 情報 | Primary SSOT | Derived From |
|------|-------------|--------------|
| 誕生日 | `child.birthDate` | — |
| 年齢 | `child.age` (ただし birthDate がある場合は `calculateAge(birthDate)` で上書き) | `calculateAge(birthDate)` |
| uiMode | `child.uiMode` (ただし `!uiModeManuallySet` なら age 変化時に再計算) | `getDefaultUiMode(age)` |

### 15.5 フォーム UX バリデーション（Committed）

- `birthDate` は YYYY-MM-DD 形式のみ許可
- `birthDate` が未来日の場合はエラー
- `age` は 0 〜 18 の整数のみ許可
- `birthDate` と `age` は**どちらか片方の入力が必須**（#1380, PR #1462 で実装済み）
- 両方入力した場合は `birthDate` 優先（`age` は `calculateAge(birthDate)` で上書き）
- `birthDate` 入力時は `age` 欄を disabled にして自動計算値を表示

### 15.6 誕生日ボーナスの演出（Committed）

| 要素 | 仕様 |
|------|------|
| バナー | 誕生日から 3 日以内にホーム画面上部に表示 |
| モーダル | 初回アクセス時に特別メッセージ |
| 請求期間 | 誕生日当日〜2 日後（`CLAIM_WINDOW_DAYS = 3`） |
| ボーナス | `age × 100 × birthdayBonusMultiplier` ポイント |
| 重複防止 | `lastBirthdayBonusYear` で同一年の二重 claim を防止 |

コンポーネント: `BirthdayBanner.svelte`, `BirthdayModal.svelte`
サービス: `birthday-bonus-service.ts`

### 15.7 年齢自動インクリメント（Committed / #1381）

毎日 00:00 JST に `/api/cron/age-recalc` を実行:

- 全テナントの全 child を `listAllTenants → findAllChildren` の 2 段階走査で取得
- `birthDate` を持つ child について `calculateAge(birthDate, today)` を計算
- 計算値が `child.age` と異なる場合のみ更新:
  - `age` を新しい年齢に更新
  - `uiModeManuallySet=false` の場合: `recalcUiMode()` で `uiMode` も再計算
  - `uiModeManuallySet=true` の場合: `uiMode` は変更しない（保護者の手動設定を維持）
- 同日 2 回実行しても 2 回目は `updated=0`（冪等性保証）
- `dryRun=true` で実際の DB 更新なしに更新件数を確認可能

スケジュール: `schedule-registry.ts` に登録済み（`utcCronExpression: cron(0 15 * * ? *)`）

### 15.8 実装ファイル参照

| ファイル | 役割 |
|---------|------|
| `src/lib/domain/validation/age-tier.ts` | `getDefaultUiMode()` / `recalcUiMode()` / `UiMode` 型 |
| `src/lib/server/services/child-service.ts` | `editChild()` — `uiModeManuallySet` を考慮した `recalcUiMode()` 呼出し |
| `src/lib/server/services/birthday-bonus-service.ts` | `calculateAge()` / `claimBirthdayBonus()` — uiMode 更新（age 更新は age-recalc cron に移譲, #1522） |
| `src/lib/server/services/age-recalc-service.ts` | `recalcAllChildrenAges()` — 年齢自動インクリメント cron サービス (#1381) |
| `src/routes/api/cron/age-recalc/+server.ts` | cron エンドポイント — verifyCronAuth 認証 + dryRun 対応 (#1381) |

### 15.9 birthday-bonus-service との責務分担

`birthday-bonus-service.claimBirthdayBonus()` と `child-service.editChild()` は uiMode 更新ロジックをそれぞれ持つ。以下のルールで責務を分離する。

| 操作 | 担当サービス | age 更新 | uiModeManuallySet の考慮 |
|------|------------|---------|------------------------|
| 保護者が管理画面で age / uiMode を編集 | `child-service.editChild()` | あり | 考慮する。`recalcUiMode()` 経由 |
| 誕生日ボーナス claim (子供画面 / cron) | `birthday-bonus-service.claimBirthdayBonus()` | **なし**（#1522 で削除。age 更新は age-recalc cron に移譲） | **考慮しない**。誕生日は成長の節目のため `getDefaultUiMode()` で常に上書き |
| 年齢自動インクリメント cron (#1381) | `age-recalc-service.recalcAllChildrenAges()` | あり | **考慮する**。`recalcUiMode()` を使用。保護者の手動設定を維持 |

**ポリシー根拠**: 誕生日ボーナス claim 時は成長の節目であるため、uiMode を自動追従させることを優先する（`getDefaultUiMode()` で常に上書き）。age 自体の更新は毎日 00:00 JST の `age-recalc` cron（#1381）が担うため、`claimBirthdayBonus()` 内での age 更新は重複となる。#1522 にて削除済み。保護者が uiMode を特定年齢帯に固定したい場合は、ボーナス受取後に管理画面から再設定する。

## 18. バトルアドベンチャー（#605）

バトルは **L1 副次** として §4 並列フィードバック系で位置づけられる。活動記録の結果として発生する体験であり、単独で起動できる遊びではない。

### 18.1 概要

日々の活動がRPGステータスに反映され、毎日自動マッチングされた敵とターン制バトルを行う機能。
活動カテゴリごとのXPがRPGステータスに変換され、活動量がそのまま戦闘力に直結する設計。

| 項目 | 値 |
|------|-----|
| バトル頻度 | 1日1回（毎日新しい敵がマッチング） |
| バトル方式 | ターン制・自動進行 |
| 勝敗 | 勝利/敗北（ペナルティなし、慰めポイントあり） |
| 対象年齢モード | elementary / junior / senior のみ（baby/preschool はバトル機能なし） |

### 18.2 カテゴリ → RPGステータス変換

活動カテゴリのXPがそのままRPGステータスに変換される。

| カテゴリ | RPGステータス | 意味 |
|---------|------------|------|
| うんどう | HP（たいりょく） | 体力 |
| べんきょう | ATK（こうげき） | 攻撃力 |
| せいかつ | SPD（すばやさ） | 素早さ |
| こうりゅう | DEF（ぼうぎょ） | 防御力 |
| そうぞう | REC（かいふく） | 回復力 |

**変換式**: `stat = BASE + floor(xp^0.6)`

| ベースステータス | 値 |
|--------------|-----|
| HP | 50 |
| ATK | 10 |
| DEF | 8 |
| SPD | 10 |
| REC | 5 |

ソース: `src/lib/domain/battle-stat-calculator.ts`

### 18.3 敵マスタデータ

10体以上の敵をレアリティ別に定義。曜日出現制限あり。

| レアリティ | 出現率 | 勝利ポイント | 慰めポイント | 代表例 |
|-----------|--------|------------|-----------|--------|
| common | 高 | 10 | 3 | スライム、コウモリ、キノコおばけ |
| uncommon | 中 | 20 | 5 | ゴブリン、おおカエル、ゴーレム |
| rare | 低 | 40 | 10 | ドラゴンパピー |
| boss | 週末限定 | 80 | 20 | まおうのかげ |

ソース: `src/lib/domain/battle-enemies.ts`

### 18.4 年齢スケーリング

敵ステータスは年齢モードに応じてスケーリングされる。

| 年齢モード | 係数 | 備考 |
|-----------|------|------|
| baby | — | バトル機能なし（親の準備モード） |
| preschool | — | バトル機能なし |
| elementary | ×0.8 | バトル開始 |
| junior | ×1.0 | 基準値 |
| senior | ×1.2 | 高難度 |

ソース: `src/lib/domain/battle-stat-calculator.ts` の `getAgeScaling()`

### 18.5 バトルエンジン

ターン制の自動バトル。先攻はSPDで決定。

```
ターン開始 → 先攻判定(SPD) → 攻撃(ATK - DEF/2) → HP判定
  → 相手ターン → 攻撃 → HP判定 → 回復(REC/2) → ループ
  → HPが0以下で勝敗確定
```

| 項目 | 仕様 |
|------|------|
| 最大ターン数 | 20（超過で引き分け＝勝利扱い） |
| ダメージ計算 | `max(1, atk - def / 2)` |
| 回復 | 毎ターン `rec / 2` 回復（HP上限を超えない） |
| クリティカル | なし（子供向けにシンプル） |

ソース: `src/lib/domain/battle-engine.ts`

### 18.6 データ構造

テーブル: `daily_battles`

| reward_points | INTEGER | 獲得ポイント |
| created_at | TEXT | 作成日時 |

UNIQUE制約: `(child_id, battle_date)` — 1子供1日1バトル
