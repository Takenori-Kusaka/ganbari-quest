## 1. 設計背景

### なぜこの設計書が必要か

本設計書は、「がんばりクエスト」ゲーミフィケーション機構の **Single Source of Truth** として機能する。

設計書が必要になった背景は、2026-04-21 の B6 ブレスト（Issue #1328）で以下の構造的問題が顕在化したことにある:

| 問題 | 詳細 |
|------|------|
| 並列フィードバック系の独立成長 | おみくじ・レベル・称号・実績・バトルが明示的な設計意図なく独立成長し、ポイント経済が歪む状態が続いていた |
| 設計意図の不在による drift | 明示的な設計原則がなかったため、LP に「シールガチャ」文言が混入（#1307 B9）するなど、実装と説明が乖離した |
| ポイント経済の未定義 | L1/L2/L3 3 層モデルが未定義のため、各機構が「ポイント供給か表現層か」の判断基準がなかった |

本設計書は以上の問題を解消するため、設計原則（why）と仕様（what）を 3 部構成で明文化する。

### 企画書 01 との関係

本設計書は `docs/design/01-企画書.md` §2 の設計原則 7〜10 を **ゲーミフィケーション専門設計書として展開** したものである。

|-------------|--------------|

## 2. 設計原則

### 2.1 基本原則

1. **活動記録の動機づけ以外は有害** — コレクション閲覧・着せ替え等のアプリ内滞在目的機能は禁止（ADR-0012）
2. **親が褒める仕組み** — アプリはきっかけを作るだけ。最終的な報酬は親からの言葉。アプリが主役になってはならない
3. **ポジティブトーンのみ** — 罰則・減点・ネガティブフィードバックは一切設けない
4. **年齢に応じた体験** — 5 つの年齢モード（baby / preschool / elementary / junior / senior）で UI とメッセージを最適化する（ADR-0015）
5. **L2/L3 は L1 を補強するのみ** — おみくじ・バトル等は「活動記録した結果として発生する体験」。記録なしに得られるゲーム体験は設けない
6. **Variable-ratio 適正活用** — おみくじスタンプの確率変動はログイン連動（日 1 回 cap）に限定する。連続ガチャ・無制限スタンプは Anti-engagement 原則違反

> **good/bad 例**:
>
> bad: 「スタンプ獲得回数を無制限にして子供がもっとアプリを開くようにする」（L2 が目的化しており Anti-engagement 原則違反）
>
> good: 「ログイン（= 活動記録開始）に紐づいた日 1 回のおみくじスタンプで、記録行動のきっかけを作る」

### 2.2 親子ポイント経済モデル要約

ポイント経済は **親（供給側）と子（需要側）の二者間市場** として設計する。

| 側面 | 親の役割（供給側） | 子の役割（需要側） |
|------|----------------|----------------|
| 活動 | 「やってほしい活動」をプリセット登録（活動→ポイント レート設定）。マーケットプレイスの活動パック（12 件）から選択可能で、各活動には `mustDefault` フラグが付与され、import 時に「今日のおやくそく推奨を採用」を ON にすると `priority='must'` で展開される（#1755 / #1758） | プリセットから選んで実行（能動的選択）。「今日のおやくそく」は子供画面で 0/N バーで進捗表示される |
| 評価 | 定量化された努力量を見る | 自分の蓄積を見て達成感 |
| ごほうび | 「渡してもよいごほうび」をショップに陳列（実物・お小遣い・特権） | 欲しいごほうびを「あと X pt」と自分の動機源にする |

**差別化核心**: 親は褒める必要がない（定量化が代替）、子は言うことを聞く必要がない（自分の欲しい報酬が待つ）。ショップが**親の「許可した欲望」と子の「本物の欲望」の交点**。

詳細は `docs/design/01-企画書.md` §2.7 を参照。

### 2.3 コアループ 3 層モデル

| 層 | 名称 | 強化対象 | 代表機能 | 出力 |
|----|------|---------|---------|------|
| **L1** | 活動層 | 個別活動完了という日常行動 | 活動記録（2タップ） | ポイント付与 |
| **L2** | メタ習慣層 | 「毎日継続して入力する」習慣 | スタンプカード（rarity 変動） | ポイント付与（変動） |
| **L3** | 経済層 | 累積ポイント → 実ごほうびへの交換 | ごほうびショップ | 実物/お小遣い/特権 |

各層は独立しており、L2/L3 は L1 の記録行動を補強する役割のみを持つ（L2/L3 自体が目的化することは Anti-engagement 原則違反）。

### 2.4 ポイント統一原則

- **唯一の通貨**: ポイントのみ（独自通貨・スタンプコイン等は設けない）
- **唯一の出口**: ごほうびショップのみ（他の消費経路を設けない）
- **レベル・称号・実績 = 表現のみ**: ポイント累積の視覚化であり、独立した報酬を付与しない
- **スタンプカード redeem の出力もポイント**: 例外なし

### 2.5 Anti-engagement 境界（ADR-0012 参照）

| 許容 | 禁止 |
|------|------|
| 日 1 回 cap のおみくじスタンプ | 連続ガチャ・無制限スタンプ |
| 活動記録後のバトル 1 回 | バトル連打・自動再生バトル |
| L2 によるメタ習慣形成 | L2 の目的化（スタンプ集め自体が目的になる設計） |
| レベルアップ演出（5秒・自動閉じ） | インフィニットスクロール・通知連打 |

### 2.6 Anti-habituation 原則

スタンプカードのレア度分散（N/R/SR/UR）は **anti-habituation（習慣化への飽き対策）** として設計される。

**設計意図**:
- 同じレアリティのスタンプが連続すると子供が飽きる（habituation）
- レア度の変動によって毎回のスタンプが「どのレアリティか」という小さなサプライズを生み出す
- このサプライズは **variable-ratio reinforcement** として機能し、習慣形成を助ける

**射幸心訴求との違い（retrofit 禁止）**:
- 本機構の目的は「アプリ滞在時間の延長」ではなく「継続記録習慣の形成支援」
- UR（3%）が出なかったことによる不満・再挑戦欲求を意図的に生む設計は禁止
- 1 日 1 回の cap が本原則のガード機構として機能する

詳細は `docs/rationale/02-core-loop-rationale.md` §採用案とその理由 を参照。

## 4. 並列フィードバック系の位置づけ

本章は §2.4 ポイント統一原則の具体的な適用表。新機能追加時は本表に位置づけを必ず明記すること。

| 機構 | 分類 | 役割 | 出力 |
|---|---|---|---|
| 活動記録 | L1 | ポイント取得主軸 | ポイント |
| スタンプカード | L2 | メタ習慣強化 + anti-habituation | ポイント (rarity 変動) |
| おみくじ | L1 副次 | デイリー動機付け | ポイント |
| コンボ・ストリーク | L1 修飾 | レート倍率 | ポイント倍率 |
| バトル | L1 副次 | RPG 演出 + 副次ポイント | ポイント (10-80pt) |
| レベル | 表現 | ポイント累積の視覚化 | 称号変化 (独立報酬なし) |
| 称号・実績 | 表現 | ポイント履歴の装飾 | 表示のみ (独立報酬なし) |
| ごほうびショップ | L3 | 唯一の出口 | 実物/お金/特権 |

**運用ルール**:
- 「表現」に分類される機構に独立したポイント供給を追加することは §2.4 ポイント統一原則違反
- 新規機構は本表に追加してから実装に着手すること（ADR-0003 準拠）

## 4a. スタンプカード — L2 メタ習慣層の詳細設計（#1338）

### 4a.1 L2 メタ習慣層としての位置づけ

スタンプカードは **L2 メタ習慣層**（§2.3）の実装機構であり、L1（個別活動完了）とは異なる行動を強化する。

| 層 | 強化する行動 | 代替可否 |
|----|-----------|---------|
| **L1** | 個別活動（「今日のうんどうを記録する」） | — |
| **L2 スタンプカード** | 継続習慣（「毎日アプリを開いて入力する」） | L1 で代替不可 |

**L2 を L1 で代替不可な理由**: L1 は「何かを記録した」という完了行動に対する報酬だが、L2 は「継続して来た」という習慣パターンそのものを強化する。1 週間の連続入力という行動は、個別の L1 記録の積み上げとは質的に異なる価値を持つ。

### 4a.2 Anti-engagement 境界（ADR-0012 準拠）

スタンプカードの **1 日 1 タップ離脱** が Anti-engagement 原則（ADR-0012）との境界ガード機構として機能する。

| 操作 | 説明 |
|------|------|
| **スタンプ押印** | 1 日 1 回のみ（`stampToday()` の `ALREADY_STAMPED` ガード）。数秒で完了 |
| **カード完成確認** | 週 1 回の redeem。アプリ内滞在のための仕掛けではない |

スタンプを目当てに「もっとアプリを開く」設計（無制限スタンプ・連続試行）は Anti-engagement 原則違反である。1 日 1 回の cap がこの境界を保証する。

### 4a.3 Anti-habituation 設計意図（#1338）

**目的**: rarity 変動（N/R/SR/UR）は **anti-habituation（習慣化への飽き対策）** として設計されている。

**variable-ratio reinforcement の仕組み**:
- 同じ結果が毎回確定すると子供が飽きる（habituation）
- rarity 変動によって毎回「どのレアリティか」という小さな予測不確実性が生まれる
- これは variable-ratio 強化スケジュールとして機能し、継続記録習慣の形成を助ける

**射幸心訴求との違い**:

| variable-ratio reinforcement（本機構） | 射幸心訴求 |
|--------------------------------------|----------|
| 目的: 継続記録習慣の形成支援 | 目的: アプリ滞在時間の延長・課金誘導 |
| 1 日 1 回の cap で再試行を禁止 | 繰り返し試行を誘導する設計 |
| UR が出なかったことによる再挑戦欲求を意図的に生まない | 確率への不満を再挑戦欲求に転換する |
| 無料・追加操作不要 | 課金・追加操作を誘う |

**LP・UI での禁止表現**: 「ガチャ」「抽選」「引く」等の語彙は使用禁止（#1311, #1313, `scripts/measure-lp-dimensions.mjs` `FORBIDDEN_TERMS`）。

### 4a.4 実装 SSOT と rarity 分布

| 要素 | 実装場所 |
|------|---------|
| rarity 確率・ポイント倍率 | `src/lib/server/services/stamp-card-service.ts` — `RARITY_WEIGHTS`, `RARITY_MULTIPLIER` |
| 押印ロジック・1 日 1 回 cap | 同ファイル `stampToday()` |
| スタンプマスタ | `stamp_masters` テーブル / §7.4 |

**実装上の rarity 分布**（`RARITY_WEIGHTS` 定数より — 実装が SSOT）:

| レアリティ | 出現率 | 即時押印ポイント | 週間 redeem 寄与 |
|-----------|--------|----------------|----------------|
| N (Normal) | 60% | 5pt (×1) | 固定 10pt/枚 |
| R (Rare) | 25% | 10pt (×2) | 固定 10pt/枚 |
| SR (Super Rare) | 12% | 20pt (×4) | 固定 10pt/枚 |
| UR (Ultra Rare) | 3% | 40pt (×8) | 固定 10pt/枚 |

**ポイント付与の二段構成**:
1. **即時押印ポイント**: スタンプ押印時に rarity に応じた即時ポイントを付与（`stamp_instant` type、`point_ledger` に記録）
2. **週間 redeem ポイント**: カード完成時にスタンプ数 × 10pt + 5 枚コンプリートボーナス 50pt（rarity 無関係の固定値、`stamp_card` type）

> **注**: rarity 分布を変更する場合は本表と §7.2 を同時に更新すること。`RARITY_WEIGHTS` 定数が SSOT。

## 4b. レベル・称号・実績 — 表現層位置づけの明文化（#1341）

### 4b.1 表現層位置づけの確定

§2.4 ポイント統一原則・§4 分類表を実装で確認し、以下を明文化する。

| 機構 | 現状実装 | 独立ポイント付与 | 根拠 |
|------|---------|--------------|------|
| **レベル** | `calcLevelFromXp(totalXp)` でポイント累積を視覚化 | **なし** | `status-service.ts` — 計算のみ。`insertPointEntry` 呼び出しなし |
| **称号** | `levelTitle` フィールドとして表示するのみ | **なし** | `status.ts` `LEVEL_TABLE` の `title` 文字列。ポイント付与コードなし |
| **実績** | `unlockCustomAchievement()` で解除フラグを立てるのみ | **なし** | `custom-achievement-service.ts` — unlock 時に `insertPointEntry` 呼び出しなし |

### 4b.2 実装整合の確認

レベルアップしてもポイントは付与されない。具体的な実装根拠:

- `src/lib/domain/validation/status.ts`: `calcLevelFromXp()` は XP → レベル変換のみ。ポイント付与なし
- `src/lib/server/services/status-service.ts`: `getChildStatus()` はレベル・称号情報を返すのみ。`insertPointEntry` 呼び出しなし
- `src/lib/server/services/custom-achievement-service.ts`: `unlockCustomAchievement()` はフラグ更新のみ。ポイント付与なし

これは §2.4 ポイント統一原則の設計意図どおりである。

### 4b.3 将来変更に関するポリシー

「レベルアップ報酬（ポイント付与）」を追加したい場合は:
1. 別 Issue を立てる（本設計書を改訂する前に PO の判断が必要）
2. §2.4 ポイント統一原則との整合を確認する ADR を起票する
3. 本節 §4b と §6・§13 を同時に更新する

> **実装者へ**: 「レベルアップに感動を出したい」という動機でポイント付与を追加することは §2.4 ポイント統一原則違反となる。演出は `LevelUpOverlay.svelte` の視覚エフェクトで完結させること。

## 4c. バトル報酬 — ポイント統一原則との整合確認（#1342）

### 4c.1 バトルのポイント統一原則との整合

バトルは **L1 副次**（§4）として定義され、活動記録の結果として発生する体験である。

**ポイント統一原則との整合確認**:

| 確認項目 | 現状実装 | 整合性 |
|---------|---------|------|
| バトル報酬はポイントのみ | `enemy.dropPoints` (勝利) / `enemy.consolationPoints` (敗北) | 整合 |
| 独立した別通貨の発行なし | `daily_battles.reward_points` に記録 | 整合 |
| 追加の独立報酬（スタンプ・バッジ等）なし | `executeDailyBattle()` — ポイント計算のみ | 整合 |

**「RPG 演出 + 副次ポイント」の役割**:
- **RPG 演出**: 活動カテゴリの XP がバトルステータス（HP/ATK/DEF/SPD/REC）に変換され、活動量がそのまま戦闘力に直結するフィードバックを提供
- **副次ポイント**: バトル結果に応じた追加ポイント（勝利: 10-80pt、慰め: 3-15pt）が付与される

### 4c.2 バトル報酬ポイント範囲（実装値）

報酬ポイントは `src/lib/domain/battle-enemies.ts` の `dropPoints` / `consolationPoints` で定義される。

| 敵レアリティ | 勝利ポイント | 慰めポイント |
|-----------|-----------|-----------|
| common | 10 | 3 |
| uncommon | 20 | 5 |
| rare | 40 | 10 |
| boss（週末限定） | 80 | 15 |


### 4c.3 #1320 B4+5（年齢別解放方針）との整合

バトルは elementary / junior / senior のみで利用可能（§18.1）。年齢別解放方針との整合:

| 年齢モード | バトル利用可否 | 根拠 |
|-----------|------------|------|
| baby | 不可（404） | 親の準備モード。ゲーミフィケーション非適用（ADR-0011） |
| preschool | 不可（404） | 幼児向け。複雑なバトル演出は Anti-engagement 原則違反リスク |
| elementary | 可（×0.8 スケーリング） | バトル開始。易しい難度でゲーミフィケーション体験 |
| junior | 可（×1.0 スケーリング） | 標準難度 |
| senior | 可（×1.2 スケーリング） | 高難度 |

## 4d. おみくじ — L1 副次トリガーの位置づけ明文化（#1340）

### 4d.1 おみくじの位置づけ確認

おみくじは §4 並列フィードバック系表において **L1 副次 / デイリー動機付け / ポイント出力** と定義される。本節はこの位置づけを実装調査で確認し、逸脱の有無を明記する。

**位置づけの定義**:
- **L1 副次**: 活動記録（L1 コアループ）に連動してトリガーされる。単独では起動しない
- **デイリー動機付け**: 「今日もアプリを開こう」というログイン習慣の補助。スタンプカード（L2）と異なり、連続習慣そのものは強化しない
- **ポイント出力のみ**: 称号・スタンプ（スタンプカードのスロット埋めを除く）・バッジ等の独立報酬を発行しない

### 4d.2 トリガー

| トリガー | 状態 | 実装箇所 |
|---------|------|---------|
| ログイン（デイリー初回） | **ログイン時のみ（活動完了時はトリガーされない）** | `home/+page.svelte` の `triggerLoginBonus()` — `loginBonusStatus.claimedToday` を確認し自動発火 |

実装上、`loginStamp` フォームアクション（`home/+page.server.ts`）が呼ばれる。このアクションは内部で以下を順番に実行する:
1. `claimLoginBonus(childId, tenantId)` — おみくじ抽選 + ポイント付与
2. `stampToday(childId, tenantId)` — スタンプカード押印（L2 系統への記録）

**重要**: おみくじポイントとスタンプカード押印は同一の `loginStamp` アクションで統合されている。スタンプカード（L2）と同じ 1 日 1 回 cap が適用される。

### 4d.3 出力: ポイントのみ（独立報酬なし）

| 確認項目 | 実装値 | 整合性 |
|---------|--------|------|
| 出力はポイントのみか | `login_bonus` type の point_entry として記録 | 整合 |
| 独立報酬（称号・バッジ等）を発行しないか | `claimLoginBonus()` はポイント計算のみ。称号・バッジの付与なし | 整合 |
| 独立した別通貨を発行しないか | `daily_login_bonuses.total_points` に記録。別 DB テーブルのみ | 整合 |

**基本ポイント範囲**（`src/lib/domain/validation/login-bonus.ts` の `OMIKUJI_RANKS` が SSOT）:

| おみくじランク | 出現率（重み比） | 基本ポイント |
|-------------|--------------|------------|
| 大大吉 | 約 1% | 30 pt |
| 大吉 | 約 5% | 15 pt |
| 中吉 | 約 15% | 7 pt |
| 小吉 | 約 25% | 5 pt |
| 吉 | 約 34% | 3 pt |
| 末吉 | 約 20% | 2 pt |

最終ポイントは `基本ポイント × 連続ログイン倍率 × ロイヤルティ倍率`。  
連続ログイン倍率は `LOGIN_MULTIPLIERS`（3日: ×1.5、7日: ×2.0、14日: ×2.5、30日: ×3.0）が SSOT。

### 4d.4 頻度制限と Anti-engagement 整合

| 確認項目 | 実装 | Anti-engagement 整合性 |
|---------|------|----------------------|
| 1 日 1 回 cap | `findTodayBonus()` で既受取を確認。競合時は `onConflictDoNothing` で 2 重付与防止 | 整合（ADR-0012 §4.1 準拠） |
| ユーザー自発操作なし | ページロード時に自動発火（手動ボタン不要） | 整合（離脱を促進しない）|
| 演出時間 | スタンプ押印演出（`StampPressOverlay.svelte`）で数秒。連続ガチャ不可 | 整合（Anti-engagement 原則） |
| ストーリー演出 | おみくじランクをテキスト表示（「大吉！」等）。ナラティブ装飾として OK | 整合 |

**独立報酬禁止ルール**: ストーリー演出（ランク名の表示）は許容するが、おみくじランクに応じた追加報酬（ランク専用スタンプ・ランク専用称号等）は §2.4 ポイント統一原則違反となるため禁止。

### 4d.5 逸脱確認

**逸脱なし。** 2026-04-26 の実装調査（`login-bonus.ts` / `login-bonus-service.ts` / `home/+page.server.ts` / `home/+page.svelte`）の結果、§4 の位置づけ定義と一致することを確認した。

> **実装者へ**: 「おみくじ演出を豪華にしたい」という動機でランクごとの独立スタンプ・称号付与を追加することは §2.4 ポイント統一原則違反となる。ナラティブテキスト（「大吉！」表示）での装飾は許容するが、ポイント以外の報酬を追加する場合は本節と §4 位置づけ表を同時に更新し PO 承認を得ること。

## 15. 年齢・誕生日設計

> **#1379 Sub B-1**: 実装から仕様を復元 (ADR-0001 準拠)。Committed（実装済み）と Aspirational（未実装）を明確に区別する (ADR-0013)。

### 15.1 データモデル

| カラム | 型 | 役割 |
|--------|-----|------|
| `child.birthDate` | `TEXT` (YYYY-MM-DD), nullable | **誕生日 SSOT**。入力された場合は年齢の唯一の真実の源 |
| `child.age` | `INTEGER` | 年齢キャッシュ。`birthDate` がある場合は `calculateAge(birthDate)` で計算。ない場合はユーザー直接入力値 |
| `child.uiMode` | `TEXT` ('baby'\|'preschool'\|'elementary'\|'junior'\|'senior') | 年齢 UI モード。`getDefaultUiMode(age)` から導出。誕生日ボーナス claim 時に自動更新 |
| `child.uiModeManuallySet` | `INTEGER` (0 or 1), default 0 | **Committed (#1382, PR #1463 実装済み)**。親が uiMode を明示的に選択したフラグ。`1` の場合、age 変更時の `uiMode` 自動再計算をスキップする |

### 15.2 年齢から uiMode へのマッピング

`getDefaultUiMode(age: number)` (`src/lib/domain/validation/age-tier.ts`):

| 年齢 | uiMode | 日本語名 |
|------|--------|---------|
| 0 〜 2 | `baby` | 乳幼児 |
| 3 〜 5 | `preschool` | 幼児 |
| 6 〜 12 | `elementary` | 小学生 |
| 13 〜 15 | `junior` | 中学生 |
| 16 以上 | `senior` | 高校生 |

### 15.3 状態遷移（Committed / 現行実装）

#### 子供登録時 (`/admin/children` POST ?/create)

```
birthDate 入力あり:
  child.birthDate = birthDate
  child.age       = calculateAge(birthDate)   ← (0 〜 18 クランプ)
  child.uiMode    = getDefaultUiMode(age)

birthDate 入力なし (年齢のみ):
  child.birthDate = null
  child.age       = Number(ageStr)            ← 0 〜 18 の範囲でバリデーション
  child.uiMode    = getDefaultUiMode(age)
```

#### 子供更新時 (`/admin/children` POST ?/update を経由した `child-service.editChild()`)

```
uiMode が明示指定された場合:
  child.uiMode            = 指定値
  child.uiModeManuallySet = 1          ← 手動設定フラグを立てる

age が変更された場合 (uiMode 未指定):
  child.age = Number(ageStr)
  child.uiMode = recalcUiMode(existing, newAge):
    uiModeManuallySet = 1 → 既存 uiMode を維持 (保護者 overrides 尊重)
    uiModeManuallySet = 0 → getDefaultUiMode(newAge) で自動再計算

birthDate が変更された場合:
  child.birthDate = birthDate (または null)
  birthDate あり → child.age = calculateAge(birthDate) → uiMode も recalcUiMode で更新
  birthDate なし → child.age = Number(ageStr) (入力があれば) → uiMode も recalcUiMode で更新
```

#### 誕生日ボーナス claim 時 (`birthday-bonus-service.claimBirthdayBonus()`)

```
claimBirthdayBonus():
  child.age                   = (更新しない — age-recalc cron (#1381) に移譲, #1522)
  child.uiMode                = getDefaultUiMode(newAge)      ← uiModeManuallySet に関わらず常に上書き
                                                                 (ポリシー: 誕生日は節目、保護者が必要なら再設定)
  child.lastBirthdayBonusYear = currentYear                   ← 重複 claim 防止
  PointEntry 追加             = age × 100 × birthdayBonusMultiplier
```

### 15.4 SSOT 優先度表

| 情報 | Primary SSOT | Derived From |
|------|-------------|--------------|
| 誕生日 | `child.birthDate` | — |
| 年齢 | `child.age` (ただし birthDate がある場合は `calculateAge(birthDate)` で上書き) | `calculateAge(birthDate)` |
| uiMode | `child.uiMode` (ただし `!uiModeManuallySet` なら age 変化時に再計算) | `getDefaultUiMode(age)` |

### 15.5 フォーム UX バリデーション（Committed）

- `birthDate` は YYYY-MM-DD 形式のみ許可
- `birthDate` が未来日の場合はエラー
- `age` は 0 〜 18 の整数のみ許可
- `birthDate` と `age` は**どちらか片方の入力が必須**（#1380, PR #1462 で実装済み）
- 両方入力した場合は `birthDate` 優先（`age` は `calculateAge(birthDate)` で上書き）
- `birthDate` 入力時は `age` 欄を disabled にして自動計算値を表示

### 15.6 誕生日ボーナスの演出（Committed）

| 要素 | 仕様 |
|------|------|
| バナー | 誕生日から 3 日以内にホーム画面上部に表示 |
| モーダル | 初回アクセス時に特別メッセージ |
| 請求期間 | 誕生日当日〜2 日後（`CLAIM_WINDOW_DAYS = 3`） |
| ボーナス | `age × 100 × birthdayBonusMultiplier` ポイント |
| 重複防止 | `lastBirthdayBonusYear` で同一年の二重 claim を防止 |

コンポーネント: `BirthdayBanner.svelte`, `BirthdayModal.svelte`
サービス: `birthday-bonus-service.ts`

### 15.7 年齢自動インクリメント（Committed / #1381）

毎日 00:00 JST に `/api/cron/age-recalc` を実行:

- 全テナントの全 child を `listAllTenants → findAllChildren` の 2 段階走査で取得
- `birthDate` を持つ child について `calculateAge(birthDate, today)` を計算
- 計算値が `child.age` と異なる場合のみ更新:
  - `age` を新しい年齢に更新
  - `uiModeManuallySet=false` の場合: `recalcUiMode()` で `uiMode` も再計算
  - `uiModeManuallySet=true` の場合: `uiMode` は変更しない（保護者の手動設定を維持）
- 同日 2 回実行しても 2 回目は `updated=0`（冪等性保証）
- `dryRun=true` で実際の DB 更新なしに更新件数を確認可能

スケジュール: `schedule-registry.ts` に登録済み（`utcCronExpression: cron(0 15 * * ? *)`）

### 15.9 birthday-bonus-service との責務分担

`birthday-bonus-service.claimBirthdayBonus()` と `child-service.editChild()` は uiMode 更新ロジックをそれぞれ持つ。以下のルールで責務を分離する。

| 操作 | 担当サービス | age 更新 | uiModeManuallySet の考慮 |
|------|------------|---------|------------------------|
| 保護者が管理画面で age / uiMode を編集 | `child-service.editChild()` | あり | 考慮する。`recalcUiMode()` 経由 |
| 誕生日ボーナス claim (子供画面 / cron) | `birthday-bonus-service.claimBirthdayBonus()` | **なし**（#1522 で削除。age 更新は age-recalc cron に移譲） | **考慮しない**。誕生日は成長の節目のため `getDefaultUiMode()` で常に上書き |
| 年齢自動インクリメント cron (#1381) | `age-recalc-service.recalcAllChildrenAges()` | あり | **考慮する**。`recalcUiMode()` を使用。保護者の手動設定を維持 |

**ポリシー根拠**: 誕生日ボーナス claim 時は成長の節目であるため、uiMode を自動追従させることを優先する（`getDefaultUiMode()` で常に上書き）。age 自体の更新は毎日 00:00 JST の `age-recalc` cron（#1381）が担うため、`claimBirthdayBonus()` 内での age 更新は重複となる。#1522 にて削除済み。保護者が uiMode を特定年齢帯に固定したい場合は、ボーナス受取後に管理画面から再設定する。

## 18. バトルアドベンチャー（#605）

バトルは **L1 副次** として §4 並列フィードバック系で位置づけられる。活動記録の結果として発生する体験であり、単独で起動できる遊びではない。

### 18.1 概要

日々の活動がRPGステータスに反映され、毎日自動マッチングされた敵とターン制バトルを行う機能。
活動カテゴリごとのXPがRPGステータスに変換され、活動量がそのまま戦闘力に直結する設計。

| 項目 | 値 |
|------|-----|
| バトル頻度 | 1日1回（毎日新しい敵がマッチング） |
| バトル方式 | ターン制・自動進行 |
| 勝敗 | 勝利/敗北（ペナルティなし、慰めポイントあり） |
| 対象年齢モード | elementary / junior / senior のみ（baby/preschool はバトル機能なし） |

### 18.2 カテゴリ → RPGステータス変換

活動カテゴリのXPがそのままRPGステータスに変換される。

| カテゴリ | RPGステータス | 意味 |
|---------|------------|------|
| うんどう | HP（たいりょく） | 体力 |
| べんきょう | ATK（こうげき） | 攻撃力 |
| せいかつ | SPD（すばやさ） | 素早さ |
| こうりゅう | DEF（ぼうぎょ） | 防御力 |
| そうぞう | REC（かいふく） | 回復力 |

**変換式**: `stat = BASE + floor(xp^0.6)`

| ベースステータス | 値 |
|--------------|-----|
| HP | 50 |
| ATK | 10 |
| DEF | 8 |
| SPD | 10 |
| REC | 5 |

ソース: `src/lib/domain/battle-stat-calculator.ts`

### 18.3 敵マスタデータ

10体以上の敵をレアリティ別に定義。曜日出現制限あり。

| レアリティ | 出現率 | 勝利ポイント | 慰めポイント | 代表例 |
|-----------|--------|------------|-----------|--------|
| common | 高 | 10 | 3 | スライム、コウモリ、キノコおばけ |
| uncommon | 中 | 20 | 5 | ゴブリン、おおカエル、ゴーレム |
| rare | 低 | 40 | 10 | ドラゴンパピー |
| boss | 週末限定 | 80 | 15 | まおうのかげ |

ソース: `src/lib/domain/battle-enemies.ts`

### 18.4 年齢スケーリング

敵ステータスは年齢モードに応じてスケーリングされる。

| 年齢モード | 係数 | 備考 |
|-----------|------|------|
| baby | — | バトル機能なし（親の準備モード） |
| preschool | — | バトル機能なし |
| elementary | ×0.8 | バトル開始 |
| junior | ×1.0 | 基準値 |
| senior | ×1.2 | 高難度 |

ソース: `src/lib/domain/battle-stat-calculator.ts` の `getAgeScaling()`

### 18.5 バトルエンジン

ターン制の自動バトル。先攻はSPDで決定。

```
ターン開始 → 先攻判定(SPD) → 攻撃(ATK - DEF/2) → HP判定
  → 相手ターン → 攻撃 → HP判定 → 回復(REC/2) → ループ
  → HPが0以下で勝敗確定
```

| 項目 | 仕様 |
|------|------|
| 最大ターン数 | 20（超過で引き分け＝勝利扱い） |
| ダメージ計算 | `max(1, atk - def / 2)` |
| 回復 | 毎ターン `rec / 2` 回復（HP上限を超えない） |
| クリティカル | なし（子供向けにシンプル） |

ソース: `src/lib/domain/battle-engine.ts`

### 18.6 データ構造

テーブル: `daily_battles`

| カラム | 型 | 説明 |
|--------|-----|------|
| id | TEXT PK | バトルID |
| child_id | TEXT | 子供ID |
| tenant_id | TEXT | テナントID |
| battle_date | TEXT | バトル日（YYYY-MM-DD） |
| enemy_id | INTEGER | 敵マスタID |
| status | TEXT | pending / won / lost |
| player_stats | TEXT | JSON: プレイヤーステータス |
| battle_log | TEXT | JSON: バトルログ |
| reward_points | INTEGER | 獲得ポイント |
| created_at | TEXT | 作成日時 |
| completed_at | TEXT | 完了日時 |

UNIQUE制約: `(child_id, battle_date)` — 1子供1日1バトル

### 18.7 API

| エンドポイント | メソッド | 説明 |
|-------------|---------|------|
| `/api/v1/battle/[childId]` | GET | 今日のバトル情報取得（敵マッチング含む） |
| `/api/v1/battle/[childId]` | POST | バトル実行（サーバー側で計算） |

**セキュリティ**: POST ではクライアントからの battleId/enemyId を受け取らない。サーバー側で childId + tenantId から今日のバトルを取得して実行する。

## §4e 初月価値プレビュー体験（#1600 / ADR-0023 I9）

### 4e.1 位置づけ

L1 コアループ（活動記録 → ポイント獲得）と L3 表現層（レベル・称号・実績）の **間にある "進捗の可視化" レイヤ**として機能する。
ポイント付与は伴わない（ADR-0012 Anti-engagement 準拠）。

### 4e.2 役割

ADR-0023 §5 I9 / C-Q13 を解決するための「最初の 30 日体験」。

- **親 P1 への効果**: 「あと N 日続ければ 1 ヶ月の節目」という時間的ゴール感の提供
- **親 P1 への効果**: カテゴリ別 bar chart で「子供がどの分野に偏っているか」が見える
- **子供への効果**: 「7 日連続できた」等の達成バナーで小さな達成感（ただし 1 件のみ・自動消滅なし）

### 4e.3 マイルストーン定義

| id | kind | threshold | 意味 |
|----|----|----|----|
| `first_record` | count | 1 | 最初の活動記録 |
| `records_5` | count | 5 | 5 件記録 |
| `records_10` | count | 10 | 10 件記録 |
| `streak_7` | streak | 7 | 1 週間連続 |
| `streak_14` | streak | 14 | 2 週間連続 |
| `streak_30` | streak | 30 | 1 ヶ月連続 |

マスタ: `src/lib/server/services/value-preview-service.ts` の `MILESTONES` 定数。

### 4e.4 Anti-engagement (ADR-0012) 境界

| 観点 | 守っていること |
|----|----|
| 過剰祝福 | バナーは fade-in 0.2s のみ。自動 BGM・紙吹雪・連続演出は不採用 |
| 自動消滅 | しない（× ボタン明示）。タイマーで自動消える挙動は不採用 |
| 通知連打 | バナーは画面内表示のみ。push 通知連動なし |
| 滞在時間延伸 | 1 件のみ表示・閲覧済みは localStorage で永続抑制 |
| baby モード | 親向け準備モードのため非対象（ADR-0011 整合） |
