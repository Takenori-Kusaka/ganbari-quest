## 1. リリース判定基準

SaaS版リリース前に以下の全項目を確認し、すべて合格であることを判定者（開発者本人）が確認する。

### 1.1 品質基準

| # | カテゴリ | 基準 | 合格条件 | 確認方法 |
|---|---------|------|---------|---------|
| 1 | テスト | E2Eテスト全通過 | 100% | `npx playwright test` |
| 2 | テスト | ユニットテスト全通過 | 100% | `npx vitest run` |
| 3 | Lint | Biome チェック | エラー 0 | `npx biome check .` |
| 4 | 型 | TypeScript strict | エラー 0 | `npx svelte-check` |
| 5 | 品質 | 重大バグ（P0/P1） | 0 件 | チケット一覧確認 |
| 6 | CI | GitHub Actions 全ジョブ成功 | 直近の main ビルドが green | `gh run list` |

### 1.2 セキュリティ基準

| # | 基準 | 合格条件 | 確認方法 |
|---|------|---------|---------|
| 1 | テナント分離 | 他テナントのデータにアクセス不可 | E2Eテスト（テナント分離シナリオ） |
| 2 | 認証・認可 | 未認証でAPI呼び出し不可 | E2Eテスト（認証テスト） |
| 3 | HTTPS | 全通信がTLS | CloudFront + ACM 証明書確認 |
| 4 | 依存関係 | 既知の脆弱性なし | `npm audit`、GitHub Dependabot |
| 5 | Secrets | .env/認証情報がリポジトリに含まれない | `.gitignore` + `git log` 確認 |
| 6 | CodeQL | セキュリティスキャン通過 | `.github/workflows/codeql.yml` |

### 1.3 法務基準

| # | 基準 | 合格条件 | 確認方法 |
|---|------|---------|---------|
| 1 | 利用規約 | 公開済み | `site/terms.html` がアクセス可能 |
| 2 | プライバシーポリシー | 公開済み | `site/privacy.html` がアクセス可能 |
| 3 | 特定商取引法に基づく表記 | 公開済み | `site/tokushoho.html` がアクセス可能 |
| 4 | SLA | 公開済み | `site/sla.html` がアクセス可能 |

### 1.4 インフラ基準

| # | 基準 | 合格条件 | 確認方法 |
|---|------|---------|---------|
| 1 | ヘルスチェック | `/api/health` が 200 | `curl https://ganbari-quest.com/api/health` |
| 2 | SSL証明書 | 有効期限 30日以上 | ACM コンソール確認 |
| 3 | ドメイン | ganbari-quest.com が正しく解決 | `nslookup ganbari-quest.com` |
| 4 | バックアップ | AWS Backup 日次バックアップ稼働中 | AWS Backup コンソール確認 |
| 5 | 監視 | CloudWatch アラーム全設定済み | OpsStack の 10 アラーム確認 |
| 6 | ロールバック | deploy.yml にロールバック手順あり | ヘルスチェック失敗時の自動ロールバック |
| 7 | コスト | Budget アラート設定済み（$5/月） | AWS Budgets コンソール確認 |

## 2. ローンチ当日チェックリスト

リリース当日に以下を順番に実行する。

### 2.1 リリース前（当日朝）

- [ ] `gh run list` で直近の main ビルドが成功していることを確認
- [ ] `curl -s https://ganbari-quest.com/api/health` で 200 応答確認
- [ ] AWS CloudWatch ダッシュボード（`ganbari-quest-ops`）で異常なしを確認
- [ ] SSL証明書の有効期限を確認（ACMコンソール）
- [ ] DynamoDB テーブルのステータス確認（ACTIVE）
- [ ] AWS Backup の直近バックアップが成功していることを確認

### 2.2 LP・外部サービス確認

- [ ] `https://www.ganbari-quest.com/` が正しく表示される
- [ ] LP 内のリンク（ログイン、デモ、利用規約等）が全て正常に動作
- [ ] サポートメール（ganbari.quest.support@gmail.com）の受信確認
- [ ] GitHub リポジトリが公開状態

### 2.3 アプリ動作確認

- [ ] サインアップフローが完了する（テスト用メールで確認）
- [ ] デモモード（`/demo`）が正常動作
- [ ] 子供画面の各年齢モードが表示される
- [ ] 管理画面（`/admin`）にPINでログインできる
- [ ] 活動記録 → ポイント付与が動作する

### 2.4 リリース実行

- [ ] リリースノートを作成（`gh release create`）
- [ ] GitHub Releases にリリース告知を投稿
- [ ] LP のリリースノートセクションを更新（必要に応じて）

### 2.5 リリース後（当日夕方）

- [ ] CloudWatch ダッシュボードでエラー率を確認
- [ ] CloudWatch Logs でエラーログがないか確認
- [ ] GitHub Issues に問い合わせや不具合報告がないか確認
- [ ] AWS Cost Explorer で異常なコスト増がないか確認

## 3. 運用手順書

### 3.1 日次運用（所要時間: 約10分）

#### CloudWatch ダッシュボード確認

```
1. AWS Console → CloudWatch → ダッシュボード → ganbari-quest-ops
2. 以下を確認:
   - Lambda Errors: 0 であること
   - Lambda Duration P99: 10秒以下であること
   - DynamoDB Throttles: 0 であること
   - CloudFront 5xx: 0% であること
3. アラーム状態のウィジェットが全て OK（緑）であること
```

#### 問い合わせ確認

```
1. GitHub Issues の新規 Issue を確認
2. メール（ganbari.quest.support@gmail.com）の問い合わせを確認
3. 対応が必要なものがあれば返信・チケット化
```

### 3.2 週次運用（所要時間: 約30分、毎週月曜）

#### エラーログレビュー

```bash
# 直近7日のエラーログを確認（AWS CLI）
aws logs filter-log-events \
  --log-group-name "/aws/lambda/ganbari-quest-app" \
  --start-time $(date -d '7 days ago' +%s000) \
  --filter-pattern "ERROR" \
  --query 'events[].message' \
  --output text \
  --region us-east-1
```

#### コスト確認

```
1. AWS Console → Billing → Cost Explorer
2. 当月の累計コストを確認
3. Budget（$5/月）に対する消化率を確認
4. 異常なサービスコスト増がないか確認
```

#### ユーザーフィードバック集約

```
1. GitHub Issues とメールから1週間分のフィードバックを収集
2. 対応が必要なものをチケット化（GitHub Issues）
```

### 3.3 月次運用（所要時間: 約1時間、毎月1日）

#### KPI確認

```
1. DynamoDB でテナント数を確認（MAU の代替指標）:
   aws dynamodb scan \
     --table-name ganbari-quest \
     --filter-expression "begins_with(PK, :pk)" \
     --expression-attribute-values '{":pk":{"S":"TENANT#"}}' \
     --select COUNT \
     --region us-east-1

2. Stripe ダッシュボードで MRR（月次経常収益）を確認
3. 解約率（Churn Rate）を確認
```

#### バックアップ・リストアテスト

```
1. AWS Console → AWS Backup → バックアップボールト → ganbari-quest-vault
2. 直近のバックアップが成功していることを確認
3. 四半期に1回: テスト用テーブルへのリストアを実行して検証
   AWS Backup → リカバリポイント → 復元 → 新しいテーブル名で復元
```

#### セキュリティパッチ適用

```bash
# 依存関係の脆弱性チェック
npm audit

# Dependabot アラートの確認
gh api repos/Takenori-Kusaka/ganbari-quest/dependabot/alerts --jq '.[].security_advisory.summary'

# 必要に応じてパッチ適用
npm update
npx vitest run
npx playwright test
# テスト通過後にコミット & プッシュ
```

## 4. 障害対応フロー

### 4.1 検知

| 検知方法 | 通知先 | 対応 |
|---------|--------|------|
| CloudWatch Alarm → SNS → メール | 開発者メール | P0: 1時間以内に初動 |
| CloudWatch Alarm → SNS → メール | 開発者メール | P1: 4時間以内 |
| ユーザーメール報告 | ganbari.quest.support@gmail.com | 状況に応じて |
| GitHub Actions 失敗 | GitHub メール通知 | デプロイ失敗時は自動ロールバック |

### 4.2 現在のアラーム一覧（ops-stack.ts）

| # | アラーム名 | 条件 | 優先度 |
|---|-----------|------|--------|
| 1 | Lambda Errors | 5分間に3回以上 | P0 |
| 2 | Lambda Throttles | 5分間に1回以上 | P0 |
| 3 | Lambda Duration P99 | > 10秒 | P1 |
| 4 | Lambda Concurrent Executions | > 50 | P1 |
| 5 | DynamoDB Throttles | 5分間に1回以上 | P1 |
| 6 | DynamoDB System Errors | 5分間に1回以上 | P0 |
| 7 | Lambda URL 5xx | 5分間に5回以上 | P0 |
| 8 | Lambda URL 4xx spike | 5分間に50回以上 | P1 |
| 9 | CloudFront 5xx Error Rate | > 5% | P0 |
| 10 | DynamoDB Consumed Capacity | 1時間で10,000超 | P1 |

### 4.3 障害対応手順

```
1. アラーム検知（メール or CloudWatch コンソール）
   └── 影響範囲の確認（全体停止 / 一部機能 / 性能劣化）

2. 一次対応（15分以内）
   ├── CloudWatch Logs でエラー内容を確認
   ├── GitHub Releases / サイトお知らせ に障害発生を投稿（ユーザー向け）
   └── 影響を受けるユーザーの規模を推定

3. 原因調査
   ├── CloudWatch Logs Insights でエラーパターンを分析
   │   aws logs start-query \
   │     --log-group-name "/aws/lambda/ganbari-quest-app" \
   │     --start-time $(date -d '1 hour ago' +%s) \
   │     --end-time $(date +%s) \
   │     --query-string 'filter @message like /ERROR/ | stats count() by bin(5m)'
   │
   ├── 直近のデプロイが原因か確認
   │   gh run list --limit 5
   │
   └── DynamoDB / S3 / Cognito の状態を確認

4. 復旧
   ├── デプロイ起因の場合:
   │   → deploy.yml の自動ロールバック or 手動ロールバック
   │   aws lambda update-function-code \
   │     --function-name ganbari-quest-app \
   │     --image-uri <previous-image-uri> \
   │     --region us-east-1
   │
   ├── データ起因の場合:
   │   → AWS Backup からリストア（§3.3 参照）
   │
   └── 外部サービス起因（AWS障害等）の場合:
       → AWS Health Dashboard で状況確認、復旧を待つ

5. 復旧確認
   ├── curl https://ganbari-quest.com/api/health
   ├── CloudWatch ダッシュボードでエラー率が 0 に戻ったことを確認
   └── 主要機能の手動確認

6. 事後対応
   ├── GitHub Releases / サイトお知らせ に復旧報告を投稿
   ├── 障害報告をチケット化（GitHub Issues）
   │   - 発生日時、影響範囲、原因、対応内容、再発防止策
   └── 必要に応じて再発防止の修正をコミット
```

## 5. バックアップ・リストア手順

### 5.1 現在のバックアップ構成

| 対象 | 方式 | 保持期間 | 設定ファイル |
|------|------|---------|------------|
| DynamoDB テーブル | AWS Backup 日次 | 3日間 | `infra/lib/storage-stack.ts` |
| S3（アバター等） | S3 バージョニング | 無期限（明示的削除のみ） | `infra/lib/storage-stack.ts` |
| S3 バックアップ | S3 ライフサイクル | 30日で自動削除 | `infra/lib/storage-stack.ts` |
| S3 ログ | S3 → Glacier | 1日後に Glacier | `infra/lib/storage-stack.ts` |
| コードベース | Git（GitHub） | 無期限 | GitHub リポジトリ |
| インフラ定義 | CDK（IaC） | Git 管理 | `infra/` ディレクトリ |

### 5.2 DynamoDB リストア手順

```
1. AWS Console → AWS Backup → バックアップボールト → ganbari-quest-vault
2. リカバリポイント一覧から復元したい日付を選択
3. 「復元」をクリック
4. 設定:
   - 新しいテーブル名: ganbari-quest-restored-YYYYMMDD
   - その他はデフォルト
5. 「バックアップを復元」をクリック
6. 復元完了後（約10-30分）、データ内容を確認
7. 本番テーブルとの差し替えが必要な場合:
   a. Lambda を停止（同時実行数を 0 に設定）
   b. 復元テーブルを本番テーブル名にリネーム（CDK再デプロイ）
   c. Lambda を復帰（同時実行数を元に戻す）
```

### 5.3 S3 オブジェクト復元手順

```
1. S3 バージョニングが有効なため、削除されたファイルも復元可能
2. AWS Console → S3 → ganbari-quest-assets-* バケット
3. 「バージョンの表示」をオンにする
4. 復元したいオブジェクトの以前のバージョンを選択 → 「復元」
```

### 5.4 コードベースのロールバック

```bash
# Lambda を前のイメージに戻す（deploy.yml の自動ロールバックと同じ）
PREVIOUS_DIGEST=$(aws ecr describe-images \
  --repository-name ganbari-quest \
  --query 'sort_by(imageDetails,&imagePushedAt)[-2].imageDigest' \
  --output text --region us-east-1)

aws lambda update-function-code \
  --function-name ganbari-quest-app \
  --image-uri <ECR_REGISTRY>/ganbari-quest@$PREVIOUS_DIGEST \
  --region us-east-1

aws lambda wait function-updated \
  --function-name ganbari-quest-app \
  --region us-east-1

# 復旧確認
curl -s https://ganbari-quest.com/api/health
```

## 6. コスト管理

### 6.1 予算設定

| 項目 | 設定値 |
|------|--------|
| 月間予算 | $5 USD |
| 50% 到達アラート | $2.50 で通知 |
| 80% 到達アラート | $4.00 で通知 |
| 100% 予測アラート | 予測超過時に通知 |

### 6.2 コスト異常時の対応

```
1. Budget アラートを受信
2. AWS Cost Explorer で原因サービスを特定
3. 対応:
   - DynamoDB: 不要なスキャン・クエリがないか確認
   - Lambda: 無限ループや高頻度呼び出しがないか確認
   - CloudFront: 異常なトラフィックがないか確認
   - S3: 大容量ファイルのアップロードがないか確認
4. 必要に応じてリソースの制限を設定
```
