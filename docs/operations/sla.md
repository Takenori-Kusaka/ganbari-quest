## 1. 設計背景

### なぜ SLA を策定するか

本サービスは個人運営の有料 SaaS であり、正式ローンチ後は保護者ユーザに対して「どこまで保証され、どこからは best-effort か」を明示する責任を負う。SLA がない状態でローンチすると、以下のリスクが顕在化する:

- 障害発生時に「何時間以内に復旧するのか」という保護者の期待値が無秩序に膨張し、クレーム対応コストが Pre-PMF 段階のキャパを超える
- 計画メンテナンス通知のリードタイムが定義されておらず、唐突なダウンタイムで信頼を失う
- 補償ポリシーが未定義のため、「お金返してほしい」と言われた際の判断基準が運営者個人の感覚に依存する
- 利用規約・特商法表記との整合性が取れず、消費者保護法・特商法上の説明義務違反リスクが残る

旧 Issue #150（0075 SLA 策定）は CLOSED されていたが `docs/operations/sla.md` 実体が存在せず、`site/sla.html` 単独が情報源だったため、運用観点（障害対応 / バックアップ / 通知）と法務観点（利用規約 §12 免責との整合）を 1 つの SSOT に揃える必要がある。Issue #2400 で本ドキュメントを正本として策定する。

### 関連ペルソナ

| ペルソナ | この SLA との関係 |
|---------|--------------|
| P1 保護者ユーザ（有料プラン契約者） | 障害時の復旧時間 / 通知経路 / 補償可否の主要受益者 |
| 運営者（日下武紀） | 個人運営の現実的キャパシティで持続可能な約束を設定する責務 |
| 法務観点 | 利用規約 §12 / 特商法表記との整合担保 |

## 2. 設計原則

1. **個人運営の現実を反映した best-effort 表明** — エンタープライズ SaaS と同等の冗長性・SLA クレジット（返金）を約束しない。「目標値」と明記し、未達時は誠実な事後対応で応える
2. **数値で具体化する** — 「速やかに対応します」のような曖昧表現を禁止。可用性 = 99.5% / 初回応答 = 48h / RTO = 4h / RPO = 24h のように測定可能な数値で記述
3. **計画停止は事前通知 + ゼロダウンタイム CI/CD を活用** — 通常デプロイは無中断、インフラ変更時のみ「計画メンテナンス」として 24h 前通知
4. **障害通知経路の二重化** — Discord 公開ステータスチャンネル（即時）+ 1h 超過時のメール通知（登録アドレス宛）
5. **補償は Pre-PMF では非金銭** — SLA クレジット（返金・減額）は提供しない。重大障害（24h 超ダウン）時のみ有料プランの期間延長を申請ベースで対応
6. **利用規約 §12（免責）に整合** — 本 SLA で約束する内容は利用規約の免責範囲内に収め、損害賠償義務を伴わない目標値として運用
7. **改定は 14 日前事前通知** — SLA 内容変更時は登録ユーザに 14 日前までに通知し、改定履歴を本ドキュメントに残す

> **good/bad 例**:
> - bad: 「障害時は速やかに復旧します」（時間目標なし）
> - good: 「障害発生から 4 時間以内の復旧を目標とする（RTO 4h）」

## 3. SLA 仕様

### 3.1 適用範囲

| 項目 | 内容 |
|------|------|
| 適用対象 | 本サービスの SaaS 版（https://ganbari-quest.com） |
| 適用外 | セルフホスト版（利用者自身の環境で動作するもの） |
| 法的性質 | 運営者が誠実に達成を目指す目標値（法的保証ではない） |

### 3.2 可用性目標

| 指標 | 目標値 | 補足 |
|------|--------|------|
| **月間可用性** | **99.5%** | 月間約 3.6 時間の計画外ダウンタイムに相当 |

**計画外ダウンタイムに含まない事象**:

- 事前に通知された計画メンテナンス
- 天災・戦争等の不可抗力による停止
- AWS など利用しているクラウド基盤の障害
- 利用者側の環境に起因する接続障害（インターネット接続不良 / ブラウザ非互換等）

### 3.3 デプロイおよび計画メンテナンス

| 種別 | 通知リードタイム | ダウンタイム想定 |
|------|----------------|----------------|
| 通常デプロイ（CI/CD） | 不要 | 0 秒（ゼロダウンタイム） |
| インフラ変更（CDK スタック / DB マイグレーション） | **24 時間前まで** Discord ステータスチャンネル告知 | 数分〜数十分 |
| 緊急セキュリティパッチ | 可能な限り速やか（事後通知許容） | 状況依存 |

事前通知時には影響範囲と想定中断時間を併記する。

### 3.4 データ保護

| 指標 | 内容 |
|------|------|
| バックアップ頻度 | 日次自動バックアップ |
| 通信暗号化 | TLS 1.2 以上 |
| 保存データ暗号化 | AES-256 |
| **RTO（復旧目標時間）** | **4 時間以内** |
| **RPO（復旧時点目標）** | **24 時間以内**（日次バックアップ間隔） |

### 3.5 障害通知

| 状況 | 通知経路 |
|------|---------|
| 障害発生時（即時） | Discord 公開ステータスチャンネル |
| ダウンタイム 1 時間超過時 | 登録メールアドレスへの個別通知を追加 |

障害検知はデプロイ時の自動検証および定期ヘルスチェック（準備中）により行い、異常検知後速やかに対応・通知する。

### 3.6 サポート対応

| 項目 | 内容 |
|------|------|
| 受付経路 | GitHub Issues / メール（ganbari.quest.support@gmail.com） |
| 受付時間 | 24 時間（自動受付） |
| **初回応答目標** | **48 時間以内（営業日ベース）** |
| 対応言語 | 日本語 |
| 注意 | 個人運営のため応答が遅れる場合あり |

### 3.7 SLA 未達時の対応

| 状況 | 対応 |
|------|------|
| 目標値未達（軽微） | 原因調査と再発防止策の社内記録 |
| SLA クレジット（返金・減額） | **提供なし**（Pre-PMF 段階の方針） |
| **重大障害**（連続 24 時間以上のサービス停止等） | 有料プラン利用者は障害期間に相当する日数分のサービス期間延長を申請可能。延長可否は運営者判断 |

### 3.8 免責事項

1. 本サービスは個人開発であり、エンタープライズ SaaS と同等の可用性・冗長性は保証しない
2. SLA 目標値未達でも損害賠償義務は負わない（[利用規約](../../site/terms.html) 第 12 条免責に従う）
3. SLA 内容変更時は 14 日前までに通知する

## 4. 障害対応との連携

SLA で約束した RTO 4h / RPO 24h を満たすための実運用手順は [`runbook.md`](runbook.md) を参照。Lambda / AWS / DDoS の 3 系統について Level 1（自動アラート） → Level 4（事後対応）の判断フローが定義されている。

通知系統の動作確認手順は [`notification-runbook.md`](notification-runbook.md) を参照。

## 5. LP との同期

本 SLA の主要項目は LP 法務文書 [`site/sla.html`](../../site/sla.html) に保護者向け文章で記載されている。SLA 改定時は本ドキュメントと sla.html の両方を更新し、`<section>` 内の本文と本ドキュメントの「3. SLA 仕様」表が整合することを確認する。

`site/sla.html` 内の `data-lp-key` fallback テキスト SSOT 化は別 Issue（#2394）の領域であり、本ドキュメントの守備範囲外。
