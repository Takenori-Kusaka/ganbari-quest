## 1. 本書の目的

がんばりクエストの SaaS 運用において、**ユーザーが最初にサービスを知ってから利用を終了するまでの全ライフサイクル**を設計する。

各フェーズで「誰が」「どこで」「何をして」「何が起き」「誰に何を通知し」「どんな証跡を残すか」を明確にする。

### 1.1 対象読者

- 開発者（自分自身）
- 将来の運用担当者
- セキュリティ・法務レビュー担当

### 1.2 関連設計書

| 設計書 | 関連箇所 |
|--------|---------|
| 03-ユースケース設計書 | UC-01〜UC-13（子供・親の操作） |
| 07-API設計書 | エンドポイント定義 |
| 08-データベース設計書 | テーブル構造 |
| 10-SaaS展開ロードマップ | フェーズ計画 |
| 13-AWSサーバレスアーキテクチャ設計書 | インフラ構成 |
| 14-セキュリティ設計書 | 認証・認可・暗号化 |

### 3.11 データ削除フェーズ

#### 削除対象と実行順序

```
データ削除バッチ（EventBridge → Lambda 日次実行）

前提: テナントステータスが grace_period かつ expiresAt < now()

実行順序:
  1. Stripe サブスクリプションキャンセル（未キャンセルの場合）
  2. S3 テナントデータ削除
     ├─ avatars/{tenantId}/ 配下の全オブジェクト
     ├─ generated/{tenantId}/ 配下の全オブジェクト
     ├─ voices/{tenantId}/ 配下の全オブジェクト
     └─ その他 {tenantId}/ プレフィックスのオブジェクト
  3. DynamoDB テナントデータ削除
     ├─ PK = TENANT#{tenantId}#CHILD#* (全子供・全活動ログ・全ステータス)
     ├─ PK = TENANT#{tenantId}#ACTIVITY#* (活動マスタ)
     ├─ PK = TENANT#{tenantId}#SETTING (設定)
     ├─ PK = TENANT#{tenantId}#LICENSE (ライセンス)
     └─ PK = TENANT#{tenantId}#* (その他全て)
  4. Cognito ユーザー属性クリア
     ├─ custom:tenantId をクリア
     └─ ユーザーアカウント自体は残す（再サインアップ可能にするため）
  5. テナントメタデータのステータスを deleted に更新
  6. 通知送信
     ├─ ユーザーへ: 削除完了メール（SES）
     └─ 運用者へ: Discord #退会 に削除完了通知
  7. CloudWatch にログ出力
```

#### S3 パス設計（テナントプレフィックス）

| ファイル種別 | 現在のパス | 変更後のパス |
|------------|----------|------------|
| アバター画像 | `uploads/avatars/avatar-{childId}-{ts}.{ext}` | `tenants/{tenantId}/avatars/{childId}/{uuid}.{ext}` |
| AI生成画像 | `generated/avatar-{childId}-{ts}.{ext}` | `tenants/{tenantId}/generated/{childId}/{uuid}.{ext}` |
| 音声ファイル | (未実装) | `tenants/{tenantId}/voices/{childId}/{uuid}.mp3` |
| その他ファイル | — | `tenants/{tenantId}/files/{uuid}.{ext}` |

> **テナントプレフィックスの導入理由**:
> - テナント削除時に `s3.listObjectsV2({ Prefix: 'tenants/{tenantId}/' })` で全ファイルを列挙 → 一括削除が可能
> - テナント間のファイルアクセスを S3 キーレベルで分離
> - IAM ポリシーで `tenants/${tenantId}/*` に限定することも可能

#### 子供削除時のファイルクリーンアップ

```
子供削除（removeChild）時:

1. S3 ファイル削除
   ├─ tenants/{tenantId}/avatars/{childId}/ 配下全て
   ├─ tenants/{tenantId}/generated/{childId}/ 配下全て
   └─ tenants/{tenantId}/voices/{childId}/ 配下全て
2. DynamoDB カスケード削除（既存実装）
   ├─ CHILD#{childId} の全 SK レコード
   └─ 関連テーブル（activity_logs, statuses 等）
3. character_images テーブルのレコード削除
```

#### 現状と課題

| 項目 | 現状 | 課題 |
|------|------|------|
| テナント一括削除 | **未実装** | #0132 で設計済みだが実装なし |
| S3 テナントプレフィックス | **未導入** | 現在は `uploads/avatars/avatar-{childId}-*` でテナントIDなし |
| 子供削除時のファイル削除 | 実装済み (`deleteChildFiles()`) | S3 パス変更に伴う修正が必要 |
| データ削除バッチ Lambda | **未実装** | EventBridge + Lambda スケジュール未構築 |
| 削除完了通知 | **未実装** | メール・Discord 両方 |
| 孤立ファイルの検出・削除 | **未実装** | DB レコードなしでS3にだけ残るファイルの定期クリーンアップ |

## 5. 静的ファイル管理設計

### 5.1 S3 バケット構成

```
ganbari-quest-assets-{accountId}/
├── tenants/
│   └── {tenantId}/
│       ├── avatars/
│       │   └── {childId}/
│       │       ├── {uuid}.png       ← アップロードされたアバター画像
│       │       └── {uuid}.webp
│       ├── generated/
│       │   └── {childId}/
│       │       └── {uuid}.png       ← AI生成キャラクター画像
│       ├── voices/
│       │   └── {childId}/
│       │       └── {uuid}.mp3       ← カスタム音声ファイル
│       └── files/
│           └── {uuid}.{ext}         ← その他のファイル
├── backups/                          ← 30日で自動削除（既存）
└── logs/                             ← 1日後 Glacier（既存）
```

### 5.2 S3 ライフサイクルルール

| プレフィックス | ルール | 目的 |
|-------------|--------|------|
| `backups/` | 30日で削除 | 既存 |
| `logs/` | 1日後 Glacier | 既存 |
| `tenants/` | なし（明示的削除のみ） | ユーザーデータは自動削除しない |

### 5.3 アクセス制御

- S3 バケットはパブリックアクセス完全ブロック（既存設定）
- ファイルアクセスは Lambda IAM ロール経由のみ
- API エンドポイントでテナントID検証後に S3 からファイル取得して返却
- 署名付き URL の発行は将来検討（パフォーマンス改善時）

### 5.4 ファイルアップロード制限

| ファイル種別 | 最大サイズ | 許可形式 | バリデーション |
|------------|----------|---------|-------------|
| アバター画像 | 5 MB | JPEG, PNG, WebP | Magic bytes + 拡張子 |
| AI生成画像 | 10 MB | PNG | システム内部生成のみ |
| 音声ファイル | 10 MB | MP3, WebM, OGG | Magic bytes + 拡張子 + 長さ制限（30秒） |
