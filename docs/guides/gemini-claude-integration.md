# ガイド: Claude Code と Gemini CLI の連携

## 1. 概要

このドキュメントは、本プロジェクトにおいて Claude Code セッションから Gemini CLI を呼び出し、タスクを委任するためのガイドラインを定めます。

開発・QAワークフローにおいて、軽微な実装や定型的なレビューなどのタスクは Gemini CLI に委任することが推奨されます。この連携をスムーズかつ安全に行うため、以下のルールに従ってください。

## 2. 基本原則

- **役割**: Gemini CLI は、「YOLOモードで動作する非対話的なエージェント」として扱います。指示は明確かつ完結させ、人間による対話的な介入は想定しません。
- **安全性**: ツール実行は `-y` (`--yolo`) オプションで自動承認されるため、プロジェクトで定義されたポリシー (`policy` / `admin-policy`) を必ず適用し、危険な操作が実行されないように制御します。
- **入出力**: Claude との通信は、標準入力（stdin）、コマンドライン引数（prompt）、そしてJSON形式の標準出力（stdout）を介して行います。

## 3. 連携パターン

### 3.1. ファイルの修正

既存のファイルを修正させる場合の連携フローです。

**ステップ 1: Claude 側でコマンドを組み立てる**

修正対象のファイル内容を標準入力で渡し、プロンプトで具体的な修正指示を与えます。

```sh
# テンプレート
cat {file_path} | gemini -p "このファイルを次のように修正してください: {修正指示}" -y -o json
```

- `{file_path}`: 修正対象のファイルパス。
- `{修正指示}`:
    - 簡潔かつ明確に指示します。（例:「`foo` を `bar` に変更してください」）
    - 複雑な修正の場合、修正前後のコードスニペットを提示することも有効です。

**ステップ 2: Gemini CLI の実行と結果の解釈**

Claude は上記コマンドを実行し、標準出力で返されるJSONを解釈します。

**成功時のJSON出力（例）**

Gemini CLIが `replace` ツールなどを実行して成功した場合、以下のようなJSONが返ってくることが想定されます。（※実際の出力はバージョンにより変化する可能性があります）

```json
{
  "status": "success",
  "tool_calls": [
    {
      "tool_name": "replace",
      "parameters": {
        "file_path": "src/example.ts",
        "old_string": "...",
        "new_string": "..."
      },
      "result": "Successfully modified file: src/example.ts"
    }
  ],
  "final_response": "ファイルの修正が完了しました。"
}
```

Claude は `status: "success"` と `tool_calls` の結果を見て、タスクが成功したことを確認します。

**エラー時のJSON出力（例）**

```json
{
  "status": "error",
  "error_message": "Tool 'replace' failed: old_string not found in file.",
  "final_response": "ファイルの修正に失敗しました。指定された文字列が見つかりません。"
}
```

Claude は `status: "error"` と `error_message` を見て、失敗の原因を分析し、リトライや別の方法を検討します。

### 3.2. テストコードの生成

特定のソースコードに対するテストコードを生成させる場合の連携フローです。

**ステップ 1: Claude 側でコマンドを組み立てる**

テスト対象のソースコードを標準入力で渡し、プロンプトでテストコードの生成を指示します。

```sh
# テンプレート
cat {source_file_path} | gemini -p "このコードに対するVitestのユニットテストを作成し、'{test_file_path}' に保存してください。" -y -o json
```

- `{source_file_path}`: テスト対象のソースファイル。
- `{test_file_path}`: 生成されるテストファイルの保存先パス。

**ステップ 2: Gemini CLI の実行と結果の解釈**

Gemini CLI は `write_file` ツールを実行してテストファイルを作成します。Claude は成功/失敗をJSON出力から判断します。

## 4. プロンプトのベストプラクティス

- **具体的かつ直接的に**: 「〜してください」という命令形で、行うべき操作を明確に指示します。
- **コンテキストの提供**: 必要な情報は標準入力やプロンプト内で十分に提供します。ファイルパスは常に正確に記述します。
- **期待する成果物を明記**: 「`{file_path}`に保存してください」「diff形式で変更点を教えてください」のように、期待するアウトプットを明確に指示します。

## 5. 注意事項

- **冪等性**: Gemini CLI に与える指示は、可能な限り冪等性（何度実行しても同じ結果になること）を意識してください。
- **監視**: Claude は Gemini CLI の実行結果を必ず確認し、予期せぬ変更が加えられていないかを検証する責任を負います。`git status` や `git diff` を実行して差分を確認することが推奨されます。
