# NUC → Web 家族データ移管 runbook

NUC（セルフホスト / SQLite / 全機能解放）から Web 版（AWS / DynamoDB / プラン制限あり）へ、1 家族分のデータを移管する手順。

**関連 Issue**: #3080（本 runbook）/ #3077（静的ファイルの ZIP 移管、実装済）・#3078（checklistLogs 往復、実装済）・#3083（ZIP export/import）・#3104（日本語名 export、実装済）・#3106 / #3107（checklist log backup 完全性、**未マージ — 修正 PR #3113 対応予定**。§5 既知の欠落を参照）
**関連**: `src/lib/server/services/{export,import}-service.ts` / `src/routes/api/v1/{export,import}/+server.ts` / `/admin/settings/data` / [#1254 家族全体 export/import]

---

## 1. 前提と移管範囲

移管は **NUC の家族全体エクスポート → Web の家族全体インポート（置換）** で行う。フォーマットは JSON で DB 非依存のため SQLite(NUC) → DynamoDB(Web) を跨げる。

### 移管されるもの（`exportFamilyData`）

- 子供 / 活動定義 / ステータス（XP・レベル）+ ステータス履歴
- **活動ログ・ポイント履歴・ログインボーナス**（過去の記録）
- チェックリスト定義 / **チェックリスト完了ログ**（#3078 で往復実装済。ただし archive 済 template の完了ログのスキップ #3106・同名 template の取り違え #3107 は未解消 — §5「既知の欠落」を参照）
- 特別ごほうび
- **アバター画像・音声などの静的ファイル**（#3077 / #3083 の **ZIP エクスポート/インポート**で実体バイナリごと移管。詳細は §4）

> **注（#3104）**: 日本語名のチェックリスト等も export/import で正しく扱える（旧 `Content-Disposition` ByteString 500 は #3104 で解消済）。日本語名データが移管対象から漏れることはない。

### 静的ファイルを含めるには ZIP を使う（移管前に把握必須）

| 形式 | 静的ファイル（アバター画像 / 音声）の扱い |
|---|---|
| **JSON エクスポート** | `child.avatarUrl` 等の文字列は含まれるが、画像/音声の**実体バイナリは含まれない**（参照のみ） |
| **ZIP エクスポート（推奨）** | JSON + 静的ファイル実体を同梱。インポート時に実体ごと復元される（#3077 / #3083） |

→ 静的ファイルも移管したい場合は **Step 1 で ZIP を選んでエクスポート**し、Step 3 で同じ ZIP をインポートする。

## 2. 手順

### Step 1 — NUC でエクスポート

NUC は全機能解放（`canExport=true`）のためエクスポートはそのまま使える。

1. NUC の `/admin/設定 > データ`（`/admin/settings/data`）を開く
2. 「エクスポート」を実行 → `ganbari-quest-backup-YYYY-MM-DD.json`（または ZIP）を保存
3. **保存した JSON の `dataSummary` 件数を控える**（後の検証で照合）

### Step 2 — Web でプレビュー（必ず先に dry-run）

1. Web 版の `/admin/設定 > データ` を開く
2. 「インポート」で Step 1 の JSON を選択 → **プレビュー（`mode=preview`）で件数・重複を確認**
3. プレビュー件数が Step 1 の `dataSummary` と一致することを確認

### Step 3 — Web で置換インポート

1. プレビューに問題なければ「置換（`mode=replace`）」で実行
   - 置換は **Web 側の既存データをクリアしてから** インポートする（移管先が空 or 破棄してよい前提）
2. 完了後、主要画面（子供一覧 / 活動 / ポイント履歴 / ステータス）を目視確認

---

## 3. プラン上限の注意（重要）

Web 無料プランには上限がある（子供 2・カスタム活動 3・履歴保持 90 日）。

- インポート自体はプラン制限を受けない（role チェックのみ）ため、上限超過データも**投入はできる**
- ただしインポート後、**上限超過分は自動アーカイブ**される（削除ではなく、有料化で復元可能）
- **フル移管したい場合は、インポート前に Web 側を有料化**（スタンダード以上）しておくこと

---

## 4. 静的ファイル（アバター画像 / 音声）の移管

#3077 / #3083 実装済のため、**ZIP エクスポート/インポートで実体ごと移管される**（旧来の手動再設定は不要）。

- **手順**: Step 1 で **ZIP** を選んでエクスポート → Step 3 で同じ ZIP をインポートすると、アバター画像・音声が実体ごと復元される
- **JSON で移管した場合**: 静的ファイル実体は含まれないため、移管後に各子供のアバター・音声を Web 側で再設定する（ZIP を使えば不要）
- **セキュリティ注（#3105）**: 復元された静的ファイルはラスタ画像のみ inline 配信、SVG 等は `attachment` 配信される（stored XSS 防止）。表示（`<img>`）には影響しない

---

## 5. 既知の欠落 / 注意（移管前に必ず確認）

現状 develop で出荷済の export-service には以下の欠落がある。**移管後に元 NUC のデータを破棄する前に**、これらを把握しておくこと（不可逆な家族ログ消失を避けるため、移管完了の目視確認が済むまで NUC 側のバックアップを保持する）。

- **archive 済 template の完了ログは現状 export でスキップされる**（#3106 / 修正 #3113 対応予定、未マージ）。アーカイブしたチェックリストに紐づく過去の完了ログは移管 JSON / ZIP に含まれない
- **同名 template が複数ある場合、checklistLog の round-trip キーが template 名のため取り違えの可能性がある**（#3107 / 修正 #3113 対応予定、未マージ）。同じ名前のチェックリストを複数運用している家族は、移管後にログの紐付きを目視確認する

> いずれも修正 PR #3113 は未マージのため、本 runbook は #3113 マージ後に「解消済」へ更新する。
